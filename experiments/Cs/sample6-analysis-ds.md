# 程序基本功能

这是一个"秘密保管"程序，用户可以通过菜单选择来管理三种不同大小的秘密：

1. **add()** - 添加秘密
   - 小秘密：40字节
   - 大秘密：4000字节  
   - 巨大秘密：400000字节（只能分配一次）

2. **del()** - 删除秘密
   - 只能删除小秘密和大秘密
   - 巨大秘密无法删除

3. **update()** - 更新秘密内容
   - 可以重新输入小秘密或大秘密的内容

4. **main()** - 主循环，提供用户交互界面

# 关键结构体

程序没有显式定义结构体，但通过全局变量管理秘密：
- `f_ptr` - 指向小秘密的指针
- `s_ptr` - 指向大秘密的指针  
- `q_ptr` - 指向巨大秘密的指针
- `f_flag`, `s_flag`, `q_flag` - 对应秘密的分配状态标志

# 漏洞成因

**漏洞类型：Use-After-Free (UAF)**

**漏洞位置：** `del()` 函数和 `update()` 函数的组合使用

**漏洞原因：**

1. 在 `del()` 函数中（第85-104行），释放内存后只将标志位置0，但没有将指针设置为NULL：
   ```c
   case 1:
       free(f_ptr);    // 释放内存
       f_flag = 0;     // 但f_ptr仍然指向已释放的内存
       break;
   ```

2. 在 `update()` 函数中（第107-128行），只检查标志位而不检查指针有效性：
   ```c
   case 1:
       if(f_flag)      // 如果标志位为1
       {
           puts("Tell me your secret: ");
           read(0, f_ptr, BASE);  // 向已释放的内存写入数据
       }
       break;
   ```

**漏洞利用场景：**
- 用户先分配一个小秘密或大秘密
- 然后删除该秘密（指针变为悬空指针）
- 接着再次调用update功能，由于标志位检查通过，程序会向已释放的内存写入数据
- 这可能导致堆元数据被破坏，进而实现任意地址读写或代码执行

**额外问题：**
- 巨大秘密分配后无法释放，存在内存泄漏
- 删除功能缺少输入验证，可能造成无效释放
