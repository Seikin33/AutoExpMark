# 程序基本功能

这是一个简单的堆内存管理程序，提供基本的堆块操作功能：

- **main()**: 程序主循环，显示菜单并根据用户选择调用相应功能
- **menu()**: 显示功能选项菜单
- **get_int()**: 读取用户输入的整数
- **init()**: 初始化程序，设置缓冲、映射内存、初始化seccomp沙箱和超时处理
- **read_n()**: 安全的输入读取函数
- **add()**: 分配新的堆块（最大4096字节）
- **edit()**: 编辑指定索引的堆块内容
- **delete()**: 释放指定索引的堆块
- **show()**: 显示指定索引的堆块内容

# 关键结构体

程序使用一个全局数组来管理堆块信息：
```c
// ptrs指向的内存区域存储堆块管理信息
// 每个条目占16字节：
// - 前8字节：堆块指针
// - 后4字节：堆块大小
// - 填充4字节（对齐）
```

# 漏洞成因

**漏洞位置**: `edit()` 函数中的这一行代码：
```c
*(_BYTE *)(*((_QWORD *)ptrs + 2 * (int)v1)
         + (int)read_n(*((void **)ptrs + 2 * (int)v1), *((_DWORD *)ptrs + 4 * (int)v1 + 2))) = 0;
```

**漏洞类型**: 单字节堆溢出 (Off-by-one)

**漏洞原因**:

1. `read_n()` 函数读取用户输入到堆块中，最大读取长度为堆块的大小
2. `read_n()` 返回实际读取的字节数
3. 程序在 `read_n()` 返回的位置（即用户输入的末尾）写入一个零字节作为字符串终止符
4. 如果用户输入恰好填满了整个堆块（读取字节数等于堆块大小），那么写入零字节的位置将在堆块边界之外

**具体分析**:
- 假设堆块大小为n，用户输入n个字符
- `read_n()` 读取n个字节，返回n
- 程序在 `堆块地址 + n` 的位置写入零字节
- 这导致在堆块边界后一个字节被覆盖为零，造成单字节堆溢出

这种off-by-one漏洞可以被利用来破坏堆的元数据，可能导致堆布局被破坏，进而实现任意地址读写或代码执行。
