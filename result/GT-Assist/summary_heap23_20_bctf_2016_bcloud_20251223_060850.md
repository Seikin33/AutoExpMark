# 漏洞成因
## 程序关键结构体
程序通过以下三个全局数组管理堆块：
- `dword_804B120[10]`: 存储`note`堆块的用户指针。
- `dword_804B0A0[10]`: 存储对应的`note`内容大小。
- `dword_804B0E0[10]`: 同步标记（在本漏洞利用中未使用）。
此外，`dword_804B0C8`和`dword_804B148`分别存储`get_org_host`函数中分配的`Host`和`Org`字符串的堆指针。

## 漏洞定位
程序包含两个关键漏洞：
1.  **`get_name`函数堆地址泄露漏洞**：
    函数`sub_80487A1`（即`get_name`）中，栈变量`s`大小为64字节，使用`sub_804868D`读取最多64个字节（不追加NULL）。随后`malloc(0x40)`得到一个0x48大小的chunk，并使用`strcpy`将`s`的内容复制到该chunk。如果`s`被填满64个非`\x00`字符，`strcpy`会继续拷贝栈上紧随`s`之后的内容（即`v2`指针自身），从而泄露堆地址。
    ```c
    // sub_80487A1 片段
    char s[64]; // [esp+1Ch] [ebp-5Ch] BYREF
    char *v2; // [esp+5Ch] [ebp-1Ch]  // 此变量紧邻 s
    ...
    sub_804868D(s, 64, 10); // 读取最多64字节，不添加终止符
    v2 = (char *)malloc(0x40u);
    dword_804B0CC = (int)v2;
    strcpy(v2, s); // 若s填满64字节，会连带拷贝v2指针的值（堆地址）
    sub_8048779(v2); // 打印内容，泄露堆地址
    ```

2.  **`get_org_host`函数top chunk size域溢出漏洞**：
    函数`sub_804884E`（即`get_org_host`）中，先为`Host`和`Org`字符串各分配一个0x48大小的chunk。`Org`字符串的chunk (`v2`) 紧邻top chunk。在随后的`strcpy(v2, s)`中，`s`为栈上的`Org`字符串缓冲区，若`strcpy`源字符串长度足够，可以覆盖`v2` chunk之后top chunk的`size`域。
    ```c
    // sub_804884E 片段
    char s[64]; // [esp+1Ch] [ebp-9Ch] BYREF
    ...
    sub_804868D((int)s, 64, 10); // 读取Org字符串
    ...
    v2 = (char *)malloc(0x40u); // 此chunk紧邻top chunk
    ...
    strcpy(v2, s); // 可溢出覆盖top chunk的size域
    ```

# 漏洞利用过程：
本漏洞利用结合**堆地址泄露**与**House of Force**攻击。首先泄露堆地址以计算top chunk位置，然后溢出修改top chunk的`size`为一个极大值（`0xffffffff`）。接着通过精心计算分配大小，使后续`malloc`从top chunk切割时，可以分配到`.bss`段上的`ptr_array`（堆指针数组）附近。通过控制`ptr_array`的内容，实现任意地址读写（劫持GOT表），最终将`free`函数替换为`system`并传入`/bin/sh`参数，从而获取shell。

- **Step 1**: 利用`get_name`泄露堆地址，计算后续top chunk地址。
- **Step 2**: 利用`get_org_host`溢出覆盖top chunk的`size`为`0xffffffff`。
- **Step 3**: 实施House of Force攻击。计算从当前top chunk到目标`ptr_array`（`0x804b120`）的偏移，并通过`malloc`分配相应大小的chunk，使得top chunk扩展到`ptr_array`附近。
- **Step 4-6**: 分配多个chunk进行堆布局，并利用已控制的`ptr_array`修改`free@got`为`puts@plt`，为泄露libc地址做准备。
- **Step 7-8**: 触发`free`（实际为`puts`）泄露`printf`的libc地址，并计算出`system`函数地址。
- **Step 9-11**: 再次篡改`ptr_array`，将`/bin/sh`字符串地址放入数组，并将`free@got`改为`system`地址，最终调用`free`（即`system("/bin/sh")`）获取shell。

## Step 1
- **栈内存变化**：向`get_name`函数栈变量`s`写入64个`'a'` (`0x61`)，覆盖了其后的指针变量`v2`（指向新分配的堆块`0x95a4008`）的低字节。`strcpy`后，堆块`0x95a4008`的内容变为`'a'*64` 拼接上堆地址`0x95a4008`本身（小端存储）。随后`put_info`打印该堆块内容，泄露出堆地址。
- **关键计算**：泄露的堆地址为`heap_leak = 0x95a4008`。根据堆布局，后续`get_org_host`中分配的chunk (`v2`) 地址为`heap_leak + 0x40 + 0x48 = heap_leak + 0x88`（因为第一个chunk `0x48`大小，加上填充头）。该chunk紧邻top chunk，故`top_chunk_addr = heap_leak + 0x88 + 0x48 = heap_leak + 0xd0`。

## Step 2
- **输入内容**：`Org`输入为64个`'b'`（`0x62`），`Host`输入为`p32(0xffffffff) + (0x40-4)*b'c'`。
- **堆内存变化**：`Host`字符串被拷贝到`heap_leak+0x48`处的chunk。`Org`字符串被拷贝到`heap_leak+0x88`处的chunk (`v2`)。由于`Org`字符串长度为64字节，`strcpy`会复制末尾的NULL字节，该NULL字节恰好覆盖了`v2` chunk之后top chunk `size`域的最低字节。但在此之前，`Host` chunk中`p32(0xffffffff)`的内容（即`ff ff ff ff`）由于内存布局，其最后一个`0xff`字节位于`v2` chunk用户空间的末尾，紧接着top chunk的`size`域。`strcpy(v2, s)`在复制64个`'b'`后追加的NULL字节 (`0x00`) 会覆盖这个`0xff`，导致top chunk的`size`域变为`0xffffff00`（注意小端序）。然而，利用脚本中传入的是`p32(0xffffffff)`，期望直接覆盖为`0xffffffff`。这里存在一个偏差：实际覆盖后`size`为`0x00ffffff`（考虑小端序），但仍是一个很大的值，足以进行House of Force攻击。动态调试中，此步骤后top chunk的`size`被成功篡改为一个极大值。

## Step 3
- **关键计算**：目标地址 `target = ptr_array = 0x804b120`。计算需要分配的大小：`size_to_malloc = target - top_chunk_addr - 2*sizeof(size_t) - sizeof(chunk_header)`。在32位系统中，简化计算通常为 `margin = ptr_array - top_chunk_addr - 0x10`（绕过top chunk头和对齐）。利用脚本中计算为`margin - 20`，经过调试确定该值能使下一次`malloc`后，top chunk正好移动到`ptr_array`之前，从而后续分配可以覆盖`ptr_array`。
- **堆内存变化**：调用`new_note(margin-20, "")`分配一个巨大chunk。此后，top chunk的地址被更新到`ptr_array`附近，为后续分配到`.bss`段创造条件。

## Step 4
- **堆内存与.bss段变化**：连续调用四次`new_note(0x40, 'aa')`。由于Step 3已将top chunk移至`ptr_array`附近，这四次分配会从该top chunk切割内存。第一次分配（note 1）的chunk会覆盖`ptr_array`数组起始部分。具体地，`ptr_array[0]`和`ptr_array[1]`被设置为note 1 chunk的用户空间地址（位于`.bss`段上，即`0x804b120`之后）。`ptr_size`数组也被相应设置。
- **结果**：此时`ptr_array[1]`指向note 1 chunk的用户空间（地址约为`0x804b124`），而我们可以通过`edit_note(1, ...)`来修改这个区域，从而间接修改`ptr_array`数组本身（因为`ptr_array[1]`指向`ptr_array[0]`所在的区域）。

## Step 5
- **.bss段变化（ptr_array）**：通过`edit_note(1, ...)`修改note 1 chunk的内容，即修改了`ptr_array`数组。具体写入：`p32(0x804b120)*2 + p32(free_got) + p32(printf_got)`。
  - `ptr_array[0]` 被设置为 `0x804b120` (自身地址，为后续写入`/bin/sh`做准备)。
  - `ptr_array[1]` 被设置为 `0x804b120` (note 1 chunk地址不变)。
  - `ptr_array[2]` 被设置为 `free@got` 地址 (`0x804b014`)。
  - `ptr_array[3]` 被设置为 `printf@got` 地址 (`0x804b010`)。
- **效果**：现在，`edit_note(2, ...)`将向`free@got`写入数据，`del_note(3)`将调用`free(printf_got)`（实际触发对`printf_got`的读写）。

## Step 6
- **.bss段变化（GOT表）**：通过`edit_note(2, p32(puts_plt))`，向`ptr_array[2]`（即`free@got`）写入`puts@plt`的地址 (`0x8048520`)。从而将`free`函数的调用劫持到`puts`。

## Step 7
- **触发与泄露**：调用`del_note(3)`。程序根据`ptr_array[3]`找到地址`printf_got`，并调用`free(ptr)`。由于`free@got`已被改为`puts@plt`，实际执行`puts(printf_got)`，打印出`printf`函数在libc中的实际地址（4字节）。
- **接收数据**：从接收到的消息中解析出`printf_addr`。

## Step 8
- **计算**：根据泄露的`printf_addr`和libc中`printf`与`system`的偏移，计算出`system`函数在内存中的实际地址：`system_addr = printf_addr - offset`。

## Step 9
- **.bss段变化（ptr_array 与 /bin/sh）**：再次通过`edit_note(1, ...)`修改`ptr_array`数组。写入：`p32(0x804b130) * 2 + p32(free_got) * 2 + b'/bin/sh'`。
  - `ptr_array[0]` 被设置为 `0x804b130` (`.bss`段上一个可写地址，用于存放`/bin/sh`字符串)。
  - `ptr_array[1]` 被设置为 `0x804b130`。
  - `ptr_array[2]` 被设置为 `free_got` (`0x804b014`)，保持篡改目标。
  - `ptr_array[3]` 被设置为 `free_got` (`0x804b014`)，后续不再使用。
  - 从`ptr_array[4]`开始写入字符串`/bin/sh`（地址约为`0x804b140`）。
- **效果**：现在`ptr_array[0]`指向`0x804b130`，该地址内容为`free_got`地址，其后面接着`/bin/sh`字符串。

## Step 10
- **.bss段变化（GOT表）**：通过`edit_note(2, p32(system_addr))`，再次向`free@got`写入数据，这次将其替换为`system`函数的地址。

## Step 11
- **触发getshell**：调用`del_note(0)`。程序根据`ptr_array[0]`找到地址`0x804b130`，并调用`free(0x804b130)`。由于`free@got`已被改为`system`，实际执行`system(0x804b130)`。而`0x804b130`处存放的是`free_got`地址 (`0x804b014`)，但`system`会将其后的内容（`/bin/sh`）解析为字符串，因为`system`会一直读取直到遇到NULL字节。`0x804b130`处的`free_got`地址 (`0x14b00408` 小端) 包含`0x00`字节(`0x14`)，因此`system`实际接收到的是`p32(free_got)`中第一个字节`0x14`之前的内容（即空字符串），这不会getshell。**这里存在一个关键错误**。
- **修正**：正确的payload应该确保`/bin/sh`字符串位于一个`system`能正确解析的地址。通常做法是将`ptr_array[0]`直接指向存放`/bin/sh`字符串的地址（例如`0x804b140`），或者将`/bin/sh`字符串放在一个单独分配的堆块中并将其地址放入`ptr_array[0]`。原始的write-up中的exp步骤9的payload `p32(0x804b130) * 2 + p32(free_got) * 2 + b'/bin/sh'` 可能依赖于特定的内存布局和`system`的实现细节（例如忽略前面的非打印字符），或者调试环境与真实环境有差异。一个更可靠的payload是：`p32(0x804b140) + ...` 并将`/bin/sh`放在`0x804b140`。
