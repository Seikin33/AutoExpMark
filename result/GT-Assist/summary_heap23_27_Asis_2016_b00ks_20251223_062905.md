# 漏洞成因
## 程序关键结构体
程序使用一个数组`book_pointers`（位于.bss段）来管理最多20个`book`结构体的指针。每个`book`结构体大小为0x20字节，由`malloc`分配，定义如下：
```c
struct book
{
    int id;                 // 图书ID，自增
    char *name;             // 指向书名缓冲区
    char *description;      // 指向描述缓冲区
    int size;               // 描述缓冲区的大小
};
```
书名和描述缓冲区的大小由用户输入，并使用`malloc`分配相应内存。

## 漏洞定位
程序的核心漏洞存在于`readline`函数中。该函数用于读取用户输入，但在处理字符串终止符时存在 **单字节空字符溢出（off-by-one）** 漏洞。
```c
signed __int64 __fastcall readline(_BYTE *a1, int a2)
{
  int i; // [rsp+14h] [rbp-Ch]
  if ( a2 <= 0 )
    return 0;
  for ( i = 0; ; ++i )
  {
    if ( (unsigned int)read(0, a1, 1u) != 1 )
      return 1;
    if ( *a1 == 10 )       // 遇到换行符'\n'则跳出循环
      break;
    ++a1;
    if ( i == a2 )         // 当已读取的字符数 i 达到传入的长度 a2 时，跳出循环
      break;
  }
  *a1 = 0;                // 在此处写入空字符终止符。问题：当 i == a2 时，a1 已经指向缓冲区[a2]的位置，这超出了用户可控制的 a2 字节范围。
  return 0;
}
```
当`i == a2`时，循环跳出，此时指针`a1`已经向后移动了`a2`次，指向了缓冲区起始地址偏移`a2`字节的位置。随后的`*a1 = 0`会将一个`\x00`字节写入到缓冲区之后，造成了单字节溢出。

此漏洞在多个地方被调用，尤其是在`set_author_name`函数中，它允许用户输入最多32字节的`author_name`（位于.bss段）。如果恰好输入32个非换行符，`readline(author_name, 32)`会在第33个字节（即`author_name[32]`）的位置写入`\x00`，而这个位置恰好是`book_pointers`数组的起始位置。这会覆盖`book_pointers[0]`指针的最低有效字节（LSB），为后续利用创造条件。

# 漏洞利用过程：
利用思路主要分为两种，都基于上述off-by-one漏洞。第一种（既有的write-up）通过mmap大块来获取libc地址；第二种（调试记录中的“简洁方案”）通过构造堆布局和unlink攻击实现任意地址读写，进而泄漏libc。以下详细分析第二种方案，它更具通用性。
- **Step 1**: 利用author name的off-by-one，为泄露堆地址创造条件。
- **Step 2**: 创建第一本书，使其`book`结构体指针被写入`book_pointers[0]`，由于Step1的溢出，该指针的最低字节被`\x00`覆盖，导致打印author name时连带泄露出一个堆地址。
- **Step 3-4**: 创建并删除两个小chunk，在fastbin中构造两个空闲块，为后续堆布局做准备。
- **Step 5-6**: 精心创建三个chunk，构造一个堆溢出场景。`chunk4`（`overflow`）用于溢出，`chunk5`（`target`）是溢出的目标，`chunk6`（`/bin/sh`）用作后续任意读写操作的“傀儡”。
- **Step 7-8**: 在`chunk4`中伪造一个已释放chunk的元数据，然后释放`chunk5`，触发`unlink`攻击。成功后将修改`book_pointers`中一个指针，使其指向自身附近，从而获得一个**原语**：通过编辑这本书，可以修改`book_pointers`数组中的任意条目。
- **Step 9**: 利用Step8获得的原语，修改`book_pointers`数组，使其中的一本书的`name`和`description`指针指向我们想要读写的任意地址（通过控制`chunk6`的`book`结构体实现）。
- **Step 10**: 利用任意读原语，读取堆上已有的一个libc地址（例如，从之前释放的chunk中残留的`main_arena`指针），计算出libc基址。
- **Step 11-12**: 利用任意写原语，将`__free_hook`的内容覆盖为`system`函数的地址。然后释放一个内容为`/bin/sh`的chunk（`chunk6`的`description`），触发`system(“/bin/sh”)`，获得shell。

## Step1-Step2: 泄露堆地址
- **内存变化**:
  - **BSS段 (`author_name` 和 `book_pointers` 区域)**:
    - **Step1后**: `author_name`区域（`0x58077f5eb080`）被字符串`”x”*27 + “leak:”`填充，共32字节。由于`readline`的off-by-one漏洞，`book_pointers`数组起始地址（`0x58077f5eb0a0`？注：地址需根据上下文调整，调试记录Step2中`0x58077f5eb080`是author_name，`book_pointers`应紧随其后）的最低字节被写入了`\x00`。
    - **Step2后**: 创建第一本书（ID 1）。其`book`结构体的地址（例如`0x5602c0b1a2a0`）被存入`book_pointers[0]`。由于Step1的溢出，这个指针与`author_name`的32字节内容在内存中连续。当打印`author_name`时，程序会一直打印到遇到空字符，从而将`book_pointers[0]`的值也作为字符串的一部分输出，泄露了堆地址`heap_leak`。根据调试记录，`heap_base = heap_leak - 0x1080`。

## Step3-Step4: 堆布局准备
- **操作**: 创建两本小书（ID 2, 3）后立即删除。这将在fastbin（大小0x30）中留下两个空闲chunk。
- **目的**: 为后续创建特定大小的chunk（`chunk4`和`chunk5`）提供可预测的堆布局，确保它们物理相邻。

## Step5-Step6: 构造溢出场景与“傀儡”chunk
- **操作**:
  - 创建`chunk4`（ID 4），描述大小`0x108`（实际分配`0x110`的chunk，加上`0x10`的chunk头）。这块内存将被用作**溢出源**。
  - 创建`chunk5`（ID 5），描述大小`0xf0`（实际分配`0x100`的chunk）。这块内存紧跟在`chunk4`的描述缓冲区之后，是**溢出目标**。
  - 创建`chunk6`（ID 6），描述大小`0x200`，内容包含`/bin/sh`。其`book`结构体将被后续利用。
- **堆布局**: 此时堆上`chunk4`的描述缓冲区（`overflow`）与`chunk5`的描述缓冲区（`target`）物理相邻。`chunk5`的`book`结构体位于它们之前。

## Step7-Step8: 触发Unlink攻击，获取写原语
- **操作**:
  1. **Step7**: 编辑`chunk4`的描述，写入精心构造的payload。该payload在`chunk5`的描述缓冲区起始处伪造了一个已释放的小chunk（例如大小`0x101`）的元数据，其`fd`和`bk`指针被设置为指向`book_pointers`数组内部的特定位置（`ptr - 0x18`和`ptr - 0x10`，其中`ptr`是`chunk5`的`book`结构体地址）。
  2. **Step8**: 删除`chunk5`（ID 5）。由于`chunk5`的描述缓冲区大小（`0xf0`）属于smallbin范围，`free`会尝试与前后的空闲chunk合并。它检查到前一个chunk（即我们伪造的chunk）处于“已释放”状态（通过`PREV_INUSE`位判断），于是触发`unlink`操作。
- **`unlink`攻击效果**: 在`unlink`宏执行时，会发生以下关键内存写操作：
  `FD->bk = BK` 且 `BK->fd = FD`。
  根据我们伪造的`fd`和`bk`，这会导致`book_pointers`数组中对应条目（存放`chunk5`的`book`结构体指针的那个slot）的内容被改写为指向`book_pointers`数组自身内部的某个地址（`ptr - 0x18`）。
- **结果**: 现在，通过编辑`book` ID为5（其指针已被篡改），我们实际上是在修改`book_pointers`数组本身的内容，从而获得了篡改其他`book`指针的能力。

## Step9: 建立任意地址读写原语
- **操作**: 编辑`book` ID 5（即已被unlink篡改的指针），将其内容修改为`p64(0x30) + p64(4) + p64(heap_base + 0x11a0) + p64(heap_base + 0x10c0)`。这相当于：
  1. 在`book_pointers`数组的某个位置（原`chunk5`的`book`指针位置）写入了`0x30`（可能是新的假`book` ID？这里需要结合exp代码解释，实际是修改了`chunk6`的`book`结构体的`name`和`description`指针）。
  2. 更关键的是，它修改了`book_pointers`数组中`chunk6`（ID 6）对应的`book`结构体指针，使其`name`和`description`指针指向我们指定的地址（例如`heap_base + 0x11a0`和`heap_base + 0x10c0`）。
- **获得原语**: 现在，通过编辑`book` ID 6，我们可以向`heap_base + 0x11a0`写入任意内容（写`name`），或向`heap_base + 0x10c0`写入任意内容（写`description`）。通过先将其`description`指针改为目标地址，再调用`edit`功能，即可实现**向任意地址写**。通过将其`description`指针改为目标地址，然后调用`print`功能，即可实现**从任意地址读**。

## Step10: 泄漏libc地址
- **操作**: 使用Step9建立的任意读原语，读取堆上一个已知包含libc地址的位置（例如，之前释放的chunk中残留的`main_arena`指针，在调试记录中是`heap_base + 0x11e0`）。读取到的值减去一个固定的偏移（如`libc.sym[‘__malloc_hook’] + 0x10 + 0x58`），即可得到libc的基地址`libc_base`。

## Step11-Step12: 劫持`__free_hook`并获取shell
- **操作**:
  1. **Step11**: 使用任意写原语，将`libc.sym[‘__free_hook’]`的内容覆盖为`libc.sym[‘system’]`。
  2. **Step12**: 删除`chunk6`（ID 6）。程序会`free`掉`chunk6`的`description`缓冲区，而该缓冲区的内容正是字符串`”/bin/sh”`。由于`__free_hook`已被劫持，实际调用的是`system(“/bin/sh”)`，从而获得shell。
