# 漏洞成因
## 程序关键结构体
程序使用一个全局数组（基地址由`mmap`随机分配）来管理堆块。每个数组元素（索引）是一个大小为24字节的结构体，用于记录一个堆块的状态。
```c
struct chunk_info {
    int is_used;            // 该索引是否已被使用 (4字节)
    long long size;         // 堆块申请的大小 (8字节)
    long long *chunk_addr;  // 堆块的地址 (8字节)
}
```

## 漏洞定位
漏洞位于`Fill`函数中。该函数允许用户向指定索引的堆块写入任意长度的数据，但并未检查用户输入的`Size`是否超过了该堆块实际分配的大小。
```c
// sub_E7F (Fill函数)
printf("Size: ");
result = readint();
v3 = result;
if ( (int)result > 0 )
{
    printf("Content: ");
    // 关键漏洞：v3为用户输入的长度，直接用于写入，未与 chunk_info.size 比较
    return sub_11B2(*(_QWORD *)(24LL * v2 + a1 + 16), v3);
}
```
`sub_11B2`函数内部是一个循环读取输入并写入堆块的函数，未做长度限制。因此，通过向一个较小的堆块写入过量的数据，可以覆盖其相邻堆块的内容，造成**堆溢出**。结合程序的`Free`和`Allocate`（使用`calloc`）功能，可以实现Use-After-Free和堆布局。

# 漏洞利用过程：
利用思路是通过堆溢出修改被释放堆块的`size`域，构造一个大的`fake chunk`。释放这个`fake chunk`使其进入`unsorted bin`，其`fd`指针会指向`main_arena`中的一个地址。通过堆块重叠，利用仍在使用的堆块`Dump`出该地址，从而泄露`libc`基址。最后，再次利用堆溢出在`__malloc_hook`附近伪造一个`fastbin chunk`，并通过分配将其`malloc_hook`覆盖为`one_gadget`地址，从而在调用`malloc`时获得shell。

- Step1: 进行初始堆布局，分配5个不同大小的堆块，为后续的溢出和伪造做准备。
- Step2~4: 利用堆溢出，通过填充`chunk 0`和`chunk 2`，在`chunk 1`的位置伪造一个`size`为`0x51`的`fastbin chunk`。释放并重新申请它，为后续修改其后方堆块`size`创造条件。
- Step5~7: 利用`UAF`，修改`chunk 2`的`size`为`0x91`，使其成为一个大的`fake chunk`。释放它使其进入`unsorted bin`，并通过`chunk 1`打印其`fd`指针，泄露`libc`地址。
- Step8~10: 释放一个`size`为`0x70`的`fastbin chunk` (`chunk 4`)，并通过堆溢出修改`chunk 3`的内容，在`__malloc_hook`附近伪造一个`size`为`0x71`的`fastbin chunk`。通过两次分配，第二次分配即可获得该伪造块。
- Step11~12: 向获得的伪造块中写入`one_gadget`地址，覆盖`__malloc_hook`。最后，通过一次新的分配触发`malloc`调用，从而执行`one_gadget`获取shell。

## Step1
- 连续调用`allocate(0x10)`, `allocate(0x10)`, `allocate(0x30)`, `allocate(0x40)`, `allocate(0x60)`。
- 在堆上依次创建了5个`chunk`（索引0~4）。此时，`chunk 1`紧邻`chunk 2`，`chunk 3`紧邻`chunk 4`，为后续的溢出操作奠定了基础。

## Step2~4
- `fill(0, p64(0x51)*4)`：向`chunk 0`（大小为0x10）写入32字节数据，造成溢出。`chunk 0`的用户区只有16字节，多余的16字节会覆盖到`chunk 1`的堆头。其中`p64(0x51)*2`覆盖了`chunk 1`的`prev_size`和`size`字段，将其伪造成一个`size`为`0x50`的`fastbin chunk`。
- `fill(2, p64(0x31)*6)`：向`chunk 2`（大小为0x30）写入48字节数据，同样造成溢出，为后续操作做准备。
- `free(1)`：释放被我们伪造了`size`的`chunk 1`。由于它的`size`（0x50）在`fastbin`范围内，它被链入`fastbin[4]`（对应`size`为0x50~0x58）。
- `allocate(0x40)`：申请一个大小为0x40的`chunk`。`glibc`会从`fastbin[4]`中取出刚刚释放的`chunk 1`分配给我们，但其管理信息中的`size`记录为0x40。**关键点**：`calloc`会将被分配内存的内容清零，这清除了`chunk 1`原有的`fd`指针，但保留了之前通过溢出写入的伪造`size`（0x51）。此时，索引1的`chunk_info`记录的`size`是0x40，但堆上该块的`size`域仍是0x51。

## Step5~7
- `fill(1, p64(0x91)*4)`：向`chunk 1`（用户区大小为0x40，即64字节）写入`p64(0x91)*4`（32字节）。由于`chunk 1`在堆上的实际`size`是0x51（包含堆头共0x60字节），这个写入不会溢出到`chunk 3`，但会覆盖`chunk 2`。这32字节数据正好覆盖了`chunk 2`的用户区前32字节，其中包括它的`size`字段。`chunk 2`的`size`被从`0x40`改为了`0x90`（注意：`size`字段的低3位是标志位，`0x91`表示`PREV_INUSE`位为1）。这样，我们就在`chunk 1`之后（包含了部分`chunk 1`的用户区和整个`chunk 2`）伪造了一个`size`为`0x90`的`large chunk`。
- `free(2)`：释放`chunk 2`。此时`glibc`检查到其`size`为`0x90`，超过了`global_max_fast`，因此将其放入`unsorted bin`。当一个`chunk`被放入`unsorted bin`，其`fd`和`bk`指针会被设置为指向`main_arena`中的特定位置（对于非`main_arena`的第一个`chunk`，`fd`指向`main_arena+88`）。
- `dump(1)`：打印`chunk 1`的内容。由于`chunk 1`和`chunk 2`存在重叠（我们伪造的`large chunk`包含了`chunk 2`），打印`chunk 1`会输出从`chunk 1`起始到其管理`size`（0x51）范围内的所有数据，这其中就包含了已进入`unsorted bin`的`chunk 2`的`fd`指针。通过解析接收到的数据，可以计算出`main_arena+88`的地址，进而得到`libc`基址和`__malloc_hook`的地址。

## Step8~10
- `free(4)`：释放索引为4、大小为0x60的`chunk`。这个`chunk`的`size`为`0x70`（包含堆头），被链入`fastbin[6]`。
- `fill(3, payload)`：其中`payload = p64(0)*9 + p64(0x71) + p64(malloc_hook-0x23)`。向`chunk 3`（大小为0x40）写入该载荷，造成溢出并覆盖了已释放的`chunk 4`的`fd`指针。`p64(0x71)`覆盖了`chunk 4`的`size`字段（确保其仍是合法的fastbin size），`p64(malloc_hook-0x23)`覆盖了其`fd`指针，指向`__malloc_hook`之前0x23字节的位置。这是因为在`__malloc_hook-0x23`这个地址，可以找到一个值为`0x7f`的字节（在64位系统下），可以作为一个`size`为`0x70`的`fastbin chunk`的`size`字段。
- `allocate(0x60)`：第一次分配，从`fastbin[6]`中取出了原本的`chunk 4`。
- `allocate(0x60)`：第二次分配，由于`fastbin`是单链表，`glibc`会顺着`chunk 4`被修改后的`fd`指针去寻找下一个`chunk`，此时它找到了我们伪造的、位于`__malloc_hook-0x23`处的“chunk”。只要这个地址满足`fastbin`的分配检查（主要是`size`检查），分配就会成功。至此，我们获得了一个可以写入`__malloc_hook`区域的“堆块”（索引为4，但指向`__malloc_hook`附近）。

## Step11~12
- `fill(4, shellcode)`：其中`shellcode = b'a'*0x13 + p64(one_gadget)`。向我们获得的伪造`chunk`（索引4）写入数据。`b'a'*0x13`用于填充从`chunk`用户区开始到`__malloc_hook`之间的0x13字节偏移，`p64(one_gadget)`则直接覆盖了`__malloc_hook`。
- `allocate(1)`：执行一次新的分配。这会调用`malloc`，进而触发`__malloc_hook`。由于`__malloc_hook`已被覆盖为`one_gadget`的地址，程序将转而执行`one_gadget`，获得shell。
