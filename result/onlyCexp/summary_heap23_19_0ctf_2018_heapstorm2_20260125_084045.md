# 漏洞成因
## 程序关键结构体
程序通过 `mmap` 在固定地址 `0x13370000` 开辟了一块内存作为全局管理结构。`0x13370800` 开始的区域存储了一个加密的 `chunk` 数组，用于管理最多 16 个堆块。每个 `chunk` 元素包含两个 8 字节的加密值：加密后的堆块指针和加密后的用户请求大小。`0x13370810` 和 `0x13370818` 存放了两个随机密钥，用于加解密操作。
```c
struct Chunk {
    QWORD encrypted_ptr;  // 堆块地址 ^ key1
    QWORD encrypted_size; // 用户大小 ^ key2
};
```
## 漏洞定位
漏洞位于 `Update` 功能中。当更新一个 `chunk` 时，程序在读取用户输入后，会使用 `strcpy` 强制追加字符串 `"HEAPSTORM_II"`（12字节 + 1字节终止符）。如果用户输入的长度恰好等于 `(chunk->size - 12)`，那么 `strcpy` 写入的 13 个字节（12字符+`\x00`）将恰好写满 `chunk` 的用户区并溢出 1 个 `NULL` 字节到下一个物理相邻堆块的 `size` 字段最低位，造成 **off-by-one (NULL byte overflow)** 漏洞。
```c
// sub_F21 函数片段
v3 = sub_1551(); // 用户输入的更新长度
if ( v3 <= 0 || v3 > (unsigned __int64)(sub_BCC((__int64)a1, a1[2 * v2 + 5]) - 12) )
    return puts("Invalid Size");
...
v4 = sub_BB0(a1, a1[2 * v2 + 4]); // 获取解密后的堆地址
sub_1377(v4, v3); // 读取用户输入
strcpy((char *)(v3 + v4), "HEAPSTORM_II"); // 漏洞点：强制追加，造成off-by-one
```

# 漏洞利用过程：
利用 `off-by-one` 漏洞修改相邻堆块的 `size`，构造堆块重叠，进而控制 `large bin` 和 `unsorted bin`。通过 **House of Storm** 技术，在全局管理区 (`0x13370800`) 伪造 `chunk` 并将其成功链接到 `bin` 中，从而获得对该区域的写权限。随后，通过修改该区域的控制结构，实现任意地址读写，最终覆写 `__free_hook` 为 `system` 并触发以获取 shell。

- Step1~2: 初始堆布局，利用 `off-by-one` 制造第一个 `chunk overlap`，为后续构造 `large bin chunk` 和 `unsorted bin chunk` 做准备。
- Step3~6: 通过释放与分配操作，将特定 `chunk` 送入 `unsorted bin` 和 `large bin`，完成攻击所需的内存状态。
- Step7~8: 在全局管理区伪造 `large bin chunk` 和 `unsorted bin chunk`，并通过一次分配触发 `House of Storm`，将伪造的 `chunk` 链接到 `bin` 中，从而获得对管理区的控制权。
- Step 9~15: 利用获得的控制权修改管理结构，实现任意地址读写，泄露堆与 `libc` 地址，最终劫持 `__free_hook` 执行 `system(“/bin/sh”)`。

## Step1~2
- **Step1**: 分配 `chunk0(0x18)`, `chunk1(0x508)`, `chunk2(0x18)`, `chunk3(0x18)`, `chunk4(0x508)`, `chunk5(0x18)`。更新 `chunk1` 和 `chunk4`，在其用户数据偏移 `0x4f0` 处写入 `0x500`，伪造 `prev_size` 字段，为后续构造 `fake chunk` 做准备。
- **Step2**: 释放 `chunk1`。更新 `chunk0`，利用 `off-by-one` 漏洞将物理相邻的下一个 `chunk` (`chunk2`) 的 `size` 从 `0x21` 修改为 `0x20`。重新分配 `chunk1(0x18)` 和 `chunk6(0x4d8)`。此时，`chunk6` 与之前释放的 `chunk1` 原始空间部分重叠，且 `chunk2` 的 `size` 被错误减小。

## Step3~6
- **Step3**: 释放 `chunk1` (即 step2 中新分配的 `0x18` chunk)，它实际上位于 `chunk6` 的头部，此操作将一个 `0x4f1` 大小的 `chunk` 放入了 `unsorted bin`。
- **Step4**: 释放 `chunk2`。分配 `chunk7(0x38)` 和 `chunk8(0x4e8)`。`chunk8` 与 `chunk6` 重叠。释放 `chunk4`。
- **Step5**: 类似于 step2，更新 `chunk3`，利用 `off-by-one` 将 `chunk5` 的 `size` 从 `0x21` 修改为 `0x20`。分配 `chunk4(0x18)` 和 `chunk9(0x4d8)`。释放 `chunk4` 和 `chunk5`。分配 `chunk10(0x48)`，它占据了之前 `chunk4` 和 `chunk5` 的空间。
- **Step6**: 释放 `chunk2` (即 step4 中分配的 `0x4e8` chunk)。分配 `chunk11(0x4e8)` (与 `chunk8` 重叠)。再次释放 `chunk2` (即 `chunk11`)。至此，我们拥有一个在 `large bin` 中的 `chunk` (原始 `chunk4` 释放后形成) 和一个在 `unsorted bin` 中的 `chunk` (最后释放的 `chunk2`)。

## Step7~8
- **Step7**: 更新 `chunk7`，在全局管理区 `0x133707e0` (`storage - 0x20`) 伪造一个 `large bin chunk`，设置其 `size=0x4f1`, `bk_nextsize = fake_chunk`。
- **Step8**: 更新 `chunk8`，在 `0x133707e8` (`fake_chunk+8`) 伪造一个 `unsorted bin chunk`，设置其 `size=0x4e1`, `bk = fake_chunk+8`, `fd = fake_chunk-0x18-5` (用于绕过 `size` 与 `prev_size` 校验)。
- **触发 House of Storm**: 分配一个 `0x48` 大小的 `chunk`。该操作会触发 `unsorted bin` 与 `large bin` 的合并检查，由于我们伪造的 `chunk` 满足 `House of Storm` 的所有条件，`glibc` 会将 `unsorted bin` 中的 `chunk` 与我们的伪造 `chunk` (`0x133707e0`) 合并。最终，这次分配会从伪造的 `chunk` 中切出一块，使得 `0x13370800` (管理结构开始地址) 落入一个已分配的 `chunk` 的用户数据区内，从而我们可以通过 `update` 该 `chunk` (`chunk2`) 来修改整个管理结构。

## Step 9~15
- **Step9**: 通过 `update(2, ...)` 修改管理结构，在 `0x13370800` 处设置一个 `chunk` 管理条目 (`idx=0`)，其 `encrypted_ptr` 和 `encrypted_size` 被设置为可读写 `storage` 区域 (`0x13370800`) 本身，并设置 `magic: 0x13377331` 以通过 `view` 的检查。
- **Step10**: 通过 `update(0, ...)` 进一步设置 `storage` 区域的控制结构，使其成为一个“任意读写器”：在固定偏移处存放着 `target_addr` 和 `length`，为后续泄露做准备。
- **Step11**: 通过 `view(1)` 泄露堆地址。
- **Step12**: 修改控制结构中的 `target_addr` 为 `main_arena` 地址，通过 `view(1)` 泄露 `libc` 地址，并计算出 `__free_hook` 和 `system` 的地址。
- **Step13**: 修改控制结构，将 `target_addr` 指向 `__free_hook`，并在 `storage` 中写入 `/bin/sh\x00`。
- **Step14**: 通过 `update(1, ...)` 向 `__free_hook` 写入 `system` 地址。
- **Step15**: 释放 `chunk2`，其 `data` 部分指向 `/bin/sh`，从而触发 `system(“/bin/sh”)`。
