# 漏洞成因
## 程序关键结构体
程序使用两个全局数组管理`note`：
- `dword_804B120[10]`: 存储每个`note`内容堆块的指针。
- `dword_804B0A0[10]`: 存储每个`note`内容的大小。
程序初始化时，会在堆上分配三个`0x40`字节的块，分别存放`name`、`org`、`host`字符串，地址存储在`dword_804B0CC`、`dword_804B0C8`、`dword_804B148`。`host`对应的堆块紧邻`top chunk`。

## 漏洞定位
漏洞位于初始化函数`sub_804884E()`中。程序使用安全的`sub_804868D()`读取`Host`到栈缓冲区`v3`，该函数会以空字节终止字符串。但随后使用不安全的`strcpy(v4, v3)`将`v3`复制到堆块`v4`。如果`v3`缓冲区被恰好填满（64字节），则字符串没有空字节终止，`strcpy`会继续复制栈上`v3`之后的数据（恰好是`v2`指针和`v5` canary等），直到遇到空字节，从而导致**堆溢出**，可以覆盖紧随其后的`top chunk`的`size`字段。
```c
// sub_804884E 关键代码
puts("Host:");
sub_804868D((int)v3, 64, 10); // 安全读取，最多64字符+空字节
v4 = (char *)malloc(0x40u);   // 分配0x40(64)字节堆块
// ...
strcpy(v4, v3);               // 不安全复制！若v3长度为64，则无终止符，造成溢出。
```

# 漏洞利用过程：
利用思路分为几个阶段：
1.  **信息泄露**：利用`name`输入后的`sub_8048779`函数，泄露一个堆地址。
2.  **堆溢出**：利用`Host`输入的`strcpy`堆溢出，将`top chunk`的`size`修改为极大值`0xffffffff`。
3.  **堆布局与计算**：通过计算，使下一次`malloc`恰好从`note`指针数组(`0x804b120`)开始分配，从而获得篡改整个管理数组的能力。
4.  **GOT劫持与信息泄露**：篡改`note`指针指向`free@got`，通过编辑`note`内容修改`free@got`为`puts@plt`，然后`delete`一个`note`触发`puts`泄露`libc`中`printf`的地址，从而计算`libc`基址。
5.  **getshell**：再次篡改指针，将`free@got`修改为`system`地址，并在一个`note`内容中写入`/bin/sh`，最后`delete`该`note`触发`system("/bin/sh")`。

## Step1: 泄露堆地址
- 堆内存`leak_heap_addr`处：在执行`p.sendafter(b'Input your name:\n', b'a' * 0x40);`后，`name`被复制到堆块`v2`。`sub_8048779(v2)`打印欢迎语时，会输出`v2`（即堆块）的内容。由于我们输入了64个`'a'`，`strcpy`会连带复制堆块`v2`之后`org`堆块的`prev_size`和`size`字段的一部分（在glibc 2.23中，`size`的低字节通常为`0x?1`），这部分内容被作为字符串的一部分打印出来。通过接收这些字节并解析，可以得到一个堆地址（实际上是`org`堆块的起始地址）。
- **变化原因**：`strcpy`没有长度限制，复制完`'a'*0x40`后，继续复制了相邻堆块的管理数据，这些数据被`printf`作为字符串内容输出。

## Step2: 溢出修改top chunk size
- 堆内存`top_chunk_addr`（即`leak_heap_addr + 0xd0`）处：在输入`Org:`和`Host:`后，程序分配两个堆块。当发送`Host:`为`p32(0xffffffff) + (0x40 - 4) * b'c'`时，`strcpy`发生溢出，覆盖了`host`堆块之后`top chunk`的`size`字段。
- 具体变化：`top chunk`的`size`从原来的`0x20fa1`（或类似值）被覆盖为`0xffffffff`。
- **变化原因**：`Host`输入恰好64字节，`strcpy`读取时没有遇到空字节，会继续复制栈上后续的`v2`指针和canary等数据。精心构造的payload（`0xffffffff`）恰好覆盖了`top chunk`的`size`字段。

## Step3: 计算并分配以覆盖指针数组
- 计算`margin = ptr_array - top_chunk_addr`，即从当前`top chunk`头到目标地址`0x804b120`（note指针数组）的偏移。
- 调用`new_note(margin - 20, "")`。由于`top chunk`的`size`被改为极大值，`malloc`会从`top chunk`切分出`margin - 20`大小的块。`margin - 20`的计算是为了在`malloc`的内部开销（`chunk header`和对齐）后，使下一次`malloc`返回的地址正好是`0x804b120`。
- **关键变化**：执行此步骤后，`top chunk`的指针被更新到`0x804b120`附近，为下一步直接从该地址分配做好了准备。

## Step4: 连续分配四个note进行布局
- 连续调用四次`new_note(0x40, b'aa')`。由于上一步的布局，第一次`malloc(0x40)`会从`0x804b120`开始分配，这正好是全局`dword_804B120`数组的位置。因此：
  - `note0`的指针(`dword_804B120[0]`)被写入到`0x804b120`，即它**指向了自己**（`0x804b120`）。
  - `note1`的指针(`dword_804B120[1]`)被写入到`0x804b124`，指向`0x804b124`。
  - `note2`的指针(`dword_804B120[2]`)被写入到`0x804b128`，指向`0x804b128`。
  - `note3`的指针(`dword_804B120[3]`)被写入到`0x804b12c`，指向`0x804b12c`。
- 这些`note`的内容区域也重叠在指针数组及其后的内存上。此时，通过编辑`note1`，可以修改从`0x804b124`开始的内存，包括`note2`和`note3`的指针。

## Step5 & Step6: 篡改指针，准备GOT写
- `edit_note(1, p32(0x804b120) * 2 + p32(free_got) + p32(printf_got))`
  - 从`0x804b124`（`note1`的内容开始处）写入数据。
  - 覆盖了`0x804b128`（原`note2`指针）为`free@got`地址。
  - 覆盖了`0x804b12c`（原`note3`指针）为`printf@got`地址。
- `edit_note(2, p32(puts_plt))`
  - 此时`note2`的指针指向`free@got`，所以编辑`note2`即向`free@got`写入`puts@plt`的地址。
- **内存变化**：`free`函数的GOT表项从原来的`libc`中`free`的地址被修改为`puts@plt`的地址。

## Step7 & Step8: 触发泄露，计算libc基址
- `del_note(3)`
  - 此时`note3`的指针指向`printf@got`。`delete`操作会调用`free(note3_ptr)`。
  - 由于`free@got`已被改为`puts@plt`，实际执行的是`puts(printf@got)`，从而打印出`libc`中`printf`函数的真实地址。
- 接收泄露的地址`printf_addr`，并计算`libc.address = printf_addr - libc.sym['printf']`。

## Step9 & Step10: 再次篡改指针，准备getshell
- `edit_note(1, p32(0x804b130) * 2 + p32(free_got) * 2 + b'/bin/sh')`
  - 重新布局指针数组。使`0x804b128`（`note2`指针）再次指向`free@got`，并使`0x804b12c`（`note3`指针）指向`0x804b13c`（即`'/bin/sh'`字符串所在处）。
- `edit_note(2, p32(libc.sym['system']))`
  - 再次编辑`note2`，将`free@got`修改为`system`的地址。

## Step11: 触发system("/bin/sh")
- `del_note(0)`
  - 删除`note0`。其指针(`dword_804B120[0]`)仍指向`0x804b120`，这是指针数组的头部。
  - 程序调用`free(0x804b120)`，此时`free`已被劫持为`system`。
  - 但`0x804b120`处的内存内容经过上一步编辑，其`0x804b13c`偏移处是字符串`"/bin/sh"`。
  - **关键点**：在glibc 2.23的`system`实现中，它会对传入的指针进行一些检查，但`0x804b120`并非一个合法的`chunk`指针。然而，`system`的参数是从`free`的参数直接传递的，即`system(0x804b120)`。`system`会尝试将`0x804b120`作为命令字符串执行，这显然不行。这里需要特别注意，原exp中`del_note(0)`触发的是`system(0x804b120)`，而不是`system("/bin/sh")`，这通常无法getshell。
  - **修正**：实际上，应该删除`note3`（其指针指向`0x804b13c`，即`"/bin/sh"`字符串）来触发`system("/bin/sh")`。原exp的step11应为`del_note(3)`。这可能是原write-up的一个笔误或利用链的另一条路径（通过覆盖`note0`内容为`/bin/sh`）。根据提供的exp代码和逻辑，正确的触发方式是在step9将`/bin/sh`写入一个note内容区域，并将某个note指针指向该区域，最后删除该note。让我们重新审视exp：step9将`/bin/sh`写入了`0x804b13c`，并将`note3`的指针（`0x804b12c`处）设置为了`0x804b13c`。因此，最终的step11应该是`del_note(3)`来调用`system("/bin/sh")`。此处根据标准利用逻辑进行修正。
