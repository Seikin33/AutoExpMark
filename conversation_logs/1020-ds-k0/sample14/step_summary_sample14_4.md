## 内存变化总结

基于整个调试过程，以下是关键的内存变化分析：

### 初始状态（程序启动时）
- `qword_602090` = 未知初始值（非0）
- `buf`数组 = 空指针或未初始化
- 堆chunk = 未分配状态
- fastbins = 空

### Exp执行后的内存变化

#### 1. 堆分配操作
```
add(0x40); add(0x20);
```
- 分配了两个chunk：
  - chunk 0: 地址0x3c73d000，大小0x50（包含元数据）
  - chunk 1: 地址0x3c73d050，大小0x30（包含元数据）
- `buf[0]` = 0x3c73d010（指向chunk 0的用户数据区）
- `buf[1]` = 0x3c73d060（指向chunk 1的用户数据区）

#### 2. 释放操作
```
delete(0)
```
- chunk 0被释放到fastbin（0x50大小）
- 但`buf[0]`指针未被清空，产生UAF漏洞
- fastbin状态：0x50 → 0x3c73d000

#### 3. UAF利用
```
edit(0, p64(0x602080))
```
- 通过UAF漏洞修改已释放chunk 0的fd指针
- chunk 0的fd从原来的next chunk指针改为0x602080（数据段地址）
- 内存变化：0x3c73d010处的内容从正常数据变为0x602080

#### 4. 堆布局操控
```
add(0x40); add(0x40)
```
- 第一次add：从fastbin分配chunk 0（fd指向0x602080）
- 第二次add：从fastbin分配，但由于fd被修改，实际分配到数据段地址0x602080
- 内存变化：
  - `buf[2]` = 0x3c73d010（重新分配chunk 0）
  - `buf[3]` = 0x602090（分配到数据段，指向`qword_602090`）

#### 5. 关键变量修改
```
edit(3, p64(0))
```
- 通过`buf[3]`（指向数据段）写入数据
- 直接修改`qword_602090` = 0x0
- 这是触发shell的关键修改

### 最终内存状态

#### 关键变量
- `qword_602090` = 0x0 ✅（成功置零，满足shell触发条件）

#### 堆状态
- chunk 0 (0x3c73d000): fd = 0x602080（被修改指向数据段）
- chunk 1 (0x3c73d050): 正常状态
- fastbin: 0x50大小链表中有一个chunk

#### 全局数组`buf`
- `buf[0]` = 0x3c73d010（指向chunk 0）
- `buf[1]` = 0x3c73d060（指向chunk 1）
- `buf[2]` = 0x3c73d010（指向chunk 0，重复指针）
- `buf[3]` = 0x602090（指向数据段中的`qword_602090`）

### 漏洞利用链总结
1. **UAF漏洞**：释放后未清空指针 → 允许编辑已释放内存
2. **堆元数据污染**：修改fd指针指向数据段
3. **任意地址分配**：通过污染后的fastbin分配到数据段
4. **关键变量覆盖**：直接修改`qword_602090`为0
5. **权限提升**：满足shell启动条件

这个利用过程展示了典型的UAF漏洞利用技术，通过堆布局操控实现任意地址写入，最终达到代码执行的目的。
