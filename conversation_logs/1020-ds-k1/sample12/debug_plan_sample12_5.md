基于提供的全局内存信息和exp执行历史，我设计了一个调试计划来探索当前程序的局部内存状态。程序当前在`__read_nocancel+7`系统调用中暂停，等待用户输入。以下命令将帮助分析栈状态、寄存器上下文和堆chunk的详细信息，以验证漏洞利用的进展。

### 调试计划
1. **检查寄存器状态**：使用`regs`命令查看当前寄存器值，特别是RSP、RIP和系统调用相关寄存器，以了解执行上下文和系统调用参数。
2. **分析栈内存**：使用`stack`命令查看当前栈帧内容，识别函数返回地址、局部变量和输入缓冲区残留数据。
3. **详细检查堆chunk**：使用`malloc_chunk`命令分析特定堆chunk的结构，确认unlink攻击是否成功设置伪造的chunk元数据。
4. **望远镜查看栈内存**：使用`telescope`命令以指针链形式查看栈内存，捕捉可能的关键地址或数据。
5. **执行continue**：如果关键信息确认完毕，恢复程序执行。

现在，我将逐步执行这些命令。

首先，执行`regs`命令查看寄存器状态：
```python
RunPwndbgCommand("regs")
```

**理由**：寄存器状态能揭示系统调用的参数（如RAX为系统调用号、RDI/RSI/RDX为参数），以及当前执行点。这对于理解程序在等待输入时的上下文至关重要。

接下来，执行`stack`命令查看栈内容：
```python
RunPwndbgCommand("stack 20")
```

**理由**：栈分析可以显示当前栈帧的布局，包括返回地址、局部变量和格式化字符串攻击的残留数据。这有助于验证栈完整性并识别潜在的控制流修改。

然后，使用`malloc_chunk`命令详细检查堆chunk：
```python
RunPwndbgCommand("malloc_chunk 0x5905958c7000")
```

**理由**：从全局内存信息中，堆chunk在`0x5905958c7000`的fd/bk指针被修改为指向全局变量区域（`key`附近），这可能表示unlink攻击已设置。详细检查chunk元数据能确认伪造是否成功。

接着，使用`telescope`命令查看栈内存的指针链：
```python
RunPwndbgCommand("telescope $rsp 20")
```

**理由**：望远镜命令能以指针解析方式显示内存，帮助识别栈上的代码指针、堆地址或libc地址，验证格式化字符串攻击的泄露数据是否残留。

最后，如果上述命令确认了关键状态（如unlink攻击准备就绪），执行`continue`恢复程序：
```python
RunPwndbgCommand("continue")
```

**理由**：程序当前处于等待输入状态，且exp可能已设置好后续利用步骤。继续执行允许exp进行下一步操作（如触发unlink修改全局变量）。
