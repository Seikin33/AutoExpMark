基于整个对话过程，我总结内存变化如下：

## 内存变化时间线

### 初始状态（程序启动后）
- **heap数组**: 全0
- **len数组**: 全0  
- **key1**: 0 (可编辑2次)
- **key2**: 0 (show功能不可用)
- **pro变量**: 未初始化
- **堆内存**: 未分配

### 执行exp第一阶段后（malloc操作）
```python
malloc(0,0xf8,'aaaa')
malloc(32,0xf8,'bbbb') 
malloc(1,0xf8,'cccc')
malloc(31,0xf8,'dddd')
```

**变化**:
- **heap数组**:
  - 索引0: `0x71fc010` (chunk0用户数据)
  - 索引32: `0x71fc110` (chunk32用户数据，存储在pro变量位置)
  - 索引1: `0x71fc210` (chunk1用户数据)  
  - 索引31: `0x71fc310` (chunk31用户数据)

- **len数组**:
  - 索引0、1、31、32: 大小设为0xf8

- **堆布局**:
  - chunk0 (0x71fc000): 内容'aaaa'
  - chunk32 (0x71fc100): 内容'bbbb'
  - chunk1 (0x71fc200): 内容'cccc' 
  - chunk31 (0x71fc300): 内容'dddd'

### 执行exp第二阶段后（edit操作）
```python
edit(32,py)  # py包含伪造堆块数据
```

**关键变化**:

1. **chunk32内容变化**:
   - 前16字节: `0x0` + `0xf1` (伪造size)
   - fd: `0x6021c8` (指向heap+232)
   - bk: `0x6021d0` (指向heap+240)
   - 填充0x00到偏移0xf0
   - 偏移0xf0: `0xf0`

2. **off-by-one溢出影响**:
   - chunk1的prev_size被覆盖为`0xf0`

3. **全局变量变化**:
   - **key1**: 从0变为1 (编辑次数计数器递增)
   - **key2**: 从0变为`0x0000000100000000` (权限提升，show功能可用)

4. **heap数组异常**:
   - 索引0的值从`0x71fc010`变为`0xf8` (可能被覆盖)

### 最终内存状态

#### 堆内存状态
```
0x71fc000: chunk0 ('aaaa') - 可能受影响
0x71fc100: chunk32 (伪造堆块) - 成功伪造
0x71fc200: chunk1 ('cccc') - prev_size被覆盖为0xf0
0x71fc300: chunk31 ('dddd') - 正常
```

#### 全局数据结构
- **heap数组**: 索引0异常，其他正常
- **len数组**: 索引0异常(0x000000f8000000f8)，其他正常
- **key1**: 1 (剩余1次编辑)
- **key2**: 非零 (show功能可用)
- **pro变量**: 0x71fc110 (指向伪造堆块)

#### 利用准备状态
- ✅ 伪造堆块设置成功
- ✅ off-by-one溢出成功
- ✅ 权限提升(key2非零)
- ⚠️ heap数组索引0异常可能影响后续操作
- ⚠️ 剩余编辑次数有限(key1=1)

## 关键漏洞利用进展

1. **数组越界利用**: 通过索引32成功操纵pro变量
2. **堆布局操纵**: 四个连续堆块为off-by-one提供有利条件  
3. **伪造堆块**: 设置fd/bk指向heap数组内部，为后续攻击做准备
4. **权限绕过**: key2被意外设置为非零，获得show功能权限

这些变化为后续的漏洞利用（如unsorted bin攻击、GOT覆盖等）奠定了重要基础。
